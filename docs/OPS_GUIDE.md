## Часть I. Что вообще запускается

Бот — это не одна программа, а **связка из двух «коробок»**, которые поднимает Docker Compose (проект `aemr-bot`):

1. **`db`** — база данных Postgres 16. Здесь физически лежат все обращения жителей, операторы, настройки. Данные хранятся в «сейфе»-volume `db_data`, который переживает любую пересборку.
2. **`bot`** — сам процесс на Python, который разговаривает с мессенджером MAX.

Внутри коробки `bot` при старте выполняется `alembic upgrade head && python -m aemr_bot.main` ([Dockerfile:64](infra/Dockerfile)): сначала накатываются миграции базы, потом запускается бот. Бот зовёт базу по имени `db` через внутренний «коридор»-сеть.

Два факта, которые определяют всё остальное:

- **Бот работает по «long polling»** — сам ходит наружу на `platform-api.max.ru` и спрашивает «есть новые сообщения?». Поэтому **никаких входящих портов открывать не надо** — нужен только *исходящий* 443. Health-сервер на порту 8080 виден лишь локально (`127.0.0.1`), наружу не торчит.
- **Минимум для старта** ([config.py:15,23](bot/aemr_bot/config.py)): `BOT_TOKEN` (токен из max.ru/business) и `POSTGRES_PASSWORD` (из него Compose сам соберёт `DATABASE_URL`). Оба — в файле `infra/.env`.

Аналогия: `db` — склад, `bot` — продавец, который к складу ходит, Compose — менеджер, который держит обоих живыми и связывает их.

## Часть II. Пять страховок: что кого поднимает

У бота **пять уровней страховки, и каждый закрывает свою беду**. Их легко перепутать, и достаточно выключить два-три, чтобы бот «лёг на два дня» — именно это и случилось в инциденте июня-2026. Вот сводка, к которой стоит возвращаться:

| Что случилось | Кто поднимает |
|---|---|
| Процесс бота **упал** (любой выход) | docker `restart: unless-stopped` |
| Процесс **жив, но завис** (unhealthy) | ТОЛЬКО вотчдог healthwatch-cron (если установлен) |
| **Перезагрузка** сервера | systemd-юнит → `docker compose up -d` (если docker.service включён) |
| docker выключен **и** cron'ы сняты | **НИКТО.** Это и есть состояние инцидента 2026-06 |

### 1. Docker и Compose — «коробки» с приложением

**Простыми словами.** Docker запаковывает программу со всем, что ей нужно (Python, библиотеки, утилиты), в изолированную «коробку»-контейнер. Её можно перенести на любой сервер — внутри всё работает одинаково. Compose из одного файла поднимает сразу несколько коробок и соединяет их.

**Аналогия.** Контейнер — собранный набор IKEA в коробке: мебель плюс все винтики внутри, докупать нечего. Образ (image) — чертёж этой коробки (его пишет `Dockerfile`). Volume — сейф, прикрученный снаружи: коробку выбрасывают и собирают заново, а содержимое сейфа остаётся. У нас два сейфа: `db_data` (вся база) и `backups` (резервные копии).

**Как эксплуатировать** (из папки `~/aemr-bot/infra`):
```bash
docker compose up -d            # поднять обе коробки в фоне
docker compose up -d --build    # то же, но с пересборкой после обновления кода
docker compose ps               # кто жив (ждём Up и healthy, НЕ Restarting)
docker compose logs --tail 200 bot
docker compose down             # погасить (данные в сейфах НЕ удаляются)
```
**Запрещено:** `docker system prune -a --volumes` — сотрёт сейфы вместе с базой.

**Как ломается.** Выключен сам Docker-демон → ни одна коробка не поднимается (ядро инцидента 2026-06). Сделали `down` и забыли поднять → коробки лежат, пока их не поднимут руками или страховкой №5.

### 2. restart-policy — нянька ловит упавшего, но не зависшего

**Простыми словами.** `restart: unless-stopped` в конце описания каждой коробки говорит Docker: «процесс внутри **упал** — немедленно подними заново; и поднимай после каждой перезагрузки. Но если человек *сознательно* остановил через `stop`/`down` — не воскрешай».

**Аналогия.** Автомат в электрощитке: сам взводится после скачка, но если ты *рукой* опустил рубильник — уважает решение. И главное: реагирует только на полное КЗ (процесс умер). Лампочка «еле светит, но ток идёт» (процесс завис, но не разбился) автомат не трогает — вот это наш главный враг.

**Как у нас.** Это **единственный настоящий супервизор** в обычной жизни бота. Тонкость: при битом токене бот сам себя завершает `sys.exit(1)`, а в коде есть комментарий «контейнер выйдет, чтобы избежать restart-loop» — **это неправда**: `unless-stopped` честно поднимет коробку снова, она снова упадёт, и получится бесконечная карусель с одной красной строкой ❌ на круг.

```bash
docker inspect aemr-bot-bot-1 --format '{{.RestartCount}} {{.State.OOMKilled}}'
```
Большое растущее число рестартов = что-то систематически роняет процесс; `OOMKilled=true` = сервер убивал коробку за переедание памяти.

**Чего не лечит.** Если процесс жив, но завис (event-loop встал, цикл к MAX замер) — Docker этого *не видит*, коробка просто красится `unhealthy`, но работает. Это «живой труп»: формально есть, фактически мёртв. Чинит его только вотчдог (№4).

### 3. healthcheck vs вотчдог — один красит, другой лечит

Два механизма «про здоровье», которые путают.

- **healthcheck в Compose** — встроенный термометр Docker: раз в 30 секунд ходит внутрь коробки и спрашивает `/livez`. Три отказа подряд → меняет ярлык с `healthy` на `unhealthy`. **И всё.** Только цвет ярлыка, без перезапуска.
- **вотчдог healthwatch** (№4) — живёт *снаружи* Docker, в расписании cron, и умеет нажать кнопку перезапуска и позвать человека.

**Аналогия.** healthcheck — лампочка «Check Engine»: загорается, но машину не чинит. Вотчдог — механик, который на эту лампочку смотрит, лезет под капот и перезапускает, а не помогло — звонит хозяину. Лампочка без механика бесполезна, механик без лампочки слеп. Нужны оба.

Разнесено это *намеренно*: тяжёлое решение о перезапуске принимает отдельный, простой, легко выключаемый механизм, а не Docker вслепую. Симптом «коробка `unhealthy`, но висит и не перезапускается» = верный признак, что **выключен вотчдог** (№4), а не термометр.
<sub>[docker-compose.yml:75-84](infra/docker-compose.yml) · [healthwatch.sh:40-63](scripts/healthwatch.sh)</sub>

### 4. вотчдог healthwatch — механик и сигнал наружу

**Простыми словами.** `healthwatch.sh` — сторожевой скрипт, который сервер запускает **сам каждые 5 минут** (строка `*/5 * * * *` в root-cron). Живёт снаружи Docker, стучится боту на `/livez` и ведёт счётчик неудач в файле `/var/lib/aemr-bot-watchdog/state`.

**Аналогия.** Ночной сторож с обходом раз в 5 минут и блокнотом. Дверь не открыли — палочка. Три палочки подряд (15 минут) — сам отпирает запасным ключом и перезапускает. Восемь палочек (40 минут) — звонит хозяину по *отдельному* телефону, не через того, кто за дверью молчит.

**Точные ступеньки.** Порог рестарта = 3, порог алёрта = 8. На *ровно третьем* провале (условие `-eq`, не «3 и больше») — **один раз** делает `docker compose restart bot`. Если этот единственный рестарт не удался (Docker выключен, коробки удалены через `down`) — пишет «restart failed» и больше *никогда* не пробует, только копит до алёрта. На восьмом и далее — достаёт `BOT_TOKEN`/`ADMIN_GROUP_ID` из `infra/.env`, строго проверяет регэкспом и шлёт «⛑️ Бот не отвечает…» **прямым curl на platform-api.max.ru** в админ-группу. Это «зов мимо бота»: долетает, даже когда сам бот мёртв — лишь бы был жив cron.

```bash
journalctl -t aemr-bot-watchdog -n 50      # что делал
cat /var/lib/aemr-bot-watchdog/state       # текущий счётчик провалов
sudo bash scripts/install-healthwatch.sh   # поставить (идемпотентно)
sudo crontab -l | grep -v aemr-bot-healthwatch | sudo crontab -   # снять
```

**Как ломается.** Удалили строку из root-crontab (как в инциденте) — исчезает *единственный* лекарь «живого трупа» и *единственный* внешний алёрт о 40-минутной недоступности. Зависший бот висит сколько угодно, никто не узнаёт.
<sub>[healthwatch.sh:19-103](scripts/healthwatch.sh) · [install-healthwatch.sh:27-42](scripts/install-healthwatch.sh)</sub>

### 5. systemd-юнит — будильник на перезагрузку

**Простыми словами.** systemd — штатный диспетчер служб в Linux. Наш юнит `aemr-bot.service` **не запускает python**. Он делает одно после загрузки сервера: `docker compose up -d` — «подними обе коробки». Тип `oneshot` + `RemainAfterExit`: «выполнился один раз, поставил галочку, спит». Это **не супервизор**: за ботом не следит, зависший не лечит.

**Аналогия.** Таймер на кофеварке: каждое утро (каждая загрузка сервера) жмёт «варить». Но рядом весь день не стоит. И у той же кнопки есть положение «выкл» — см. ловушку.

**Зачем нужен.** `restart-policy` после перезагрузки поднимает только те коробки, что были запущены на момент остановки демона. Если их положили в `down` — сами не вернутся. Юнит — «вторая страховка»: идемпотентный `up -d` поднимет в любом случае.

```bash
sudo bash scripts/install-systemd-autostart.sh   # поставить
systemctl status aemr-bot.service
systemctl is-enabled aemr-bot.service            # ждём enabled
```

**⚠️ Ловушка.** У юнита `ExecStop=docker compose down`. Значит `systemctl stop aemr-bot` — это **не «остановить наблюдателя», а «погасить ОБЕ коробки, включая базу»**. Чтобы отключить только автозапуск, не гася бота: `systemctl disable aemr-bot.service` (без `stop`).

**Как ломается.** Юнит требует `docker.service`. Если демон выключен — даже после reboot юнит ничего не поднимет. «systemd есть, а бот не встаёт после reboot» = почти всегда выключенный `docker.service` (`systemctl is-enabled docker.service`).

### 6. /livez /readyz /healthz — три вопроса о здоровье

Внутри бота крутится крошечный веб-сервер «здоровья» на `127.0.0.1:8080` (наружу не торчит). Три адреса задают разные вопросы, каждый отвечает 200 (ок) или 503 (плохо):

- **`/livez` — «ты вообще живой и шевелишься?»** Базу не трогает. Проверяет два сигнала: *heartbeat* (фоновый «тук» каждые 30с, протухает за 120с) И *poll_watch* (отметка на каждом успешном запросе к MAX, протухает за 60с). Два сигнала нужны потому, что heartbeat остаётся зелёным даже при зависшем цикле к MAX — а poll_watch именно этот завис и ловит.
- **`/readyz` — «готов работать, база на связи?»** То же, что `/livez`, плюс живой `SELECT 1` к Postgres (кэш 10с).
- **`/healthz`** — старое имя-синоним `/readyz`.

**Кто их дёргает** (ключ ко всей супервизии): по `/livez` ходят **трое** — Docker-термометр (красит `unhealthy`), вотчдог (перезапускает и алёртит), автодеплой (60-секундный гейт при выкатке). А `/readyz`/`/healthz` **не управляют ни рестартом, ни откатом нигде** — только для ручной диагностики.

```bash
curl -fsS http://127.0.0.1:8080/livez     # пульс
curl -fsS http://127.0.0.1:8080/readyz    # пульс + база
```

**Важнейший случай — «livez зелёный, readyz красный».** Процесс жив, но база недоступна. Бот «жив», но не сохраняет обращения, обработчики падают по одному. Автоматически его *никто не перезапускает* (правильно — рестарт базу не воскресит), но и **алёрта об этом нет** — бот может сутками отвечать 503 на `/readyz`, и оператор не узнает (это дыра, см. Часть V). При таком симптоме — лезть в `docker compose logs db`, а не ждать сигнала.

## Часть III. Деплой, данные, наблюдение

### 7. Автодеплой — сервер сам тащит код с GitHub

**Простыми словами.** Скрипт, который root-cron запускает раз в 10 минут. Смотрит на GitHub: появились новые коммиты в `main`? Нет — тихо выходит. Да — выкачивает и пересобирает контейнер.

**Аналогия.** Вахтёр каждые 10 минут заглядывает в почтовый ящик. Пусто — спит. Пришла посылка (новый код) — распаковывает, ставит, проверяет, что новая вещь включается, и только если включилась — оставляет; задымилась — выкидывает, возвращает старую.

**Как у нас, по шагам:** read-only deploy-key читает репозиторий → `reset --hard origin/main` (локальные правки на сервере при этом *затираются*, кроме `.env`) → переустановка обёрток в `/usr/local/bin` → `up -d --build` → **health-gate**: до 60 секунд стучится на `/livez`; поднялся → деплой засчитан; не поднялся за 60с → **авто-откат** на предыдущий коммит, пересборка, и ненулевой код для алёрта.

```bash
sudo bash scripts/install-auto-deploy.sh      # поставить (печатает pubkey для GitHub)
sudo /usr/local/bin/aemr-bot-autodeploy       # выкатить руками сейчас
journalctl -t aemr-bot-deploy -n 50           # лог (new commits / deploy ok / ROLLBACK)
```

**Когда опасен / не работает:**
1. **Закрытая гос-сеть.** Если сервер не ходит к github.com — `git fetch` падает, автодеплой мёртв, выкатка только руками.
2. **`reset --hard` затирает** ручные правки на сервере. Чинить прод «по-живому» нельзя, только коммитом в main.
3. **Footgun-воскрешение.** На *не-каноническом* хосте автодеплой поднимает лишний экземпляр бота — это часть причины инцидента (см. RUNBOOK §7). На VPS/выводимом хосте автодеплой **не ставить**.
<sub>[auto-deploy.sh:18-147](scripts/auto-deploy.sh) · [install-auto-deploy.sh:30-83](scripts/install-auto-deploy.sh)</sub>

### 8. Два вида расписаний: серверный cron vs 17 внутренних задач

Будильников два слоя, и их постоянно путают.

- **Серверный cron** (root, снаружи контейнера) — ровно две строки: `*/10` автодеплой и `*/5` вотчдог. Живут в ОС хоста и **работают, даже когда процесс бота лежит**. Поэтому отключать их нельзя: без вотчдога зависший «живой труп» висит до ручного вмешательства.
- **17 внутренних задач APScheduler** (внутри процесса бота) — пульс, health-selfcheck, операторские напоминания, ретенции по 152-ФЗ (обезличивание ПДн, сроки хранения), бэкап вс 03:00. **Зависят от живого процесса**: бот умер — все 17 молчат разом, включая пульс. Тишина в админ-группе MAX = лёг весь процесс.

**Аналогия.** Серверный cron — сторож снаружи здания, обходит по часам. Внутренние задачи — сотрудники внутри, у каждого свой будильник. Обесточили здание — сотрудники спят, работает только внешний сторож.
<sub>[cron_registry.py:27-119](bot/aemr_bot/services/cron_registry.py) · [install-healthwatch.sh:32](scripts/install-healthwatch.sh)</sub>

### 9. Бэкапы — слепок базы под шифром

**Простыми словами.** Раз в неделю (вс 03:00) бот сам снимает слепок базы, шифрует и кладёт в защищённую папку, удаляя самые старые.

**По шагам:** `pg_dump` (пароль через env, не через командную строку) → шифрование `gpg --symmetric AES256` → файл `aemr-<дата>.sql.gpg` → хранится 8 последних копий в volume `backups`. **Защита:** если парольная фраза короче 12 символов — бот *отказывается* писать незашифрованный дамп (там ПДн по 152-ФЗ). Провал бэкапа громкий — алёрт в админ-группу.

```bash
# восстановить (расшифровать дамп):
gpg --batch --passphrase '<BACKUP_GPG_PASSPHRASE>' -d aemr-<дата>.sql.gpg > dump.sql
```

**Чего не хватает** (честно):
1. **Offsite выключен** по умолчанию — все 8 копий на том же диске, что и база. Потеря сервера = потеря базы *и* всех копий разом. (rclone-выгрузка в коде есть, включается заданием `BACKUP_S3_*`.)
2. **DR-учение не репетировалось.** Бэкап, который ни разу не восстанавливали, — это надежда, а не страховка.
3. **Одна GPG-фраза.** Потеряли — все копии превращаются в мусор. Хранить отдельно от сервера.

### 11. Один токен — один бот

`BOT_TOKEN` — это и логин, и пароль, и адрес бота в MAX одновременно. Если токен крутят **два процесса**, оба представляются одним ботом — и начинается тихий хаос.

**Аналогия.** Один почтовый ящик, два курьера с одинаковым ключом. Письмо достаётся тому, кто успел первым; второй уходит пустой. Жители видят, что бот отвечает «через раз». И никто не ругается: ключ-то у обоих настоящий.

**Почему молча.** Каждый процесс держит позицию «что прочитал» только у себя в памяти, общего журнала нет. Конфликта `409` код MAX-библиотеки не обрабатывает (знает только `401`); любой другой ответ → INFO-лог и вечный retry. Защиты на старте (advisory-lock «я уже запущен») **нет** — это структурная дыра (Часть V). Единственная точка, где два процесса реально убивают друг друга — порт 8080 (`EADDRINUSE`), но только если оба на *одной* машине; на разных хостах конфликт остаётся тихим (так и было между VPS и рабочим хостом).

**Диагностика и лечение:**
```bash
ss -ltnp | grep 8080            # кто слушает health-порт
ps aux | grep aemr_bot          # сколько процессов
systemctl list-units | grep -i aemr   # нет ли постороннего юнита с прямым python
```
Гасить лишний экземпляр. Если не найти, кто ещё крутит токен — **ротация токена**: новый `BOT_TOKEN` через max.ru/business, вписать в `.env` канонического хоста. Старый мгновенно протухает, любая забытая копия отваливается с `401` и сама останавливается. Это самая надёжная гарантия «ровно один бот».

## Часть IV. Запуск на разных ОС

Главный архитектурный факт: весь стек написан под **Docker на Linux** (`python:3.12-slim` + `postgres:16-alpine` — оба линуксовые). От этого пляшет вся матрица.

### Ubuntu/Debian — эталон
Родная среда. Docker Engine + плагин `docker compose` v2 (не старый `docker-compose` через дефис; не snap-версия — она ломает volumes). Непривилегированный пользователь `aemr` в группе `docker` → `git clone` → `cp infra/.env.example infra/.env`, вписать `BOT_TOKEN` + `POSTGRES_PASSWORD`, `chmod 600` → `docker compose up -d --build` → проверить `docker compose ps` (оба Up, bot `healthy`) и `curl 127.0.0.1:8080/livez`. Автозапуск: `install-systemd-autostart.sh`. Страховки: `install-healthwatch.sh`, `install-auto-deploy.sh`. Требования: 1-2 vCPU, 2-4 ГБ RAM, 20 ГБ SSD. Время на хосте должно быть верным (NTP) — иначе расписания (бэкап, ретенции) уедут.

### FreeBSD через Linux-VM (bhyve) — **рекомендованный путь**
FreeBSD не умеет нативный Docker (нет Linux-ядра), а образы линуксовые. Честный способ запустить *ровно тот* стек — поднять внутри FreeBSD виртуалку с Linux:
1. Включить bhyve (родной гипервизор FreeBSD): `pkg install vm-bhyve grub2-bhyve`, настроить пул и сетевой мост.
2. Создать VM с Ubuntu 24.04 LTS Server (2 vCPU, 4 ГБ RAM, 30 ГБ диск).
3. **Внутри Ubuntu-гостя** — выполнить раздел «Ubuntu» дословно.
4. Автозапуск двойной: bhyve поднимает VM при загрузке FreeBSD (`vm_enable=YES` в rc.conf), systemd внутри VM поднимает Docker-стек при загрузке гостя.

Подводные камни: проброс DNS в VM (gateway должен резолвить `platform-api.max.ru`); TZ `Asia/Kamchatka` + NTP в VM; место на диске VM (образы копятся при rebuild); firewall FreeBSD (`pf`/`ipfw`) должен пропускать исходящий 443 от моста VM. **Плюс пути:** работают все hardening'и compose и все 4 серверные страховки — как на эталоне.

### FreeBSD нативно (без Docker) — запасной путь, честно про риски
Только если виртуализация запрещена регламентом. Вы руками пересобираете то, что Docker давал из коробки. Из pkg: `python312`, `postgresql16-server`+`client` (libpq для бэкапов), `gnupg`, `geos` (обязательно до сборки `shapely`, иначе wheel не соберётся), `git`, `uv`. `asyncpg`/`shapely`/`uvicorn[standard]` на FreeBSD собираются *при наличии* geos/libpq. Postgres ставить рядом (`initdb`, `postgresql_enable=YES`, в `shared_preload_libraries` добавить `pg_stat_statements`). `DATABASE_URL` задать руками, демонизировать через `rc.d`-скрипт.

**Честные потери (большие):** ни одна из 4 страховок репо не переносится (все на bash+systemd+`docker compose`) — watchdog и автодеплой придётся писать с нуля; теряется весь hardening compose (read-only rootfs, лимиты памяти/процессов) — компенсировать средствами FreeBSD (jail, rctl); возможен дрейф версии Postgres. Путь рабочий, но это ручная пересборка эксплуатационного контура с понижением защиты — оправдан только при полном запрете виртуализации.

### RHEL / Astra / РЕД ОС (гос-Linux РФ)
Стек запускается, две специфики. **(а) SELinux/PARSEC и volumes:** при включённом мандатном доступе named-volume'ы и bind-mount'ы могут ловить `permission denied` даже при верных правах — из-за меток контекста. Лечение: суффикс `:Z` для bind-mount'ов, `semanage`/`container_file_t`, либо мандатные атрибуты на каталог; смотреть `ausearch -m avc`. **(б) podman вместо docker:** понимает `docker-compose.yml`, но rootless иначе мапит UID (а образ фиксирует UID/GID 1000 ради прав на volume), и автозапуск делается через quadlet/`generate systemd`, а не через юнит репо; скрипты страховок жёстко зашиты на `docker compose` — под podman их надо править. Часто проще поставить `docker-ce` из совместимого репо и идти эталонным путём.

### Windows / macOS — только разработка
Образ линуксовый → на Windows/macOS он крутится лишь через Docker Desktop (Linux-VM под капотом), не для серверной нагрузки 24/7; все страховки Linux-only не работают. Для прода исключено архитектурно. Для dev: тесты `cd bot && uv sync --extra dev && uv run pytest tests/ -q` (на SQLite, без Postgres). **⚠️ Не запускать локального бота с боевым `BOT_TOKEN`** — это мгновенно устроит гонку с продом (см. раздел 11). Для локалки — отдельный тестовый токен.

### За UserGate (для любой ОС)
Боту нужно пропустить наружу **ровно один адрес** — `platform-api.max.ru:443` (зашит в коде). Плюс DNS, чтобы его резолвить.
- **Allowlist:** правило исходящий → `platform-api.max.ru:443`, лучше по FQDN (IP у MAX могут меняться).
- **SSL-инспекция — главная мина.** Если UserGate расшифровывает HTTPS (подмена сертификата корп-CA), Python отклонит соединение (TLS verify fail) — бот не выйдет в MAX, симптом неотличим от «нет сети». Лечение по предпочтению: **(1) лучше — bypass инспекции** для этого FQDN на UserGate (и правильнее юридически: через канал идут ПДн жителей по 152-ФЗ); (2) если нельзя — вшить корп-CA в trust store *и контейнера, и хоста*. Тонкость: вотчдог-алёрт тоже шлёт curl на max.ru без `--cacert` — при инспекции сломается и он, и оператор не получит ⛑️.
- **⚠️ Прокси не работает без правки кода.** `HTTP_PROXY`/`HTTPS_PROXY` MAX-библиотека **игнорирует** (создаёт HTTP-сессию без `trust_env`). Если в сети наружу пускают *только через прокси* — бот молча не подключится. Нужна либо правка кода (`trust_env=True`), либо прямой исходящий 443 (allowlist).
- **Air-gap несовместим.** MAX — облачный мессенджер, бот сам ходит на `platform-api.max.ru`. Без выхода в интернет к этому домену бот не работает в принципе. Согласовать контролируемое окно исходящего 443 **до** развёртывания.

### Рекомендация для FreeBSD-хоста в гос-сети
**Путь bhyve-VM (Ubuntu 24.04 + Docker внутри), не нативная FreeBSD-сборка.** Потому что: VM даёт настоящий Linux с ровно протестированным в CI стеком (без ручной пересборки и дрейфа версий); сохраняются все hardening'и и все 4 страховки; обновления — штатным автодеплоем внутри VM; VM изолирует линуксовый стек от FreeBSD-серверной. На UserGate — allowlist на `platform-api.max.ru:443` + **bypass SSL-инспекции** для этого FQDN (снимает разом TLS бота, TLS алёрта и не требует поддержки CA в двух местах). Нативный путь — задокументированный fallback только при прямом запрете виртуализации, с письменным перечнем потерь.
