Администратору: установка, настройка, обслуживание

Этот раздел для ИТ-специалиста, который ставит бота на сервер и потом его обслуживает. Здесь всё по шагам: от чистого сервера до рабочего бота, что лежит в настройках, как делать резервные копии (backup) и восстанавливаться, как выкатывать новые версии и откатываться, куда смотреть при сбоях. Команды можно копировать как есть.

Подсказка

Бот «Администрация Елизовского муниципального округа. Обратная связь» — это один процесс в мессенджере MAX, который сам опрашивает сервер MAX (режим long polling — длинный опрос). Он живёт в двух Docker-контейнерах на одной виртуальной машине (VPS): сам бот плюс база данных PostgreSQL. Никаких облаков, никаких входящих портов наружу. Связь только исходящая на серверы MAX.

Что вообще происходит (картинка целиком)

Чтобы понимать, что чинить, держите в голове общую схему:

  1. Шаг. Житель пишет боту в личке. Бот ведёт его по анкете (имя, населённый пункт, адрес, тема, суть) и создаёт обращение №N.
  2. Шаг. Карточка обращения №N уходит в служебную группу MAX (это отдельный чат, где сидят операторы).
  3. Шаг. Оператор отвечает на карточку (свайпом или командой /reply). Бот проверяет ответ и доставляет его жителю в личку.
  4. Шаг. В служебную группу приходит подтверждение «ответ отправлен жителю».

Бот отличает жителя от оператора по идентификатору чата (chat_id). Сообщение из чата, у которого chat_id совпадает с переменной ADMIN_GROUP_ID, считается операторским. Любой другой чат — это житель в личке. Поэтому правильный ADMIN_GROUP_ID в файле настроек — обязательное условие.

Подсказка

Канонический репозиторий проекта: github.com/gaben1488/aemr-bot. Все пути в командах ниже даны для рабочего пользователя aemr и каталога /home/aemr/aemr-bot (в инструкции по первой установке встречается также вариант /opt/aemr-bot — это одно и то же, выберите один путь и держитесь его).

Установка с нуля по шагам

Шаг 1. Получить токен бота через MAX для бизнеса

Боты в MAX регистрируются через корпоративный портал, а не через @MasterBot (он не работает):

  1. Шаг. Откройте https://max.ru/business, войдите под учёткой администратора организации. Юрлицо должно быть верифицировано (это заранее закрывает ИТ-отдел).
  2. Шаг. Раздел «Боты» → «Создать бота» (или откройте уже заведённого).
  3. Шаг. Заполните имя, описание, аватар.
  4. Шаг. Скопируйте Bot API token — длинную строку. Это значение для BOT_TOKEN. Префикс «Bearer» не нужен.

Опасно

Токен — это полный доступ к боту. Храните в защищённом месте (менеджер паролей либо файл .env с правами 600 на сервере). В git не коммитьте. Если токен утёк или уволился человек, у которого был доступ к серверу, — выпустите новый токен на той же странице, старый сразу отзовите.

Шаг 2. Подготовить систему и поставить Docker

Создаём отдельного пользователя для бота, ставим базовые пакеты, включаем брандмауэр (firewall — межсетевой экран):

# Пользователь для бота, добавляем в группу docker
sudo adduser --disabled-password --gecos "" aemr
sudo usermod -aG docker aemr

# Базовые пакеты и firewall: открыт только SSH, всё остальное закрыто
sudo apt update && sudo apt upgrade -y
sudo apt install -y ca-certificates curl gnupg git ufw
sudo ufw default deny incoming
sudo ufw default allow outgoing
sudo ufw allow 22/tcp
sudo ufw enable

Подсказка

Порт бота (8080) слушает только внутри сервера (127.0.0.1) и наружу не публикуется — отдельное правило firewall для него заводить не нужно. Установка Docker Engine и плагина Compose — по официальной инструкции Docker (на старых Ubuntu в стандартных репозиториях лежит устаревший пакет без docker compose). После установки проверьте: docker --version и docker compose version.

Шаг 3. Склонировать репозиторий и подготовить настройки

sudo -u aemr -i
git clone https://github.com/gaben1488/aemr-bot.git
cd aemr-bot

# Скопировать образец настроек и закрыть права (только владелец читает)
cp infra/.env.example infra/.env
chmod 600 infra/.env

Откройте infra/.env и заполните как минимум обязательные поля (полный справочник — ниже в разделе «Файл настроек .env»). На первом запуске ADMIN_GROUP_ID оставьте пустым — его узнаем позже.

Шаг 4. Первый запуск

cd /home/aemr/aemr-bot/infra
docker compose pull
docker compose up -d --build
docker compose ps

Через 30–60 секунд у обоих контейнеров в колонке STATUS должно быть Up (healthy). В логах бота (docker compose logs -f bot) должна появиться строка Starting in long polling mode. Бот уже отвечает в личке жителям. На сообщения в группе он пока не реагирует — ADMIN_GROUP_ID ещё не задан.

Осторожно

После первого запуска нужен один лишний рестарт базы, чтобы заработала статистика медленных запросов (pg_stat_statements подхватывается из настройки shared_preload_libraries только при старте PostgreSQL): docker compose restart db.

Шаг 5. Создать служебную группу и узнать её chat_id

  1. Шаг. В MAX создайте группу (например, «АЕМО · Обратная связь · служебка»). Добавьте туда бота и операторов. В группы добавляются только верифицированные боты — если ошибка, проверьте, что юрлицо подтверждено.
  2. Шаг. В группе напишите /whoami. Бот ответит вашим max_user_id, именем и chat_id группы. chat_id группы — это отрицательное число вида -74181728103785. Сохраните его и свой max_user_id.

Подсказка

Команда /whoami отвечает только в служебной группе. В личке у жителя она тихо игнорируется — это сделано намеренно, чтобы обычные жители не видели служебных команд. Пусть каждый будущий оператор напишет /whoami в группе — соберёте их max_user_id в табличку для массовой регистрации.

Шаг 6. Прописать ADMIN_GROUP_ID и перезапустить

nano /home/aemr/aemr-bot/infra/.env
# вписать: ADMIN_GROUP_ID=-74181728103785
docker compose up -d --force-recreate bot

Перезапуск обязателен — файл .env читается только при старте контейнера. С этого момента сообщения в группе с этим chat_id обрабатываются как операторские.

Шаг 7. Создать первого ИТ-оператора

Команда регистрации операторов доступна только роли it, а при первом запуске в базе нет ни одного it. Чтобы развязать эту петлю, бот умеет создать первого ИТ из переменных окружения. Впишите в .env свой max_user_id и ФИО:

BOOTSTRAP_IT_MAX_USER_ID=<ваш_max_user_id>
BOOTSTRAP_IT_FULL_NAME=Ваше ФИО
docker compose up -d --force-recreate bot

При старте, если в таблице operators нет ни одной активной записи с ролью it, бот вставит её из этих переменных. На повторных запусках — ничего не делает (no-op), переменные можно оставить навсегда. Запасной путь, если автоинициализация не сработала, — напрямую через psql:

docker compose exec db psql -U aemr -d aemr -c \
  "INSERT INTO operators (max_user_id, full_name, role, is_active) \
   VALUES (<ваш_max_user_id>, 'Ваше ФИО', 'it', true);"

Дальше всех остальных операторов удобнее заводить кнопочным мастером «👥 Операторы» в меню /op_help или командой /add_operators — это уже работа оператора, описана в разделе для оператора.

Шаг 8. Установить страховочные службы (один раз)

Эти службы запускаются от root и держат бота живым между перезапусками сервера и при зависаниях:

exit   # выйти из-под пользователя aemr обратно в root

# Auto-deploy: каждые 10 минут тянет новый код из main и пересобирает
sudo bash /home/aemr/aemr-bot/scripts/install-auto-deploy.sh
# выдаст публичный ключ — добавьте его в GitHub: Settings → Deploy keys (read-only)

# Watchdog: каждые 5 минут проверяет, что бот жив
sudo bash /home/aemr/aemr-bot/scripts/install-healthwatch.sh

# Автозапуск контейнеров после перезагрузки сервера
sudo bash /home/aemr/aemr-bot/scripts/install-systemd-autostart.sh

Подробнее про каждую службу — в разделах «Автодеплой и откат», «Сторожевые механизмы», «Автозапуск после reboot» ниже.

Шаг 9. Проверочный прогон

Финальная проверка, что всё работает (полный сценарий со стороны жителя и оператора — в разделе «Чек-лист первого боевого запуска» в конце):

docker compose ps                              # оба контейнера healthy
docker compose logs --tail 50 bot             # последние строки без ERROR
curl -fsS http://127.0.0.1:8080/livez && echo  # 200
curl -fsS http://127.0.0.1:8080/readyz && echo  # 200

Webhook-режим (по умолчанию НЕ используется)

Боевая сборка работает в режиме long polling. Webhook оставлен на случай будущего перехода. Если включаете (BOT_MODE=webhook), нужны:

BOT_MODE=webhook
WEBHOOK_URL=https://feedback.elizovomr.ru/max/webhook
WEBHOOK_SECRET=<≥16 символов: python -c 'import secrets; print(secrets.token_urlsafe(32))'>

Осторожно

Без WEBHOOK_URL и WEBHOOK_SECRET длиной ≥16 символов процесс бота просто не стартует в webhook-режиме (валидатор в config.py остановит запуск). В режиме polling эти поля не нужны.

Repo-sync — синхронизация настроек в репозиторий (опционально)

Фича для роли it: из меню «⚙️ Настройки бота» одной кнопкой создать Pull Request с обновлённым seed/runtime_config.json. Если не нужна — оставьте переменные пустыми, кнопка покажет «не настроено», бот не падает.

GITHUB_PAT=<fine-grained токен: Contents Read/Write + Pull requests Read/Write>
GITHUB_REPO=Gaben1488/aemr-bot
GITHUB_PR_BASE_BRANCH=main
COMMIT_AUTHOR_NAME=<имя автора коммитов>
COMMIT_AUTHOR_EMAIL=<email>

Токен передаётся только в HTTP-заголовке Authorization, не логируется и не появляется в текстах PR. Каждое нажатие создаёт отдельную ветку bot-config-sync-YYYYMMDD-HHMMSS и отдельный PR; в main напрямую бот не пишет. Срок жизни токена ставьте 90 дней и заводите напоминание о ротации.

Осторожно

В старых production-файлах .env встречаются мёртвые переменные COORDINATOR_MAX_USER_ID и BACKUP_TMP_DIR — остатки ранних версий. Бот их не читает (конфиг игнорирует всё, чего нет в схеме). Их можно и нужно удалить, чтобы не путать того, кто откроет файл через год.

Порты и проверки здоровья

Бот слушает порт 8080, но только на петлевом адресе 127.0.0.1 — то есть он доступен только внутри сервера, из интернета его не видно. PostgreSQL на хост вообще не пробрасывается — доступен только боту по внутренней сети Docker.

Технологический стек и образы

Базовые Docker-образы закреплены не только тегом, но и цифровым отпечатком (digest) — чтобы docker compose pull не утянул втихую новый минорный выпуск с поломанными скриптами. Актуальные digest всегда смотрите в infra/docker-compose.yml (не по памяти).

ОбразТегЗачем
python (для бота)3.12-slimсреда исполнения бота
postgres16-alpineбаза данных
nginx1.27-alpineтолько профиль webhook, TLS-терминация
certbotlatestтолько профиль webhook, выпуск сертификатов

Чтобы обновить digest вручную: docker pull postgres:16-alpine, скопировать значение из строки «Digest: sha256:…» в docker-compose.yml. Автообновление выключено намеренно — сломанный или скомпрометированный минор не должен попасть в production молча.

Внутри образа бота дополнительно ставятся apt-пакеты: postgresql-client, gnupg, ca-certificates, curl, tzdata, unzip, rclone (для S3-выгрузки). Python-зависимости фиксируются через ~= в bot/pyproject.toml (патчи разрешены, скачки минорных версий — нет). Ключевые из них:

Подсказка

Точные согласованные версии всех пакетов (включая транзитивные) лежат в bot/uv.lock — он закоммичен в репозиторий. Обновление любой зависимости делается отдельным Pull Request с проверкой каждого слоя, процедура — в docs/DEPS.md и docs/MAXAPI_UPGRADE_PROCEDURE.md.

Фоновые задачи (cron внутри бота)

Внутри процесса бота крутится планировщик APScheduler — это «cron» (запуск по расписанию), но не системный, а встроенный. Все расписания — в часовом поясе Asia/Kamchatka. Подсказка

Несколько тонкостей расписания. Напоминалки операторам не дёргают в обеденный перерыв (12:00–13:00 выпадает из расписания) и в субботу. «Пульс» же идёт 24/7 — это технический сигнал «процесс жив», и непрерывность мониторинга важнее. Напоминалки по SLA молчат в государственные праздники РФ (даты берутся из seed/holidays.json). Все задачи имеют «окно прощения» опоздания (misfire_grace_time) 120 секунд — это покрывает паузу 30–90 сек, пока контейнер пересобирается при деплое, так что пропущенный тик догоняется. Дублей не будет: настройки coalesce=True и max_instances=1 сливают просроченные тики в один и не дают запустить второй экземпляр.

Осторожно

Планировщик хранит расписание в памяти. Если сервер был выключен ровно в момент еженедельного backup — этот запуск не наверстается (в отличие от тиков, пропущенных на короткую пересборку: их покрывает окно прощения). Перед опасными операциями (миграции, выкладка) делайте /backup руками.

Резервное копирование и восстановление

Что и как бэкапится

Каждое воскресенье в 03:00 по Камчатке бот делает pg_dump всей базы, шифрует его GPG (алгоритм AES-256) с паролем BACKUP_GPG_PASSPHRASE и кладёт файл в Docker-том backups (внутри контейнера путь /backups/aemr-YYYYMMDD_HHMMSS.sql.gpg, права 0600). Хранятся последние 8 файлов (≈2 месяца). Опционально — дополнительная копия в S3.

Опасно

Без BACKUP_GPG_PASSPHRASE (или с фразой короче 12 символов) backup отказывается выполняться и шлёт алёрт в служебную группу. Это требование закона 152-ФЗ: дамп содержит телефоны жителей, имена и тексты обращений — это персональные данные (ПДн), хранить их на диске в открытом виде нельзя. Для dev-машины открыть незашифрованный режим можно явным BACKUP_ALLOW_UNENCRYPTED=1, но в production это запрещено.

Снять backup вручную и посмотреть список

/backupиз служебной группы (только роль it) — внеплановый pg_dump прямо сейчас
# Список файлов
docker compose exec bot ls -lah /backups

# Снять backup из shell (альтернатива кнопке)
docker compose exec bot python -m aemr_bot.services.db_backup

Восстановление из backup

Опасно

Никогда не восстанавливайте backup прямо в боевую базу aemr. Всегда сначала в отдельную тестовую базу aemr_restore_test, сверьте, и только потом принимайте решение. Восстановление из backup — терминальный шаг: теряются данные, появившиеся между моментом backup и моментом сбоя.

  1. Шаг. Найдите свежий файл: sudo ls -lah /var/lib/docker/volumes/aemr-bot_backups/_data/
  2. Шаг. Создайте тестовую базу: 
    cd /home/aemr/aemr-bot/infra
. ./.env
docker compose exec -T db psql -U "$POSTGRES_USER" -d postgres -c 'DROP DATABASE IF EXISTS aemr_restore_test;'
docker compose exec -T db psql -U "$POSTGRES_USER" -d postgres -c 'CREATE DATABASE aemr_restore_test;'
  3. Шаг. Расшифруйте и залейте (для зашифрованного .sql.gpg): 
    BACKUP=/var/lib/docker/volumes/aemr-bot_backups/_data/aemr-YYYYMMDD_HHMMSS.sql.gpg
sudo cat "$BACKUP" \
  | gpg --batch --passphrase "$BACKUP_GPG_PASSPHRASE" --decrypt \
  | docker compose exec -T db psql -U "$POSTGRES_USER" -d aemr_restore_test
  4. Шаг. Проверьте, что таблицы есть и счётчики правдоподобны: 
    docker compose exec -T db psql -U "$POSTGRES_USER" -d aemr_restore_test -c '\dt'
docker compose exec -T db psql -U "$POSTGRES_USER" -d aemr_restore_test -c 'select count(*) from users;'
docker compose exec -T db psql -U "$POSTGRES_USER" -d aemr_restore_test -c 'select count(*) from appeals;'
  5. Шаг. Удалите тестовую базу: docker compose exec -T db psql -U "$POSTGRES_USER" -d postgres -c 'DROP DATABASE IF EXISTS aemr_restore_test;'

Опасно

Если пароль BACKUP_GPG_PASSPHRASE утерян — зашифрованный backup восстановить невозможно. Поэтому фразу храните отдельно от сервера (в менеджере паролей). Полный регламент проверки восстановления — в docs/BACKUP_RESTORE_TEST.md.

Тест восстановления (раз в квартал)

Backup, который ни разу не восстанавливали, официально считается непроверенным. Раз в квартал (в первый рабочий день января, апреля, июля, октября) проходите процедуру восстановления из docs/BACKUP_RESTORE_TEST.md в тестовую базу и записывайте результат в журнал эксплуатации (кто, когда, итог). Проверка занимает ~15 минут. Дополнительно — после каждой новой миграции схемы.

Осторожно

Локальный backup защищает от логической ошибки в базе, но не от потери самой VPS. Если сгорит диск — потеряются и база, и backup, лежащие на нём. Цель восстановления по времени при потере всего сервера (RPO) — до 1 недели (backup раз в воскресенье). Чтобы это закрыть, либо включите BACKUP_S3_*, либо настройте внешний rsync/scp по cron на отдельный хост. Сейчас offsite-копии нет — это документированный риск.

Выкладка новой версии и откат

Автодеплой (auto-deploy)

Скрипт scripts/auto-deploy.sh запускается из cron от root каждые 10 минут. Логика:

  1. Шаг. git fetch origin main. Если новых коммитов нет — тихо выходит, контейнер не трогает.
  2. Шаг. Запоминает текущий коммит как «предыдущий рабочий» (для отката).
  3. Шаг. git reset --hard origin/main и docker compose up -d --build.
  4. Шаг. Health-gate: до 60 секунд опрашивает /livez каждые 5 секунд.
  5. Шаг. Успех → запись в журнал «deploy ok». Провал → авто-откат на предыдущий коммит, пересборка, запись «DEPLOY FAILED: ROLLBACK».

Подсказка

Откат управляется только проверкой /livez, а не /readyz — кратковременная проблема с базой не должна откатывать корректный код. Авто-деплой тянет код только из ветки main; ветки Pull Request на сервер не попадают. После мержа PR изменения доезжают на сервер в течение 10 минут. Логи деплоя: journalctl -t aemr-bot-deploy -n 80 --no-pager.

Ручная выкладка

sudo -u aemr -i
cd aemr-bot
git fetch origin main && git checkout main && git pull --ff-only
cd infra
docker compose up -d --build bot
docker compose logs -f bot     # убедиться, что стартовал без ошибок

Миграции базы (alembic) применяются автоматически при старте контейнера (команда запуска — alembic upgrade head && python -m aemr_bot.main). Если миграция падает — контейнер уходит в перезапуск; смотрите docker compose logs --tail 200 bot | grep -A20 alembic. После выкладки пройдите чек-лист docs/VPS_SMOKE_CHECKLIST.md.

Откат при сбое (цель — за ≤10 минут)

Подробный playbook — docs/ROLLBACK.md. Главное: бот «health-down» НЕ значит «потеря данных» — база живёт независимо от приложения, откатываем только код.

  1. Шаг. Сначала диагностика, 30 секунд логов: docker logs aemr-bot-bot-1 --tail 50. Если ошибка явно временная (сеть, таймаут MAX) — не откатывайте, дайте 5 минут на самовосстановление.
  2. Шаг. Откат кода на сервере: на коммит назад через git reset --hard HEAD~1, chown -R aemr:aemr ., затем docker compose up -d --build.
  3. Шаг. Сразу сделайте revert-коммит в репозитории на машине разработчика (git revert HEAD --no-edit && git push origin main) — иначе через 10 минут авто-деплой снова подтянет сломанный main и снова упадёт.
  4. Шаг. Если виновата миграция и она обратима — docker exec aemr-bot-bot-1 alembic downgrade -1. Но не каждая миграция обратима: проверьте, есть ли реальный код в downgrade(). На проде даунгрейд миграций — крайняя мера, обычно проще восстановить из backup.
  5. Шаг. После восстановления убедитесь, что /livez отвечает 200, и дождитесь «пульса» в служебной группе.

Сторожевые механизмы и автозапуск

Watchdog (внешний сторож)

Внутри контейнера уже есть Docker healthcheck и политика restart: unless-stopped — если процесс упал, Docker сам поднимет контейнер. Но Docker не перезапускает контейнер, который «жив, но не отвечает» (висит). Для этого случая есть скрипт scripts/healthwatch.sh в cron от root каждые 5 минут:

  • успешный /livez → счётчик провалов сбрасывается;
  • 3 провала подряд (≈15 минут) → docker compose restart bot;
  • 8 провалов подряд (≈40 минут) → пишет алёрт прямо в служебную группу через MAX API (минуя бота — он же завис), отправляя POST на platform-api.max.ru/messages с заголовком Authorization.

Подсказка

Не путайте watchdog и health-gate авто-деплоя. Health-gate ловит сломанный новый коммит сразу после выкладки. Watchdog ловит зависший живой контейнер в любой момент. Логи сторожа: journalctl -t aemr-bot-watchdog -n 80 --no-pager.

Автозапуск после перезагрузки сервера

Контейнеры с restart: unless-stopped поднимаются вместе с Docker при штатной перезагрузке. Но если до reboot контейнеры были остановлены вручную (docker compose down) — Docker сам их не вернёт. На этот случай есть systemd-юнит infra/aemr-bot.service, который при старте системы делает docker compose up -d. Операция идемпотентная (если уже подняты — ничего не меняет). Устанавливается один раз: sudo bash /home/aemr/aemr-bot/scripts/install-systemd-autostart.sh. Проверка: sudo systemctl is-enabled aemr-bot.service (должно быть enabled).

Логи

Все три контейнера пишут логи в формате json-file с ротацией: 10 МБ × 3 файла = до 30 МБ на контейнер. Бесконечного роста логов не будет.

cd /home/aemr/aemr-bot/infra
docker compose logs --tail 300 bot                                   # последние 300 строк бота
docker compose logs --since 24h bot | grep -Ei 'error|exception|misfire|pulse'  # за сутки, отфильтровано
docker compose logs --tail 200 db                                    # логи PostgreSQL

Осторожно

Docker отдельно хранит сырой stdout контейнера в файлах /var/lib/docker/containers/<id>/<id>-json.log. В них попадают псевдоидентификаторы жителей (max_user_id), а команда /erase чистит только базу, не эти файлы. После запроса жителя на удаление данных вручную обнулите docker-логи бота за последние 7 дней, не удаляя сам файл (Docker держит его открытым): sudo truncate -s 0 /var/lib/docker/containers/<id>/*-json.log. Это компромисс под 152-ФЗ, чтобы не строить отдельный пайплайн ради одного запроса в месяц.

Диагностика типичных сбоев

СимптомКуда смотреть / что делать
Бот молчит, в служебной группе тишинаСначала curl http://127.0.0.1:8080/livez. Если 200 — процесс жив, проблема в сессии MAX или сети. Если 5xx или таймаут — docker compose logs --tail 200 bot. Контейнер упал → docker compose up -d bot. Жив, но молчит → docker compose restart bot.
Бот в группе игнорирует сообщенияПроверьте, что ADMIN_GROUP_ID в .env совпадает с chat_id из /whoami в группе (минус на месте?). Бот после правки перезапущен? Бот не покинул группу? Если /whoami в группе ещё отвечает — бот видит группу, но chat_id не совпадает.
Оператор пишет, а бот не доставляетОтвет должен быть свайпом на карточку (иначе бот промолчит) и не длиннее 300 символов. Оператор зарегистрирован в operators с is_active=true? Логи: docker compose logs bot --tail 100 | grep -i error (частые причины: житель заблокировал бота, MAX рвёт TLS).

База растёт быстрее ожидаемогоdocker compose exec db psql -U aemr aemr -c "\dt+" — размеры таблиц. Обычно самая большая events (защита от повторов), её сдерживает retention 30 дней.
/readyz падает, а /livez окПроцесс жив, но база лежит или зависла. Не рестартуйте бота — сначала docker compose logs --tail 100 db и проверьте DATABASE_URL.

Аудит сервера одной командой

Скрипт собирает срез состояния сервера для передачи в чат или тикет:

sudo bash /home/aemr/aemr-bot/scripts/audit_vps.shвыгрузка в /tmp/aemr_audit_<timestamp>.tar.gz

Внутри: системные данные, состояние git, .env с замаскированными секретами, docker compose ps, проверки здоровья, последние логи бота и базы. Скрипт не выводит реальные значения BOT_TOKEN, POSTGRES_PASSWORD, BACKUP_GPG_PASSPHRASE, WEBHOOK_SECRET и ключей S3 — все заменены на ***HIDDEN***, поэтому архив безопасно пересылать.

Ротация секретов

Чтобы скомпрометированный секрет не жил годами, держите плановую ротацию. Заведите напоминание в календарь и проходитесь по списку.

Сетевая изоляция (опционально)

В боевой сборке бот делает исходящие запросы строго на ограниченный список доменов: MAX API (botapi.max.ru, platform-api.max.ru), GitHub (для авто-деплоя), пинг-сервис (если задан HEALTHCHECK_URL) и источники угроз для антифишинга (urlhaus.abuse.ch, threatfox.abuse.ch, data.phishtank.com). Всё остальное — потенциальная утечка ПДн или признак компрометации.

Если нужен жёсткий исходящий whitelist через iptables — порядок такой: запретить весь исходящий трафик (OUTPUT DROP), затем точечно разрешить loopback, установленные соединения, DNS (порт 53), перечисленные домены на 443/TCP и NTP (порт 123). Точная схема — в docs/SYSADMIN.md §13a.

Осторожно

Подводные камни: IP-адреса CDN меняются — правила надо пересоздавать по расписанию либо использовать nftables с динамическим резолвингом. Авто-деплой дёргает github.com каждые 10 минут — его блокировать нельзя, иначе деплои встанут. Если включён S3-backup — добавьте endpoint S3 в whitelist. В текущей инсталляции исходящий whitelist не включён: на VPS работает только бот, риск утечки минимальный. Включать стоит при появлении других сервисов на той же машине.

SBOM и CI

Если нужен отчёт о составе компонентов (SBOM — Software Bill of Materials), сгенерируйте его из контейнера:

docker compose exec bot pip-audit --format=cyclonedx-json > sbom-$(date +%Y%m%d).json

Утилита pip-audit уже в dev-зависимостях. SBOM публикуйте в защищённую зону репозитория, не в публичный README.

На каждый push в main и каждый Pull Request гоняется .github/workflows/ci.yml: проверка стиля (ruff, mypy, bandit), аудит уязвимостей (pip-audit --strict — жёсткое падение на любой CVE), shellcheck на все скрипты; тесты с порогом покрытия --cov-fail-under=67, проверка миграций (alembic upgrade head, drift между моделями и миграциями, round-trip upgrade→downgrade→upgrade); смок-сборка образа. На сервер попадает только то, что прошло CI и смержено в main.

Частые вопросы

Чем менять настройки бота — файлом или через бота?

Зависит от того, что меняете. Технические параметры (токен, пароль базы, ADMIN_GROUP_ID, расписание backup, webhook) — только в infra/.env, с перезапуском контейнера. Содержательные настройки (тематики обращений, населённые пункты, тексты приветствия и согласия, контакты служб, ссылки) — в таблице settings, через меню «⚙️ Настройки бота» в /op_help (роль it) или командой /setting, без редеплоя. Этих редактируемых ключей 17. Список операторов — в таблице operators через мастер «👥 Операторы».

Нужно ли открывать порты в firewall?

Нет. В режиме long polling бот сам исходящими запросами опрашивает MAX, входящих соединений снаружи нет. Откройте только SSH (22). Порт 8080 слушает на 127.0.0.1 и наружу не виден. Порты 80/443 нужны только в webhook-режиме, который в боевой сборке не используется.

Где физически лежат backup?

В Docker-томе backups, на диске сервера по пути /var/lib/docker/volumes/aemr-bot_backups/_data/. Том переживает пересборку контейнера. Уносить копии с сервера на отдельный хост — отдельная задача ИТ-отдела (S3 или внешний rsync), без неё локальный backup не спасёт от потери всей VPS.

Бот ушёл в restart-loop после деплоя — что делать?

Если включён авто-деплой, он сам должен был откатиться по health-gate (ищите «DEPLOY FAILED: ROLLBACK» в journalctl -t aemr-bot-deploy). Если не откатился — откатитесь вручную (см. «Откат при сбое») и обязательно сделайте revert-коммит в репозитории, иначе авто-деплой через 10 минут снова подтянет сломанный код.

Почему в служебной группе у бота нет списка команд в меню «/»?

Так задумано. MAX не умеет показывать разные команды в личке и в группе. Раньше 7 команд жителя были видны и операторам, что путало. Поэтому при старте бот очищает /-меню для всех чатов (отправляет в MAX пустой список команд). Сами команды по-прежнему работают, если набрать их руками; жители работают через кнопочное меню, операторы — через /op_help.

Можно ли восстановить backup, если потерян GPG-пароль?

Нет. Зашифрованный backup без пароля не расшифровать — это by design. Именно поэтому BACKUP_GPG_PASSPHRASE храните отдельно от сервера (в менеджере паролей), а тест восстановления проводите раз в квартал, чтобы вовремя заметить проблему.

Что будет, если сервер выключится в момент еженедельного backup?

Этот запуск не наверстается — планировщик хранит расписание в памяти. На следующее воскресенье цикл продолжится. Перед опасными операциями делайте /backup руками. Короткие пропуски (на пересборку контейнера 30–90 сек) при этом догоняются за счёт окна прощения опоздания (120 сек) — это касается частых задач вроде «пульса», но не еженедельного backup.
