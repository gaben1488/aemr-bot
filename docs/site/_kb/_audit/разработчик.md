> Источник истины — код (`bot/aemr_bot`) + выверенные факты `docs/site/_kb/_LEDGER.md`.
> Сверяемый раздел вики — `docs/site/_kb2/developer.html` (на 2026-06-01 — 830 строк, очень полный: стек, deps/uv, архитектура, дерево, БД+миграции, локальный запуск, тесты, CI, рецепты, ADR-обоснования, грабли maxapi, лимиты, деплой, словарь, FAQ).
> Пометки: ✓ отвечен в вики · ~ частично · ✗ пробел.
> Код-сверка проведена по: `handlers/__init__.py` (register_handlers, middleware), `callback_router.py` + `callback_payloads.py` (+ контракт-тест), `main.py` (bootstrap), `services/admin_bus.py`, `utils/menu_tracker.py`, `services/cron.py`/`cron_registry.py`, `config.py`, `infra/Dockerfile`, `.github/workflows/ci.yml`.

Раздел `developer.html` — один из самых сильных в вике: он закрывает почти весь повседневный жизненный цикл контрибьютора (запуск, deps-дисциплина, тесты, CI, рецепты «как добавить X», грабли maxapi, ADR-обоснования). Большинство вопросов — ✓. Реальные пробелы сосредоточены в нескольких зонах, и одна из них — **прямое противоречие кода и вики**:

1. **Типизированный callback-payload — УЖЕ реализован, но вика называет его «отложенным на будущее».** `handlers/callback_payloads.py` (константы + типизированные билдеры) + `handlers/callback_router.py` (реестр маршрутов с группой/`admin_allowed`) + контракт-тест `tests/test_callback_payloads_contract.py` — это шипнутая, центральная для добавления кнопки система. А developer.html в свёрнутом блоке «Нереализованный потенциал maxapi» пишет, что `CallbackPayload` с `pack()/unpack()` «сделал бы регрессии невозможными» — как о несделанном. Это самый опасный пробел: разработчик, добавляющий кнопку, по вике будет парсить `.split(":")` руками, хотя в коде есть готовый контракт. ✗
2. **Sacred event log и freshness глазами разработчика** (как добавить вывод в админ-чат, не сломав edit-vs-send): механика `admin_bus` + dual-tracker `menu_tracker` в developer.html почти не описана со стороны «как НЕ сломать». ✗/~
3. **Middleware-слой как точка расширения** (`register_outer_middleware`, порядок, ловушка `isinstance(MessageCreated)`): упомянут фрагментарно. ~
4. **Полнота рецептов**: «добавить callback-кнопку оператора» (а не только пункт меню жителя), «добавить роль оператора», «добавить статус обращения», «добавить поле в карточку» — этих рецептов в таблице «как добавить X» нет. ~/✗
5. **Точечные ~**: отладка без Docker, доступ к приватному репо/секретам, работа с seed, локальный smoke без админ-группы, обновление guard-теста при бампе maxapi, поведение фоновых задач (`spawn_background_task`).

## 11. Архитектурные обоснования (почему так)

116. Почему JSONB, а не Redis для состояния анкеты? — ✓ (ADR: десятки воронок в пик, Postgres проглатывает; триггер пересмотра >500 одновременных)
117. Почему обязателен слой защиты от повторов? — ✓ (без него webhook даёт двойные карточки; юридический смысл — одно действие ≠ два обращения)
118. Почему лимит ответа 300 символов? — ✓ (ANSWER_MAX_CHARS; верхняя граница требования куратора 200–300; не ломает шаблоны 250–280)
119. Почему подписка — отдельное согласие, а скорость — половина лимита? — ✓ (opt-in отдельная цель ПДн; снимок аудитории; 1 msg/sec = половина платформенного лимита 2 rps; «Экстренно остановить» — любому оператору)
120. Почему один экземпляр и что сломает масштабирование? — ✓ (asyncio.Lock + wizard-словари в памяти; нужен advisory lock/Redis)
121. Почему таймаут опроса — это не интервал между запросами? — ✓ (серверное время удержания getUpdates; на 0 — хлещет API; реализовано подменой `bot.get_updates` в main.py)
122. Зачем вообще подменять `bot.get_updates`? — ✓ (`Dispatcher.start_polling` не пробрасывает таймаут внутрь; первый кандидат на форк maxapi)
123. Почему SLA в коде 4 часа, а в Регламенте иначе? — ✓ (единый базовый 4ч, нет поля важности/маршрутизации; дифференциацию ведёт Куратор ЦУР вручную)
124. Инженерное следствие отсутствия приоритета по теме? — ✓ (чтобы ввести разные сроки — сперва добавить поле важности/маршрута в схему)
125. Почему 365 дней на audit_log, а не 5 лет? — ✓ (журнал событий безопасности, ФСТЭК РСБ.1; отдельный 5-летний ретеншн обращений не привязан)
126. Где зафиксированы эти решения, чтобы их не переоткрывали? — ✓ (в ADR и в коде; раздел «Архитектурные обоснования»)
