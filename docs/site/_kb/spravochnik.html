Справочник: переменные, схема БД, расписание, секреты, словарь

Это «сухой» раздел-справочник. Здесь собраны точные значения, по которым настраивается и работает бот: переменные окружения (env — настройки, которые читаются из файла infra/.env при запуске), все таблицы базы данных, список миграций, расписание фоновых задач, ключи настроек, реестр секретов и словарь терминов. Все значения сверены с реальным кодом репозитория, а не только с документацией. Если в старых документах встретится другое число — верьте этому разделу и коду.

Подсказка

Главное правило: env-переменные задаются один раз в файле infra/.env и применяются при перезапуске бота (docker compose restart bot). А «настройки» (ссылки, тексты, контакты, тематики) меняются прямо на ходу через меню «Настройки бота» в служебной группе и хранятся в таблице settings — перезапуск им не нужен. Это два разных механизма, не путайте.

Полный реестр переменных окружения

Ниже все переменные из файла-образца infra/.env.example. Большинство необязательны: если переменная не задана или закомментирована знаком #, бот берёт встроенное значение по умолчанию из кода. Обязательные перечислены первыми в каждой группе. Значения сверены с bot/aemr_bot/config.py (класс Settings).

Имя и почта автора коммитов берутся не из env, а из настроек в базе (ключи commit_author_name / commit_author_email, меню «Автор коммитов»). Есть ещё одна необязательная переменная окружения — PHISHTANK_APP_KEY: если её задать, к источникам антифишинга добавится PhishTank. Её нет в файле-образце, потому что URLhaus и ThreatFox работают и без ключей.

Схема базы данных

База — PostgreSQL. Все таблицы описаны в файле bot/aemr_bot/db/models.py. Ниже — 11 таблиц с их полями, связями и важными правилами (инвариантами). Тип jsonb — это «гибкое» поле, куда кладут структуру (список вложений, произвольный словарь). Тип timestamptz — дата и время с часовым поясом. «PK» — первичный ключ (primary key, уникальный идентификатор строки), «FK» — внешний ключ (foreign key, ссылка на строку другой таблицы), «UK» — ограничение уникальности.

Как таблицы связаны

  1. users → appeals. Один житель — много обращений (appeals.user_id ссылается на users.id). При удалении жителя его обращения удаляются каскадом.
  2. appeals → messages. Одно обращение — много сообщений переписки (messages.appeal_id). Каскадное удаление.
  3. operators → appeals. Оператор может быть назначен на обращение (appeals.assigned_operator_id). При удалении оператора ссылка обнуляется.
  4. operators → messages. Сообщение оператора помнит, кто его написал (messages.operator_id).
  5. broadcasts → broadcast_deliveries. Одна рассылка — много записей о доставке (по одной на получателя).
  6. operators → broadcasts / broadcast_templates. Рассылку и шаблон помнят, какой оператор их создал.

Ключевые правила (инварианты)

  • users.max_user_id уникален — дублей жителей не возникает, при следующем /start находят существующую запись.
  • events.idempotency_key уникален — повтор того же события MAX тихо отбрасывается ограничением уникальности.
  • appeals.admin_message_id до публикации карточки пусто; после — это якорь, по которому оператор отвечает свайпом или командой /reply.
  • Подписчики рассылки определяются условием subscribed_broadcast=true и is_blocked=false (плюс наличие согласия на рассылку). После полного удаления данных оба флага переключаются так, чтобы житель больше не получал сообщений без нового согласия.
  • broadcasts.subscriber_count_at_start — снимок на момент старта, в ходе рассылки не меняется.
  • Особая «техническая» запись жителя с max_user_id = -1 (anonymous) — на неё переподвешиваются обращения после полного удаления персональных данных, чтобы статистика «было обращение №N» сохранилась, а сам житель исчез.

Подсказка

Сигнальные задачи (pulse, напоминания) и алёрты по-разному реагируют на «тихий режим» (quiet hours — настройка admin_quiet_hours_enabled). Обычный пульс и не-срочные уведомления в тихие часы молчат, а критические алёрты (упавший бэкап, ошибки очистки данных, авто-деактивация операторов, аномалия в воронке) пробивают тишину — оператор узнаёт о реальном инциденте сразу.

Частые вопросы

Где задать пароль базы — в одном месте или в двух?

Пароль придумывается один раз и попадает в POSTGRES_PASSWORD. Эту же строку нужно вписать внутрь DATABASE_URL (часть USER:PASS@db:5432/DB). То есть пароль фигурирует в файле дважды и должен совпадать, иначе бот не зайдёт в базу.

Чем «переменная окружения» отличается от «настройки»?

Переменная окружения (env) — это техническая настройка из файла infra/.env: токен, пароль, лимиты, расписание бэкапа. Меняется редко и требует перезапуска бота. «Настройка» (таблица settings) — это содержательные значения: ссылки, тексты, контакты, тематики. Меняется на ходу через меню бота, перезапуск не нужен.

Почему бэкап иногда «не создаётся»?

По умолчанию бот отказывается делать незашифрованную копию, потому что в дампе лежат персональные данные. Чтобы бэкап заработал, задайте парольную фразу BACKUP_GPG_PASSPHRASE (рабочий вариант) либо явно разрешите незашифрованную копию через BACKUP_ALLOW_UNENCRYPTED=1 (только на машине разработчика). При отказе бот пишет понятное сообщение в служебную группу.

Почему в коде осталось «AEMR», хотя теперь округ?

«АЕМР» — прежнее название (район). Оно зашито как технический идентификатор: имя проекта aemr-bot, роль оператора aemr, переменные. Переименовывать это в коде рискованно, поэтому в интерфейсе для жителей используется актуальное «округ» (АЕМО), а внутри техники остаётся «aemr».

Сколько хранятся данные?

События MAX (таблица events) — 30 дней. Журнал действий (audit_log) — по умолчанию 365 дней (AUDIT_LOG_RETENTION_DAYS). Персональные данные жителя после отзыва согласия обезличиваются через 30 дней. Тексты обращений старше 5 лет обнуляются, но метаданные (даты, статусы, номера) сохраняются для статистики. Резервных копий хранится последние BACKUP_KEEP_COUNT штук (по умолчанию 8 ≈ два месяца).
