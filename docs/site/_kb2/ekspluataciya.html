Бот лёг — реанимация по шагам

Если бот перестал отвечать, не нужно паниковать и не нужно ничего удалять. Сначала смотрим, что происходит, и только потом действуем. База данных живёт в Docker независимо от самого бота, поэтому «бот не отвечает» почти никогда не значит «данные потеряны».

Главная ловушка, из-за которой бот лежал двое суток

Не выключайте Docker, чтобы «починить бота»: в Docker живёт не только бот, но и база данных Postgres. Выключенный Docker — это выключенная база, бот не сможет работать ни в каком виде. И не снимайте оба cron-задания: вотчдог — единственное, что поднимает зависший процесс. Если выключить и Docker, и cron, бота не поднимет уже никто (это и был инцидент: «лежит двое суток»).

  1. Понять, что происходит (четыре команды, ничего не меняют): 
    systemctl is-active docker     # ждём active; inactive — причина №1: Docker выключен
docker compose ps              # из ~/aemr-bot/infra; ждём два контейнера Up
ss -ltnp | grep 8080           # кто держит health-порт (должен один docker-proxy)
sudo crontab -l | grep aemr    # живы ли страховки-cron'ы
  2. Включить Docker, если он выключен: sudo systemctl enable --now docker.
  3. Убедиться, что нет второго бота вне Docker (он тихо ворует сообщения, см. ниже): ps aux | grep aemr_bot | grep -v grep и systemctl list-units | grep -i aemr. Любой посторонний процесс или юнит — погасить.
  4. Поднять стек: cd ~/aemr-bot/infra && docker compose up -d.
  5. Проверить через 30–60 секунд: docker compose ps (оба Up, bot healthy), curl -s http://127.0.0.1:8080/readyz (должно {"ok": true}), docker compose logs --tail 50 bot — в логе ищем строки «preflight: токен валидный — бот …» и «Starting in long polling mode». Если контейнер в Restarting — идём в форс-мажоры, первым делом docker compose logs bot | grep 'BOT_TOKEN неверный'.
  6. Контроль: в служебной группе MAX в течение часа придёт «🟢 Пульс: бот работает». Пришёл — инцидент закрыт.

Почему бот «умирал сам» при живом сервере — запуск на переднем плане

Если стек запущен командой docker compose up / podman compose up на переднем плане (процессом в терминале, а не фоном и не службой), он живёт ровно пока открыт терминал. Закрыли SSH-сессию, разлогинились, оборвалась связь — процесс получает сигнал, и контейнеры гаснут. В логах это видно как bot-1 | Terminated и db-1 | received fast shutdown request. Это и есть «бот падал, когда сисадмин закрывал терминал».

Лечение — запускать не на переднем плане. Минимум: docker compose up -d (фоном). Правильно: как службу, которая переживает логаут и перезагрузку — под Podman это Quadlet-юнит или aemr-bot-compose.service (см. «Запуск под Podman как systemd-служба»), под Docker — systemd-юнит поверх compose up -d. nohup/screen — не постоянное решение, а костыль на один раз. Логи при этом всё равно сохраняются на диск (см. «Логи»), так что разбор «почему лёг» не зависит от живого контейнера.

Кто кого перезапускает

У бота несколько уровней страховки, и каждый закрывает свою беду. Подробно про вотчдог и автозапуск — в разделе «Сторожевые механизмы»; здесь короткая сводка, кто реагирует на что.

Что случилосьКто поднимает
Процесс бота упал (любой выход)Docker, политика restart: unless-stopped
Процесс жив, но завис («живой труп», unhealthy)только вотчдог-cron healthwatch (если установлен)
Перезагрузка сервераsystemd-юнит → docker compose up -d (если docker.service включён)
Docker выключен и cron'ы снятыникто — это и есть «лежит двое суток»

Важно не путать: systemctl stop aemr-bot выполняет docker compose down и гасит и бота, и базу. Перезапуск самого бота — это docker compose restart bot, а не systemctl. Логи при down не теряются — они на диске хоста, см. «Логи» ниже.

Логи: где лежат и почему переживают перезапуск

Важная тонкость: логи, которые показывает docker compose logs / podman logs, по умолчанию хранятся внутри контейнера и удаляются вместе с ним. Сделали docker compose down или пересобрали образ — и история пропала. Для бота, который отвечает жителям и ведёт журнал по 152-ФЗ, это недопустимо. Поэтому логи дублируются на диск сервера двумя независимыми слоями, и оба переживают удаление контейнера.

Слой 1 — файл приложения bot.log. Бот сам пишет читаемый лог в каталог /var/log/aemr-bot, примонтированный с хоста, с автоматической ротацией (по умолчанию 10 файлов по 10 МБ, дальше самые старые удаляются). Это главный человекочитаемый журнал, и он не зависит от жизни контейнера:

tail -f /var/log/aemr-bot/bot.log         # следить в реальном времени
grep ERROR /var/log/aemr-bot/bot.log      # найти ошибки

В файл попадает всё Python-логирование, включая саму библиотеку maxapi (обработчик висит на корневом логгере), а не только наш код.

Слой 2 — journald. Драйвер логов контейнера переключён на journald, поэтому весь вывод stdout/stderr уходит в системный журнал хоста и тоже переживает удаление контейнера. Этот слой ловит то, что до файлового логгера приложения не доходит: вывод alembic при старте и аварийные трейсбэки до инициализации логирования.

journalctl CONTAINER_NAME=aemr-bot -f     # Docker
journalctl -u aemr-bot.service -f         # Podman/Quadlet
docker compose logs -f bot                # тоже работает (читает journald)

Разовая подготовка хоста

Чтобы оба слоя писали на диск, на хосте нужно один раз создать каталог логов с владельцем UID 1000 (пользователь внутри контейнера, иначе read-only-контейнер не сможет в него писать) и включить персистентный journald (иначе журнал живёт только в памяти и гибнет при перезагрузке):

sudo mkdir -p /var/log/aemr-bot && sudo chown 1000:1000 /var/log/aemr-bot
sudo mkdir -p /var/log/journal && sudo systemctl restart systemd-journald

Для Podman это делает установщик install-podman-quadlet.sh автоматически; для Docker выполните вручную перед docker compose up. Каталог, размер и число файлов настраиваются переменными LOG_DIR, LOG_FILE_MAX_BYTES, LOG_FILE_BACKUPS в infra/.env; пустой LOG_DIR выключает файловый слой (например, на dev-машине).

Тонкость для SELinux-систем (Astra / РЕД ОС): к bind-mount каталога логов в Podman добавьте суффикс :Z, иначе метки контекста дадут «permission denied».

Два бота на один токен и вывод лишней копии

Токен BOT_TOKEN — это одновременно логин, пароль и адрес бота в MAX. Если этот токен крутят два процесса (например, на старом и на новом сервере), оба представляются одним ботом, и начинается тихий хаос: они делят входящие сообщения между собой, половина нажатий жителей теряется, а в логах ни единой ошибки. На одном хосте второй экземпляр умер бы сам на занятом порту 8080, но на разных хостах такого столкновения нет — поэтому всё тихо.

Восстановление: выбрать один канонический хост, на всех остальных погасить бота (docker compose down), снять там страховки (иначе вернётся в течение десяти минут — это и был эффект «поработал пять минут и умер»), подождать минуту, чтобы MAX забыл лишнюю polling-сессию, и проверить, что канонический в одиночку получает все сообщения. Самая надёжная гарантия — перевыпустить токен в max.ru/business и вписать новый только на канонический хост: старый мгновенно умирает, любая забытая копия отваливается сама.

Как навсегда обезвредить старую копию на сервере

Чтобы старый сервер никогда не поднял второй экземпляр, на нём убирают страховки (нужен root):

# снять оба cron-задания (это они воскрешают бота)
sudo crontab -l | grep -v aemr-bot | sudo crontab -
# удалить обёртки и состояние вотчдога
sudo rm -f /usr/local/bin/aemr-bot-autodeploy /usr/local/bin/aemr-bot-healthwatch
sudo rm -rf /var/lib/aemr-bot-watchdog
# на всякий случай снять автозапуск, если он был
sudo systemctl disable --now aemr-bot.service 2>/dev/null
# убедиться, что бот не запущен
cd ~/aemr-bot/infra && docker compose down

После этого на сервере остаётся репозиторий и infra/.env с токеном, но запускать их уже нечему. Полностью убрать копию (rm -rf ~/aemr-bot) стоит только после ротации токена и с бэкапом .env.

Запуск на разных операционных системах

Главный факт, от которого всё зависит: стек написан под Docker на Linux (образы python:3.12-slim и postgres:16-alpine оба линуксовые).

Ubuntu / Debian — эталон. Родная среда: Docker Engine плюс плагин docker compose v2 (не snap-версия — она ломает тома). Дальше — установка по разделу «Установка по шагам». Требования к серверу: 1–2 vCPU, 2–4 ГБ RAM, 20 ГБ SSD, верное время (NTP), иначе расписания бэкапа и ретенций уедут.

FreeBSD через виртуалку с Linux (bhyve) — рекомендованный путь для серверной Администрации. FreeBSD не умеет нативный Docker, поэтому внутри неё поднимают bhyve-VM с Ubuntu 24.04 (2 vCPU, 4 ГБ RAM, 30 ГБ диск) и гоняют Docker уже в ней. Автозапуск двойной: bhyve поднимает VM при загрузке FreeBSD (vm_enable=YES в rc.conf), а systemd внутри VM поднимает стек при загрузке гостя. Подводные камни: проброс DNS в VM (gateway должен резолвить platform-api.max.ru), часовой пояс Asia/Kamchatka и NTP в VM, место на диске VM, и firewall FreeBSD (pf/ipfw) должен пропускать исходящий 443 от моста VM. Плюс этого пути: работают все защиты контейнера и все страховки, как на эталоне.

FreeBSD нативно, без Docker — запасной путь, честно про потери. Оправдан только если виртуализацию прямо запретит регламент. Из портов ставят python312, postgresql16-server+client, gnupg и обязательно geos (без него не соберётся shapely). Postgres поднимают рядом, DATABASE_URL задают руками, демонизируют через rc.d-скрипт. Потери большие: ни одна из четырёх серверных страховок не переносится (все на bash, systemd и docker compose), теряется весь hardening контейнера — вотчдог и автодеплой пришлось бы писать заново, а изоляцию компенсировать средствами FreeBSD (jail, rctl).

Astra Linux / РЕД ОС (гос-Linux). Стек запускается, две тонкости. При включённом мандатном доступе (SELinux/PARSEC) тома могут ловить «permission denied» из-за меток контекста — лечится суффиксом :Z на bind-mount'ах или настройкой контекста; смотреть ausearch -m avc. И если штатно стоит podman вместо docker — он понимает docker-compose.yml, но автозапуск и скрипты страховок завязаны на docker compose и их пришлось бы править; часто проще поставить docker-ce из совместимого репозитория.

Windows / macOS — только разработка, для прода исключено архитектурно. Локально тесты гоняются на SQLite без Postgres. Категорически нельзя запускать локального бота с боевым токеном — это мгновенно устроит гонку с продом.

Запуск за российским межсетевым экраном (UserGate, Континент, ALTELL, Рубикон, ИКС, Ideco), вопросы администратору и рецепт sidecar cntlm — отдельно в «За корпоративным межсетевым экраном».

Запуск под Podman как systemd-служба (Debian, без Docker)

Наш случай: на рабочем сервере (Debian) бот запускают не на Docker, а на Podman, и нужна служба, которая переживает отключение терминала, сама перезапускается и стартует при загрузке. Правильный современный способ — Quadlet: декларативные юниты, из которых systemd сам делает службы. Это рекомендованный путь Podman, начиная с версии 4.4.

Сначала проверьте версию Podman

podman --version. Quadlet есть только в Podman 4.4 и новее. На Debian 13 (Trixie) штатный Podman 5.x — Quadlet есть. На Debian 12 (Bookworm) штатный Podman 4.3 — Quadlet нет, тогда либо запасной путь (ниже), либо установка Podman 5 из стороннего репозитория.

Способ 1 — Quadlet (рекомендуется, Podman 4.4+). В репозитории лежат готовые юниты infra/podman/aemr-bot.network, aemr-bot-db.container, aemr-bot.container — это построчный перенос docker-compose.yml (та же база Postgres, тот же hardening: неизменяемый rootfs, сброс capabilities, лимиты памяти и процессов, порт только на 127.0.0.1). Установка одной командой:

sudo bash infra/podman/install-podman-quadlet.sh

Скрипт проверит версию Podman, соберёт образ бота из Dockerfile, положит юниты в /etc/containers/systemd/ (подставив реальный путь репозитория), перегенерирует службы и поднимет стек. Поскольку это системные службы, они не зависят от вашей сессии — переживают логаут и стартуют при загрузке автоматически, без всякого nohup или screen.

Одно отличие от Docker Compose — DATABASE_URL

Compose выводил DATABASE_URL из POSTGRES_* автоматически. Quadlet так не умеет, поэтому добавьте в infra/.env строку с реальным паролем:

DATABASE_URL=postgresql+asyncpg://aemr:ВАШ_ПАРОЛЬ@aemr-bot-db:5432/aemr

Хост здесь — имя контейнера базы (aemr-bot-db), оно резолвится по DNS внутри сети aemr-bot. Установщик предупредит, если строки нет.

Порядок запуска: бот зависит от базы (Requires= + After=). Если база ещё инициализируется на первом старте, бот пару раз перезапустится (alembic не достучался до базы) и поднимется, как только она будет готова — это нормальное самоисцеление, как depends_on в compose. Управление:

systemctl status aemr-bot.service        # состояние
podman ps                                 # контейнеры
curl -s http://127.0.0.1:8080/readyz      # здоровье (ждём {"ok": true})
journalctl -u aemr-bot.service -f         # логи
systemctl restart aemr-bot.service        # после пересборки образа
systemctl stop aemr-bot.service           # остановить бота (база останется жить)

Способ 2 — запасной, для старого Podman (Debian 12). Если Quadlet нет и ставить Podman 5 не хочется, есть юнит infra/podman/aemr-bot-compose.service — он гоняет тот же самый docker-compose.yml через podman compose (единый источник истины, дублировать ничего не надо):

sudo apt install podman-compose
sudo cp infra/podman/aemr-bot-compose.service /etc/systemd/system/
sudo systemctl daemon-reload
sudo systemctl enable --now aemr-bot-compose.service

Это тоже системная служба и так же переживает логаут. Минус — podman compose это обёртка с менее полной поддержкой compose, чем нативный Quadlet.

За прокси с NTLM/Kerberos

Если наружу пускают только через корпоративный прокси с доменной аутентификацией, к Quadlet добавляется sidecar cntlm — юнит infra/podman/aemr-bot-cntlm.container и переменные в .env (подробно — в «За корпоративным межсетевым экраном»).

Rootful или rootless. Установщик ставит rootful (юниты в /etc/containers/systemd/) — это проще всего и сразу переживает логаут, потому что служба системная. Для большей безопасности на гос-сервере Podman можно гонять rootless под отдельным пользователем: юниты кладут в ~/.config/containers/systemd/ и включают loginctl enable-linger aemr, чтобы службы пользователя жили после выхода из сессии и поднимались при загрузке. Сам бот в любом случае работает внутри контейнера под непривилегированным пользователем (UID 1000).

Прокси и SSL-инспекция за межсетевым экраном. Раньше бот не умел ходить через прокси и не доверял корпоративному сертификату при SSL-инспекции — за UserGate он молча не подключался. Высокий приоритет (важно для гос-данных):

  • Восстановимость бэкапа не проверяется автоматически, полное учение по восстановлению не проводилось. Бэкап, который ни разу не разворачивали, — это надежда, а не страховка. Нужен ежемесячный авто-restore во временную базу с алёртом при провале и хотя бы одно учение с фиксацией времени восстановления.
  • Offsite-копии нет: все восемь копий лежат на том же диске, что и база; потеря сервера уносит и базу, и копии. Включить выгрузку в защищённое хранилище вне периметра либо настроить регламентный вынос зашифрованного дампа.
  • Единственность экземпляра не закреплена кодом: нет глобального замка «один бот на токен», поэтому два процесса молча делят апдейты. Лечится ротацией токена и снятием лишних копий; по-хорошему — взять на старте advisory-lock и при неудаче выходить с явной ошибкой.
  • Затяжное падение базы не алёртится: бот может сутками отвечать ошибками при зелёном /livez, а оператор не узнает. Добавить в вотчдог отдельный сигнал по затяжному /readyz.
  • Внешний «мёртвый человек» выключен (HEALTHCHECK_URL пуст): полную смерть сервера или сети не заметит никто, кроме человека по тишине пульса. Поднять внутренний приёмник пингов на отдельном хосте.
  • Токен живёт на лишних хостах, права на .env не форсятся install-скриптами. Ротировать токен, обнулить .env на не-канонических хостах, рассмотреть docker secrets.

Средний приоритет: автодеплой не работает в закрытой сети и опасен на не-каноническом хосте (воскрешает лишний экземпляр); git reset --hard автодеплоя молча затирает ручные правки на сервере; у страховок нет машинной проверки «я канонический хост?» (помог бы guard-файл); единый «однонаправленный шлюз» (data-diode) для бота не подходит — нужен дуплекс; вводящий в заблуждение комментарий про restart-loop в коде.

Гигиена: в репозитории не должно оставаться .env.example с реальными секретами (стоит добавить секрет-сканер в CI). Битые ссылки на упразднённые отдельные документы переписаны на эту вику.

Чего не делать

  • Не выключать Docker, чтобы «починить бота»: там база, станет только хуже.
  • Не снимать оба cron-задания на работающем сервере: вотчдог — единственный, кто лечит зависший процесс.
  • Не путать systemctl stop aemr-bot (гасит всё, включая базу) с перезапуском бота docker compose restart bot.
  • Не запускать compose up на переднем плане в терминале — при закрытии сессии (логаут, разрыв SSH) бот и база умрут. Только -d или служба (см. Podman).
  • Не запускать второй экземпляр с тем же токеном нигде, включая свой компьютер.
  • Не коммитить infra/.env.example с заполненными секретами.
  • Не выполнять docker system prune -a --volumes — снесёт том с базой.
