Как развёрнут бот

Бот «ЧАТ-БОТ / АДМИНИСТРАЦИЯ ЕМО» — это один процесс в мессенджере MAX, который сам опрашивает серверы MAX (режим long polling — длинный опрос: бот периодически спрашивает «нет ли новых сообщений»). Он живёт в двух Docker-контейнерах (изолированных коробках с приложением) на одной виртуальной машине (VPS — арендованный сервер): сам бот плюс база данных PostgreSQL. Никаких облачных сервисов, никаких входящих портов наружу. Связь только исходящая, на серверы MAX. Это сознательная модель развёртывания: self-host, длинный опрос, без публичной точки входа — и потому без класса атак на открытый порт.

Что вообще происходит (картинка целиком)

Чтобы понимать, что чинить, держите в голове общую схему движения обращения:

  1. Житель пишет боту в личке. Бот ведёт его по анкете (имя, населённый пункт, адрес проблемы, тема, суть) и создаёт обращение №N. На шаге населённого пункта житель может нажать штатную кнопку «📍 Поделиться геолокацией» — бот сам определит поселение и адрес, жителю не нужно присылать координаты вложением.
  2. Карточка обращения №N уходит в служебную группу MAX. Это отдельный чат, где сидят операторы Администрации.
  3. Оператор отвечает на карточку (свайпом-ответом на сообщение или командой /reply). Бот проверяет ответ и доставляет его жителю в личку.
  4. В служебную группу приходит подтверждение «ответ отправлен жителю», карточка дополняется лентой переписки.

Бот отличает жителя от оператора по идентификатору чата (chat_id — числовой номер чата). Сообщение из чата, чей chat_id совпадает с переменной ADMIN_GROUP_ID, считается операторским и попадает в обработчик ответов операторов. Любой другой чат — это житель в личке. Поэтому правильный ADMIN_GROUP_ID в файле настроек — обязательное условие: без него операторская часть просто не работает (бот молчит в группе, потому что в конфиге admin_group_id is None и весь блок группы пропускается).

Установка с нуля по шагам

Шаг 1. Получить токен бота через MAX для бизнеса

Боты в MAX регистрируются через корпоративный портал, а не через @MasterBot (он для MAX не работает):

  1. Откройте https://max.ru/business, войдите под учётной записью администратора организации. Юридическое лицо должно быть верифицировано (это заранее закрывает ИТ-отдел Администрации).
  2. Раздел «Боты» → «Создать бота» (или откройте уже заведённого).
  3. Заполните имя, описание, аватар.
  4. Скопируйте Bot API token — длинную строку. Это значение для переменной BOT_TOKEN. Префикс «Bearer» добавлять не нужно.

Опасно

Токен — это полный доступ к боту. Храните его в защищённом месте (менеджер паролей либо файл .env с правами 600 на сервере). В git не коммитьте. Если токен утёк или уволился человек, у которого был доступ к серверу, — выпустите новый токен на той же странице, старый сразу отзовите.

Шаг 2. Подготовить систему и поставить Docker

Создаём отдельного пользователя для бота, ставим базовые пакеты, включаем брандмауэр (firewall — межсетевой экран, фильтр сетевых соединений):

# Пользователь для бота, добавляем в группу docker
sudo adduser --disabled-password --gecos "" aemr
sudo usermod -aG docker aemr

# Базовые пакеты и firewall: открыт только SSH, всё остальное закрыто
sudo apt update && sudo apt upgrade -y
sudo apt install -y ca-certificates curl gnupg git ufw
sudo ufw default deny incoming
sudo ufw default allow outgoing
sudo ufw allow 22/tcp
sudo ufw enable

Порт 8080 закрыт наружу; установка Docker

Порт бота (8080) слушает только внутри сервера (127.0.0.1) и наружу не публикуется — отдельное правило firewall для него заводить не нужно. Установка Docker Engine и плагина Compose — по официальной инструкции Docker (на старых Ubuntu в стандартных репозиториях лежит устаревший пакет без подкоманды docker compose: подключают официальный репозиторий Docker, ставят docker-ce docker-ce-cli containerd.io docker-buildx-plugin docker-compose-plugin). Чтобы не писать sudo перед каждой командой Docker — добавьте пользователя в группу docker (sudo usermod -aG docker $USER && newgrp docker). После установки проверьте: docker --version (27.x) и docker compose version (v2.x).

Осторожно: членство в группе docker = root

Любой, кто состоит в группе docker, фактически имеет права root на хосте (через монтирование файловой системы в контейнер). Поэтому в эту группу добавляют только владельца бота. Если на той же машине заведётся другой сервис со своим обслуживающим пользователем — его в группу docker к боту не сажают.

Шаг 4. Первый запуск

cd /home/aemr/aemr-bot/infra
docker compose pull
docker compose up -d --build
docker compose ps

Через 30–60 секунд у обоих контейнеров в колонке STATUS должно быть Up (healthy). В логах бота (docker compose logs -f bot) появится строка о старте в режиме long polling. Бот уже отвечает жителям в личке. На сообщения в группе он пока не реагирует — ADMIN_GROUP_ID ещё не задан.

Осторожно

После первого запуска полезен один лишний рестарт базы, чтобы заработала статистика медленных запросов: расширение pg_stat_statements создаётся миграцией 0010, но реально начинает собирать данные только когда модуль подгружен из shared_preload_libraries ещё до старта PostgreSQL. Поэтому: docker compose restart db. После этого top-N медленных запросов виден через select * from pg_stat_statements.

Шаг 5. Создать служебную группу и узнать её chat_id

  1. Создайте группу в MAX (например, «АДМИНКА / ЧАТ-БОТ»). Добавьте туда бота и операторов. В группы добавляются только верифицированные боты — если ошибка, проверьте, что юрлицо подтверждено.
  2. Напишите в группе /whoami. Бот ответит вашим max_user_id, именем и chat_id группы. chat_id группы — это отрицательное число вида -74181728103785. Сохраните его и свой max_user_id.

/whoami отвечает только в служебной группе

В личке у жителя команда /whoami тихо игнорируется — это сделано намеренно, чтобы обычные жители не видели служебных команд. Пусть каждый будущий оператор напишет /whoami в группе (его сначала нужно туда добавить) — соберёте их max_user_id в табличку для массовой регистрации.

Шаг 6. Прописать ADMIN_GROUP_ID и перезапустить

nano /home/aemr/aemr-bot/infra/.env
# вписать: ADMIN_GROUP_ID=-74181728103785
docker compose up -d --force-recreate bot

Перезапуск обязателен — файл .env читается только при старте контейнера. С этого момента сообщения в группе с этим chat_id обрабатываются как операторские.

Шаг 7. Создать первого ИТ-оператора

Команда регистрации операторов доступна только роли it, а при первом запуске в базе нет ни одного it. Получается «петля»: чтобы зарегистрировать ИТ, нужно быть ИТ. Чтобы её развязать, бот умеет создать первого ИТ из переменных окружения. Впишите в .env свой max_user_id и ФИО:

BOOTSTRAP_IT_MAX_USER_ID=<ваш_max_user_id>
BOOTSTRAP_IT_FULL_NAME=Ваше ФИО
docker compose up -d --force-recreate bot

При старте, если в таблице operators нет ни одной активной записи с ролью it, бот вставит её из этих переменных. На повторных запусках — ничего не делает (no-op, безопасно), переменные можно оставить навсегда. Запасной путь, если автоинициализация не сработала, — напрямую через psql (консоль PostgreSQL):

docker compose exec db psql -U aemr -d aemr -c \
  "INSERT INTO operators (max_user_id, full_name, role, is_active) \
   VALUES (<ваш_max_user_id>, 'Ваше ФИО', 'it', true);"

Дальше всех остальных операторов удобнее заводить кнопочным мастером «👥 Операторы» в меню /op_help или командой /add_operators — это уже работа оператора, описана в разделе «Оператору». Роли бота — it, coordinator (Куратор ЦУР), aemr (специалист АЕМО), egp (специалист Елизовского городского поселения). Роль it подразумевает все права координатора: если один человек одновременно ИТ-специалист и координатор, заводите ему одну запись с role=it, две не нужны.

Webhook-режим (по умолчанию НЕ используется)

Боевая сборка работает в режиме long polling. Webhook (вебхук — обратный вызов, когда MAX сам шлёт событие на ваш URL) оставлен в коде на случай будущего перехода, но для него нужен публичный домен с TLS-сертификатом, которого у проекта сейчас нет. Если всё же включаете (BOT_MODE=webhook), нужны:

BOT_MODE=webhook
WEBHOOK_URL=https://<ваш-домен>/max/webhook
WEBHOOK_SECRET=<≥16 символов: python -c 'import secrets; print(secrets.token_urlsafe(32))'>

Осторожно

Без WEBHOOK_URL и WEBHOOK_SECRET длиной ≥16 символов процесс бота просто не стартует в webhook-режиме (валидатор в config.py остановит запуск). В режиме polling эти поля не нужны вовсе. Домен feedback.elizovomr.ru в старых черновиках — плейсхолдер с тестового стенда, не боевой адрес. Под профилем webhook compose дополнительно поднимает nginx (TLS-терминация, порты 80/443) и certbot (выпуск и продление сертификатов) — в боевой polling-сборке оба контейнера не запускаются.

Осторожно: мёртвые переменные

В старых production-файлах .env встречаются переменные COORDINATOR_MAX_USER_ID и BACKUP_TMP_DIR — остатки ранних версий, бот их не читает. Конфиг построен на pydantic-settings с режимом «игнорировать лишнее», поэтому любая переменная вне схемы (в том числе опечатка в имени) молча пропускается. COORDINATOR_MAX_USER_ID — реликт эпохи, когда координатор был один и зашит в конфиг; сейчас роли живут в таблице operators и назначаются через /add_operators. BACKUP_TMP_DIR — реликт ранней схемы backup; сейчас дамп пишется сразу в BACKUP_LOCAL_DIR. Обе строки можно и нужно удалить, чтобы не путать того, кто откроет файл через год. Риск нулевой; перед удалением убедитесь, что их не подхватывает сторонний shell-скрипт на хосте (в репозитории — не подхватывает).

Почему сторож смотрит /livez, а не /readyz

Docker и watchdog проверяют именно /livez, а не /readyz. Это сделано специально: кратковременная проблема с базой не должна перезапускать живой процесс — иначе теряется возможность спокойно разобраться в причине. С локального адреса (127.0.0.1, localhost, ::1) бот отдаёт полную диагностику — свежесть пульса, возраст последнего удара в секундах, статус базы; внешним запросам — только короткое {"ok": …}, чтобы не раскрывать внутреннее состояние.

Почему у /livez два сигнала живости

Одного «пульса» мало. Heartbeat-таск только спит и периодически «бьёт», поэтому если завис не он, а обработчик события или backup (а события бот разбирает параллельными задачами), пульс продолжал бы идти и /livez оставался бы ложно-зелёным — авто-рестарт не сработал бы. Поэтому добавлен второй, независимый сигнал: свежесть опроса MAX. Пока бот реально опрашивает серверы MAX, таймстемп последнего успешного get_updates обновляется; стоит poll-циклу встать — этот таймстемп протухает за LIVEZ_POLL_STALE_FACTOR × POLLING_TIMEOUT_SECONDS (3× по умолчанию: запас на один потерянный long-poll плюс сетевой джиттер, чтобы не поднимать ложную тревогу на здоровом простаивающем боте), и /livez краснеет. Тогда watchdog перезапустит зависший контейнер. Это защита по принципу «в глубину» (defense-in-depth): два разных датчика ловят разные классы зависаний. В webhook-режиме опроса нет, и второй сигнал не учитывается.

Как собираются зависимости: hash-enforced сборка из uv.lock

Диапазоны ~= выше — это то, что разрешено в pyproject.toml. А ставится в образ ровно то, что закреплено в закоммиченном bot/uv.lock (точные версии всех пакетов, включая транзитивные, плюс SHA256-хэш каждого wheel). infra/Dockerfile читает именно lock: на этапе сборки uv export --frozen --no-emit-project --no-dev разворачивает lock в requirements.txt с хэшем на каждый пакет, а pip install --require-hashes отказывается ставить что-либо, чей скачанный артефакт не совпал с зафиксированным хэшем. Сам пакет бота добавляется отдельным шагом editable и без зависимостей (pip install -e /app --no-deps) — дерево зависимостей уже установлено и хэш-проверено.

Что это даёт. Во-первых, детерминизм: в прод приезжают ровно те версии, что в lock-файле, без пересборки на момент сборки. Это закрывает реальный дрейф «у меня работает, на проде падает», который однажды уже положил бота (разошлись версии maxapi между локальной средой и Docker). Для авто-деплоя, пересобирающего образ каждые 10 минут, это критично: без закрепления образ молча менялся бы без единой строки в git diff, подтянув новый патч транзитивной зависимости. Во-вторых, проверка целостности (supply-chain): --require-hashes работает по принципу fail-closed — один несовпавший хэш останавливает сборку, поэтому подмена пакета на зеркале или MITM при скачивании не пройдут незаметно. uv тянется в образ пиннутым по digest бинарём через COPY --from, без curl … | bash (тот же принцип, что для rclone), и в финальный рантайм не попадает. Совместимо с read-only рантаймом: всё ставится в build-time, готовое дерево пакетов в рантайме неизменно.

Тонкость про CI: hash-enforced живёт именно в сборке Docker-образа. Job'ы стиля и тестов в .github/workflows/ci.yml ставят dev-окружение по диапазонам (pip install -e .[dev]) — это сознательно, тестам нужны и dev-инструменты, а супла-чейн-гарантию даёт отдельный шаг сборки образа плюс pip-audit. Развёрнутое обоснование и полный дифф Dockerfile — в docs/DEPS.md (раздел «Зависимости»).

Тонкости расписания cron

Напоминалки операторам не дёргают в обеденный перерыв (12:00–12:59 выпадает из расписания, отсюда «9–11 и 13–17») и в субботу — обе настроены на mon-fri. «Пульс» же идёт 24/7 — это технический сигнал «процесс жив», непрерывность мониторинга важнее, госпраздники в нём не учитываются. Напоминалки по SLA дополнительно молчат в государственные нерабочие праздники РФ: даты берутся из seed/holidays.json (покрыты 2026 и 2027 годы, с учётом переносов выходных). Ретеншн-задачи (обезличивание, очистка) к календарю праздников не привязаны — закон не делает каникул, 30-дневный срок идёт и в выходные. Все задачи имеют «окно прощения» опоздания (misfire_grace_time) 120 секунд — оно покрывает паузу 30–90 сек, пока контейнер пересобирается при деплое, так что пропущенный тик догоняется. Дублей не будет: настройки coalesce=True и max_instances=1 сливают просроченные тики в один и не дают запустить второй экземпляр.

Осторожно

Планировщик хранит расписание в памяти. Если сервер был выключен ровно в момент еженедельного backup — этот запуск не наверстается (в отличие от тиков, пропущенных на короткую пересборку: их покрывает окно прощения). На следующее воскресенье цикл продолжится. Перед опасными операциями (миграции, выкладка) делайте /backup руками.

Защита от перегрузки и анти-флуд

Бот сам себя бережёт от двух разных видов перегрузки: чтобы не превысить лимит частоты запросов на стороне MAX (и не получить отказ 429 Too Many Requests), и чтобы один шумный отправитель не завалил процесс. Админу это знать полезно: благодаря этим механизмам всплеск активности не роняет бота, а сглаживается.

Глобальный лимитер исходящих (темп к MAX)

У MAX есть потолок частоты исходящих (порядка 2 запросов в секунду). Чтобы в него не упереться, в боте стоит один на весь процесс ограничитель темпа (token-bucket — «корзина токенов»): через него проходит каждое исходящее сообщение бота — ответ оператора жителю, новая карточка обращения в служебную группу, «пульс» и алёрты, и особенно массовая рассылка. Все эти источники делят общий бюджет около 1,5 сообщения в секунду (с небольшим запасом до потолка MAX), плюс небольшой задел на короткий всплеск подряд (несколько сообщений сглаживаются без задержки). Поэтому рассылка и обычная работа бота не складываются в превышение лимита: лимитер выстраивает отправки в очередь и разносит их во времени.

Важная деталь для понимания логов: лимитер работает по принципу fail-open — он может слегка притормозить отправку, но никогда её не блокирует. Если внутри него что-то пойдёт не так, сообщение всё равно уйдёт: для гос-бота доступность (например, успеть отправить оповещение о ЧС) важнее идеально ровного темпа. Отдельная настройка BROADCAST_RATE_LIMIT_PER_SEC (по умолчанию 1) задаёт темп внутри рассылки, но даже он проходит поверх общего лимитера — то есть рассылка ограничена дважды и точно не пробьёт общий бюджет.

Анти-флуд на входе (троттл жителя и окно параллельности)

На входящей стороне есть два рубежа, которые срабатывают раньше, чем событие дойдёт до базы или до вызова MAX, — поэтому машинный флуд гасится дёшево, не нагружая базу и не тратя бюджет исходящих:

  • Персональный троттл жителя. На каждого отправителя (по его max_user_id) заведена своя «корзина токенов»: ёмкость 20, пополнение 5 токенов в секунду. Этого с запасом хватает живому человеку, который быстро проходит анкету (десяток нажатий подряд), но machine-spam (десятки событий в секунду от одного отправителя) упирается в пустую корзину, и лишние события тихо отбрасываются ещё до открытия сессии БД и до похода в MAX (у callback-кнопки при этом гасится «крутилка», чтобы у жителя не висел спиннер). Операторов и служебные события без user_id троттл не трогает — операторов ограничивать нельзя.
  • Окно параллельности обработчиков. На пути опроса (polling) число одновременно работающих обработчиков событий ограничено сверху (семафор на 32). Без этого окна шквал апдейтов мог бы породить тысячи параллельных задач, каждая из которых берёт соединение из пула БД — пул исчерпывается, память растёт под лимитом контейнера 512 МБ, и плохо становится всем. Ограниченное окно держит число параллельных обработок в узде (тот же приём, что и для входящих webhook-запросов).

Итог: персональный троттл режет всплеск конкретного отправителя на самом входе, окно параллельности страхует процесс целиком от перегрузки задачами, а глобальный лимитер не даёт превысить темп уже на исходящей стороне. Подробный разбор модели угроз перегрузки (DoS) — в разделе «Безопасность».

Резервное копирование и восстановление

Что и как бэкапится

Каждое воскресенье в 03:00 по Камчатке бот делает pg_dump всей базы, шифрует его GPG (алгоритм AES-256, парольная фраза подаётся через канал os.pipe, а не через аргументы командной строки, чтобы не светиться в списке процессов) и кладёт файл в Docker-том backups. Внутри контейнера путь — /backups/aemr-YYYYMMDD_HHMMSS.sql.gpg (дата без дефисов, со временем по часовому поясу бота; права файла после записи ужесточаются до 0600). Хранятся последние 8 файлов (≈2 месяца), старые удаляются по дате изменения. Опционально — дополнительная копия в S3.

aemr-20260601_030000.sql.gpgпример имени файла бэкапа (дата + время по часовому поясу бота)

Опасно: без GPG-фразы backup не создаётся

Без BACKUP_GPG_PASSPHRASE (или с фразой короче 12 символов) backup отказывается выполняться и шлёт categorized-алёрт в служебную группу (категория провала config). Это требование закона 152-ФЗ: дамп содержит телефоны жителей, имена и тексты обращений — это персональные данные (ПДн), хранить их на диске или в S3 в открытом виде нельзя. Для dev-машины открыть незашифрованный режим можно явным BACKUP_ALLOW_UNENCRYPTED=1, но в production это запрещено. Если фраза от 1 до 11 символов — она сбрасывается в пустую с логом-ошибкой, и backup идёт уже под проверкой «разрешён ли незашифрованный», то есть в проде всё равно откажет. Сама функция backup никогда не бросает исключение: она возвращает категорию провала (config / pg_dump / gpg / unknown), а cron-обёртка читает её и шлёт точечный critical-алёрт.

Зависший backup не молчит: таймауты дочерних процессов

Backup запускает внешние процессы — pg_dump (снять дамп), gpg (зашифровать), rclone (выгрузить в S3). Каждый из них ограничен по времени: pg_dump и gpg — щедрее (по умолчанию 600 секунд, большая база может дампиться минутами), сетевой rclone — жёстче (300 секунд). Точные значения настраиваются и собраны в «Справочнике» (образец — в infra/.env.example). Зачем это нужно: раньше повисший внешний ресурс (недоступный S3-эндпоинт, зависший Postgres, заблокированный диск) подвешивал задачу backup навсегда — и, что хуже, алёрт о провале тогда вообще не уходил, потому что cron-задача просто не возвращалась. Теперь по истечении таймаута процесс убивается и гарантированно «прибирается» (kill + reap — чтобы не осталось процесса-зомби и утечки файловых дескрипторов), backup возвращает категорию провала (fail_kind), и в служебную группу штатно приходит critical-алёрт. Иными словами: backup может не получиться, но «тихо зависнуть без сигнала» он больше не может.

Восстановление из backup

Опасно

Никогда не восстанавливайте backup прямо в боевую базу aemr. Всегда сначала в отдельную тестовую базу aemr_restore_test, сверьте, и только потом принимайте решение. Восстановление — терминальный шаг: теряются данные, появившиеся между моментом backup и моментом сбоя.

  1. Найдите свежий файл на диске сервера: sudo ls -lah /var/lib/docker/volumes/aemr-bot_backups/_data/
  2. Создайте тестовую базу: 
    cd /home/aemr/aemr-bot/infra
. ./.env
docker compose exec -T db psql -U "$POSTGRES_USER" -d postgres -c 'DROP DATABASE IF EXISTS aemr_restore_test;'
docker compose exec -T db psql -U "$POSTGRES_USER" -d postgres -c 'CREATE DATABASE aemr_restore_test;'
  3. Расшифруйте и залейте (для зашифрованного .sql.gpg): 
    BACKUP=/var/lib/docker/volumes/aemr-bot_backups/_data/aemr-YYYYMMDD_HHMMSS.sql.gpg
sudo cat "$BACKUP" \
  | gpg --batch --passphrase "$BACKUP_GPG_PASSPHRASE" --decrypt \
  | docker compose exec -T db psql -U "$POSTGRES_USER" -d aemr_restore_test
    Для незашифрованного .sql (только dev) — без шага с gpg, напрямую cat … | psql.
  4. Проверьте, что таблицы есть и счётчики правдоподобны, а версия миграций совпадает с последней в main: 
    docker compose exec -T db psql -U "$POSTGRES_USER" -d aemr_restore_test -c '\dt'
docker compose exec -T db psql -U "$POSTGRES_USER" -d aemr_restore_test -c 'select count(*) from users;'
docker compose exec -T db psql -U "$POSTGRES_USER" -d aemr_restore_test -c 'select count(*) from appeals;'
docker compose exec -T db psql -U "$POSTGRES_USER" -d aemr_restore_test -c 'select * from alembic_version;'
  5. Удалите тестовую базу: docker compose exec -T db psql -U "$POSTGRES_USER" -d postgres -c 'DROP DATABASE IF EXISTS aemr_restore_test;'

Опасно

Если пароль BACKUP_GPG_PASSPHRASE утерян — зашифрованный backup восстановить невозможно, тест восстановления провален. Поэтому фразу храните отдельно от сервера (в менеджере паролей). Полный регламент проверки восстановления с минимальным отчётом — в docs/BACKUP_RESTORE_TEST.md.

Осторожно: ПДн после восстановления

После восстановления старого backup могут вернуться данные, уже удалённые в production после даты этого backup (по /erase или авто-обезличиванию). Это нормальное свойство backup, но его надо учитывать регламентно: после восстановления боевой базы из копии старше факта удаления ПДн нужно повторно применить операции удаления по журналу заявок. Один только restore не завершает восстановление после инцидента.

Тест восстановления (раз в квартал)

Backup, который ни разу не восстанавливали, официально считается непроверенным. Раз в квартал (в первый рабочий день января, апреля, июля, октября) проходите процедуру восстановления из docs/BACKUP_RESTORE_TEST.md в тестовую базу и записывайте результат в журнал эксплуатации (кто, когда, итог). Проверка занимает ~15 минут. Дополнительно — после каждой новой миграции схемы. Если квартальный тест пропущен — backup официально считаются непроверенными, и об этом должен знать владелец проекта.

Осторожно: offsite-копии нет

Локальный backup защищает от логической ошибки в базе, но не от потери самой VPS. Если сгорит диск — потеряются и база, и backup, лежащие на нём. Цель восстановления по времени при потере всего сервера (RPO — допустимая потеря данных) сейчас до 1 недели (backup раз в воскресенье); восстановление на новом хосте — 2–4 часа (новый сервер, восстановить .env руками, восстановить базу из последнего backup). Чтобы это закрыть, либо включите BACKUP_S3_*, либо настройте внешний rsync/scp по cron на отдельный хост. Сейчас offsite-копии нет — это документированный риск.

Восстановление на БОЕВУЮ базу (после реальной потери данных)

Всё, что выше, — отработка на тестовой базе aemr_restore_test. Это та репетиция, которую вы проводите раз в квартал, чтобы быть уверенным, что копии живые. Но когда боевая база действительно испорчена или потеряна (битый том после сбоя диска, ошибочный массовый DELETE, переезд на новый сервер), нужно залить backup уже в рабочую базу aemr. Это другой, более опасный сценарий — и делать его надо строго по шагам.

Опасно: это терминальный шаг

Восстановление на боевую базу стирает её текущее содержимое и заменяет состоянием на момент backup. Всё, что появилось между моментом backup и сейчас, теряется безвозвратно. Делайте это только когда боевые данные уже потеряны или заведомо испорчены — то есть терять нечего. Если база ещё жива и сомнения есть — сначала снимите свежий /backup или хотя бы pg_dump текущего состояния, и только потом перезаписывайте. Перед заливкой убедитесь, что тест восстановления этого же файла на aemr_restore_test (процедура выше) прошёл — нельзя обнаружить, что backup битый, уже после DROP боевой.

  1. Остановите только бота, базу оставьте поднятой (она нужна, чтобы выполнять psql). Так в базу никто не пишет во время заливки: 
    cd /home/aemr/aemr-bot/infra
. ./.env
docker compose stop bot
  2. Пересоздайте боевую базу aemr начисто. Подключаемся к служебной базе postgres, потому что нельзя удалить базу, к которой сам подключён: 
    docker compose exec -T db psql -U "$POSTGRES_USER" -d postgres -c 'DROP DATABASE IF EXISTS aemr;'
docker compose exec -T db psql -U "$POSTGRES_USER" -d postgres -c 'CREATE DATABASE aemr OWNER '"$POSTGRES_USER"';'
    Если DROP ругается «database is being accessed by other users» — значит бот не до конца остановлен или открыта чужая сессия psql; закройте их и повторите.
  3. Расшифруйте и залейте выбранный backup в боевую базу (для зашифрованного .sql.gpg): 
    BACKUP=/var/lib/docker/volumes/aemr-bot_backups/_data/aemr-YYYYMMDD_HHMMSS.sql.gpg
sudo cat "$BACKUP" \
  | gpg --batch --passphrase "$BACKUP_GPG_PASSPHRASE" --decrypt \
  | docker compose exec -T db psql -U "$POSTGRES_USER" -d aemr
    Для незашифрованного .sql (только dev) — без шага с gpg, напрямую cat … | psql … -d aemr.
  4. Проверьте, что данные на месте и версия миграций совпадает с последней в main (число alembic_version сверьте с разделом «Справочник» → миграции; на момент написания последняя — 0018_users_trigram_search_indexes): 
    docker compose exec -T db psql -U "$POSTGRES_USER" -d aemr -c '\dt'
docker compose exec -T db psql -U "$POSTGRES_USER" -d aemr -c 'select count(*) from users;'
docker compose exec -T db psql -U "$POSTGRES_USER" -d aemr -c 'select count(*) from appeals;'
docker compose exec -T db psql -U "$POSTGRES_USER" -d aemr -c 'select * from alembic_version;'
  5. Поднимите бота обратно. При старте он сам прогонит alembic upgrade head — если backup был снят на чуть более старой схеме, чем текущий код, недостающие миграции доедут автоматически: 
    docker compose up -d bot
docker compose logs -f bot         # убедиться, что стартовал без ошибок
curl -fsS http://127.0.0.1:8080/readyz && echo   # 200 = база жива и подключена
  6. Если версия backup НОВЕЕ кода на сервере (восстанавливаете на откатившийся сервер) — alembic upgrade head ничего не сделает, но бот может не знать о столбцах из будущего. Это редкий случай; сначала подтяните код до той же версии, на которой снимался backup (см. «Выкладка»), потом восстанавливайте.

Осторожно: ПДн после восстановления боевой

Та же оговорка, что и для тестового восстановления, но здесь она боевая: если backup старше факта удаления ПДн (житель просил /forget или сработало авто-обезличивание после даты этого backup), восстановление вернёт уже удалённые данные. После заливки боевой базы из старой копии пройдитесь по журналу действий (audit_log: записи erase и self_erase с датой позже backup) и повторно примените удаления командой /erase по тем же жителям. Один restore не закрывает обязанность по 152-ФЗ — разбор правовой стороны в разделе «Удаление ПДн».

Выкладка новой версии и откат

Авто-деплой (auto-deploy)

Скрипт scripts/auto-deploy.sh запускается из cron от root каждые 10 минут. Логика:

  1. Проверка обновлений. git fetch origin main. Если новых коммитов нет — тихо выходит, контейнер не трогает.
  2. Запоминает текущий коммит как «предыдущий рабочий» (для отката).
  3. Обновляет и пересобирает. git reset --hard origin/main и docker compose up -d --build.
  4. Health-gate. До 60 секунд опрашивает /livez каждые 5 секунд.
  5. Решение. Успех → запись в журнал «deploy ok». Провал → авто-откат на предыдущий коммит, пересборка, запись «DEPLOY FAILED: ROLLBACK».

Откат смотрит /livez; ветки PR не деплоятся

Откат управляется только проверкой /livez, а не /readyz — кратковременная проблема с базой не должна откатывать корректный код. Авто-деплой тянет код только из ветки main; ветки Pull Request на сервер не попадают. После мержа PR изменения доезжают на сервер в течение 10 минут. Логи деплоя: journalctl -t aemr-bot-deploy -n 80 --no-pager. Имя compose-проекта (name: aemr-bot) закреплено прямо в docker-compose.yml, а не вставляется скриптом — иначе рабочее дерево после каждого git reset числилось бы изменённым.

Необязательная проверка подписи коммита перед деплоем

У авто-деплоя есть опциональный предохранитель: проверка криптоподписи входящего коммита. По умолчанию он выключен — поведение прежнее, ничего настраивать не нужно. Если его включить, то прежде чем тронуть рабочее дерево, скрипт сверяет подпись origin/main (git verify-commit) с доверием, заведённым на сервере. Коммит без подписи или с непроверяемой подписью не деплоится (fail-closed): рабочее дерево остаётся нетронутым, контейнер продолжает крутить прежний доверенный коммит, и непроверенный код на сервер не приезжает. Смысл — закрыть сценарий «в main прилетел коммит не от доверенного автора» (например, при компрометации чьего-то доступа к репозиторию).

Как включить владельцу. Два шага, оба на сервере. (1) Завести доверие к ключу, которым подписываются коммиты: для GPG-подписей — импортировать открытый ключ в keyring пользователя, под которым идёт деплой; для SSH-подписей — указать ключ в файле gpg.ssh.allowedSignersFile (git config gpg.ssh.allowedSignersFile …). (2) Выставить переменную REQUIRE_SIGNED_COMMITS=1 в окружении cron-юнита авто-деплоя (или в /etc/default). После этого все коммиты в main должны быть подписаны доверенным ключом, иначе деплой остановится — поэтому включайте вместе с настройкой подписания у тех, кто пушит в main. Та же мера упомянута со стороны ИБ в разделе «Безопасность».

Ручная выкладка

sudo -u aemr -i
cd aemr-bot
git fetch origin main && git checkout main && git pull --ff-only
cd infra
docker compose up -d --build bot
docker compose logs -f bot     # убедиться, что стартовал без ошибок

Миграции базы (alembic) применяются автоматически при старте контейнера (команда запуска — alembic upgrade head && python -m aemr_bot.main). Если миграция падает — контейнер уходит в перезапуск; смотрите docker compose logs --tail 200 bot | grep -A20 alembic. После выкладки пройдите чек-лист docs/VPS_SMOKE_CHECKLIST.md (сверить, что HEAD на сервере совпадает с main, контейнеры Up, все три health-эндпоинта отвечают, watchdog и deploy-журналы чисты, в служебной группе работают /op_help, /diag, /open_tickets).

Откат при сбое (цель — за ≤10 минут)

Подробный playbook — docs/ROLLBACK.md. Главное: «бот health-down» НЕ значит «потеря данных» — база живёт независимо от приложения, откатываем только код. Цель — вернуть последнее работавшее состояние за ≤10 минут.

  1. Сначала диагностика, 30 секунд логов: docker logs aemr-bot-bot-1 --tail 50. Если ошибка явно временная (сеть, таймаут MAX) — не откатывайте, дайте 5 минут на самовосстановление через restart: unless-stopped.
  2. Откат кода на сервере: на коммит назад через git reset --hard HEAD~1, затем chown -R aemr:aemr . и docker compose up -d --build.
  3. Сразу сделайте revert-коммит в репозитории на машине разработчика (git revert HEAD --no-edit && git push origin main) — иначе через 10 минут авто-деплой снова подтянет сломанный main и снова упадёт на той же ошибке.
  4. Если виновата миграция и она обратима — docker exec aemr-bot-bot-1 alembic downgrade -1. Но не каждая миграция обратима: проверьте, есть ли реальный код в функции downgrade() или там заглушка. На проде даунгрейд миграций — крайняя мера, обычно проще восстановить из backup.
  5. После восстановления убедитесь, что /livez отвечает 200, и дождитесь «пульса» в служебной группе.

Перенос бота на новый сервер

Рано или поздно VPS придётся менять: сменился хостер, кончилось место, нужен сервер помощнее или с поддерживаемой ОС. Поскольку весь бот — это два контейнера, один файл настроек и одна база, переезд сводится к «поднять стек на новом сервере и залить туда backup». Никакой магии, но порядок важен, чтобы не потерять данные и не остаться с двумя живыми ботами на один токен.

Что именно нужно унести

Из всего сервера переезжают ровно три вещи: (1) файл infra/.env — все секреты и настройки развёртывания; (2) свежий зашифрованный backup базы из тома backups; (3) знание парольной фразы BACKUP_GPG_PASSPHRASE, иначе backup не расшифровать. Сам код тащить не нужно — он клонируется заново из репозитория. Тома Docker (db_data, build-кэш) копировать тоже не нужно и не стоит: базу переносим через backup→restore, а не копированием сырого тома (копия тома между разными версиями PostgreSQL ненадёжна). Если включён S3-backup — пункт (2) уже лежит в облаке, достаточно фразы и доступа к bucket.

  1. На старом сервере снимите свежий backup прямо сейчас, чтобы переехать с минимальной потерей: /backup из служебной группы (или docker compose exec bot python -m aemr_bot.services.db_backup). Скопируйте на свою машину файл .env и последний backup: 
    scp aemr@OLD_SERVER:/home/aemr/aemr-bot/infra/.env ./env.backup
scp aemr@OLD_SERVER:/var/lib/docker/volumes/aemr-bot_backups/_data/aemr-YYYYMMDD_HHMMSS.sql.gpg ./
  2. На новом сервере пройдите установку с нуля до конца шага 6 включительно («Установка по шагам»): создать пользователя aemr, поставить Docker, склонировать репозиторий. Вместо заполнения .env с нуля — положите туда перенесённый файл (ADMIN_GROUP_ID, токен, пароль базы и фраза уже в нём; новый /whoami не нужен, группа та же).
  3. Поднимите стек на новом сервере — база поднимется пустой, бот прогонит миграции до актуальной схемы: 
    cd /home/aemr/aemr-bot/infra
docker compose up -d --build
docker compose ps                       # дождаться healthy у обоих
  4. Залейте backup в боевую базу нового сервера по процедуре «Восстановление на боевую базу» (остановить бота → DROP/CREATE aemr → расшифровать и залить → проверить счётчики и alembic_version → поднять бота). Перенесённой фразы хватит для расшифровки.
  5. Переключение «в один момент». Бот в режиме long polling: два процесса с одним BOT_TOKEN начнут драться за обновления MAX (дубли сообщений, гонки). Поэтому сначала остановите старого (docker compose down на старом сервере), и только потом дайте работать новому. Перекрытия быть не должно. Если есть сомнения — на минуту остановите обоих, это безопаснее, чем оба разом живые.
  6. Поставьте страховочные службы заново на новом сервере (они привязаны к хосту, не переносятся): авто-деплой (новый deploy key в GitHub), watchdog, автозапуск systemd — шаг 8 установки. Старый deploy key из GitHub удалите.
  7. Проверка по чек-листу docs/VPS_SMOKE_CHECKLIST.md и финальному чек-листу запуска: оба контейнера Up, все три health-эндпоинта 200, в группе работают /op_help, /diag, пришёл «пульс».
  8. Выведите старый сервер из строя только после суток уверенной работы нового. Перед уничтожением старой VPS не забудьте: на ней лежат старые backup с ПДн — диск надо затереть, а не просто «удалить инстанс». Правовая сторона уничтожения носителя — в разделе «Безопасность».

Апгрейд ОС под живым ботом и дрейф времени (NTP)

Поскольку бот изолирован в контейнерах, операционная система хоста для него — это лишь Docker, cron и SSH. Поэтому обновлять ОС можно «под живым ботом», не трогая сам бот. Но два момента требуют внимания.

  1. Обычные обновления безопасности (sudo apt update && sudo apt upgrade -y) бот переживает без остановки — обновляются пакеты хоста, контейнеры продолжают работать. Если в составе обновлений был Docker или ядро — после sudo reboot контейнеры поднимутся сами (если установлен systemd-автозапуск, см. ниже). Перед перезагрузкой всё же снимите /backup руками.
  2. Крупный апгрейд релиза (do-release-upgrade, 22.04 → 24.04) — это уже риск: перепрыгивать через релиз нельзя, а сам процесс может перезаписать конфиг Docker. Безопаснее не апгрейдить на месте, а поднять чистый новый сервер на свежей ОС и переехать по процедуре «Перенос на новый сервер» выше — это и есть способ «обновить ОС» без риска для боевой машины. Старый сервер остаётся рабочим откатом, пока новый не подтвердит сутки аптайма.

Осторожно: дрейф времени ломает SLA и расписания

Бот считает все сроки по часовому поясу Asia/Kamchatka: SLA, окна «тихого режима», cron-расписание backup и напоминаний. Если системные часы хоста «уплывут» (типично после долгого аптайма, миграции виртуалки или севшей батарейки RTC), бот будет считать просрочки и слать «пульс» по неверному времени, а backup уйдёт не в 03:00. Поэтому на сервере обязательно должна работать синхронизация времени по NTP (Network Time Protocol — протокол сетевого времени). Проверка: timedatectl — строки System clock synchronized: yes и NTP service: active. Если no — включите: sudo timedatectl set-ntp true. После переноса на новый сервер или апгрейда ОС эту проверку делайте обязательно. Часовой пояс самого хоста при этом не важен — бот живёт по TZ из .env; важна именно точность абсолютного времени (UTC), которую и держит NTP. Если включён исходящий whitelist по iptables — порт 123 (NTP) должен быть разрешён, иначе синхронизация молча перестанет работать (см. «Сетевая изоляция»).

Сторожевые механизмы и автозапуск

Watchdog (внешний сторож)

Внутри контейнера уже есть Docker healthcheck и политика restart: unless-stopped — если процесс упал, Docker сам поднимет контейнер. Но Docker не перезапускает контейнер, который «жив, но не отвечает» (висит, отдаёт unhealthy). Для этого случая есть скрипт scripts/healthwatch.sh в cron от root каждые 5 минут:

  • успешный /livez → счётчик провалов сбрасывается;
  • 3 провала подряд (≈15 минут) → docker compose restart bot;
  • 8 провалов подряд (≈40 минут) → пишет алёрт прямо в служебную группу через MAX API (минуя бота — он же завис), отправляя POST на platform-api.max.ru/messages с заголовком Authorization.

Watchdog и health-gate ловят разные сбои

Health-gate ловит сломанный новый коммит сразу после выкладки. Watchdog ловит зависший живой контейнер в любой момент. Логи сторожа: journalctl -t aemr-bot-watchdog -n 80 --no-pager. Тестовый запуск без ожидания cron: sudo /usr/local/bin/aemr-bot-healthwatch && echo OK.

Автозапуск после перезагрузки сервера

Контейнеры с restart: unless-stopped поднимаются вместе с Docker при штатной перезагрузке. Но если до reboot контейнеры были остановлены вручную (docker compose down) или после неуспешного апгрейда — Docker сам их не вернёт. На этот случай есть systemd-юнит infra/aemr-bot.service, который при старте системы делает docker compose up -d в каталоге infra. Операция идемпотентная (если уже подняты — ничего не меняет, если упали — поднимаются). Устанавливается один раз: sudo bash /home/aemr/aemr-bot/scripts/install-systemd-autostart.sh. Проверка: sudo systemctl is-enabled aemr-bot.service (должно быть enabled), а после sudo reboot — docker compose ps показывает все контейнеры в Up.

Прогрев геоданных при старте

Когда житель на шаге населённого пункта жмёт «📍 Поделиться геолокацией», бот определяет поселение и адрес локально — по картам поселений, улиц и домов (это ~2,6 МБ данных GeoJSON и расчёт «точка внутри многоугольника» через библиотеку shapely). Эти карты подгружаются и индексируются лениво — при первом обращении. Загрузка занимает сотни миллисекунд, а процесс у бота одно-петлевой (single event-loop): если бы она впервые случилась прямо внутри обработки первого жителя, на это время «подморозило» бы всех сразу.

Чтобы первая геолокация не была медленной, бот на старте запускает фоновый прогрев геоиндексов: заранее, в отдельном потоке, подгружает карты и один раз прогоняет реальное определение адреса по центру Елизово — так прогревается весь путь «поселение → дом → улица». Дальше первый же запрос жителя попадает в уже тёплый кеш и отвечает быстро. Прогрев полностью необязателен (best-effort): если seed-файлов нет или они битые, бот просто пишет это в лог и работает дальше — индексы тогда прогреются лениво при первом запросе, а определение адреса в крайнем случае мягко деградирует, не роняя бота (fail-open).

Логи

Все контейнеры пишут логи в формате json-file с ротацией: 10 МБ × 3 файла = до 30 МБ на контейнер. Бесконечного роста логов не будет.

cd /home/aemr/aemr-bot/infra
docker compose logs --tail 300 bot                                   # последние 300 строк бота
docker compose logs --since 24h bot | grep -Ei 'error|exception|misfire|pulse'  # за сутки, отфильтровано
docker compose logs --tail 200 db                                    # логи PostgreSQL

Сторож и авто-деплой пишут через системный logger в journald:

journalctl -t aemr-bot-watchdog -n 100 --no-pager
journalctl -t aemr-bot-deploy   -n 100 --no-pager

Осторожно: docker-логи и 152-ФЗ

Docker отдельно хранит сырой stdout контейнера в файлах /var/lib/docker/containers/<id>/<id>-json.log. В них попадают псевдоидентификаторы жителей (max_user_id), а команда /erase чистит только базу, не эти файлы. Поэтому ротация (10 МБ × 3) настроена в docker-compose.yml и для бота, и для базы — без неё json-лог рос бы неограниченно и через год мог занять десятки гигабайт и положить диск. После запроса жителя на удаление данных, помимо /erase в боте, вручную обнулите docker-логи бота за последние дни, не удаляя сам файл (Docker держит его открытым): sudo truncate -s 0 /var/lib/docker/containers/<id>/*-json.log. Это компромисс под 152-ФЗ, чтобы не строить отдельный пайплайн ради одного запроса в месяц. Для расследования инцидента нужен журнал действий в базе (хранится 365 дней), а не docker-stdout, поэтому дольше держать сырые логи бессмысленно.

О наблюдаемости (observability)

Внешнего сбора ошибок (Sentry) в коде нет — его откатили целиком (разбор — в разделе «Как устроено»). Единственный механизм внешнего мониторинга — пинг на HEALTHCHECK_URL, по умолчанию выключенный (задача-пинг добавляется в расписание, только если URL задан). Вся остальная диагностика — логи Docker плюс «пульс» и алёрты в служебную группу.

Повторяющийся — мало RAM на VPS (добавьте до 4 ГБ) или утечка (зафиксируйте docker stats --no-stream в динамике и в тикет).

База не стартует: порча данных или нехватка места

Если контейнер db висит в Restarting, причину покажет docker compose logs --tail 200 db — и развилка тут принципиальная, потому что лечится по-разному:

  • Нет места на диске. В логе — could not extend file, No space left on device, PANIC: could not write to file. Это не порча, данные целы — PostgreSQL специально останавливается, чтобы не писать в обрезанные файлы. Освободите место (см. «Диск кончился» ниже) и поднимите базу обратно: docker compose up -d db. Чаще всего этого достаточно.
  • Порча data-каталога. В логе — invalid page in block, could not read block, database files are incompatible, PANIC: could not locate a valid checkpoint record. Это повреждение тома db_data (внезапное выключение питания, сбой диска). Чинить «руками» через pg_resetwal на боевой базе — крайняя мера, легко добить данные. Правильный путь — восстановление из backup на боевую базу (см. выше): данные на испорченном томе всё равно ненадёжны.

Как отличить быстро: если df -h /var/lib/docker показывает 100% — это почти наверняка место, а не порча. Сначала проверяйте диск, потом подозревайте повреждение.

Диск кончился: аварийное освобождение и регламент

Полный диск — едва ли не самый частый способ «уронить» self-host бота, и виновник почти всегда один: Docker копит образы при каждой пересборке. Авто-деплой пересобирает контейнер каждые 10 минут командой docker compose up -d --build — старый образ не удаляется, а становится «висячим» (dangling — образ без тега, на который никто не ссылается). За недели таких слоёв набегает на гигабайты. Когда диск заполняется под 100%, PostgreSQL переходит в режим «только чтение» (перестаёт принимать запись, чтобы не испортить файлы), бот не может писать обращения, и backup тоже падает.

Сначала понять, что съело диск

df -h /                                  # сколько занято на корневом разделе
sudo du -sh /var/lib/docker/* 2>/dev/null | sort -rh | head   # что внутри docker крупнее всего
docker system df                         # сводка: образы / контейнеры / тома / build-кэш

Почти всегда самым большим окажется /var/lib/docker/overlay2 (слои образов) и build-кэш — это и есть накопленные пересборки.

Аварийно освободить место

Безопасный порядок — от самого безобидного к более радикальному. Висячие образы и build-кэш удалять можно всегда, на работающий бот это не влияет:

# 1) Удалить ТОЛЬКО висячие образы и build-кэш — самое безопасное, бьёт по главному виновнику
docker image prune -f
docker builder prune -f

# 2) Если мало — почистить остановленные контейнеры и неиспользуемые сети
docker container prune -f
docker network prune -f

# 3) Обрезать docker-логи, не удаляя файлы (Docker держит их открытыми)
sudo truncate -s 0 /var/lib/docker/containers/*/*-json.log

Опасно: не трогайте тома и не используйте prune -a --volumes

Команда docker system prune -a --volumes удаляет всё неиспользуемое, включая тома — а в томах живут и база (db_data), и резервные копии (backups). Если запустить её при остановленных контейнерах, можно стереть боевую базу. Для рутинной чистоты используйте только docker image prune -f и docker builder prune -f — они не трогают именованные тома. Флаг --volumes на этом сервере не нужен никогда.

После чистки поднимите то, что упало, и проверьте, что PostgreSQL вышел из режима «только чтение»:

cd /home/aemr/aemr-bot/infra
docker compose up -d                                # поднять всё, что не работало
curl -fsS http://127.0.0.1:8080/readyz && echo       # 200 = база снова принимает запись
docker compose exec -T db psql -U aemr -d aemr -c 'select now();'   # запись/чтение восстановлены

Чтобы не повторялось: профилактика и контроль

  1. Авто-уборка висячих образов. Скрипт авто-деплоя пересобирает образ постоянно, поэтому раз в сутки имеет смысл выметать висячие слои. Добавьте в cron от root (например, рядом с прочими страховочными задачами): docker image prune -f && docker builder prune -f --filter 'until=168h' — это убирает безымянные образы и кэш старше недели, не задевая активный.
  2. Следить за свободным местом. Раз в неделю (или в эксплуатационном календаре, см. «Эксплуатационный календарь») смотрите df -h /. Тревожный порог — заполнение выше 80%: пора чистить, не дожидаясь 100%. Если у вас подключён внешний пинг-мониторинг (HEALTHCHECK_URL), помните: он ловит «бот завис», но не «диск кончается» — за местом следит человек по календарю.
  3. Помнить про backup на том же диске. 8 еженедельных зашифрованных копий (≈400 МБ) и логи (до 60 МБ) уже учтены в минимальных 20 ГБ из «Требований к серверу». Если включили S3-выгрузку — локальные копии всё равно лежат на диске, их число задаёт BACKUP_KEEP_COUNT (по умолчанию 8).

Почему это важно именно для авто-деплоя

Без периодической уборки авто-деплой — медленная мина: он молча копит образы, пока в одно воскресенье backup не упадёт с «нет места», а следом база не встанет в read-only. Алёрт о провале backup придёт (категория pg_dump или unknown), но к тому моменту бот уже не принимает обращения. Дешевле выметать висячие образы по расписанию, чем разбирать аварию. Диагностика «база не стартует из-за места» — во врезке «База не стартует» выше.

Владение секретами и ротация

Кто держит секреты и что будет, если этот человек уйдёт (bus-factor)

«Bus-factor» (фактор автобуса) — это вопрос «сколько человек должно одновременно выпасть, чтобы система стала неуправляемой». Для этого бота критическая точка — парольная фраза резервных копий BACKUP_GPG_PASSPHRASE: если её знает один человек и он недоступен, все зашифрованные backup превращаются в нечитаемый мусор (восстановить их без фразы невозможно — это by design, см. «Резервное копирование»). Потеря фразы хуже потери самого backup: файлы есть, а толку ноль.

BOT_TOKENИТ-специалист; перевыпускается на max.ru/business владельцем-администратором юрлицаменеджер паролей; перевыпуск всегда доступен через портал MAX

Опасно: один держатель GPG-фразы = одна точка отказа

Если парольную фразу backup знает ровно один человек и только в своей голове или на своём ноутбуке — проект в одной аварии от безвозвратной потери данных. Канон: фразу знают минимум двое (ИТ и владелец), и она лежит в двух независимых местах, ни одно из которых — не сам сервер с ботом (иначе сгорит вместе с базой). Хранить фразу в infra/.env на сервере — нормально для работы бота, но это не резервная копия фразы: при потере сервера теряется и она. Резервная копия фразы должна быть offsite.

Передача секретов при увольнении или смене ИТ-специалиста

Когда человек с доступом к серверу уходит (увольнение, перевод, конец договора с подрядчиком), все секреты, которые он мог видеть, считаются потенциально скомпрометированными — даже если нет признаков утечки. Процедура передачи дел:

  1. Отозвать доступы уходящего: удалить его SSH-ключ с сервера (~/.ssh/authorized_keys пользователя aemr и root), снять его учётную запись из менеджера паролей, отозвать его доступ к репозиторию и к порталу max.ru/business.
  2. Ротировать всё, что он знал — по таблице ротации ниже, но внепланово и сразу: BOT_TOKEN, POSTGRES_PASSWORD, BACKUP_GPG_PASSPHRASE. Это та же работа, что строка «при увольнении» в таблице ротации.
  3. Передать секреты преемнику через менеджер паролей (не в мессенджере, не по почте открытым текстом), убедиться, что преемник может: зайти по SSH, поднять стек, снять /backup и расшифровать тестовый backup новой фразой.
  4. Зафиксировать передачу в эксплуатационном журнале: кто сдал, кто принял, дата, какие секреты ротированы. Эта же запись закрывает вопрос «кто отвечает за бота сейчас».

Орг-сторона преемственности (договор с подрядчиком, кто назначен ответственным, что делать при уходе единственного ИТ) разобрана в разделе «Концепция» — здесь только техническая передача секретов. ИБ-сторона смены ответственного (передача роли по безопасности) — в ИБ-календаре.

Сетевая изоляция (опционально)

В боевой сборке бот делает исходящие запросы строго на ограниченный список доменов: MAX API (botapi.max.ru, platform-api.max.ru), GitHub (для авто-деплоя — github.com, api.github.com, codeload.github.com), пинг-сервис (если задан HEALTHCHECK_URL) и источники угроз для антифишинга (urlhaus.abuse.ch, threatfox.abuse.ch, data.phishtank.com). Всё остальное — потенциальная утечка ПДн или признак компрометации.

Если нужен жёсткий исходящий whitelist через iptables — порядок такой: запретить весь исходящий трафик (OUTPUT DROP), затем точечно разрешить loopback, установленные соединения, DNS (порт 53), перечисленные домены на 443/TCP и NTP-синхронизацию времени (порт 123). Готовая схема с циклом по доменам — в docs/SYSADMIN.md §13a.

Осторожно

Подводные камни: IP-адреса CDN меняются — правила надо пересоздавать по расписанию либо использовать nftables с динамическим резолвингом. Авто-деплой дёргает github.com каждые 10 минут — его блокировать нельзя, иначе деплои встанут. Если включён S3-backup — добавьте endpoint S3 в whitelist. После любых изменений правил проверьте /livez, /readyz и /diag в группе. В текущей инсталляции исходящий whitelist не включён: на VPS работает только бот, риск утечки минимальный. Включать стоит при появлении других сервисов на той же машине.

Осторожно: соседний проект на том же хосте

Если на той же машине, что и бот с персональными данными граждан, живёт другой проект (например, веб-витрина dash с публичным caddy на 0.0.0.0:80/443) — учтите модель угроз. На уровне сети изоляция уже есть: бот и соседний проект — в разных Docker-сетях, Postgres бота не имеет host-привязки порта и недоступен снаружи и из соседних контейнеров, HTTP бота — только на 127.0.0.1. Остаточный риск узкий: общий хост и общий Docker-демон, то есть теоретический побег из скомпрометированного соседнего контейнера на хост, где лежат тома бота. Это высокий барьер, но не нулевой. Надёжнее всего — вынести соседний проект на отдельный VPS (решение владельца); как минимум — держать его под отдельным непривилегированным пользователем, не в группе docker вместе с ботом, с cap_drop: ALL и no-new-privileges.

SLA и приоритизация обращений

Постановление Администрации №2129 (от 19.12.2025) ставит бота в рамки МЦУ/ЦУР (Центр управления регионом) и задаёт сроки ответа. Часть этих сроков живёт в коде как порог напоминаний, часть остаётся организационной нормой, которую бот не считает автоматически — это разделение труда между программой и людьми, а не противоречие.

В коде задан один базовый срок: SLA_RESPONSE_HOURS = 4 (рабочих часа). По нему бот считает просрочку и шлёт напоминания операторам (overdue-reminder-workhours). Укороченные сроки Регламента (§62: 1/2/4 часа на возврат сведений профильным подразделением; §58: 30 минут на «не наша компетенция») в коде отдельно не отслеживаются — бот меряет всё по единому порогу 4 часа. Приоритизация по важности организационная: «высокую важность» назначает Куратор ЦУР вручную, поля важности у обращения в коде нет, по теме («вода/газ/отопление») бот срочность не повышает. Полный разбор, как 4 часа в коде сочетаются с 2 часами по важности, — в разделе для оператора.

59-ФЗ и Стандарт качества

К сообщениям через бот 59-ФЗ не применяется — обработка идёт по 152-ФЗ (разбор — в разделе «Правовое»). Стандарт качества ответов (Приложение 2 Постановления) — норма для операторов, а не код: требует действительного залога, запрещает канцелярит и страдательные обороты, ограничивает длину сообщения лимитами Telegram (≤1024 символа). Бот при этом держит свой, более жёсткий лимит ответа оператора — по умолчанию 300 символов (ANSWER_MAX_CHARS), и при превышении просит сократить, жителю ничего не уходит.

Это эксплуатация, а не безопасность

Здесь — рутинная техническая эксплуатация бота. Отдельный ИБ-календарь (ревью журнала действий на аномалии, аудит доступов, учебная отработка реагирования на инцидент, передача роли ответственного по безопасности) живёт в разделе «Безопасность» → «ИБ-календарь» — там свои сроки и свой ответственный. Общие пункты (ротация секретов, тест восстановления) не дублируются: за них отвечает этот календарь, ИБ-сторона лишь проверяет, что они выполнены. Ведите единый журнал эксплуатации (кто, когда, что сделал) — он же закрывает вопрос «кто отвечает за бота сейчас» и служит доказательством регулярного обслуживания при проверке.

Чек-лист первого боевого запуска

Пройдите перед публикацией бота жителям. Отметки сохраняются в браузере:

  • docker compose ps — оба контейнера Up (healthy).
  • curl http://127.0.0.1:8080/livez и /readyz — оба 200.
  • /start в личке у тестового жителя — приходит приветствие и главное меню из 7 кнопок.
  • Полная анкета (согласие → контакт → имя → населённый пункт → адрес → тема → суть с фото) проходится без сбоев, бот отвечает «Обращение #N принято в работу».

  • В служебной группе появляется карточка #N с кнопками действий; фото жителя приходит ответом к карточке.
  • Свайп-ответ на карточку и команда /reply 42 Тест доставляют жителю письмо с шапкой Администрации.
  • /op_help присылает панель из 4 / 7 / 11 элементов (3 / 6 / 10 действий + «Памятка оператора») в зависимости от роли вызвавшего.

  • /whoami в группе возвращает данные, в личке у жителя — тихо игнорируется.

  • /backup создаёт файл в /backups (зашифрованный .sql.gpg, если задана GPG-фраза).
  • /policy доставляет PDF политики ПДн.
  • «Пульс» приходит по расписанию; алёрт о просрочке — при искусственном создании просроченного обращения.
  • Установлены и проверены страховочные службы: авто-деплой (deploy key в GitHub), watchdog (cron */5), автозапуск systemd (enabled).

Частые вопросы

Чем менять настройки бота — файлом или через бота?

Зависит от того, что меняете. Технические параметры (токен, пароль базы, ADMIN_GROUP_ID, расписание backup, webhook) — только в infra/.env, с перезапуском контейнера. Содержательные настройки (тематики обращений, населённые пункты, тексты приветствия и согласия, контакты служб, ссылки) — в таблице settings, через меню «⚙️ Настройки бота» в /op_help (роль it) или командой /setting, без редеплоя. Этих редактируемых ключей 17. Список операторов — в таблице operators через мастер «👥 Операторы».

Нужно ли открывать порты в firewall?

Нет. В режиме long polling бот сам исходящими запросами опрашивает MAX, входящих соединений снаружи нет. Откройте только SSH (22). Порт 8080 слушает на 127.0.0.1 и наружу не виден. Порты 80/443 нужны только в webhook-режиме, который в боевой сборке не используется.

Где физически лежат backup?

В Docker-томе backups, на диске сервера по пути /var/lib/docker/volumes/aemr-bot_backups/_data/. Том переживает пересборку контейнера. Уносить копии с сервера на отдельный хост — отдельная задача ИТ-отдела (S3 или внешний rsync), без неё локальный backup не спасёт от потери всей VPS.

Бот ушёл в restart-loop после деплоя — что делать?

Если включён авто-деплой, он сам должен был откатиться по health-gate (ищите «DEPLOY FAILED: ROLLBACK» в journalctl -t aemr-bot-deploy). Если не откатился — откатитесь вручную (см. «Откат при сбое») и обязательно сделайте revert-коммит в репозитории, иначе авто-деплой через 10 минут снова подтянет сломанный код.

Какие команды бот показывает в меню «/» и почему не все?

При старте бот публикует в /-меню ровно две команды — /start и /menu (прямым PATCH /me в MAX с этим списком из двух пунктов). Специфичные жительские (/forget, /subscribe и пр.) и операторские (/op_help, /reply и пр.) команды в подсказку не выносятся. Причина — у MAX нет команд «по областям» (per-scope): один и тот же список виден и в личке жителя, и в служебной группе. Раньше, когда в подсказке светились жительские команды, операторы могли случайно тапнуть /forget или /subscribe; поэтому в меню оставлены только два безопасных универсальных входа. Все остальные команды по-прежнему работают, если набрать их руками: жители ведутся кнопочным меню, операторы — панелью /op_help.

Можно ли восстановить backup, если потерян GPG-пароль?

Нет. Зашифрованный backup без пароля не расшифровать — это by design. Именно поэтому BACKUP_GPG_PASSPHRASE храните отдельно от сервера (в менеджере паролей), а тест восстановления проводите раз в квартал, чтобы вовремя заметить проблему.

Чем восстановление на тестовую базу отличается от восстановления на боевую?

Это два разных сценария. Тест восстановления (раз в квартал) льёт backup в одноразовую базу aemr_restore_test, боевую aemr не трогает — это репетиция, чтобы убедиться, что копии живые. Восстановление на боевую делается только когда боевые данные реально потеряны или испорчены: оно стирает текущую базу и заменяет состоянием на момент backup. Порядок боевого восстановления (остановить бота, пересоздать базу, залить, проверить, поднять) — в разделе «Восстановление на боевую базу». Никогда не льют на боевую, не проверив тот же файл на тестовой.

Как перенести бота на другой сервер, ничего не потеряв?

Унести нужно только три вещи: файл infra/.env, свежий backup базы и знание GPG-фразы. Код клонируется заново из репозитория, тома Docker не копируются. На новом сервере проходите установку, кладёте перенесённый .env, заливаете backup на боевую базу. Главное правило — не держать двух живых ботов на один токен: сначала остановить старого, потом запускать нового. Полная процедура — «Перенос на новый сервер».

Что будет, если сервер выключится в момент еженедельного backup?

Этот запуск не наверстается — планировщик хранит расписание в памяти. На следующее воскресенье цикл продолжится. Перед опасными операциями делайте /backup руками. Короткие пропуски (на пересборку контейнера 30–90 сек) при этом догоняются за счёт окна прощения опоздания (120 сек) — это касается частых задач вроде «пульса», но не еженедельного backup.

Чем /livez отличается от /readyz и почему сторож смотрит только на первый?

/livez отвечает «жив ли главный цикл бота» и базу не трогает. /readyz вдобавок делает SELECT 1 к базе. Docker-healthcheck, watchdog и health-gate авто-деплоя смотрят именно /livez — иначе кратковременная проблема с базой перезапускала бы вполне живой polling-процесс, и вы теряли бы возможность спокойно разобраться, что с базой. /readyz — для ручной диагностики: если он падает, а /livez в порядке, значит процесс жив, но база лежит или зависла — смотрите логи базы и DATABASE_URL, не рестартуйте бота вслепую.
