Имя пакета и название территории

Имя Python-пакета — aemr_bot. Буквы «AEMR» расшифровываются как «Администрация Елизовского муниципального района» — это историческое название. Территория давно переименована в округ (АЕМО — Администрация Елизовского муниципального округа), и все тексты для жителей говорят «округ». Но идентификатор aemr_bot в коде, в именах миграций и в путях оставлен прежним ради обратной совместимости: переименование пакета сломало бы историю миграций и потребовало бы трогать сотни импортов без всякой пользы. Домен проекта — elizovomr.ru, репозиторий — github.com/gaben1488/aemr-bot, код бота живёт в каталоге bot/aemr_bot/.

Стек и зачем каждая библиотека

Бот — один сервис на Python 3.12. Он сам разговаривает с жителями в личных диалогах, общается с операторами в служебной группе, запускает задачи по расписанию и ходит в базу данных. Отдельных «сервера», «клиента» или мини-приложения (mini-app) нет. Коротко: один процесс на Python поверх библиотеки maxapi, рядом PostgreSQL и встроенный планировщик фоновых задач APScheduler. Связь с MAX — опросный режим (long polling, длинный опрос: бот сам периодически спрашивает сервер «нет ли новых сообщений»). Административная панель — обычный групповой чат в MAX, а не отдельное приложение.

Зависимости закреплены в bot/pyproject.toml через «совместимый выпуск» (оператор ~= разрешает патч-обновления, но не пропускает внезапные ломающие изменения — например, ~=1.1 допускает 1.1.5, но не 1.2.0). Пересматривать версии договорились раз в квартал; это задача координатора.

Управление зависимостями: uv.lock — закон

Прежде чем ставить первую зависимость, разберитесь с одним дисциплинарным правилом проекта.

Единственный источник истины по версиям — файл bot/uv.lock. Это «замороженный» список всех пакетов вместе с их транзитивными зависимостями (то, что тянется само собой), с точностью до версии. Он закоммичен в репозиторий. Файл pyproject.toml задаёт диапазоны (maxapi~=1.1 — это «любая 1.1.x»), а uv.lock фиксирует, что внутри диапазона выбрано именно 1.1.0. Смысл: локальная среда разработчика, пайплайн CI и Docker-образ должны ставить побитово одни и те же версии. Тогда не бывает ситуации «у меня на ноутбуке работает, а на проде падает».

Менеджер пакетов — uv (быстрый менеджер от Astral). Канонический рабочий цикл:

cd bot
uv sync --extra dev        # синхронизировать .venv ровно с uv.lock
uv run pytest tests/ -q    # запускать тесты ТОЛЬКО через uv run
uv run ruff check aemr_bot/

Никогда не делайте pip install мимо uv

Команда pip install <пакет> в системный Python ставит версию мимо uv.lock. Тест на ноутбуке пройдёт, а в Docker всё развалится с TypeError или ImportError на старте. Именно так однажды и положили бота: тестировали локально на одной версии maxapi, а в собранный образ попала другая — и бот падал с TypeError: ClientSession got max_retries. Тесты прогоняйте только через uv run.

Как обновить пакет правильно (на примере maxapi)

Обновление зависимости делается отдельным PR, не вперемешку с фичами — чтобы при откате не тащить за собой чужие изменения. Последовательность:

  1. Сначала обновить lock. cd bot && uv lock --upgrade-package maxapi, затем git diff uv.lock — сверить, какая теперь версия. Если нужно выйти за текущий диапазон, поднять границу в pyproject.toml (например maxapi~=1.1 → ~=1.2) и снова uv lock.
  2. Синхронизировать локальную среду. uv sync --extra dev, проверить версию: uv run python -c "import maxapi; print(maxapi.__version__)".
  3. Обновить guard-тест. В bot/tests/test_deps_environment.py поднять ожидаемую версию. Если изменилась сигнатура конструктора соединения — поправить и тест на сигнатуру.
  4. Проверить ломающие изменения. uv run pytest tests/ -q и uv run ruff check aemr_bot/. Если красное — чинить вызывающий код в том же PR. Особое внимание: main.py (создание Bot/Dispatcher), utils/event.py (ack_callback, send_or_edit_screen), все вызовы bot.send_message/bot.edit_message в handlers/*, пересылка вложений в operator_reply.py.
  5. Локальный smoke. Перед мерджем поднять как на проде: cd infra && docker compose build bot && docker compose up bot — проверить, что стартует без TypeError, /livez отвечает 200. Отправить себе тестовое сообщение, тапнуть кнопку.
  6. Merge → авто-деплой → verify. Авто-деплой на сервере подхватит образ. После деплоя проверить, что контейнер бота в статусе Up (healthy), а не Restarting, и что в логах нет TypeError/ImportError. Если бот в Restarting — git revert немедленно, разбираться потом.

Диагностика расхождения (drift), когда «у меня зелено, в Docker красно»: сперва uv run pytest tests/test_deps_environment.py — если эти guard-тесты зелёные, среда синхронна с lock; если красные, uv sync --extra dev пересоздаст .venv по lock; если кто-то поставил глобальный pip-пакет — pip uninstall <pkg> -y, потом uv sync; если разошлась версия в lock против Docker — пересобрать образ docker compose build --no-cache bot. Аудит уязвимостей запускается раз в квартал или при сообщении о CVE: uv run pip-audit.

Образ собирается через pip — и почему его решено переводить на uv

Между намерением и текущим состоянием есть расхождение, и назовём его прямо. Весь deps-аппарат проекта построен вокруг uv.lock: рабочий цикл разработчика, guard-тест, dependabot (бот, который автоматически бампит зависимости в PR) — все ориентируются на lock как на закон. А вот сам Docker-образ его сегодня не читает: infra/Dockerfile ставит зависимости командой pip install -e /app из диапазонов pyproject.toml, перерезолвивая транзитивные версии заново на момент сборки. Значит lock-файл сейчас фактически декоративный: он есть, закоммичен, dependabot его бампит — но образ собирается мимо него.

Принято решение мигрировать образ на uv sync --frozen (ставить строго из lock, не перерезолвивая). Обоснование: это закрывает реальный drift, который уже один раз положил бота в проде (тот самый инцидент с max_retries). Guard-тест test_deps_environment.py сейчас зелёный по случайности — потому что ~=1.1 пока резолвится именно в 1.1.0; в день выхода новой версии pip в Docker молча подтянет её, а lock останется на старой — и «протестировано» разойдётся с «задеплоено». При авто-деплое раз в десять минут это особенно опасно: любая пересборка может незаметно поменять патч транзитивной зависимости без единой строки в git diff. С --frozen пересборка детерминирована — меняется только то, что коммит изменил в uv.lock.

Как выглядит миграция образа на uv

Образ становится двухстадийным (multi-stage): на стадии builder ставится uv (через COPY --from=ghcr.io/astral-sh/uv:<версия> — без исполнения сетевого инсталлятора curl | bash, тот же принцип безопасности, что и взятие rclone из apt), резолвится виртуальное окружение .venv строго по uv.lock командой uv sync --frozen --no-dev --no-editable; на стадию runtime копируется только готовый .venv, без uv и кэша. В .env-окружение runtime прописывается PATH="/app/.venv/bin:$PATH", чтобы голые alembic и python в команде запуска резолвились из venv. Откат миграции тривиален: git revert, плюс авто-деплой сам откатится на предыдущий коммит, если образ не поднимет /livez. Главное правило: если uv.lock хоть на одну версию расходится с диапазонами pyproject.toml, сборка упадёт на uv sync --frozen — это фича, а не баг: рассинхрон ловится до прода, а не после. Лечение — cd bot && uv lock, закоммитить обновлённый lock, пересобрать.

Архитектура и потоки данных (шагами)

Связь с MAX в боевом режиме — опросная. Бот сам периодически спрашивает MAX, есть ли новые события. Плюсы: не нужен публичный адрес, домен и сертификат; бот поднимается одной командой; разработку можно вести с любой машины с интернетом; меньше поверхность атаки (входящих портов наружу нет). Минус: при простое процесса MAX держит очередь событий ограниченное время (до восьми часов), дальше события теряются. Для MVP с одним работающим экземпляром это приемлемо. Серверный (webhook) режим полностью написан и лежит «спящим» в опциональном профиле webhook в docker-compose (Nginx, certbot, обработчик HTTP-запросов); переключение — переменная BOT_MODE=webhook.

Путь обращения: от жителя до ответа оператора

  1. Старт. Житель открывает бота. Бот показывает приветствие и главное меню — семь кнопок прямо под сообщением (об их числе ниже отдельно).
  2. Воронка. Житель нажимает «📝 Написать обращение». Запускается пошаговая анкета (конечный автомат, FSM — finite state machine). Состояние шага хранится в поле users.dialog_state, накопленные данные — в users.dialog_data (тип JSONB — бинарный JSON в PostgreSQL). Шаги по порядку: согласие на обработку персональных данных → контакт (телефон) → имя → населённый пункт (или геолокация) → адрес → тематика → суть.
  3. Финализация. На шаге сути житель отправляет одно непустое сообщение (текст плюс необязательные фото, видео, файл). Бот создаёт запись appeals со статусом new и структурированными полями (населённый пункт, адрес, тема, суть, вложения), публикует карточку в служебную группу, сохраняет идентификатор сообщения карточки и отвечает жителю «Обращение #N принято».
  4. Ответ оператора. Оператор отвечает на карточку бота свайп-реплаем (жест ответа на сообщение) или командой /reply N текст. Бот по идентификатору карточки находит обращение, проверяет длину ответа (не больше 300 символов), оборачивает текст в официальное письмо от имени Администрации и отправляет жителю в личку.
  5. Запись. После успешной доставки бот публикует в группе подтверждение, ставит appeals.status=answered и answered_at=now(), добавляет запись в messages с направлением from_operator и пишет действие в журнал audit_log.

Путь события: от MAX до записи в базе

  1. Приём. MAX отдаёт обновление (Update) через опрос. Серверный таймаут опроса вшит подменой метода bot.get_updates в main.py::_install_polling_timeout — об этом ниже в «обоснованиях».
  2. Защита от повторов. Первый промежуточный слой (middleware — код, выполняемый до обработчика) — IdempotencyMiddleware — строит ключ из типа события и его полей (update_type, callback_id, mid, seq, timestamp, chat, user) и делает INSERT ... ON CONFLICT DO NOTHING в таблицу events. Дубликат тихо отбрасывается на уровне уникального ограничения events.idempotency_key. Без этого слоя переход на webhook давал бы двойные карточки в группе.
  3. Свежесть админ-чата. Второй слой на каждое новое входящее сообщение в служебной группе сдвигает «трекер свежести» (указатель на последнюю карточку-меню). Это часть правила «редактировать или слать новое» для меню (см. ниже).
  4. Маршрутизация. Диспетчер выбирает обработчик по типу события и по тому, откуда пришло — личка жителя или служебная группа (функция utils/event.is_admin_chat). Сообщения жителя без команды попадают в общий маршрутизатор анкеты appeal.on_message, который по справочнику _STATE_HANDLERS вызывает функцию текущего шага.

Почему важен порядок регистрации обработчиков

Диспетчер MAX берёт первый подошедший обработчик одного типа события. Поэтому команды и callback-кнопки регистрируются раньше, а общий «ловит всё» маршрутизатор анкеты — последним. Иначе он молча проглотил бы /stats, /reopen и прочие команды. Это закреплено в handlers/__init__.py::register_handlers. Любая правка этого файла требует ручного прогона из админ-группы.

Защита от гонок при финализации обращения

Два сообщения от одного жителя за миллисекунды (например, свайп с фото и текст следом) могут параллельно изменить dialog_data и привести к двойной финализации. Функция persist_and_dispatch_appeal обёрнута в блокировку на пользователя (asyncio.Lock в словаре _user_locks: dict[int, asyncio.Lock]). Внутри блокировки делается проверка на повтор по полю user.dialog_state == IDLE и повторная проверка лимита трёх новых обращений за час. Второй параллельный вызов после успешной финализации первого видит состояние IDLE и выходит без создания дубля. Хелпер _drop_user_lock удаляет блокировку после завершения, чтобы словарь не рос неограниченно. Важно: это рассчитано на один экземпляр сервиса; горизонтальное масштабирование (более одного контейнера бота) потребует pg_advisory_xact_lock или блокировку на Redis.

Карта модулей по слоям

Карта слоёв помогает держать в голове, кто кого вызывает. Правило: обработчики (handlers/) принимают событие от MAX и отвечают за «диалог», но бизнес-логику не пишут — её исполняют сервисы (services/). Сервисы знают про базу и предметную область, но ничего не знают про MAX-события. В самый низ ходят только модели и сессия (db/). Тексты (copy/) и клавиатуры (ui/) — презентационный слой, его дёргают и обработчики, и часть сервисов (карточки, прогресс). Мелкие адаптеры (utils/) живут сбоку и доступны всем; главный из них, utils/event.py, — единая точка чтения разнородных событий maxapi. Стрелка на схеме означает «вызывает» (зависит от). Зависимости идут сверху вниз: db/ ничего из верхних слоёв не импортирует, и эту границу нарушать нельзя.

Из карты видны два правила, которые легко нарушить по незнанию. Первое: обработчик не лезет в базу напрямую SELECT-ом — для этого есть сервис (например, поиск обращения по карточке — это appeals.get_by_admin_message_id, а не голый запрос в operator_reply.py). Второе: всё чтение полей события идёт через utils/event.py, поэтому на схеме стрелка от MAX входит в обработчики только через этот адаптер. Фасады keyboards.py и texts.py в корне пакета на схему не вынесены: это тонкие обёртки над ui/ и copy/ ради старых импортов, новый код импортирует домен напрямую.

Ключевые инварианты схемы

  • users.max_user_id уникален в пределах платформы MAX — по нему ищется житель при следующем старте, дубликатов не возникает.
  • events.idempotency_key уникален — это основа защиты от повторов. Дубликат тихо отбрасывается на уровне ограничения уникальности.
  • appeals.admin_message_id — идентификатор первой карточки обращения, не редактируется после создания (используется как точка привязки для пересылки вложений). appeals.last_admin_card_mid — идентификатор последней карточки, обновляется при каждом её перерисовывании; по нему определяется «устаревшая» карточка при нажатии оператора.
  • users.phone_normalized (только цифры) синхронно поддерживается через services/users.set_phone. Прямой UPDATE users SET phone=... в обход сервиса ломает инвариант и поиск по телефону (/erase phone=, поиск жителя).
  • Подписчики рассылки определяются условием subscribed_broadcast=true AND is_blocked=false (плюс на SQL-уровне фильтра — наличие согласия на рассылку и имя не «Удалено»).
  • dialog_state хранится как String(32), значения берутся из перечисления DialogState в коде. Перевод в тип Enum базы — одно из возможных направлений; для MVP оставили строкой ради скорости миграций при добавлении новых состояний.
  • appeals.attachments и messages.attachments — массивы JSONB с сериализованными вложениями MAX. При пересылке в служебную группу они восстанавливаются в pydantic-объекты Attachments через TypeAdapter.
  • По полям JSONB (users.dialog_data, appeals.attachments, audit_log.details) поиска нет — только чтение по PK/FK, поэтому GIN-индексы не нужны. Если добавляете запрос вида WHERE attachments @> '...', сперва заведите индекс отдельной миграцией.

Карта таблиц и связей (ER-диаграмма)

Ниже схема «сущность-связь» (ER, entity-relationship) всех одиннадцати таблиц с реальными внешними ключами (foreign key, FK) ровно так, как они объявлены в db/models.py. Линия со значком «много» (◇ на конце «o{») читается «один ко многим»: у одного жителя может быть много обращений. Настоящих FK в схеме всего пять, все показаны сплошными линиями. Четыре таблицы стоят особняком без единого FK: events, audit_log, settings и wizard_state. Две из них (audit_log и wizard_state) ссылаются на оператора мягко, через колонку operator_max_user_id (идентификатор оператора в MAX), а не через жёсткую связь с operators.id. Сделано нарочно: журнал действий и персистентность мастеров должны переживать удаление строки оператора, поэтому привязка там логическая, а не на уровне базы. Работа с миграциями

Миграции — это версионные изменения схемы базы. Управляет ими Alembic.

# Сгенерировать новую миграцию из изменений в models.py
docker compose exec bot alembic revision --autogenerate -m "что изменилось"

# Накатить до последней
docker compose exec bot alembic upgrade head

# Откатить на одну
docker compose exec bot alembic downgrade -1

# Текущая версия в БД
docker compose exec bot alembic current

# История ревизий и сверка моделей со схемой

Если бот не поднялся

Частые причины: тайм-аут pull от Docker Hub (повторить); забит диск (docker system prune -a -f); битый WSL2 (в Docker Desktop: Settings → Troubleshoot → Clean / Purge data). Тесты: два режима и реальный порог покрытия

Тестовая база bot/tests/ рассчитана на два режима: лёгкий локальный (без Postgres, проверка чистой логики на mock-ах) и полный CI (с реальной БД, миграциями и порогом покрытия). Канонический способ запуска, совпадающий со средой Docker, — через uv и lock-файл.

Не запускайте тесты системным pip/pytest

Это поставит другие версии пакетов мимо uv.lock. Локально тесты могут пройти, а в Docker всё развалится с TypeError/ImportError. И не пытайтесь добиться «CI-режима» локально без Postgres — упрётесь в skip-нутые тесты и покрытие ниже 67. Это ожидаемая разница между «local-pure» и «CI-with-DB», а не поломка. Известный нюанс: отдельные тесты меню могут падать RuntimeError: Event loop is closed, если предыдущий тест в той же сессии закрыл async event loop — лечится точечным мок-патчем настроек в самом тесте, чтобы не открывать новое соединение к пулу Postgres.

Реальный порог покрытия в боевом CI — --cov-fail-under=67 (источник истины — .github/workflows/ci.yml). Фактическое покрытие держится чуть выше порога; цель на будущее — постепенно поднимать до 80%. Покрытие считается по веткам (--cov-branch), отчёты выгружаются как артефакты CI (XML и HTML). Точные числа тест-файлов и процент покрытия — снимок на конкретную дату и меняются от PR к PR; единственная стабильная величина, на которую стоит опираться, — это порог 67 из ci.yml. За актуальной цифрой смотрите вывод последнего прогона CI, а не документ.

Покрывать стоит: бизнес-сценарии, граничные случаи и пути с проверками безопасности (роли, валидаторы, лимиты). Базовые сценарии в тестах: согласие → запрос контакта → меню; приём обращения серией сообщений; ответ оператора цитированием; отказ при ответе длиннее 300 символов; статистика за период; удаление ПДн. Новые тесты пишите в той же папке bot/tests/.

Что проверяет CI

Пайплайн (.github/workflows/ci.yml) запускается на push и pull request в ветку main и состоит из трёх параллельных задач (job).

Job «Lint, types, security»

  • Ruff — линтер и проверка стиля (ruff check aemr_bot/).
  • MyPy — статическая проверка типов.
  • Bandit — поиск проблем безопасности уровня medium и выше (-ll).
  • pip-audit — сканирование зависимостей на известные уязвимости (CVE). Жёсткий провал при находке. Локальный editable-пакет исключается из аудита; два конкретных CVE в цепочке fastapi → starlette игнорируются с письменным обоснованием неприменимости прямо в YAML (бот не реконструирует URL из заголовка Host и не использует FileResponse/StaticFiles).
  • Shellcheck — проверка всех shell-скриптов в репозитории.

Job «Pytest with Postgres 16»

Поднимает сервис-контейнер PostgreSQL 16, прогоняет тесты с покрытием и порогом 67%, затем тремя отдельными шагами проверяет миграции:

  • upgrade head на чистую БД — ловит «осиротевшие»/дублирующиеся ревизии, синтаксис и опечатки импортов в миграциях;
  • alembic check — сравнивает модели (Base.metadata) с последней миграцией; падает, если кто-то изменил модель и забыл сгенерировать миграцию;
  • round-trip (upgrade head → downgrade base → upgrade head) — ловит сломанный откат (забытую обратную операцию, неверную ссылку на ревизию).

Job «Docker build smoke test»

Собирает образ из infra/Dockerfile без публикации — проверяет, что Dockerfile не сломан, миграции применяются и /livez отвечает.

Путь команды /reply — от оператора до жителя по слоям

Рецепты выше станут нагляднее на одном сквозном пути по слоям: оператор пишет ответ, житель получает официальное письмо. Это самый чувствительный к безопасности маршрут в проекте (ответ от имени Администрации), и на нём хорошо видно правило «обработчик → сервис → база → обратно». Ответ можно дать двумя способами, и они сходятся в одной точке: свайп-реплай (жест ответа на карточку) и команда /reply N текст. Разница только в том, как находится номер обращения. При свайпе идентификатор исходной карточки лежит в link.message.mid (на уровень глубже, чем link.mid — это разобранная в граблях maxapi ловушка), и обработчик operator_reply.py зовёт сервис appeals.get_by_admin_message_id; при команде номер берётся прямо из аргумента и идёт в appeals.get_by_id. Дальше путь общий: проверка длины (не больше 300 символов, ANSWER_MAX_CHARS), проверка is_blocked (житель, отозвавший согласие, не получит сообщение — оператору вернётся подсказка «звоните»), упаковка текста в официальное письмо (card_format), отправка через bot.send_message и запись результата в базу одной транзакцией: строка в messages с направлением from_operator, appeals.status=answered и answered_at=now(), плюс действие action="reply" в audit_log. На схеме горизонтальные дорожки — это слои: событие приходит в обработчик, тот ходит в сервисы, сервисы — в базу, и в конце житель получает письмо.

Где истина по cron-задачам

Реестр cron_registry.py — машиночитаемый источник имён задач и их назначения для anti-drift-теста test_cron_docs_sync. Но точное расписание (минуты, дни недели) надо брать из реальных объектов CronTrigger в cron.py — именно там фактический триггер. Сами расписания подробно расписаны в разделе «Администратору»; здесь достаточно знать имена ключевых задач: pulse-hourly (heartbeat каждый час 24/7), pulse-workhours-extra (доп. пинг в рабочее время), startup-pulse, health-selfcheck, funnel-watchdog (сброс зависших анкет старше 24 ч), events-retention (30 дней), audit-log-retention (365 дней), appeals-5y-retention, pdn-retention, threat-intel-refresh, broadcast-draft-reaper, db-backup и две reminder-задачи по открытым и просроченным обращениям. Общие параметры всех задач: max_instances=1, coalesce=True, misfire_grace_time=120 секунд.

Архитектурные обоснования (почему так)

Самое ценное — не «как», а «почему». Эти решения зафиксированы в ADR и в коде, чтобы их не переоткрывали и не ломали по незнанию.

Почему JSONB, а не Redis для состояния анкеты

Состояние пошаговой анкеты (dialog_state + dialog_data) хранится прямо в PostgreSQL в колонке JSONB, а не в Redis. Население округа около 70 тысяч человек, проникновение цифровых госсервисов 3–5%. Активных воронок одновременно — десятки в часы пик, единицы ночью; пиковая нагрузка 1–2 операции записи в секунду. Postgres проглатывает это не замечая, а лишний сервис (Redis) — это лишняя точка отказа и эксплуатации. Триггер пересмотра — устойчиво более 500 одновременных воронок по метрикам /diag.

Почему форк сообщества maxapi, а не официальный

Из трёх вариантов клиента MAX для Python выбран love-apples/maxapi (сообщество, активно обновляется, последний коммит — апрель 2026). Официальный max-messenger/max-botapi-python — это снимок исходного репозитория девятимесячной давности (заморожен на ревизии 2025-07-30): переход означал бы откат с потерей нужных нам исправлений (включая process_update_webhook, TypeAdapter[Attachments], MessageLinkType). Вариант через шлюз green-api требует платную подписку, что противоречит принципу «свой код, не подписочный сервис». Решение делать собственный форк (в gaben1488/maxapi под контролем муниципалитета) принимается только при трёх и более месяцах без коммитов в исходник с открытыми критическими задачами, либо при незакрытой уязвимости в течение 14 дней. Ежеквартальная проверка активности — задача координатора.

Почему обязательный слой защиты от повторов

Дубликаты обновлений от MAX редки в опросном режиме, но частые при повторных доставках в серверном. IdempotencyMiddleware подключён как внешний слой и срабатывает на каждое обновление до любого обработчика. Без него переход на webhook сразу дал бы двойные карточки в админ-группе. Поэтому это не «страховка на будущее», а обязательный компонент. Юридический смысл тоже есть: одно действие жителя не должно превратиться в два обращения или две записи в журнале.

Почему лимит ответа оператора — 300 символов

Жёсткий лимит в 300 символов в коде (ANSWER_MAX_CHARS). При попытке ответить длиннее — отказ с сообщением в группу, без модального окна и без авто-разбивки: оператор сам решает, что делать. Цифра из требований куратора (200–300); взята верхняя граница, чтобы не ломать стандартные шаблоны ответов на 250–280 символов. Это упрощает интерфейс карточки.

Почему подписка на рассылку — отдельное согласие, а скорость — половина лимита

Рассылка спроектирована (дизайн-спека одобрена 2026-05-04) вокруг нескольких намеренных ограничений. Знать их нужно прежде, чем трогать broadcast*.py. Подписка на новости — это явный opt-in (users.subscribed_broadcast, по умолчанию false), и это отдельная цель обработки данных: согласие на ПДн внутри воронки обращения — про конкретное обращение, а согласие на рассылку — отдельный флаг, который житель ставит и снимает сам (/subscribe//unsubscribe, переключатель в меню, кнопка «отписаться» под каждым объявлением). Список получателей — это снимок условия subscribed_broadcast=true AND is_blocked=false (плюс наличие согласия на рассылку и имя не «Удалено»), снятый один раз в начале и зафиксированный, чтобы рассылка не «плыла» по мере отписок. Скорость отправки намеренно ограничена одним сообщением в секунду (BROADCAST_RATE_LIMIT_PER_SEC=1.0) — это ровно половина платформенного лимита MAX в 2 запроса в секунду, вторая половина оставлена обычной активности бота, чтобы рассылка на тысячу жителей не блокировала приём обращений. Кнопка «⛔ Экстренно остановить» под прогресс-сообщением доступна любому оператору в группе (а не только автору) — на случай компрометации учётки координатора; фоновая задача сверяет флаг status между каждой отправкой и обрывается на cancelled. Сам текст рассылки в stdout не пишется — только идентификаторы и счётчики; в audit_log уходит action="broadcast_send" с числом подписчиков, но опять же без утечки текста в логи. Незавершённый мастер рассылки живёт в памяти с TTL 300 секунд и теряется на рестарте — это приемлемо (оператор просто запускает /broadcast заново), а застрявшие черновики со статусом draft старше 30 минут добивает cron-задача broadcast-draft-reaper, переводя их в failed.

Почему один экземпляр (и что сломает масштабирование)

Финализация обращения защищена блокировкой на пользователя в памяти процесса (asyncio.Lock в словаре _user_locks) — это закрывает гонки двойной отправки и параллельных событий по одному жителю. Состояние мастеров рассылки тоже частично живёт в памяти (словари с TTL 300 секунд). Всё это рассчитано на один экземпляр бота. Горизонтальное масштабирование (несколько контейнеров) потребует перейти на блокировки уровня базы (pg_advisory_xact_lock) или Redis.

Почему таймаут опроса — это не интервал

POLLING_TIMEOUT_SECONDS (по умолчанию 30, потолок MAX — 90, диапазон 0..90) задаёт серверное время ожидания вызова getUpdates: сколько MAX держит соединение, пока нет событий. Это не пауза между запросами клиента. Опрос возвращает ответ либо при появлении события, либо по истечении этого времени. Чем больше значение — тем меньше пустых запросов при простое (важно при ограничении 2 запроса в секунду на бота, действующем с 11 мая 2026 года), но медленнее реакция на остановку. На нуле бот будет хлестать API на пределе и кончит бюджет за минуты. Реализовано подменой связанного метода bot.get_updates в main.py::_install_polling_timeout, потому что Dispatcher.start_polling не пробрасывает таймаут внутрь. Если когда-нибудь понадобится свой форк maxapi — это первый кандидат на запрос изменений.

Почему SLA в коде — 4 часа, а в Регламенте — иначе

В коде заложен единый базовый срок ответа: SLA_RESPONSE_HOURS = 4 рабочих часа (config.py). У модели обращения нет поля важности или приоритета и нет тематической маршрутизации — бот меряет попадание в единый 4-часовой SLA и шлёт напоминания о просроченных. Дифференциация сроков по важности остаётся организационной (её ведёт Куратор ЦУР вручную); полный правовой разбор этого разделения — в разделе для оператора. Инженерное следствие: чтобы ввести разные сроки по типу обращения, сначала нужно добавить в схему поле важности/маршрута.

Почему 365 дней на журнал действий, а не 5 лет

Срок хранения audit_log — 365 дней (AUDIT_LOG_RETENTION_DAYS, диапазон 30–3650), это сознательный выбор: журнал действий — это журнал событий безопасности (приказ ФСТЭК № 21, мера РСБ.1), а не архив обращений, поэтому отдельный 5-летний ретеншн обращений (appeals-5y-retention) к нему не привязан, и менять код не нужно. Полный правовой разбор норм — подробно в Правовом.

Отладка и грабли maxapi

Библиотека maxapi удобна, но местами «протекает»: модели не покрывают все поля сервера, имена полей расходятся с документацией, отдельные методы возвращают объекты, которых нет в типизации. Ниже грабли, на которые уже наступали — собраны здесь, чтобы больше никто не наступал.

  1. Свайп-реплай: Message.link.message.mid, а не Message.link.mid. Когда оператор отвечает свайпом, приходит Message.link типа LinkedMessage. Сам link.mid пустой — идентификатор исходного сообщения лежит на уровень глубже, в link.message.mid (внутри вложенного MessageBody). Извлечение — operator_reply.py::_extract_reply_target_mid и _mid_from_link. На отдельных версиях клиента pydantic-обёртка превращается в dict — отсюда двойная попытка чтения.
  2. Результат отправки — SendedMessage, а не Message. У результата bot.send_message(...) нет .message_id напрямую; идентификатор — в result.message.body.mid. Адаптер utils/event.extract_message_id снимает оба варианта через цепочку getattr и работает на None-входе. Используется при сохранении appeals.admin_message_id и messages.max_message_id.
  3. Загрузка файлов — AttachmentUpload, а не плоский dict. Старая ревизия maxapi отдавала dict, текущая — pydantic-модель. Для отправки в attachments=[...] сериализуйте через .model_dump(by_alias=True) или конкретные конструкторы (PhotoAttachment, FileAttachment). См. services/uploads.py и services/policy.py::build_file_attachment.
  4. Порядок регистрации хендлеров важен. Диспетчер берёт первый совпавший. Если зарегистрировать message_created без фильтра раньше, чем message_created(Command(...)), команда не сработает. Все универсальные обработчики анкеты регистрируются последними, команды и callback — раньше.
  5. Защита потока жителя через is_admin_chat. Обработчики для жителя (/start, главное меню, «Написать обращение») оборачиваются проверкой if is_admin_chat(event): return. Иначе оператор, написавший /start в служебной группе, попал бы в анкету как «житель», завёл бы запись в users и застрял в AWAITING_CONSENT.
  6. Никогда не лезьте к полям события напрямую. Не пишите event.chat_id или event.user.user_id — структуры событий неоднородны. Всегда через адаптер utils/event.py (get_chat_id, get_user_id).
  7. Отладочный дамп — осторожно с ПДн. При проблемах с парсингом включайте LOG_LEVEL=DEBUG — диспетчер пишет полный pydantic-дамп события. Но не дампите event.message в боевые логи: туда утекут персональные данные.
  8. Восстановление застрявших анкет. Если бот рестартанул посреди анкеты, dialog_state жителя застывает. На старте recover_stuck_funnels (через asyncio.create_task, не блокируя диспетчер) находит застрявших в AWAITING_SUMMARY старше таймаута и завершает их обращения пачками (RECOVER_BATCH_SIZE=1000). Пустые сессии (нет ни текста, ни вложений) сбрасываются в IDLE. Лимит партии защищает от патологии: 10 тысяч застрявших воронок после долгого простоя не дадут 10 тысяч вызовов API на старте.

Эксплуатационные компоненты

Проверка живости. health.py поднимает HTTP-сервер на порту WEBHOOK_PORT (по умолчанию 8080), публикуется только на 127.0.0.1 (наружу ничего не выставляется — модель самостоятельного хостинга без входящих портов). Эндпоинтов три: /livez — liveness, отдаёт 200, пока жив asyncio-цикл бота (только heartbeat, базу не трогает; на него смотрят Docker healthcheck, watchdog и авто-деплой при решении об откате); /readyz — readiness, отдаёт 200, если heartbeat свежий и база отвечает на SELECT 1 (с TTL-кэшем 10 секунд; для диагностики, не для авто-рестарта); /healthz — алиас /readyz, оставлен для обратной совместимости. Локальным запросам (с 127.0.0.1) отдаётся полная диагностика, внешним — только {"ok": ...}. Объект Heartbeat обновляется фоновой задачей; если основной цикл завис, сигнал перестаёт обновляться и эндпоинт отдаёт 503.

В старых документах фигурирует только /healthz

Это устарело. В коде эндпоинты разделены на /livez (жив ли процесс) и /readyz (жив и готов, с проверкой базы), а /healthz остался лишь как алиас readiness ради совместимости. Автоматический рестарт/откат должен смотреть на /livez, а не на /healthz.

Нереализованный потенциал maxapi (на будущее)

Это не онбординг, а каталог возможностей библиотеки на будущее — свёрнут, чтобы не отвлекать от текущего кода. Разверните, когда будете планировать развитие. Разбор установленной библиотеки maxapi 1.1.0 и её примеров вскрыл набор готовых возможностей, которые упростили бы код или дали новые фичи. Часть уже внедрена, часть осознанно отложена.

Уже внедрено по итогам разведки

  • Кнопка запроса геолокации. RequestGeoLocationButton уже стоит на шаге выбора населённого пункта (ui/citizen_keyboards.py, в localities_keyboard) — житель не пишет адрес текстом и не шлёт геопозицию вложением, а нажимает «📍 Поделиться геолокацией», и бот определяет поселение/адрес через services/geo.py, затем просит подтверждения (AWAITING_GEO_CONFIRM).
  • Кэш токена вложения. PDF-шаблон политики (он не меняется) заливается один раз, токен кэшируется в БД и переиспользуется без повторной загрузки — экономия времени на каждом /policy.
  • Индикатор «бот печатает». Долгие операции (снимок аудитории рассылки, листинг «Мои обращения», тяжёлые PDF/XLSX) обёрнуты в авто-периодический typing-индикатор (utils/typing_indicator.py) — пользователь видит, что бот не завис.
  • Обход аудитории через iter_all(). Перечисление участников группы идёт через библиотечный async-итератор с защитой от зацикливания marker-пагинации вместо ручного цикла.
  • Обработка HTTP 429 и BOT_STOPPED. Библиотека сама не парсит Retry-After и игнорирует 429 — поэтому для рассылки на 1000+ жителей добавлена ручная обработка ограничения скорости; событие BOT_STOPPED (житель остановил бота) исключает его из аудитории рассылки.

Сознательно отложено (декларативный сдвиг парадигмы)

Библиотека предлагает целый декларативный слой, который сейчас в боте написан вручную. Это не дефект, а осознанный выбор «не сейчас» — миграция большая и рискованная, а текущая ручная реализация работает.

  • FSM через StatesGroup + MemoryContext. В maxapi есть встроенное состояние per пара (chat_id, user_id): context.set_state(...), get_data()/update_data(...), фильтр по состоянию вторым аргументом декоратора, LRU-вытеснение и TTL, подменяемое хранилище (вплоть до RedisContext). У нас этот механизм фактически переизобретён руками: wizard_registry.py + wizard_persist.py + миграция 0011 + несколько module-level словарей. Переход на встроенный FSM снял бы сотни строк и убрал бы ручной catch-all диспетчер (каждое сообщение приходило бы сразу в правильный обработчик по состоянию). Это самый весомый кандидат на будущий рефакторинг, но он требует полной регрессии всей анкеты через сквозные тесты.
  • Типизированный payload кнопок (CallbackPayload). Сейчас callback-строки вида appeal:123:reply парсятся вручную через .split(":"), префиксы зашиты в строках, формат легко спутать между местом сборки кнопки и местом разбора. Pydantic-модель CallbackPayload с pack()/unpack() и фильтром по полю сделала бы такие регрессии невозможными конструктивно — особенно для самого чувствительного к безопасности пути обращений.
  • Event-helpers вместо ручного send. Примеры используют event.message.answer(text), event.message.reply(...), event.ack(); мы во многих местах пишем event.bot.send_message(chat_id=..., user_id=...) — 6 строк вместо 1. Идиома в коде смешана (часть файлов уже на helper-ах).
  • Готовый фильтр контакта и html_text/md_text. Фильтр Contact() из maxapi уже распаршивает vCard и инжектит ContactAttachment в обработчик — нашу ручную реализацию парсинга vCard можно было бы убрать. Свойства MessageBody.html_text/md_text отдают форматированный текст с правильным экранированием — сейчас в карточке «суть» оператор не видит жирный/курсив, которым писал житель.
  • Раздельные роутеры. Router со своими middleware и фильтрами, подключаемый через dp.include_routers(...) — естественные кандидаты на вынос крупные модули рассылок и настроек.

Чего нет в платформе MAX в принципе

Эти вещи отсутствуют и в swagger MAX, и в библиотеке — делать их не из чего: inline-режим (как в Telegram), реакции, опросы (polls), live-геолокация (только статичная), бесшумная копия сообщения (copy_message — есть только forward с пометкой «переслано»), пакетная загрузка и presigned-URL, заголовки max_connections/drop_pending_updates/ip_address, idempotency-key на уровне API (поэтому наш IdempotencyMiddleware — единственный слой защиты), API истории правок сообщения (приходит только новый текст — старый, если нужен, храните сами), mute/restrict участника (только kick с опциональным баном), запланированные сообщения (наш cron-cooldown остаётся). Параллельная обработка событий (use_create_task=True) в библиотеке есть, но включать её без рефакторинга module-level словарей мастеров нельзя — будут гонки.

Известные ограничения и направления оптимизации

Вещи, которые работают, но при росте нагрузки или объёма данных могут дать о себе знать. Каждый пункт — фиксация осознанного выбора «не сейчас» с триггером, при котором стоит вернуться. На MVP с десятками подписчиков и тысячами обращений в год ни один из них не блокирует запуск.

  • Рассылка открывает три session_scope на получателя (проверка флага отмены, запись доставки, периодический прогресс). На текущем масштабе это копейки. Триггер фикса: устойчиво >1000 подписчиков или несколько одновременных рассылок. Лечение: батчить запись доставок пачками по 50, флаг проверять раз в N сообщений.
  • Длинная if-цепочка в on_callback (~200 строк ветвлений по префиксам payload). Читается линейно, но новый callback добавляется в общую кучу. Триггер: 5+ новых callback-ов. Лечение: словарь {prefix: handler_fn} по образцу _STATE_HANDLERS (или переход на CallbackPayload, см. выше).
  • Анонимные обработчики _(event) в start.py и др. — удобно для регистрации, но в стектрейсе вместо имени команды будет <locals>._. Триггер: ловля сложного бага. Лечение: именовать функции по команде.
  • _user_locks в памяти процесса — не разделяется между репликами. На MVP один инстанс, это нормально. Триггер: multi-replica. Лечение: pg_advisory_xact_lock или Redis.
  • Inline русские usage-строки в admin_commands.py (десяток коротких подсказок «Используйте: /reopen <номер>») — технические, жителю не видны. Триггер: третья ревизия команд или i18n. Лечение: вынести в константы.

Несколько ограничений ранних версий уже закрыто: полный скан таблицы при поиске по телефону (закрыт индексом 0003); дублирование обвязки XLSX (общий хелпер); рост events без чистки (cron-ретеншн 30 дней); фиксированный шаг прогресс-бара рассылки (адаптивная формула); молчаливый сбой бэкапа (категоризированные алёрты); /healthz, отдававший OK при зависшей базе (добавлен SELECT 1); запуск контейнера под root без read-only (теперь непривилегированный botuser, read_only: true, tmpfs:/tmp, лимиты памяти и pids, cap_drop: ALL, no-new-privileges).

Развёртывание и рабочий цикл

Ручной деплой на сервере (полную автоматику авто-деплоя см. в разделе «Администратору»):

  1. Ветка. git checkout -b feat/название.
  2. Правки и проверка. Локальные тесты (uv run pytest) плюс ручная проверка через docker compose. Перед коммитом — зелёные линтер, типы и тесты.
  3. Коммит и PR. Сообщения в формате feat: / fix: / refactor: / docs: / chore:. Push в origin, открыть pull request на main. После одобрения — squash-merge.
  4. Деплой. На сервере: git pull && docker compose up -d --build bot && docker compose exec bot alembic upgrade head.

Соглашения по коду

  • Python 3.12, аннотации типов везде, где возможно. Ruff с длиной строки 100 (конфиг в pyproject.toml).
  • Комментарии объясняют зачем, а не что: невидимые ограничения, тонкие инварианты, ссылки на документацию. Имена переменных уже говорят, что делает код. Комментарии в проекте пишутся на русском.
  • Импорты — сверху файла. Импорт внутри функции оправдан только честной циклической зависимостью.
  • Никогда не обращайтесь к полям события напрямую — только через адаптер utils/event.py.

Словарь терминов

ТерминРасшифровка
long polling (опросный режим)Бот сам периодически спрашивает MAX о новых событиях. Не требует публичного адреса.
webhook (серверный режим)MAX сам шлёт события на HTTP-эндпоинт бота. Требует домен и сертификат. В боевом режиме не используется.
FSM (конечный автомат)Пошаговая анкета приёма обращения: состояние — в dialog_state, данные — в dialog_data.
middleware (промежуточный слой)Код, который выполняется до обработчика на каждое событие: защита от повторов и трекер свежести.
ORMОбъектно-реляционное отображение: таблицы базы описаны как классы Python (SQLAlchemy в models.py).
JSONBБинарный JSON в PostgreSQL — хранит структурированные данные (анкета, вложения, настройки) в одной колонке.
cron (планировщик)Запуск задач по расписанию через APScheduler: бэкап, отчёты, ретеншн, heartbeat, напоминания.
migration (миграция)Версионное изменение схемы базы через Alembic.
backup (резервная копия)Снимок базы через pg_dump, зашифрованный gpg, в named-volume backups.
idempotency (идемпотентность)Свойство «повторная обработка того же события не создаёт дубль». Обеспечивается уникальным ключом в таблице events.
heartbeat (сигнал живости)Периодический сигнал, что основной цикл бота жив; на нём держится /readyz.
lock-файл (uv.lock)Замороженный список точных версий всех пакетов; гарантирует одинаковую среду локально, в CI и в Docker.
drift (расхождение)Ситуация «у меня работает, на проде падает» из-за разных версий зависимостей. Лечится синхронизацией по lock.

Частые вопросы

Почему мои локальные тесты дают покрытие ниже 67%, а CI зелёный?

Без поднятого PostgreSQL тесты, которым нужна реальная база, помечаются skip, и покрытие закономерно ниже. Это не баг. Чтобы повторить условия CI, поднимите docker compose up -d db и задайте DATABASE_URL на локальную базу.

Где «истина» по списку cron-задач и их расписаниям?

Имена и назначение — в services/cron_registry.py (его соответствие документации проверяет тест test_cron_docs_sync). А точные минуты и дни недели — в реальных CronTrigger в services/cron.py: реестр человекочитаемый и в одном месте может отставать от фактического триггера.

Сколько в проекте таблиц и миграций?

11 таблиц (db/models.py) и 17 миграций (0001..0017, последняя — 0017_appeals_last_card_mid). Цифры «9 таблиц» и «5 миграций» в старых документах устарели.

Можно ли менять phone прямым SQL?

Нет. Всегда через services/users.set_phone — иначе рассинхронизируется phone_normalized и сломается поиск по телефону (/erase phone=, поиск жителя).

Почему slash-команды видны в админ-группе и путают операторов?

У MAX нет раздельных списков команд для лички и группы (нет per-scope команд). Поэтому на старте бот вычищает slash-меню целиком (прямой PATCH /me с {"commands": []}), а команды остаются рабочими как запасной путь. Жители работают кнопками, операторы знают команды из RUNBOOK или памятки /op_help.

Сколько кнопок в главном меню жителя — шесть или семь?

Семь. В коде main_menu отдаёт семь рядов: «📝 Написать обращение», «📂 Мои обращения», переключатель подписки на рассылку, «🏛 Приём граждан», «ℹ️ Полезная информация», «🛡️ Защита от мошенников», «⚙️ Настройки и помощь». В части старых текстов и в комментарии-докстринге упомянуто «шесть» — это устаревшая цифра: кнопка «🛡️ Защита от мошенников» была добавлена отдельным пунктом сверх исходных шести. Доверяйте коду.

Что делать при ошибке «password authentication failed for user "aemr"»?

При первом запуске POSTGRES_PASSWORD записывается в данные тома и больше не перечитывается. После ротации пароля либо сделайте ALTER USER aemr WITH PASSWORD '...' в живой базе, либо пересоздайте том (docker compose down -v — сотрёт все данные, только на dev).
