Имя пакета и название территории

Имя Python-пакета — aemr_bot. Буквы «AEMR» расшифровываются как «Администрация Елизовского муниципального района» — это историческое название. Территория давно переименована в округ (АЕМО — Администрация Елизовского муниципального округа), и все тексты для жителей говорят «округ». Но идентификатор aemr_bot в коде, в именах миграций и в путях оставлен прежним ради обратной совместимости: переименование пакета сломало бы историю миграций и потребовало бы трогать сотни импортов без всякой пользы. Домен проекта — elizovomr.ru, репозиторий — github.com/gaben1488/aemr-bot, код бота живёт в каталоге bot/aemr_bot/.

Стек и зачем каждая библиотека

Бот — один сервис на Python 3.12. Он сам разговаривает с жителями в личных диалогах, общается с операторами в служебной группе, запускает задачи по расписанию и ходит в базу данных. Отдельных «сервера», «клиента» или мини-приложения (mini-app) нет. Коротко: один процесс на Python поверх библиотеки maxapi, рядом PostgreSQL и встроенный планировщик фоновых задач APScheduler. Связь с MAX — опросный режим (long polling, длинный опрос: бот сам периодически спрашивает сервер «нет ли новых сообщений»). Административная панель — обычный групповой чат в MAX, а не отдельное приложение.

Зависимости закреплены в bot/pyproject.toml через «совместимый выпуск» (оператор ~= разрешает патч-обновления, но не пропускает внезапные ломающие изменения — например, ~=1.1 допускает 1.1.5, но не 1.2.0). Пересматривать версии договорились раз в квартал; это задача координатора.

Управление зависимостями: uv.lock — закон

Прежде чем ставить первую зависимость, разберитесь с одним дисциплинарным правилом проекта.

Единственный источник истины по версиям — файл bot/uv.lock. Это «замороженный» список всех пакетов вместе с их транзитивными зависимостями (то, что тянется само собой), с точностью до версии. Он закоммичен в репозиторий. Файл pyproject.toml задаёт диапазоны (maxapi~=1.1 — это «любая 1.1.x»), а uv.lock фиксирует, что внутри диапазона выбрано именно 1.1.0. Смысл: локальная среда разработчика, пайплайн CI и Docker-образ должны ставить побитово одни и те же версии. Тогда не бывает ситуации «у меня на ноутбуке работает, а на проде падает».

Менеджер пакетов — uv (быстрый менеджер от Astral). Канонический рабочий цикл:

cd bot
uv sync --extra dev        # синхронизировать .venv ровно с uv.lock
uv run pytest tests/ -q    # запускать тесты ТОЛЬКО через uv run
uv run ruff check aemr_bot/

Никогда не делайте pip install мимо uv

Команда pip install <пакет> в системный Python ставит версию мимо uv.lock. Тест на ноутбуке пройдёт, а в Docker всё развалится с TypeError или ImportError на старте. Именно так однажды и положили бота: тестировали локально на одной версии maxapi, а в собранный образ попала другая — и бот падал с TypeError: ClientSession got max_retries. Тесты прогоняйте только через uv run.

Как обновить пакет правильно (на примере maxapi)

Обновление зависимости делается отдельным PR, не вперемешку с фичами — чтобы при откате не тащить за собой чужие изменения. Последовательность:

  1. Сначала обновить lock. cd bot && uv lock --upgrade-package maxapi, затем git diff uv.lock — сверить, какая теперь версия. Если нужно выйти за текущий диапазон, поднять границу в pyproject.toml (например maxapi~=1.1 → ~=1.2) и снова uv lock.
  2. Синхронизировать локальную среду. uv sync --extra dev, проверить версию: uv run python -c "import maxapi; print(maxapi.__version__)".
  3. Обновить guard-тест. В bot/tests/test_deps_environment.py поднять ожидаемую версию. Если изменилась сигнатура конструктора соединения — поправить и тест на сигнатуру.
  4. Проверить ломающие изменения. uv run pytest tests/ -q и uv run ruff check aemr_bot/. Если красное — чинить вызывающий код в том же PR. Особое внимание: main.py (создание Bot/Dispatcher), utils/event.py (ack_callback, send_or_edit_screen), все вызовы bot.send_message/bot.edit_message в handlers/*, пересылка вложений в operator_reply.py.
  5. Локальный smoke. Перед мерджем поднять как на проде: cd infra && docker compose build bot && docker compose up bot — проверить, что стартует без TypeError, /livez отвечает 200. Отправить себе тестовое сообщение, тапнуть кнопку.
  6. Merge → авто-деплой → verify. Авто-деплой на сервере подхватит образ. После деплоя проверить, что контейнер бота в статусе Up (healthy), а не Restarting, и что в логах нет TypeError/ImportError. Если бот в Restarting — git revert немедленно, разбираться потом.

Диагностика расхождения (drift), когда «у меня зелено, в Docker красно»: сперва uv run pytest tests/test_deps_environment.py — если эти guard-тесты зелёные, среда синхронна с lock; если красные, uv sync --extra dev пересоздаст .venv по lock; если кто-то поставил глобальный pip-пакет — pip uninstall <pkg> -y, потом uv sync; если разошлась версия в lock против Docker — пересобрать образ docker compose build --no-cache bot. Аудит уязвимостей запускается раз в квартал или при сообщении о CVE: uv run pip-audit.

Как образ ставит зависимости: из uv.lock с проверкой хешей

Весь deps-аппарат проекта построен вокруг uv.lock: рабочий цикл разработчика, guard-тест, Dependabot (бот, который автоматически бампит зависимости в PR) — все ориентируются на lock как на закон. И боевой образ тоже: infra/Dockerfile собирает зависимости строго из uv.lock, а не из «плавающих» диапазонов pyproject.toml. Раньше образ действительно ставил pip install -e /app по диапазонам и перерезолвивал транзитивные версии заново на каждой сборке — отсюда был реальный drift, который однажды положил бота в проде (тот самый инцидент с max_retries). Сейчас это закрыто.

Сборка идёт в два шага и проверяет целостность каждого пакета по криптографическому хешу:

  1. uv export --frozen разворачивает uv.lock в обычный requirements.txt, где у каждого пакета проставлен SHA256-хеш wheel'а (флаги --no-emit-project --no-dev выкидывают сам aemr-bot и dev-группу). --frozen означает «не перерезолвивать»: если uv.lock хоть на одну версию расходится с диапазонами pyproject.toml, шаг падает — рассинхрон ловится при сборке, а не на проде.
  2. pip install --require-hashes --no-deps ставит ровно этот список и отказывается ставить что-либо, чей фактический хеш не совпал с зафиксированным (fail-closed: один неверный SHA256 — и build обрывается). Это защита supply-chain: подмена пакета на зеркале PyPI или MITM при скачивании не пройдёт незаметно — у подменённого артефакта будет другой хеш.

Следствие: пересборка детерминирована. Меняется только то, что коммит изменил в uv.lock; «протестировано» и «задеплоено» не расходятся даже при авто-деплое раз в десять минут. Guard-тест test_deps_environment.py сверяет установленную версию maxapi с lock как дополнительный рубеж.

Почему именно uv export + --require-hashes, а не просто uv sync --frozen

Оба способа дают детерминированную установку из lock, но выбран именно экспорт в requirements.txt с последующим pip --require-hashes, потому что он добавляет явную проверку SHA256 каждого wheel'а — гарантия целостности строже, чем у одного --frozen (тот фиксирует версии, но не сверяет хеши при скачивании). Сам uv нужен только на этапе сборки: его бинарь берётся пиннутым из официального образа (COPY --from=ghcr.io/astral-sh/uv:<версия>@sha256:…, без исполнения сетевого инсталлятора curl | bash — тот же принцип безопасности, что и взятие rclone из apt) и в финальный runtime не попадает. Пакет aemr-bot ставится отдельным шагом editable и тоже --no-deps: всё дерево зависимостей уже установлено хеш-верифицированным шагом, повторно тянуть с PyPI нечего. Если uv.lock разойдётся с диапазонами pyproject.toml, лечение — cd bot && uv lock, закоммитить обновлённый lock, пересобрать.

Dependabot: как обновляются зависимости и как ревьюить PR

Обновления зависимостей не нужно искать вручную — за этим следит Dependabot (настройка в .github/dependabot.yml). Раз в неделю, по понедельникам в 06:00 по Камчатке, он проверяет три экосистемы и заводит pull request на каждое доступное обновление:

ЭкосистемаГде смотритЛимит открытых PRПрефикс коммита / метки
pip (Python-зависимости)/bot5chore(deps) · dependencies, python
github-actions/ (файлы workflow)5chore(ci) · dependencies, github-actions
docker (базовый образ)/infra3chore(docker) · dependencies, docker

Лимит открытых PR на экосистему стоит нарочно — чтобы не завалить трекер десятком веток разом. Три пакета выведены из автообновления частично или полностью, и это самое важное в конфигурации, потому что объясняет «почему так»:

  • maxapi — исключён целиком (любая версия). Это ключевая зависимость от платформы MAX, и любой её апдейт — даже патч — может незаметно сломать создание Bot/Dispatcher, пересылку вложений или разбор событий (именно она однажды положила бота инцидентом max_retries). Поэтому обновляется только вручную, по процедуре из docs/RUNBOOK.md (см. «Как обновить пакет правильно» выше); в комментарии конфигурации стоит отсылка к истории 0.9.18 → 1.1.0.
  • sqlalchemy — игнорируется только мажор (semver-major). Минорные и патч-обновления 2.0.x безопасны и проходят автоматически; переход на 2.1+ требует ручной проверки, потому что async-API мог измениться.
  • pydantic — игнорируется только мажор (semver-major). Та же логика: минор проходит сам, major-bump требует ручной проверки моделей настроек (Settings) и ChatMembersManager.

Как ревьюить Dependabot-PR. Минорные апдейты безопасных пакетов обычно достаточно «прогнать через CI и смержить, если зелено». А вот мажорный апдейт (или любой апдейт пакета из списка ручной проверки) ревьюится как полноценная смена зависимости:

  1. Прочитать changelog обновляемого пакета на предмет ломающих изменений (breaking changes), особое внимание — публичному API, который реально вызывается в боте.
  2. Прогнать guard-тест среды. uv run pytest tests/test_deps_environment.py — он сверит установленную версию и сигнатуру ключевого API с lock. При смене мажора обновить в нём ожидаемую версию.
  3. Полный прогон и smoke. uv run pytest tests/ -q и uv run ruff check aemr_bot/; затем локальный smoke как на проде — cd infra && docker compose build bot && docker compose up bot — убедиться, что контейнер стартует без TypeError/ImportError и /livez отвечает 200.
  4. Чинить вызывающий код в том же PR, если что-то покраснело, — не отдельной веткой.

Связка с supply-chain: lock → CI → hash-enforced образ

Эти части складываются в одну цепочку, и держать её в голове полезно. Dependabot бампит uv.lock (для Python) или пин базового образа / SHA action'а — в pull request. CI на этом PR проверяет: pip-audit по сторонним зависимостям и round-trip миграций, а test_deps_environment ловит drift среды. После мерджа боевой образ собирается из обновлённого uv.lock через uv export --frozen + pip install --require-hashes — то есть в прод приедет ровно то, что было проверено и хеш-верифицировано, версия в версию. Сами GitHub Actions в пайплайне тоже запинены по commit-SHA (а не по перемещаемому тегу) — поэтому github-actions-экосистема Dependabot и нужна: она поднимает эти SHA-пины. Подробнее про SHA-пиннинг и pip-audit — в разделе «Что проверяет CI».

Архитектура и потоки данных (шагами)

Связь с MAX в боевом режиме — опросная. Бот сам периодически спрашивает MAX, есть ли новые события. Плюсы: не нужен публичный адрес, домен и сертификат; бот поднимается одной командой; разработку можно вести с любой машины с интернетом; меньше поверхность атаки (входящих портов наружу нет). Минус: при простое процесса MAX держит очередь событий ограниченное время (до восьми часов), дальше события теряются. Для MVP с одним работающим экземпляром это приемлемо. Серверный (webhook) режим полностью написан и лежит «спящим» в опциональном профиле webhook в docker-compose (Nginx, certbot, обработчик HTTP-запросов); переключение — переменная BOT_MODE=webhook.

Путь обращения: от жителя до ответа оператора

  1. Старт. Житель открывает бота. Бот показывает приветствие и главное меню — семь кнопок прямо под сообщением (об их числе ниже отдельно).
  2. Воронка. Житель нажимает «📝 Написать обращение». Запускается пошаговая анкета (конечный автомат, FSM — finite state machine). Состояние шага хранится в поле users.dialog_state, накопленные данные — в users.dialog_data (тип JSONB — бинарный JSON в PostgreSQL). Шаги по порядку: согласие на обработку персональных данных → контакт (телефон) → имя → населённый пункт (или геолокация) → адрес → тематика → суть.
  3. Финализация. На шаге сути житель отправляет одно непустое сообщение (текст плюс необязательные фото, видео, файл). Бот создаёт запись appeals со статусом new и структурированными полями (населённый пункт, адрес, тема, суть, вложения), публикует карточку в служебную группу, сохраняет идентификатор сообщения карточки и отвечает жителю «Обращение #N принято».
  4. Ответ оператора. Оператор отвечает на карточку бота свайп-реплаем (жест ответа на сообщение) или командой /reply N текст. Бот по идентификатору карточки находит обращение, проверяет длину ответа (не больше 300 символов), оборачивает текст в официальное письмо от имени Администрации и отправляет жителю в личку.
  5. Запись. После успешной доставки бот публикует в группе подтверждение, ставит appeals.status=answered и answered_at=now(), добавляет запись в messages с направлением from_operator и пишет действие в журнал audit_log.

Путь события: от MAX до записи в базе

  1. Приём. MAX отдаёт обновление (Update) через опрос. Серверный таймаут опроса вшит подменой метода bot.get_updates в main.py::_install_polling_timeout — об этом ниже в «обоснованиях».
  2. Защита от повторов. Первый промежуточный слой (middleware — код, выполняемый до обработчика) — IdempotencyMiddleware — строит ключ из типа события и его полей (update_type, callback_id, mid, seq, timestamp, chat, user) и делает INSERT ... ON CONFLICT DO NOTHING в таблицу events. Дубликат тихо отбрасывается на уровне уникального ограничения events.idempotency_key. Без этого слоя переход на webhook давал бы двойные карточки в группе.
  3. Свежесть админ-чата. Второй слой на каждое новое входящее сообщение в служебной группе сдвигает «трекер свежести» (указатель на последнюю карточку-меню). Это часть правила «редактировать или слать новое» для меню (см. ниже).
  4. Маршрутизация. Диспетчер выбирает обработчик по типу события и по тому, откуда пришло — личка жителя или служебная группа (функция utils/event.is_admin_chat). Сообщения жителя без команды попадают в общий маршрутизатор анкеты appeal.on_message, который по справочнику _STATE_HANDLERS вызывает функцию текущего шага.

Почему важен порядок регистрации обработчиков

Диспетчер MAX берёт первый подошедший обработчик одного типа события. Поэтому команды и callback-кнопки регистрируются раньше, а общий «ловит всё» маршрутизатор анкеты — последним. Иначе он молча проглотил бы /stats, /reopen и прочие команды. Это закреплено в handlers/__init__.py::register_handlers. Любая правка этого файла требует ручного прогона из админ-группы.

Защита от гонок при финализации обращения

Два сообщения от одного жителя за миллисекунды (например, свайп с фото и текст следом) могут параллельно изменить dialog_data и привести к двойной финализации. Функция persist_and_dispatch_appeal обёрнута в блокировку на пользователя (asyncio.Lock в словаре _user_locks: dict[int, asyncio.Lock]). Внутри блокировки делается проверка на повтор по полю user.dialog_state == IDLE и повторная проверка лимита трёх новых обращений за час. Второй параллельный вызов после успешной финализации первого видит состояние IDLE и выходит без создания дубля. Хелпер _drop_user_lock удаляет блокировку после завершения, чтобы словарь не рос неограниченно. Важно: это рассчитано на один экземпляр сервиса; горизонтальное масштабирование (более одного контейнера бота) потребует pg_advisory_xact_lock или блокировку на Redis.

Фоновые задачи: spawn_background_task

Иногда боту нужно запустить работу «в фоне», не блокируя обработку текущего события: разослать тысяче жителей объявление, восстановить зависшие анкеты на старте. Наивный asyncio.create_task(...) здесь — ловушка: в Python 3.11+ event loop хранит лишь слабую ссылку на задачу, и сборщик мусора (garbage collector) может прервать её посреди работы, если на task больше никто не ссылается. Для рассылки это значит «оборвалась на середине без следа».

Поэтому фоновые задачи запускаются только через хелпер utils/background.spawn_background_task(coro, name=...). Он кладёт task в модульное множество _BACKGROUND_TASKS (сильная ссылка, GC не тронет), а в done_callback вычищает её оттуда, чтобы множество не росло. Где применяется: фоновый прогон рассылки (_run_broadcast) и восстановление зависших анкет на старте (recover_stuck_funnels) — ровно те места, где обрыв на середине был бы заметнее всего.

Не теряйте исключения в фоне

Хелпер защищает task от GC, но не логирует исключения за вас: необработанная ошибка внутри фоновой корутины всплывёт в asyncio только при сборке task и легко проходит незаметно. Внутри фоновой корутины оборачивайте тело в try/except с логированием (как сделано в рассылке и восстановлении), иначе сбой «растворится». Импортировать хелпер — из aemr_bot.utils.background, а не из aemr_bot.main: utils/ — нижний слой без обратных зависимостей, импорт из main создавал бы цикл (раньше так и было, чинили).

  • events.idempotency_key уникален — это основа защиты от повторов. Дубликат тихо отбрасывается на уровне ограничения уникальности.
  • appeals.admin_message_id — идентификатор первой карточки обращения, не редактируется после создания (используется как точка привязки для пересылки вложений). appeals.last_admin_card_mid — идентификатор последней карточки, обновляется при каждом её перерисовывании; по нему определяется «устаревшая» карточка при нажатии оператора.
  • users.phone_normalized (только цифры) синхронно поддерживается через services/users.set_phone. Прямой UPDATE users SET phone=... в обход сервиса ломает инвариант и поиск по телефону (/erase phone=, поиск жителя).
  • Подписчики рассылки определяются условием subscribed_broadcast=true AND is_blocked=false (плюс на SQL-уровне фильтра — наличие согласия на рассылку и имя не «Удалено»).
  • dialog_state хранится как String(32), значения берутся из перечисления DialogState в коде. Перевод в тип Enum базы — одно из возможных направлений; для MVP оставили строкой ради скорости миграций при добавлении новых состояний.
  • appeals.attachments и messages.attachments — массивы JSONB с сериализованными вложениями MAX. При пересылке в служебную группу они восстанавливаются в pydantic-объекты Attachments через TypeAdapter.
  • По полям JSONB (users.dialog_data, appeals.attachments, audit_log.details) поиска нет — только чтение по PK/FK, поэтому GIN-индексы не нужны. Если добавляете запрос вида WHERE attachments @> '...', сперва заведите индекс отдельной миграцией.

Карта таблиц и связей (ER-диаграмма)

Ниже схема «сущность-связь» (ER, entity-relationship) всех одиннадцати таблиц с реальными внешними ключами (foreign key, FK) ровно так, как они объявлены в db/models.py. Линия со значком «много» (◇ на конце «o{») читается «один ко многим»: у одного жителя может быть много обращений. Настоящих FK в схеме всего пять, все показаны сплошными линиями. Четыре таблицы стоят особняком без единого FK: events, audit_log, settings и wizard_state. Две из них (audit_log и wizard_state) ссылаются на оператора мягко, через колонку operator_max_user_id (идентификатор оператора в MAX), а не через жёсткую связь с operators.id. Сделано нарочно: журнал действий и персистентность мастеров должны переживать удаление строки оператора, поэтому привязка там логическая, а не на уровне базы. Каскад CASCADE на appeals.user_id и на доставках означает: физическое удаление жителя унесёт за собой его обращения и записи доставок — но в норме житель не удаляется физически, а обезличивается (обращения переподвешиваются на технического anonymous-пользователя max_user_id = -1, см. раздел про удаление ПДн).

Работа с миграциями

Миграции — это версионные изменения схемы базы. Управляет ими Alembic.

# Сгенерировать новую миграцию из изменений в models.py
docker compose exec bot alembic revision --autogenerate -m "что изменилось"

# Накатить до последней
docker compose exec bot alembic upgrade head

# Откатить на одну
docker compose exec bot alembic downgrade -1

# Текущая версия в БД
docker compose exec bot alembic current

# История ревизий и сверка моделей со схемой
docker compose exec bot alembic history --verbose
docker compose exec bot alembic check

После авто-генерации всегда читайте файл миграции глазами

Авто-генератор иногда ошибается с типами JSONB, partial-индексами и enum-полями. Перед коммитом прогоните проверочный откат на тестовом стенде: «миграцию, которую не пробовали откатывать, может однажды не откатиться». Конфигурация Alembic асинхронная (db/alembic/env.py использует async_engine_from_config и берёт target_metadata = Base.metadata из моделей). URL базы подставляется из настроек — отдельной правки alembic.ini не требуется.

Подключение к живой базе и частые операции

# Открыть psql внутри контейнера db (разовые запросы)
docker compose exec db psql -U aemr -d aemr

# Один SQL без интерактива
docker compose exec db psql -U aemr -d aemr -c "SELECT count(*) FROM appeals;"

# Список таблиц, описание таблицы, индексы
docker compose exec db psql -U aemr -d aemr -c "\dt"
docker compose exec db psql -U aemr -d aemr -c "\d+ appeals"
docker compose exec db psql -U aemr -d aemr -c "\di"

Точечные правки данных, которые бот сознательно не умеет (например, смена роли оператора заблокирована в UI как защита от самоповышения):

-- Сменить роль оператора (только прямым SQL — защита от самоповышения)
UPDATE operators SET role='it' WHERE max_user_id=<id>;

-- Мягко деактивировать оператора
UPDATE operators SET is_active=false WHERE max_user_id=<id>;

-- Принудительно закрыть зависшее обращение
UPDATE appeals SET status='closed', closed_at=now() WHERE id=<N>;

-- Сбросить воронку конкретного жителя (обычно делает recover_stuck_funnels сам)
UPDATE users SET dialog_state='idle', dialog_data='{}'::jsonb WHERE max_user_id=<id>;

JSONB после ручной правки — проверяйте на валидность

После любой ручной правки users.dialog_data или appeals.attachments убедитесь, что данные валидны: приложение читает их через pydantic-модели и упадёт на битом JSON. Скрипт полной зачистки тестовых данных — scripts/reset_test_data.sql (удаляет обращения, сообщения, события, рассылки, журнал и жителей, оставляет operators, settings и схему). В нём нет защиты «вы уверены» — в production не запускать.

Локальный запуск за 5 минут

Минимальные требования: Windows 10/11, macOS 14+ или Linux с Docker (на Windows — Docker Desktop с включённым WSL2-бэкендом, памяти не меньше 4 ГБ); Git с доступом к приватному репозиторию; Python 3.12 на хосте (нужен только для скриптов и pytest, сам бот живёт в контейнере); аккаунт в MAX для тестирования.

  1. Клонировать и подготовить .env. git clone https://github.com/gaben1488/aemr-bot.git, затем cd aemr-bot/infra && cp .env.example .env. Заполните три строки: BOT_TOKEN (токен с max.ru/business → раздел «Боты»), POSTGRES_PASSWORD и DATABASE_URL (с тем же паролем). Остальное оставьте как есть; ADMIN_GROUP_ID и прочее заполните после первого запуска.
  2. Собрать и запустить. Из каталога infra/: docker compose up --build bot. Первая сборка занимает 3–5 минут (ставит зависимости, тянет python:3.12-slim), дальше пересборки — секунды. Alembic на старте сам прогонит миграции.
  3. Поговорить с ботом. Найдите своего тестового бота в MAX (боевое имя — @aemo_chat_bot, id 274913354), нажмите «Старт». Должны прилететь приветствие и кнопки главного меню. Пройдите воронку: «📝 Написать обращение» → «✅ Согласен» (PDF приложен) → «📲 Поделиться контактом» → имя → населённый пункт → адрес → тема → суть одним сообщением. Бот ответит «Обращение #1 принято».
  4. Подключить админ-группу. Чтобы карточка ушла операторам, создайте в MAX группу, добавьте туда бота и напишите в группе /whoami — бот вернёт chat_id группы и ваш max_user_id. Пропишите их в .env: ADMIN_GROUP_ID, BOOTSTRAP_IT_MAX_USER_ID, BOOTSTRAP_IT_FULL_NAME. Перезапустите: docker compose up -d --force-recreate bot. На старте бот сам впишет вас первым ИТ-оператором, если активного it ещё нет. Теперь после прохождения воронки в группу прилетит карточка; свайп-реплай на неё — ответ пойдёт жителю.

В логах успешного старта вы увидите примерно такие строки: Running upgrade -> 0001, initial schema; healthcheck listening on 0.0.0.0:8080; policy PDF uploaded; token cached; Starting in long polling mode; Бот: @aemo_chat_bot ... id=274913354.

Поставьте pre-commit — он зеркалит CI

Локальные git-хуки pre-commit повторяют lint-задачу из CI: что упало локально, упадёт и в пайплайне. Установка: pip install pre-commit && pre-commit install, разовая проверка всего репозитория — pre-commit run --all-files. Конфигурация — .pre-commit-config.yaml в корне. Что проверяется: ruff (lint+format), mypy, trailing whitespace, большие файлы >2 МБ, маркеры мерж-конфликтов, приватные ключи, случайные .env.

Если бот не поднялся

Частые причины: тайм-аут pull от Docker Hub (повторить); забит диск (docker system prune -a -f); битый WSL2 (в Docker Desktop: Settings → Troubleshoot → Clean / Purge data). Ошибка password authentication failed for user "aemr" возникает после смены пароля: при первом запуске POSTGRES_PASSWORD записывается в данные тома и больше не перечитывается — либо ALTER USER aemr WITH PASSWORD '...' в живой базе, либо пересоздать том docker compose down -v (сотрёт все данные, только на dev). Ошибка ssl negotiation packet означает, что бот стучится в Postgres до готовности контейнера — проверьте docker compose ps, у db должен быть статус healthy.

Запуск без Docker (прямо из .venv)

Контейнер удобен для «как на проде», но при отладке быстрее гонять бота прямо из локального окружения .venv — тогда работают точки останова в IDE и моментальный перезапуск. Боту по сути нужны две вещи: строка подключения к Postgres (DATABASE_URL — обязательное поле настроек, без него процесс не стартует) и токен (BOT_TOKEN). Базу проще всего поднять одним контейнером, а сам бот запустить процессом на хосте:

cd bot
uv sync --extra dev                       # окружение .venv ровно по uv.lock
# Postgres — отдельным контейнером (данные боту всё равно нужны):
cd ../infra && docker compose up -d db
# миграции и запуск процессом на хосте (не в контейнере):
cd ../bot
DATABASE_URL=postgresql+asyncpg://aemr:<пароль>@localhost:5432/aemr uv run alembic upgrade head
DATABASE_URL=postgresql+asyncpg://aemr:<пароль>@localhost:5432/aemr BOT_TOKEN=<токен> uv run python -m aemr_bot.main

Точка входа — модуль aemr_bot.main (тот же, что в контейнере: CMD образа делает alembic upgrade head && python -m aemr_bot.main). На Windows переменные окружения задаются иначе, чем в примере выше для bash: $env:DATABASE_URL="…"; uv run python -m aemr_bot.main в PowerShell. Запускать обязательно через uv run — иначе подхватится системный Python мимо uv.lock (см. предупреждение выше).

Что работает без ADMIN_GROUP_ID

ADMIN_GROUP_ID — поле необязательное (None по умолчанию). Без него бот поднимается в «citizen-only» режиме: вся гражданская сторона работает полностью — приветствие, меню, воронка приёма обращения, согласие, «Мои обращения», подписка. Не работает только всё, что адресовано в служебную группу: карточки новых обращений никуда не уходят (admin_bus.send видит пустой ADMIN_GROUP_ID и возвращает None), hook-трекер исходящих не ставится (это штатно для citizen-only), операторские команды и ответы жителю недоступны. То есть для отладки чистого диалога с жителем группа не нужна; она нужна, как только хотите проверить путь «обращение → карточка → ответ оператора» (тогда заполните ADMIN_GROUP_ID и bootstrap-переменные, как в шаге 4 выше).

Тесты: два режима и реальный порог покрытия

Тестовая база bot/tests/ рассчитана на два режима: лёгкий локальный (без Postgres, проверка чистой логики на mock-ах) и полный CI (с реальной БД, миграциями и порогом покрытия). Канонический способ запуска, совпадающий со средой Docker, — через uv и lock-файл.

Не запускайте тесты системным pip/pytest

Это поставит другие версии пакетов мимо uv.lock. Локально тесты могут пройти, а в Docker всё развалится с TypeError/ImportError. И не пытайтесь добиться «CI-режима» локально без Postgres — упрётесь в skip-нутые тесты и покрытие ниже 84. Это ожидаемая разница между «local-pure» и «CI-with-DB», а не поломка. Известный нюанс: отдельные тесты меню могут падать RuntimeError: Event loop is closed, если предыдущий тест в той же сессии закрыл async event loop — лечится точечным мок-патчем настроек в самом тесте, чтобы не открывать новое соединение к пулу Postgres.

Реальный порог покрытия в боевом CI — --cov-fail-under=84 (источник истины — .github/workflows/ci.yml). Порог уже подняли с 80 после гигиены покрытия; фактическое покрытие держится чуть выше порога. Покрытие считается по веткам (--cov-branch), отчёты выгружаются как артефакты CI (XML и HTML). Точные числа тест-файлов и процент покрытия — снимок на конкретную дату и меняются от PR к PR; единственная стабильная величина, на которую стоит опираться, — это порог 84 из ci.yml. За актуальной цифрой смотрите вывод последнего прогона CI, а не документ.

Покрывать стоит: бизнес-сценарии, граничные случаи и пути с проверками безопасности (роли, валидаторы, лимиты). Базовые сценарии в тестах: согласие → запрос контакта → меню; приём обращения серией сообщений; ответ оператора цитированием; отказ при ответе длиннее 300 символов; статистика за период; удаление ПДн. Новые тесты пишите в той же папке bot/tests/.

Что проверяет CI

Пайплайн (.github/workflows/ci.yml) запускается на push и pull request в ветку main и состоит из трёх параллельных задач (job).

Job «Lint, types, security»

  • Ruff — линтер и проверка стиля (ruff check aemr_bot/).
  • MyPy — статическая проверка типов.
  • Bandit — поиск проблем безопасности уровня medium и выше (-ll).
  • pip-audit — сканирование зависимостей на известные уязвимости (CVE). Жёсткий провал при находке. Локальный editable-пакет исключается из аудита; два конкретных CVE в цепочке fastapi → starlette игнорируются с письменным обоснованием неприменимости прямо в YAML (бот не реконструирует URL из заголовка Host и не использует FileResponse/StaticFiles).
  • Shellcheck — проверка всех shell-скриптов в репозитории.

Две supply-chain меры в самом пайплайне

Помимо того, что проверяет CI, важно как он сам собран. Во-первых, все используемые GitHub Actions запинены по commit-SHA, а не по перемещаемому тегу (например actions/checkout@34e114… # v4.3.1, с комментарием версии рядом). Это защита от компрометации мейнтейнера action или подмены тега: на тот же SHA подставить чужой код нельзя. Поднимает эти пины github-actions-экосистема Dependabot. Во-вторых, шаг pip-audit не «падает на любой найденной CVE вообще»: он берёт pip freeze --exclude-editable (сам aemr-bot из аудита исключён), а две конкретные CVE в цепочке fastapi → starlette внесены в обоснованный ignore-list — с письменным объяснением неприменимости прямо в YAML (бот не реконструирует URL из заголовка Host и не использует FileResponse/StaticFiles) и grep-проверками. Этот ignore-list стоит перечитывать при следующем подъёме fastapi/starlette — вдруг обоснование перестало быть верным.

Job «Pytest with Postgres 16»

Поднимает сервис-контейнер PostgreSQL 16, прогоняет тесты с покрытием и порогом 84%, затем тремя отдельными шагами проверяет миграции:

  • upgrade head на чистую БД — ловит «осиротевшие»/дублирующиеся ревизии, синтаксис и опечатки импортов в миграциях;
  • alembic check — сравнивает модели (Base.metadata) с последней миграцией; падает, если кто-то изменил модель и забыл сгенерировать миграцию;
  • round-trip (upgrade head → downgrade base → upgrade head) — ловит сломанный откат (забытую обратную операцию, неверную ссылку на ревизию).

Job «Docker build smoke test»

Собирает образ из infra/Dockerfile без публикации — проверяет, что Dockerfile не сломан, миграции применяются и /livez отвечает. Заодно это проверка hash-enforced сборки: если uv.lock разошёлся с pyproject.toml или хеш какого-то wheel'а не сходится, шаг pip install --require-hashes уронит сборку прямо здесь, до прода.

Контракт-тесты против дрейфа «код ↔ док ↔ реестр»

Помимо обычных юнит-тестов в наборе есть несколько контракт-тестов (anti-drift): их задача — не проверить логику, а поймать рассинхрон между двумя источниками, которые обязаны совпадать. Они дёшевы и ловят целый класс ошибок «поправил тут, забыл там» до прода. Знать их полезно, потому что многие рецепты выше заканчиваются словами «иначе контракт-тест покраснеет».

Боевой образ уважает lock, а вот lint/test-job CI ставит из диапазонов

Тонкость, о которой стоит знать, чтобы не путать два разных окружения. Боевой образ (infra/Dockerfile) собирается строго из uv.lock и сверяет SHA256 каждого пакета (uv export --frozen → pip install --require-hashes, см. раздел «Как образ ставит зависимости» выше) — то, что приедет в прод, хеш-верифицировано и совпадает с lock. А вот lint/test-задачи CI ставят зависимости проще — pip install -e ".[dev]" из диапазонов pyproject.toml (им нужна dev-группа, которой в боевом образе нет). Поэтому guard-тест test_deps_environment в CI ловит drift не сам по себе, а потому что сверяет установленную версию maxapi с закреплённой в lock; supply-chain же в CI прикрыт отдельным шагом pip-audit по pip freeze. Практический вывод прежний: локальные тесты запускайте через uv run — это среда, которая ставит ровно по lock, как боевой образ.

Путь команды /reply — от оператора до жителя по слоям

Рецепты выше станут нагляднее на одном сквозном пути по слоям: оператор пишет ответ, житель получает официальное письмо. Это самый чувствительный к безопасности маршрут в проекте (ответ от имени Администрации), и на нём хорошо видно правило «обработчик → сервис → база → обратно». Ответ можно дать двумя способами, и они сходятся в одной точке: свайп-реплай (жест ответа на карточку) и команда /reply N текст. Разница только в том, как находится номер обращения. При свайпе идентификатор исходной карточки лежит в link.message.mid (на уровень глубже, чем link.mid — это разобранная в граблях maxapi ловушка), и обработчик operator_reply.py зовёт сервис appeals.get_by_admin_message_id; при команде номер берётся прямо из аргумента и идёт в appeals.get_by_id. Дальше путь общий: проверка длины (не больше 300 символов, ANSWER_MAX_CHARS), проверка is_blocked (житель, отозвавший согласие, не получит сообщение — оператору вернётся подсказка «звоните»), упаковка текста в официальное письмо (card_format), отправка через bot.send_message и запись результата в базу одной транзакцией: строка в messages с направлением from_operator, appeals.status=answered и answered_at=now(), плюс действие action="reply" в audit_log. На схеме горизонтальные дорожки — это слои: событие приходит в обработчик, тот ходит в сервисы, сервисы — в базу, и в конце житель получает письмо.

Где истина по cron-задачам

Реестр cron_registry.py — машиночитаемый источник имён задач и их назначения для anti-drift-теста test_cron_docs_sync. Но точное расписание (минуты, дни недели) надо брать из реальных объектов CronTrigger в cron.py — именно там фактический триггер. Сами расписания подробно расписаны в разделе «Администратору»; здесь достаточно знать имена ключевых задач: pulse-hourly (heartbeat каждый час 24/7), pulse-workhours-extra (доп. пинг в рабочее время), startup-pulse, health-selfcheck, funnel-watchdog (сброс зависших анкет старше 24 ч), events-retention (30 дней), audit-log-retention (365 дней), appeals-5y-retention, pdn-retention, threat-intel-refresh, broadcast-draft-reaper, db-backup и две reminder-задачи по открытым и просроченным обращениям. Общие параметры всех задач: max_instances=1, coalesce=True, misfire_grace_time=120 секунд.

Рецепт: добавить inline-кнопку с действием (callback)

Самый частый рецепт после «поменять текст». Кнопка в MAX, у которой есть полезное действие (не просто открыть подменю), несёт строку-payload — например op:reply:42 значит «ответить по обращению №42». Чтобы кнопка не сломалась, эта строка должна собираться и разбираться в одном понятном месте, а не россыпью f-строк по файлам. В проекте за это отвечают два модуля-близнеца: handlers/callback_payloads.py (где строка собирается) и handlers/callback_router.py (где объявлен маршрут). Их совпадение пинит контракт-тест — если разойдутся, CI покраснеет до прода. Порядок действий на примере «кнопка оператора по обращению»:

  1. Объявить payload. В handlers/callback_payloads.py добавить либо EXACT-константу для кнопки без параметра (OP_FOO = "op:foo"), либо билдер-функцию для кнопки с числовым хвостом (def op_foo(appeal_id: int) -> str: return PREFIX_OP_FOO + str(appeal_id), рядом — голый префикс PREFIX_OP_FOO = "op:foo:"). Билдер принимает типизированный аргумент — это и есть защита от «подставили объект вместо id».
  2. Зарегистрировать маршрут. В handlers/callback_router.py добавить CallbackRoute в EXACT_ROUTES (точное совпадение) или PREFIX_ROUTES (по префиксу). Обязательны четыре поля: паттерн (тот же литерал, что в шаге 1), группа (CallbackGroup.OPERATOR_ADMIN для операторских), admin_allowed=True — иначе кнопка не сработает в служебной группе, — и человеко-читаемое описание.
  3. Поставить кнопку в клавиатуру. В нужном модуле ui/*_keyboards.py импортировать from aemr_bot.handlers import callback_payloads as cp и собрать кнопку через билдер: CallbackButton(text="✉️ Сделать", payload=cp.op_foo(appeal_id)). Никаких сырых строк f"op:foo:{id}" — только через cp, тогда контракт-тест видит payload.
  4. Написать обработчик. В диспетчере callback'ов разобрать хвост через готовый хелпер callback_router.parse_int_tail(payload, cp.PREFIX_OP_FOO) — он вернёт int или None для битой/устаревшей кнопки (тогда сделать ack и выйти, не выполняя действие). Не парсить вручную через .split(":").
  5. Прогнать тесты. uv run pytest tests/test_callback_payloads_contract.py tests/test_callback_coverage_contract.py -q. Первый проверит, что строка билдера байт-в-байт ложится на маршрут; второй — что кнопка не «осиротела» (есть и в UI, и в реестре).

Живой пример — кнопка «Ответить»

Кнопка оператора «✉️ Ответить и закрыть» собрана ровно по этому рецепту: билдер op_reply(appeal_id) в callback_payloads.py → маршрут CallbackRoute("op:reply:", CallbackGroup.OPERATOR_ADMIN, True, ...) в callback_router.py → кнопка в ui/operator_keyboards.py через payload=cp.op_reply(appeal_id). Рядом — близнец op_replyint («промежуточно, не закрывая»). Куда дальше уходит нажатие — это уже путь /reply, разобранный диаграммой выше.

Рецепт: добавить новый статус обращения

Статус обращения — это перечисление AppealStatus в db/models.py. Сейчас в нём четыре значения: new, in_progress, answered, closed. Поле хранится строкой (String(...)), не базовым enum-типом, — поэтому новое значение не требует миграции базы, достаточно тронуть код. Но статус читается в нескольких местах, и если забыть хоть одно — обращение «провалится» в нераспознанное состояние. Места по порядку:

  1. Само перечисление. Новое значение в AppealStatus (db/models.py), например DEFERRED = "deferred".
  2. Переход в это состояние. Логика, которая ставит статус, — воронка и операции с обращениями (handlers/appeal_runtime.py для приёма, handlers/operator_reply.py и admin_* для смены оператором). Здесь же решить, считается ли статус «открытым» (карточка operator_keyboards.py делит статусы на open_states/closed_states — новое значение надо явно отнести к одной из групп, иначе у обращения пропадут кнопки действий).
  3. Подпись для жителя. Таблица эмодзи+подпись STATUS_LABELS в copy/citizen_funnel.py (её читают карточка жителя и лента в services/card_format.py). Без новой строки статус покажется жителю «как есть», латиницей.
  4. Подпись для статистики. Отдельный маппинг _status_label в services/stats.py (для XLSX-выгрузки, без эмодзи) — это другая таблица, не та, что видит житель.
  5. Выборки и сортировка. Запросы в services/appeals.py, которые фильтруют по статусу (листинг открытых, поиск просроченных, сортировка status_priority). Решить, куда новый статус попадает в порядке.

Две таблицы подписей — не перепутайте

Подпись статуса живёт в двух местах сознательно: STATUS_LABELS (copy/citizen_funnel.py) — с эмодзи, для жителя; _status_label (services/stats.py) — без эмодзи, для Excel-отчёта. Это разные аудитории и разный формат, поэтому новое значение надо добавить в обе.

Рецепт: добавить роль оператора

Роли оператора — перечисление OperatorRole в db/models.py (сейчас четыре: coordinator, aemr, egp, it). Роль решает, что оператор видит и может. Полный разбор того, какая роль что умеет, — в разделе «Оператору»; здесь только инженерные точки правки:

  1. Перечисление. Новое значение в OperatorRole (db/models.py). Поле operators.role — строка, миграция базы не нужна.
  2. Флаги доступа. В handlers/admin_panel.py задаются производные флаги по роли (is_it = role == IT, can_broadcast = role in {IT, COORDINATOR}). Решить, какие способности даёт новая роль, и включить её в нужные множества.
  3. Гард команд. Проверка роли — handlers/_auth.py: ensure_operator пропускает любую активную роль, ensure_role(event, IT) требует точного вхождения в множество (координатор через IT-гард не проходит). Если новой роли нужен доступ к командам под ensure_role(...) — добавить её в соответствующий вызов.
  4. Кнопки меню. ui/operator_keyboards.py (op_help_keyboard) собирает набор кнопок по флагам из шага 2 — новые способности появятся в меню автоматически, если флаги расставлены верно. Picker смены роли (мастер операторов) тоже перечисляет роли — проверить, что новая там видна.

Сменить роль можно только прямым SQL

Повышение роли через интерфейс намеренно ограничено (защита от самоповышения) — отсюда команда «сменить роль оператора» делается прямым UPDATE operators SET role=..., см. блок «Подключение к живой базе» выше. Это не баг рецепта, а защитное решение.

Рецепт: добавить поле в карточку обращения

Карточка — это то, что оператор видит в служебной группе (admin_card) и что житель видит в «Моих обращениях» (user_card); обе живут в services/card_format.py. Добавление нового видимого поля — это два независимых шага: сначала завести его в схеме, потом отрисовать.

  1. Поле в модели + миграция. Колонка в нужной модели (Appeal в db/models.py), затем миграция Alembic (см. «Работа с миграциями» выше — авто-сгенерированный файл обязательно прочитать глазами). Если поле заполняется в воронке — добавить шаг (рецепт «новый шаг анкеты») или запись при финализации.
  2. Отрисовка. Вывести поле в services/card_format.py: admin_card — карточка оператора, user_card — карточка жителя. Решить, нужно ли поле обеим аудиториям (например, внутренняя пометка оператора жителю не показывается). Тексты-обёртки берутся из copy/ через презентационный слой, не зашиваются в сервис.

Скелет «модель → миграция → отрисовка» — тот же, что у любого расширения схемы: данные внизу (db/), презентация наверху (card_format + copy/), и нельзя тянуть тексты прямо в сервис в обход слоя copy/.

Рецепт: отправить что-то в служебную группу

Есть тонкость, на которой легко сделать «съезжающее меню». Любое новое сообщение боту в служебную группу надо слать через единую шину services/admin_bus.send(bot, text=...), а не напрямую bot.send_message(chat_id=ADMIN_GROUP_ID, ...). Причина — «трекер свежести меню» (utils/menu_tracker.py): он помнит, какое сообщение в чате сейчас «редактируемое меню», и по нему решает «перерисовать на месте или прислать новое». Если слать в группу мимо шины, трекер отстаёт от реального состояния чата, и следующее меню перерисуется поверх не того сообщения.

  • Новое сообщение (уведомление, алёрт, подтверждение, pulse) → admin_bus.send(bot, text=..., critical=...). Шина атомарно шлёт и двигает трекер. Возвращает mid отправленного сообщения либо None (если ADMIN_GROUP_ID не задан, отправка упала или тихий режим подавил).
  • Редактируемое меню (карточка обращения, экран мастера) → не шина, а freshness-aware пути admin_card.render / send_or_edit_screen: они сами знают, что шлют именно перерисовываемый экран.
  • Прямые bot.send_message в группу, которые остались в коде, подстрахованы общим monkey-patch hook'ом (admin_bus.install_outgoing_tracker_hook): он двигает трекер после любой отправки в группу. Но осознанно полагаться на hook не стоит — пишите через admin_bus.send.

Два трекера: editable_mid против physical_mid

Под капотом трекер хранит две отметки. physical_mid — последнее физическое сообщение в чате (двигается на каждое исходящее и входящее, в том числе через hook и admin_bus.send). editable_mid — последнее редактируемое меню (двигается только когда бот осознанно шлёт перерисовываемый экран). Историческое уведомление (pulse, алёрт, подтверждение через admin_bus.send) двигает только physical_mid и остаётся в чате навсегда — тап оператора по нему не превратит его в меню, потому что callback_mid ≠ editable_mid. Это разделение починило жалобу «меню в админ-чате редактируется при тапе на не-последнем сообщении».

Рецепт: добавить промежуточный слой (middleware)

Промежуточный слой (middleware) выполняется на каждое событие до обработчиков. Сейчас их два, оба внешние (outer): IdempotencyMiddleware (защита от повторов) и AdminChatActivityMiddleware (сдвиг трекера на входящее сообщение оператора). Оба объявлены и подключены в handlers/__init__.py.

  • Подключение — только через register_outer_middleware. В проекте обёртка _attach_outer_middleware требует именно публичный API maxapi 1.1+ и падает ясной ошибкой, если метода нет (никакого тихого фолбэка на устаревшие формы). Класс наследует BaseMiddleware из maxapi.filters.middleware и реализует async def __call__(self, handler, event_object, data).
  • Порядок важен. IdempotencyMiddleware подключается первым (отбрасывает дубли до всего остального — иначе MAX-повтор того же события сдвинул бы трекер дважды), активность-слой — вторым (двигает трекер уже на реально новое сообщение). Новый слой ставьте в эту цепочку осознанно.
  • Ловушка isinstance(event_object, MessageCreated). Если слой реагирует на входящие сообщения — обязательно сначала проверить тип. Без проверки слой поймает все события с event.message, в том числе нажатие кнопки (MessageCallback), у которого event.message — это старая карточка, на которой нажали, а не новое сообщение. Трекер съедет на старую карточку, и следующий экран перерисуется поверх неё. Это уже ловили — комментарий в AdminChatActivityMiddleware хранит разбор инцидента.

Архитектурные обоснования (почему так)

Самое ценное — не «как», а «почему». Эти решения зафиксированы в ADR и в коде, чтобы их не переоткрывали и не ломали по незнанию.

Почему JSONB, а не Redis для состояния анкеты

Состояние пошаговой анкеты (dialog_state + dialog_data) хранится прямо в PostgreSQL в колонке JSONB, а не в Redis. Население округа около 70 тысяч человек, проникновение цифровых госсервисов 3–5%. Активных воронок одновременно — десятки в часы пик, единицы ночью; пиковая нагрузка 1–2 операции записи в секунду. Postgres проглатывает это не замечая, а лишний сервис (Redis) — это лишняя точка отказа и эксплуатации. Триггер пересмотра — устойчиво более 500 одновременных воронок по метрикам /diag.

Почему форк сообщества maxapi, а не официальный

Из трёх вариантов клиента MAX для Python выбран love-apples/maxapi (сообщество, активно обновляется, последний коммит — апрель 2026). Официальный max-messenger/max-botapi-python — это снимок исходного репозитория девятимесячной давности (заморожен на ревизии 2025-07-30): переход означал бы откат с потерей нужных нам исправлений (включая process_update_webhook, TypeAdapter[Attachments], MessageLinkType). Вариант через шлюз green-api требует платную подписку, что противоречит принципу «свой код, не подписочный сервис». Решение делать собственный форк (в gaben1488/maxapi под контролем муниципалитета) принимается только при трёх и более месяцах без коммитов в исходник с открытыми критическими задачами, либо при незакрытой уязвимости в течение 14 дней. Ежеквартальная проверка активности — задача координатора.

Почему обязательный слой защиты от повторов

Дубликаты обновлений от MAX редки в опросном режиме, но частые при повторных доставках в серверном. IdempotencyMiddleware подключён как внешний слой и срабатывает на каждое обновление до любого обработчика. Без него переход на webhook сразу дал бы двойные карточки в админ-группе. Поэтому это не «страховка на будущее», а обязательный компонент. Юридический смысл тоже есть: одно действие жителя не должно превратиться в два обращения или две записи в журнале.

Почему лимит ответа оператора — 300 символов

Жёсткий лимит в 300 символов в коде (ANSWER_MAX_CHARS). При попытке ответить длиннее — отказ с сообщением в группу, без модального окна и без авто-разбивки: оператор сам решает, что делать. Цифра из требований куратора (200–300); взята верхняя граница, чтобы не ломать стандартные шаблоны ответов на 250–280 символов. Это упрощает интерфейс карточки.

Почему подписка на рассылку — отдельное согласие, а скорость — половина лимита

Рассылка спроектирована (дизайн-спека одобрена 2026-05-04) вокруг нескольких намеренных ограничений. Знать их нужно прежде, чем трогать broadcast*.py. Подписка на новости — это явный opt-in (users.subscribed_broadcast, по умолчанию false), и это отдельная цель обработки данных: согласие на ПДн внутри воронки обращения — про конкретное обращение, а согласие на рассылку — отдельный флаг, который житель ставит и снимает сам (/subscribe//unsubscribe, переключатель в меню, кнопка «отписаться» под каждым объявлением). Список получателей — это снимок условия subscribed_broadcast=true AND is_blocked=false (плюс наличие согласия на рассылку и имя не «Удалено»), снятый один раз в начале и зафиксированный, чтобы рассылка не «плыла» по мере отписок. Скорость отправки намеренно ограничена одним сообщением в секунду (BROADCAST_RATE_LIMIT_PER_SEC=1.0) — это ровно половина платформенного лимита MAX в 2 запроса в секунду, вторая половина оставлена обычной активности бота, чтобы рассылка на тысячу жителей не блокировала приём обращений. Кнопка «⛔ Экстренно остановить» под прогресс-сообщением доступна любому оператору в группе (а не только автору) — на случай компрометации учётки координатора; фоновая задача сверяет флаг status между каждой отправкой и обрывается на cancelled. Сам текст рассылки в stdout не пишется — только идентификаторы и счётчики; в audit_log уходит action="broadcast_send" с числом подписчиков, но опять же без утечки текста в логи. Незавершённый мастер рассылки живёт в памяти с TTL 300 секунд и теряется на рестарте — это приемлемо (оператор просто запускает /broadcast заново), а застрявшие черновики со статусом draft старше 30 минут добивает cron-задача broadcast-draft-reaper, переводя их в failed.

Почему один экземпляр (и что сломает масштабирование)

Финализация обращения защищена блокировкой на пользователя в памяти процесса (asyncio.Lock в словаре _user_locks) — это закрывает гонки двойной отправки и параллельных событий по одному жителю. Состояние мастеров рассылки тоже частично живёт в памяти (словари с TTL 300 секунд). Всё это рассчитано на один экземпляр бота. Горизонтальное масштабирование (несколько контейнеров) потребует перейти на блокировки уровня базы (pg_advisory_xact_lock) или Redis.

Почему таймаут опроса — это не интервал

POLLING_TIMEOUT_SECONDS (по умолчанию 20, потолок MAX — 90, диапазон 0..90) задаёт серверное время ожидания вызова getUpdates: сколько MAX держит соединение, пока нет событий. Это не пауза между запросами клиента. Опрос возвращает ответ либо при появлении события, либо по истечении этого времени. Чем больше значение — тем меньше пустых запросов при простое (важно при ограничении 2 запроса в секунду на бота, действующем с 11 мая 2026 года), но медленнее реакция на остановку. На нуле бот будет хлестать API на пределе и кончит бюджет за минуты. Реализовано подменой связанного метода bot.get_updates в main.py::_install_polling_timeout, потому что Dispatcher.start_polling не пробрасывает таймаут внутрь. Если когда-нибудь понадобится свой форк maxapi — это первый кандидат на запрос изменений.

Почему SLA в коде — 4 часа, а в Регламенте — иначе

В коде заложен единый базовый срок ответа: SLA_RESPONSE_HOURS = 4 рабочих часа (config.py). У модели обращения нет поля важности или приоритета и нет тематической маршрутизации — бот меряет попадание в единый 4-часовой SLA и шлёт напоминания о просроченных. Дифференциация сроков по важности остаётся организационной (её ведёт Куратор ЦУР вручную); полный правовой разбор этого разделения — в разделе для оператора. Инженерное следствие: чтобы ввести разные сроки по типу обращения, сначала нужно добавить в схему поле важности/маршрута.

Почему 365 дней на журнал действий, а не 5 лет

Срок хранения audit_log — 365 дней (AUDIT_LOG_RETENTION_DAYS, диапазон 30–3650), это сознательный выбор: журнал действий — это журнал событий безопасности (приказ ФСТЭК № 21, мера РСБ.1), а не архив обращений, поэтому отдельный 5-летний ретеншн обращений (appeals-5y-retention) к нему не привязан, и менять код не нужно. Полный правовой разбор норм — подробно в Правовом.

Отладка и грабли maxapi

Библиотека maxapi удобна, но местами «протекает»: модели не покрывают все поля сервера, имена полей расходятся с документацией, отдельные методы возвращают объекты, которых нет в типизации. Ниже грабли, на которые уже наступали — собраны здесь, чтобы больше никто не наступал.

  1. Свайп-реплай: Message.link.message.mid, а не Message.link.mid. Когда оператор отвечает свайпом, приходит Message.link типа LinkedMessage. Сам link.mid пустой — идентификатор исходного сообщения лежит на уровень глубже, в link.message.mid (внутри вложенного MessageBody). Извлечение — operator_reply.py::_extract_reply_target_mid и _mid_from_link. На отдельных версиях клиента pydantic-обёртка превращается в dict — отсюда двойная попытка чтения.
  2. Результат отправки — SendedMessage, а не Message. У результата bot.send_message(...) нет .message_id напрямую; идентификатор — в result.message.body.mid. Адаптер utils/event.extract_message_id снимает оба варианта через цепочку getattr и работает на None-входе. Используется при сохранении appeals.admin_message_id и messages.max_message_id.
  3. Загрузка файлов — AttachmentUpload, а не плоский dict. Старая ревизия maxapi отдавала dict, текущая — pydantic-модель. Для отправки в attachments=[...] сериализуйте через .model_dump(by_alias=True) или конкретные конструкторы (PhotoAttachment, FileAttachment). См. services/uploads.py и services/policy.py::build_file_attachment.
  4. Порядок регистрации хендлеров важен. Диспетчер берёт первый совпавший. Если зарегистрировать message_created без фильтра раньше, чем message_created(Command(...)), команда не сработает. Все универсальные обработчики анкеты регистрируются последними, команды и callback — раньше.
  5. Защита потока жителя через is_admin_chat. Обработчики для жителя (/start, главное меню, «Написать обращение») оборачиваются проверкой if is_admin_chat(event): return. Иначе оператор, написавший /start в служебной группе, попал бы в анкету как «житель», завёл бы запись в users и застрял в AWAITING_CONSENT.
  6. Никогда не лезьте к полям события напрямую. Не пишите event.chat_id или event.user.user_id — структуры событий неоднородны. Всегда через адаптер utils/event.py (get_chat_id, get_user_id).
  7. Отладочный дамп — осторожно с ПДн. При проблемах с парсингом включайте LOG_LEVEL=DEBUG — диспетчер пишет полный pydantic-дамп события. Но не дампите event.message в боевые логи: туда утекут персональные данные.
  8. Восстановление застрявших анкет. Если бот рестартанул посреди анкеты, dialog_state жителя застывает. На старте recover_stuck_funnels (через asyncio.create_task, не блокируя диспетчер) находит застрявших в AWAITING_SUMMARY старше таймаута и завершает их обращения пачками (RECOVER_BATCH_SIZE=1000). Пустые сессии (нет ни текста, ни вложений) сбрасываются в IDLE. Лимит партии защищает от патологии: 10 тысяч застрявших воронок после долгого простоя не дадут 10 тысяч вызовов API на старте.

Эксплуатационные компоненты

Проверка живости. health.py поднимает HTTP-сервер на порту WEBHOOK_PORT (по умолчанию 8080), публикуется только на 127.0.0.1 (наружу ничего не выставляется — модель самостоятельного хостинга без входящих портов). Эндпоинтов три: /livez — liveness, отдаёт 200, пока жив asyncio-цикл бота (на него смотрят Docker healthcheck, watchdog и авто-деплой при решении об откате); /readyz — readiness, отдаёт 200, если heartbeat свежий и база отвечает на SELECT 1 (с TTL-кэшем 10 секунд; для диагностики, не для авто-рестарта); /healthz — алиас /readyz, оставлен для обратной совместимости. Локальным запросам (с 127.0.0.1) отдаётся полная диагностика, внешним — только {"ok": ...}.

У /livez два независимых сигнала живости (defense-in-depth), и эндпоинт краснеет, если протух любой. Первый — Heartbeat: фоновая задача периодически бьёт таймстемп. Но heartbeat-таск только спит и бьёт — при use_create_task=True зависший await в обработчике или бэкапе мог бы заморозить обработку, а пульс остался бы ложно-зелёным. Поэтому добавлен второй сигнал — PollWatch (last_poll): polling-обёртка отмечает его на каждом успешном get_updates. Пока polling-цикл реально крутит long-poll к MAX, таймстемп свежий; если цикл встал (мёртвая aiohttp-сессия, голодание event-loop) — протухает, и /livez отдаёт 503 → внешний watchdog/Docker healthcheck перезапустит. Порог протухания — LIVEZ_POLL_STALE_FACTOR × POLLING_TIMEOUT_SECONDS (по умолчанию 3×: запас на один потерянный long-poll плюс сетевой джиттер, чтобы не поднимать ложную тревогу на здоровом простаивающем боте). В webhook-режиме long-poll'а нет (last_poll == 0.0), и этот сигнал намеренно игнорируется (fail-open) — иначе мы бы ложно краснили webhook-процесс. Ни один из сигналов базу не трогает: DB-ping живёт в /readyz.

В старых документах фигурирует только /healthz

Это устарело. В коде эндпоинты разделены на /livez (жив ли процесс) и /readyz (жив и готов, с проверкой базы), а /healthz остался лишь как алиас readiness ради совместимости. Автоматический рестарт/откат должен смотреть на /livez, а не на /healthz.

Геолокация: вынос в поток и прогрев на старте. Определение адреса по координатам (services/geo.find_address) — это CPU-bound работа на shapely поверх ленивых GeoJSON-индексов (поселения, здания, улицы, около 2.6 МБ). В одно-loop процессе синхронный вызов на горячем пути заморозил бы всех жителей и операторов на время счёта, а первая геолокация после старта ещё и платила бы за ленивую загрузку индексов. Поэтому из обработчика геошага (handlers/appeal_geo.py) find_address вызывается через asyncio.to_thread — счёт уходит в пул потоков, event-loop остаётся свободным. А на старте фоновая задача _warm_geo_indexes (тоже через asyncio.to_thread, fail-open: ошибка прогрева не валит запуск) заранее строит индексы и прогоняет один реальный find_address по центру округа (Елизово), чтобы первый живой запрос жителя не ждал прогрева. Если прогрев не успел или упал — индексы построятся лениво при первом обращении, просто чуть медленнее.

Нереализованный потенциал maxapi (на будущее)

Это не онбординг, а каталог возможностей библиотеки на будущее — свёрнут, чтобы не отвлекать от текущего кода. Разверните, когда будете планировать развитие. Разбор установленной библиотеки maxapi 1.1.0 и её примеров вскрыл набор готовых возможностей, которые упростили бы код или дали новые фичи. Часть уже внедрена, часть осознанно отложена.

Уже внедрено по итогам разведки

  • Кнопка запроса геолокации. RequestGeoLocationButton уже стоит на шаге выбора населённого пункта (ui/citizen_keyboards.py, в localities_keyboard) — житель не пишет адрес текстом и не шлёт геопозицию вложением, а нажимает «📍 Поделиться геолокацией», и бот определяет поселение/адрес через services/geo.py, затем просит подтверждения (AWAITING_GEO_CONFIRM).
  • Кэш токена вложения. PDF-шаблон политики (он не меняется) заливается один раз, токен кэшируется в БД и переиспользуется без повторной загрузки — экономия времени на каждом /policy.
  • Индикатор «бот печатает». Долгие операции (снимок аудитории рассылки, листинг «Мои обращения», тяжёлые PDF/XLSX) обёрнуты в авто-периодический typing-индикатор (utils/typing_indicator.py) — пользователь видит, что бот не завис.
  • Обход аудитории через iter_all(). Перечисление участников группы идёт через библиотечный async-итератор с защитой от зацикливания marker-пагинации вместо ручного цикла.
  • Обработка HTTP 429 и BOT_STOPPED. Библиотека сама не парсит Retry-After и игнорирует 429 — поэтому для рассылки на 1000+ жителей добавлена ручная обработка ограничения скорости; событие BOT_STOPPED (житель остановил бота) исключает его из аудитории рассылки.
  • Типизированный слой callback-payload. Идея «не парсить callback-строки руками» уже внедрена — но своим, более лёгким способом, чем pydantic-модель CallbackPayload из примеров библиотеки. Все строки кнопок собираются в одном модуле handlers/callback_payloads.py (EXACT-константы вида OP_MENU = "op:menu" и билдеры-функции вида op_reply(appeal_id: int) → "op:reply:<id>"), маршруты перечислены в реестре handlers/callback_router.py (EXACT_ROUTES + PREFIX_ROUTES), а разбор числового хвоста делает один хелпер callback_router.parse_int_tail (а не россыпь .split(":") по обработчикам). Совпадение «то, что собрала кнопка» ↔ «то, что ждёт роутер» байт-в-байт пинит контракт-тест test_callback_payloads_contract. Рецепт «как добавить кнопку оператора» — ниже в разделе рецептов. От полноценного CallbackPayload сознательно отказались: 30+ pydantic-моделей на каждую кнопку для MLP-бота читаются и проверяются тяжелее, чем простые константы и тонкие билдеры (см. докстринг callback_payloads.py).

Сознательно отложено (декларативный сдвиг парадигмы)

Библиотека предлагает целый декларативный слой, который сейчас в боте написан вручную. Это не дефект, а осознанный выбор «не сейчас» — миграция большая и рискованная, а текущая ручная реализация работает.

  • FSM через StatesGroup + MemoryContext. В maxapi есть встроенное состояние per пара (chat_id, user_id): context.set_state(...), get_data()/update_data(...), фильтр по состоянию вторым аргументом декоратора, LRU-вытеснение и TTL, подменяемое хранилище (вплоть до RedisContext). У нас этот механизм фактически переизобретён руками: wizard_registry.py + wizard_persist.py + миграция 0011 + несколько module-level словарей. Переход на встроенный FSM снял бы сотни строк и убрал бы ручной catch-all диспетчер (каждое сообщение приходило бы сразу в правильный обработчик по состоянию). Это самый весомый кандидат на будущий рефакторинг, но он требует полной регрессии всей анкеты через сквозные тесты.
  • Event-helpers вместо ручного send. Примеры используют event.message.answer(text), event.message.reply(...), event.ack(); мы во многих местах пишем event.bot.send_message(chat_id=..., user_id=...) — 6 строк вместо 1. Идиома в коде смешана (часть файлов уже на helper-ах).
  • Готовый фильтр контакта и html_text/md_text. Фильтр Contact() из maxapi уже распаршивает vCard и инжектит ContactAttachment в обработчик — нашу ручную реализацию парсинга vCard можно было бы убрать. Свойства MessageBody.html_text/md_text отдают форматированный текст с правильным экранированием — сейчас в карточке «суть» оператор не видит жирный/курсив, которым писал житель.
  • Раздельные роутеры. Router со своими middleware и фильтрами, подключаемый через dp.include_routers(...) — естественные кандидаты на вынос крупные модули рассылок и настроек.

Чего нет в платформе MAX в принципе

Эти вещи отсутствуют и в swagger MAX, и в библиотеке — делать их не из чего: inline-режим (как в Telegram), реакции, опросы (polls), live-геолокация (только статичная), бесшумная копия сообщения (copy_message — есть только forward с пометкой «переслано»), пакетная загрузка и presigned-URL, заголовки max_connections/drop_pending_updates/ip_address, idempotency-key на уровне API (поэтому наш IdempotencyMiddleware — единственный слой защиты), API истории правок сообщения (приходит только новый текст — старый, если нужен, храните сами), mute/restrict участника (только kick с опциональным баном), запланированные сообщения (наш cron-cooldown остаётся). Параллельная обработка событий (use_create_task=True) в библиотеке есть, но включать её без рефакторинга module-level словарей мастеров нельзя — будут гонки.

Известные ограничения и направления оптимизации

Вещи, которые работают, но при росте нагрузки или объёма данных могут дать о себе знать. Каждый пункт — фиксация осознанного выбора «не сейчас» с триггером, при котором стоит вернуться. На MVP с десятками подписчиков и тысячами обращений в год ни один из них не блокирует запуск.

  • Рассылка открывает три session_scope на получателя (проверка флага отмены, запись доставки, периодический прогресс). На текущем масштабе это копейки. Триггер фикса: устойчиво >1000 подписчиков или несколько одновременных рассылок. Лечение: батчить запись доставок пачками по 50, флаг проверять раз в N сообщений.
  • Длинная цепочка диспетчеризации в on_callback (ветвление по префиксам payload — какой обработчик звать). Сборка и разбор самих строк уже централизованы (callback_payloads.py + callback_router.parse_int_tail), но выбор обработчика по префиксу всё ещё линейный. Читается сверху вниз, однако новый callback добавляется в общую кучу. Триггер: 5+ новых callback-ов. Лечение: словарь {prefix: handler_fn} по образцу _STATE_HANDLERS.
  • Анонимные обработчики _(event) в start.py и др. — удобно для регистрации, но в стектрейсе вместо имени команды будет <locals>._. Триггер: ловля сложного бага. Лечение: именовать функции по команде.
  • _user_locks в памяти процесса — не разделяется между репликами. На MVP один инстанс, это нормально. Триггер: multi-replica. Лечение: pg_advisory_xact_lock или Redis.
  • Inline русские usage-строки в admin_commands.py (десяток коротких подсказок «Используйте: /reopen <номер>») — технические, жителю не видны. Триггер: третья ревизия команд или i18n. Лечение: вынести в константы.

Несколько ограничений ранних версий уже закрыто: префиксный и точный поиск по телефону за O(log n) (индекс 0003), а подстрочный поиск жителя по имени и частичному телефону (ILIKE '%…%' в меню «Аудитория») — триграммными GIN-индексами миграции 0018 (B-tree из 0003 такой поиск с ведущим % не ускоряет, поэтому раньше был seq-scan); дублирование обвязки XLSX (общий хелпер); рост events без чистки (cron-ретеншн 30 дней); фиксированный шаг прогресс-бара рассылки (адаптивная формула); молчаливый сбой бэкапа (категоризированные алёрты); /healthz, отдававший OK при зависшей базе (добавлен SELECT 1); запуск контейнера под root без read-only (теперь непривилегированный botuser, read_only: true, tmpfs:/tmp, лимиты памяти и pids, cap_drop: ALL, no-new-privileges).

Развёртывание и рабочий цикл

Ручной деплой на сервере (полную автоматику авто-деплоя см. в разделе «Администратору»):

  1. Ветка. git checkout -b feat/название.
  2. Правки и проверка. Локальные тесты (uv run pytest) плюс ручная проверка через docker compose. Перед коммитом — зелёные линтер, типы и тесты.
  3. Коммит и PR. Сообщения в формате feat: / fix: / refactor: / docs: / chore:. Push в origin, открыть pull request на main. После одобрения — squash-merge.
  4. Деплой. На сервере: git pull && docker compose up -d --build bot && docker compose exec bot alembic upgrade head.

Соглашения по коду

  • Python 3.12, аннотации типов везде, где возможно. Ruff с длиной строки 100 (конфиг в pyproject.toml).
  • Комментарии объясняют зачем, а не что: невидимые ограничения, тонкие инварианты, ссылки на документацию. Имена переменных уже говорят, что делает код. Комментарии в проекте пишутся на русском.
  • Импорты — сверху файла. Импорт внутри функции оправдан только честной циклической зависимостью.
  • Никогда не обращайтесь к полям события напрямую — только через адаптер utils/event.py.

Доступ к репозиторию и секретам

Репозиторий приватный (github.com/gaben1488/aemr-bot). Чтобы начать работать, нужен доступ от владельца — он добавляет вас как коллаборатора либо в команду организации. Дальше клонирование по HTTPS с personal access token или по SSH-ключу — как удобнее.

  • Поток через форк (fork-flow). Внешний контрибьютор без прав на запись форкает репозиторий к себе, ведёт работу в ветке форка и открывает pull request в main основного репозитория. Коллаборатор с правом записи может ветвиться прямо в основном репозитории (git checkout -b feat/…, как в шагах деплоя выше) — это быстрее, потому что CI с секретами на форк-PR по умолчанию ограничен.
  • Деплой-ключ (deploy key). На боевом сервере доступ к приватному репозиторию даётся не личным логином, а deploy key — SSH-ключом, привязанным к одному репозиторию (как правило, read-only, чтобы сервер только тянул код). Это позволяет git pull при авто-деплое без хранения чьих-то личных учёток на VPS.
  • Секреты — не в репозитории. Реальные значения (BOT_TOKEN, пароль Postgres, GPG-фраза бэкапов) живут только в infra/.env на сервере и никогда не коммитятся; в репозитории — лишь .env.example с пустыми полями. Pre-commit-хук дополнительно ловит случайно добавленные .env и приватные ключи. Кто физически владеет секретами, как передаются дела при увольнении и где резерв GPG-фразы — это организационный вопрос; канон — в разделе «Администратору» (владение секретами), здесь не дублируется.

Частые вопросы

Почему мои локальные тесты дают покрытие ниже 84%, а CI зелёный?

Без поднятого PostgreSQL тесты, которым нужна реальная база, помечаются skip, и покрытие закономерно ниже. Это не баг. Чтобы повторить условия CI, поднимите docker compose up -d db и задайте DATABASE_URL на локальную базу.

Где «истина» по списку cron-задач и их расписаниям?

Имена и назначение — в services/cron_registry.py (его соответствие документации проверяет тест test_cron_docs_sync). А точные минуты и дни недели — в реальных CronTrigger в services/cron.py: реестр человекочитаемый и в одном месте может отставать от фактического триггера.

Сколько в проекте таблиц и миграций?

11 таблиц (db/models.py) и 18 миграций (0001..0018, последняя — 0018_users_trigram_search_indexes). Цифры «9 таблиц» и «5 миграций» в старых документах устарели.

Можно ли менять phone прямым SQL?

Нет. Всегда через services/users.set_phone — иначе рассинхронизируется phone_normalized и сломается поиск по телефону (/erase phone=, поиск жителя).

Почему специфичные slash-команды не видны в подсказке, хотя работают?

У MAX нет раздельных списков команд для лички и группы (нет per-scope команд). Одно /-меню публикуется сразу на оба чата, поэтому на старте бот выкладывает прямым PATCH /me ровно две универсальные команды — /start и /menu (осмысленные в любом чате), а специфичные жительские и операторские команды (/forget, /export, /policy, /reply и пр.) в подсказку не выносит — иначе /forget мозолил бы глаза операторам в служебной группе. Все эти команды остаются рабочими как запасной путь. Жители работают кнопками, операторы знают команды из RUNBOOK или памятки /op_help.

Сколько кнопок в главном меню жителя — шесть или семь?

Семь. В коде main_menu отдаёт семь рядов: «📝 Написать обращение», «📂 Мои обращения», переключатель подписки на рассылку, «🏛 Приём граждан», «ℹ️ Полезная информация», «🛡️ Защита от мошенников», «⚙️ Настройки и помощь». В части старых текстов и в комментарии-докстринге упомянуто «шесть» — это устаревшая цифра: кнопка «🛡️ Защита от мошенников» была добавлена отдельным пунктом сверх исходных шести. Доверяйте коду.

Что делать при ошибке «password authentication failed for user "aemr"»?

При первом запуске POSTGRES_PASSWORD записывается в данные тома и больше не перечитывается. После ротации пароля либо сделайте ALTER USER aemr WITH PASSWORD '...' в живой базе, либо пересоздайте том (docker compose down -v — сотрёт все данные, только на dev).
