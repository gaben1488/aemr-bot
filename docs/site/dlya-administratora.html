Администратору: установка и обслуживание

Бот живёт на одном небольшом сервере: сам опрашивает мессенджер MAX и хранит данные у администрации, без сторонних облаков. Здесь собрано всё, что нужно тому, кто его ставит и сопровождает: как развернуть с нуля, что настраивается без программиста, как снимать резервные копии и что делать, если бот замолчал.

Страница рассчитана на технически грамотного читателя, но термины мы всё равно поясняем коротко в скобках при первом появлении. Бот и база данных запускаются в Docker (среда для запуска программ в изолированных контейнерах), всё управление идёт несколькими командами в терминале. Полные пошаговые руководства лежат в репозитории: docs/SETUP.md, docs/SYSADMIN.md, docs/RUNBOOK.md, docs/ROLLBACK.md.

Минимальный сервер и установка#

Боту достаточно скромной виртуальной машины (VPS, арендованный компьютер в дата-центре). Проверенная конфигурация:

  • Система: Ubuntu 22.04 или 24.04 LTS (Debian 12 тоже подходит). Время на сервере должно быть синхронизировано по сети (NTP), потому что бот считает сроки в часовом поясе Asia/Kamchatka.
  • Ресурсы: 1 процессорное ядро, 2 ГБ оперативной памяти, 20 ГБ диска. Сам бот ограничен 512 МБ памяти, база данных возьмёт ещё двести-четыреста мегабайт.
  • Сеть: нужен только исходящий доступ в интернет по защищённому протоколу HTTPS. Входящие порты открывать не надо: бот сам обращается к серверам MAX и ничего не ждёт снаружи. Открытые порты понадобятся только в режиме webhook (приём входящих вызовов), но он не используется по умолчанию.

Установка идёт по шагам. Подробный разбор каждого шага с пояснениями есть в docs/SETUP.md, здесь коротко по сути.

  1. Завести пользователя и поставить Docker. Создайте отдельного пользователя для бота и добавьте его в группу docker. Установите Docker Engine и плагин Compose (инструмент запуска нескольких контейнеров одной командой) по официальной инструкции Docker.
  2. Скачать код. Склонируйте репозиторий с ботом на сервер и перейдите в его папку.
  3. Заполнить файл настроек. Скопируйте образец infra/.env.example в infra/.env, ограничьте к нему доступ (chmod 600) и впишите обязательные значения. Что именно обязательно, разобрано в следующем разделе.
  4. Запустить. Из папки infra выполните сборку и запуск. 
    cd infra
docker compose up -d --build
    Бот и база данных поднимутся в двух контейнерах. Через минуту проверьте здоровье (см. раздел «Порты и проверка здоровья»).
  5. Один раз перезапустить базу. После первого старта нужно один раз перезапустить контейнер базы, чтобы подхватился модуль сбора статистики запросов: docker compose restart db.
  6. Включить авто-обновление и сторожа. Поставьте скрипт авто-деплоя (auto-deploy, автоматическое обновление кода с сервера хранения) и внешнего сторожа здоровья (watchdog) в расписание через cron (системный планировщик задач). Оба скрипта и команды установки лежат в папке scripts/ и описаны в docs/SYSADMIN.md.
Автозапуск после перезагрузки сервера

Контейнеры подняты с политикой перезапуска unless-stopped, поэтому при штатной перезагрузке сервера они стартуют вместе с Docker. На случай, если до перезагрузки они были остановлены вручную, в репозитории есть systemd-юнит (служба автозапуска) infra/aemr-bot.service. Установить его один раз поможет scripts/install-systemd-autostart.sh.

Настройка через файл .env#

Все секреты и параметры запуска бот читает из одного файла infra/.env. Файл читается только при старте контейнера, поэтому после любой правки бот нужно перезапустить: docker compose up -d --force-recreate bot. Полное описание каждого поля есть в образце infra/.env.example.

Обязательные значения#

Без этих значений бот либо не запустится, либо не сможет работать:

BOT_TOKENключ доступа бота, выдаётся на портале max.ru/business, без приставки «Bearer»
ADMIN_GROUP_IDидентификатор служебной группы в MAX (отрицательное число), узнаётся командой /whoami в самой группе
DATABASE_URLадрес подключения к базе данных Postgres; в Docker формируется автоматически из пароля и имени базы

Вместе с ними в файле задают пароль базы данных (POSTGRES_PASSWORD, не меньше 24 случайных символов), имя базы и пользователя (POSTGRES_DB, POSTGRES_USER), часовой пояс TZ=Asia/Kamchatka и данные первого ИТ-специалиста (BOOTSTRAP_IT_MAX_USER_ID и BOOTSTRAP_IT_FULL_NAME), чтобы он получил права без ручного доступа к базе. Если пароль базы оставить пустым, запуск прервётся с понятной ошибкой ещё до старта контейнеров.

Рекомендуемые значения#

Эти параметры не обязательны для запуска, но в рабочей эксплуатации их стоит задать:

BACKUP_GPG_PASSPHRASEпароль для шифрования резервных копий; в рабочем режиме обязателен (см. раздел про копии)
HEALTHCHECK_URLадрес внешнего наблюдателя (например, Healthchecks.io или Uptime Kuma), куда бот шлёт сигнал «жив»
AUDIT_LOG_RETENTION_DAYSсколько дней хранить журнал действий операторов; по умолчанию 365, допустимо от 30 до 3650
Версии библиотек не переписывайте по памяти

Точные версии всех программных библиотек, на которых работает бот, закреплены в одном месте: файл bot/pyproject.toml. Там же стоят и границы обновлений (например, клиент MAX зафиксирован записью maxapi~=1.1, а веб-каркас и инструменты тестирования имеют свои границы). Это единственный достоверный источник: при обновлении и при сверке смотрите в него, а не в старые заметки. Точные собранные версии фиксируются в bot/uv.lock и одинаковы в локальной среде, в проверке кода и в контейнере.

Настройки без программиста#

Многое из того, что меняется со временем (тексты для жителей, контакты служб, ссылки, список тематик и поселений), можно править прямо из служебной группы, без редактирования кода и без перезапуска. Эти значения живут в таблице базы данных под названием settings.

Удобный способ, рекомендуемый, открыть в панели оператора /op_help кнопку ⚙️ Настройки бота (доступна роли ИТ). Откроется иерархическое меню (вложенное, с разделами и подразделами), где значения сгруппированы по смыслу:

  • Тексты для жителей: приветствие, текст согласия на обработку данных, текст про приём граждан.
  • Внешние ссылки: политика обработки данных, электронная приёмная, два расписания транспорта.
  • Списки: тематики обращений и населённые пункты (правятся построчно).
  • Объекты: контакты экстренных служб и диспетчеров транспорта (у каждого название и телефон).
  • Прочее: лимит картинок в рассылке, тихий режим служебного чата (часы, когда не приходят некритичные уведомления), автор служебных коммитов.

Запасной способ, для длинных значений и точечных правок, команда /setting в служебной группе. Она доступна только роли ИТ.

/setting listпоказать все настраиваемые ключи с типами и текущими значениями
/setting policy_url https://elizovomr.ru/...заменить ссылку на политику обработки данных
Ровно 17 ключей, и ссылки только на официальные домены

Через меню и команду правится строго 17 ключей (заранее заданный белый список). Любой другой ключ бот отклонит. Все значения проходят проверку: тексты ограничены по длине, у контактов поле «телефон» должно выглядеть как телефон, а ссылки принимаются только на официальные ресурсы (elizovomr.ru, kamgov.ru, gosuslugi.ru, kamchatka.gov.ru). Это защита от случайной или подменённой ссылки на сторонний сайт.

Порты и проверка здоровья#

У бота есть три служебных адреса проверки здоровья (health-эндпоинты, страницы статуса). Все они отвечают на порту 8080 и привязаны к локальному адресу самого сервера (127.0.0.1), наружу не публикуются. Номер порта задаётся переменной WEBHOOK_PORT в файле .env (значение по умолчанию 8080).

Адрес Что проверяет Когда отвечает «200 ОК»
/livez жив ли главный рабочий цикл бота (не трогает базу) если внутренний сигнал жизни обновлялся недавно
/readyz готов ли бот работать с данными (делает запрос к базе) если сигнал свежий и база отвечает
/healthz то же, что /readyz, оставлен для старых инструкций при свежем сигнале и доступной базе

Проверить вручную можно так:

curl -fsS http://127.0.0.1:8080/livez && echo
curl -fsS http://127.0.0.1:8080/readyz && echo

Важная тонкость: автоматический сторож и встроенная проверка контейнера смотрят на /livez, а не на /readyz. Короткая проблема с базой не должна перезапускать живой процесс бота. Если /livez отвечает, а /readyz нет, значит процесс жив, но база недоступна или зависла.

Переменной HEALTHCHECK_PORT не существует

Номер служебного порта один и задаётся только переменной WEBHOOK_PORT. Отдельной переменной HEALTHCHECK_PORT в боте нет, и придумывать её не нужно. База данных Postgres на хост-машину не выводится вовсе: обратиться к ней можно лишь изнутри через docker compose exec db psql -U aemr aemr.

Регулярные задачи#

Внутри процесса бота работает планировщик (он же cron, выполняет задачи по расписанию). Все времена указаны в часовом поясе Asia/Kamchatka. У каждой задачи есть запас времени на «догон» (120 секунд), поэтому короткий перезапуск во время обновления не приводит к потере тика. Повторный запуск той же задачи второй экземпляр не порождает.

Ниже все регулярные задачи с точным расписанием и назначением простыми словами.

Идентификатор Когда Что делает
startup-pulse один раз, через 5 секунд после старта сообщает в служебную группу, что процесс запущен заново
pulse-hourly каждый час, минута :05, круглосуточно базовый сигнал «бот работает» дежурному
pulse-workhours-extra пн–пт, часы 09–17, минута :35 дополнительный сигнал жизни в рабочее время
health-selfcheck каждые 5 минут следит за сигналом жизни, шлёт алёрт при смене «жив ↔ завис»
healthcheck-ping каждые 5 минут, только если задан HEALTHCHECK_URL пингует внешнего наблюдателя, чтобы тот знал, что бот жив
funnel-watchdog каждый час, минута :15 сбрасывает зависшие анкеты (житель начал и не закончил)
threat-intel-refresh каждый час, минута :17 обновляет списки опасных ссылок для предупреждений в карточках
broadcast-draft-reaper каждый час, минута :37 помечает «провалившимися» черновики рассылок старше 30 минут
open-reminder-workhours пн–пт, часы 09–11 и 13–17, минута :10 напоминает операторам обо всех открытых обращениях
overdue-reminder-workhours пн–пт, часы 09–11 и 13–17, минута :40 отдельно напоминает о просроченных по сроку ответа
events-retention ежедневно, 04:00 удаляет служебные события защиты от повторов старше 30 дней
audit-log-retention ежедневно, 04:15 чистит журнал действий операторов старше срока хранения
stale-operators-cleanup ежедневно, 04:20 отключает операторов, покинувших служебную группу (кроме ИТ)
pdn-retention ежедневно, 04:30 обезличивает данные жителей через 30 дней после отзыва согласия
appeals-5y-retention ежедневно, 04:45 обнуляет текст обращений старше 5 лет, оставляя метаданные
monthly-stats 1-го числа, 09:00 присылает в группу таблицу-отчёт за прошлый месяц
db-backup каждое воскресенье, 03:00 (по умолчанию) снимает резервную копию базы, шифрует и хранит с ротацией

Сроки хранения данных (30 дней, 30 дней после отзыва, 5 лет, 365 дней журнала) заданы законом о персональных данных (152-ФЗ) и порядком работы с обращениями граждан (59-ФЗ). Время еженедельной копии можно сдвинуть переменными BACKUP_DAY_OF_WEEK, BACKUP_HOUR, BACKUP_MINUTE в файле .env.

Резервные копии и восстановление#

Резервная копия (backup, сохранённый слепок базы данных) снимается автоматически каждое воскресенье в 03:00 по Камчатке. Это полный дамп базы (выгрузка всего содержимого) средствами pg_dump. Копии складываются в постоянное хранилище контейнера (named volume) и хранятся по последним 8 файлов, примерно за два месяца. По желанию можно включить дополнительную выгрузку в облачное S3-хранилище переменными BACKUP_S3_*.

Шифрование обязательно в рабочем режиме

Копия содержит персональные данные жителей: телефоны, имена, тексты обращений. Поэтому в рабочем режиме копия шифруется (GPG, симметричное шифрование AES-256) паролем из переменной BACKUP_GPG_PASSPHRASE (не короче 12 символов). Без этого пароля копия не создаётся вовсе: задача пишет отказ в журнал и шлёт алёрт в служебную группу. Это требование 152-ФЗ: незашифрованный дамп на диске или в облаке считается утечкой. Открыть незашифрованный режим можно только явным BACKUP_ALLOW_UNENCRYPTED=1, и только на тестовой машине. Пароль шифрования храните отдельно от сервера: без него зашифрованную копию восстановить невозможно.

Снять копию вручную и посмотреть список#

Внеплановую копию ИТ-специалист снимает прямо из служебной группы командой /backup или кнопкой 💾 Снять бэкап в панели /op_help. Посмотреть, какие копии уже есть:

docker compose exec bot ls -lah /backups

Зашифрованные файлы имеют имя вида aemr-…​.sql.gpg. Имя без .gpg появляется только в незашифрованном тестовом режиме.

Восстановление из копии#

Восстановление выполняют только в отдельную тестовую базу, не затрагивая рабочую. Полный порядок с проверкой есть в docs/BACKUP_RESTORE_TEST.md. Коротко: создать новую пустую базу, расшифровать файл и залить его туда.

# создать тестовую базу
docker compose exec -T db psql -U aemr -d postgres \
  -c 'CREATE DATABASE aemr_restore_test;'

# расшифровать и залить в неё
sudo cat /var/lib/docker/volumes/aemr-bot_backups/_data/aemr-ГГГГ-ММ-ДД.sql.gpg \
  | gpg --batch --passphrase "$BACKUP_GPG_PASSPHRASE" --decrypt \
  | docker compose exec -T db psql -U aemr -d aemr_restore_test

После заливки сверьте, что таблицы на месте и счётчики правдоподобны, затем удалите тестовую базу. Проверять восстановление рекомендуется раз в квартал: только так копия считается рабочей, а не просто существующей.

Бот молчит: что делать#

Если в служебной группе тишина и обращения не приходят, разбирайтесь по порядку, от простого к сложному.

  1. Проверьте сигнал жизни. Выполните curl -fsS http://127.0.0.1:8080/livez. Если ответ есть, процесс жив, и проблема, скорее всего, в сети или в сессии с MAX. Если ответа нет или ошибка, переходите к логам.
  2. Посмотрите логи. Последние строки журнала покажут причину: 
    docker compose logs --tail 200 bot
    Частые причины тишины при живом процессе: истёк или отозван BOT_TOKEN, либо сервер MAX временно ограничивает частоту запросов. Подсказку даст /diag в служебной группе.
  3. Не было ли обновления. Если бот недавно перезапустился сам, это, как правило, авто-деплой. Загляните в его журнал: journalctl -t aemr-bot-deploy -n 50. Запись «DEPLOY FAILED» означает, что новое обновление оказалось сломанным и сработал автоматический откат: сейчас работает предыдущая, рабочая версия.
  4. Откат вручную, если нужно. Когда сломанное обновление не откатилось само, верните рабочую версию по инструкции docs/ROLLBACK.md. Важно: бот недоступен не значит потеря данных. База живёт отдельно от программы, обращения и журнал сохраняются, откатывается только код. Цель отката, вернуть рабочее состояние за 10 минут.

Если процесс зависает, а не падает, его перезапустит внешний сторож healthwatch.sh (cron каждые 5 минут): после трёх неудачных проверок подряд он перезапускает бота, после восьми пишет тревогу в служебную группу напрямую через MAX, минуя самого бота.

Регулярные мероприятия#

Несколько вещей делаются не разово, а по календарю. Их стоит поставить себе в напоминания.

Ротация секретов#

Ротация (плановая замена) секретов раз в полгода, в январе и июле. Меняются ключ бота BOT_TOKEN, пароль базы POSTGRES_PASSWORD, пароль шифрования копий BACKUP_GPG_PASSPHRASE и прочие чувствительные значения. Порядок описан в docs/SECURITY.md, раздел про ротацию секретов. Ведите журнал замен: кто, когда и что поменял.

Сетевая изоляция (по желанию)#

Бот обращается наружу только к ограниченному списку адресов: серверы MAX, сервер хранения кода, внешний наблюдатель здоровья (если задан) и источники списков опасных ссылок. При желании можно жёстко закрыть весь прочий исходящий трафик правилами межсетевого экрана (iptables), оставив доступ только нужным адресам. Готовый набор правил и подводные камни (адреса дата-центров меняются, не заблокируйте сервер обновлений) описаны в docs/SYSADMIN.md. По умолчанию такая изоляция не включена: на сервере живёт только бот, и риск утечки минимален.

Опись зависимостей и проверка на уязвимости#

Раз в квартал или при сообщении об уязвимости в какой-либо библиотеке проверьте зависимости одной командой. Инструмент pip-audit сверит установленные библиотеки с базой известных уязвимостей и заодно даст опись состава (SBOM, перечень компонентов программы):

uv run pip-auditпроверить все библиотеки бота на известные уязвимости

Та же проверка выполняется автоматически при каждой сборке кода и блокирует выпуск при обнаружении уязвимости. Обновление любой библиотеки делается отдельно, по порядку из docs/DEPS.md: сперва обновляется файл фиксации версий, затем прогоняются тесты, и лишь потом изменение уходит в работу.
