aemr-bot — handover

Документ для админа и безопасника. Self-host, MAX-мессенджер, Postgres 16.
CI: зелёный 676 tests / 58 skipped Python 3.12 Postgres 16 11 migrations

В двух предложениях

Бот «Обратная связь Администрации Елизовского муниципального округа» в MAX-мессенджере. Один процесс на одной VPS, никаких облаков, Postgres рядом в Docker Compose.

Что внутри

  • Python 3.12-slim (digest pinned)
  • Postgres 16-alpine (digest pinned)
  • FastAPI + uvicorn для webhook
  • SQLAlchemy 2.0 async, asyncpg
  • APScheduler — все cron внутри процесса
  • maxapi — MAX Bot API клиент

Что снаружи

  • опц.nginx + certbot (только webhook)
  • опц.rclone для S3-бэкапов
  • GPG для шифрования pg_dump
  • опц.Healthchecks.io / Uptime Kuma

Чего нет и не нужно

  • Никаких VK, Yandex Cloud, AWS
  • Никаких сторонних SaaS
  • Никаких Redis/Kafka — всё в Postgres
  • Никакого Telegram

Установка с нуля

# 1. Пользователь под бота
sudo adduser --disabled-password --gecos "" aemr
sudo usermod -aG docker aemr

# 2. Клонирование
sudo -u aemr -i
git clone https://github.com/gaben1488/aemr-bot.git
cd aemr-bot

# 3. .env
cp infra/.env.example infra/.env
chmod 600 infra/.env
# заполнить infra/.env (см. таб ИБ → секреты)

# 4. Старт
cd infra
docker compose up -d --build

# 5. Health-check
curl -fsS http://127.0.0.1:8080/livez && echo
curl -fsS http://127.0.0.1:8080/readyz && echo

# 6. Auto-deploy под root (раз в 10 мин)
exit
sudo bash /home/aemr/aemr-bot/scripts/install-auto-deploy.sh

# 7. Watchdog под root в cron
sudo crontab -e
# */5 * * * * /home/aemr/aemr-bot/scripts/healthwatch.sh
Один лишний шаг после первой установки: миграция 0010 создаёт pg_stat_statements, но сбор статистики стартует только после рестарта Postgres с включённым preload. docker compose restart db — один раз.

Деплой и rollback

Auto-deploy (cron */10 под root)

Логика scripts/auto-deploy.sh:

  1. git fetch origin main, если новых коммитов нет — выходит тихо
  2. Запоминает текущий HEAD как PREV_LOCAL
  3. git reset --hard origin/main, docker compose up -d --build
  4. Health-gate: до 60 сек опрашивает /livez каждые 5 сек
  5. Успех — лог «deploy ok»
  6. Фейл — auto-rollback на PREV_LOCAL, пересборка, alert в journald
Ручной деплой 
sudo -u aemr -i
cd aemr-bot
git fetch origin main && git checkout main && git pull --ff-only
cd infra
docker compose build bot
docker compose up -d bot
# затем VPS_SMOKE_CHECKLIST.md
Откат вручную 
sudo -u aemr -i
cd aemr-bot
git log --oneline -10
git checkout <предыдущий рабочий коммит>
cd infra
docker compose up -d --build bot

Миграции откатываются через alembic downgrade <revision>. На проде — крайняя мера; обычно проще restore из бэкапа.

Бэкап и восстановление

Список бэкапов:

Расшифровка:

  | gpg --batch --passphrase "$BACKUP_GPG_PASSPHRASE" --decrypt > restore.sql

psql -h ... -U ... -d aemr_restore -f restore.sql
Никогда не restore'ить в production. Только в отдельную чистую БД. После restore из старой копии — повторно прогнать pdn-retention, иначе данные удалённых жителей возвращаются.

Что часто ломается и где смотреть

Бот молчит, тишина в служебной группе

Сначала curl -fsS http://127.0.0.1:8080/livez. Если 200 — процесс жив, проблема в MAX-сессии или сети. Если 5xx или таймаут — docker compose logs --tail 200 bot.

Pulse не приходит

С misfire_grace_time=120 пропуски при рестартах догоняются. Если pulse молчит дольше — скорее всего BOT_TOKEN отозван или MAX rate-limit. /diag в служебной группе подскажет.

DEPLOY FAILED в journald

Auto-rollback сработал. Сейчас работает предыдущий коммит. Проверить новый локально, понять причину, пушнуть фикс. Никаких ручных действий не требуется.

Restart-loop

docker compose ps покажет «Restarting». docker compose logs --tail 500 bot — почему. Если новый коммит — auto-rollback должен был сработать; если не сработал, ручной откат.

БД растёт быстрее ожидаемого 
docker compose exec db psql -U aemr aemr -c "\dt+"

Обычно жирная — events (idempotency), retention 30 дней её сдерживает.

Кратко о модели угроз

Активы

  • ПДн жителей в Postgres
  • Секреты в infra/.env
  • Резервные копии (named volume)
  • audit_log и логи Docker

Угрозы

  • Компрометация VPS (украли SSH-ключ)
  • Компрометация MAX (BOT_TOKEN)
  • Внутренний злоумышленник-оператор
  • Внешний флуд / DoS
  • 152-ФЗ запросы на удаление

Container hardening

bot — самый строгий
  • read_only: true — root-fs неизменяемый
  • tmpfs: /tmp:size=128m — единственное writable место
  • mem_limit: 512m, memswap_limit: 512m
  • pids_limit: 200 — fork-bomb капается
  • cap_drop: ALL
  • no-new-privileges: true
  • UID/GID 1000 (botuser), не root
db — минимально необходимый
  • cap_drop: ALL, оставлены только CHOWN, DAC_READ_SEARCH, FOWNER, SETGID, SETUID
  • no-new-privileges: true
  • не read_only — нужен writable ${PGDATA}

Защита кода и состояния

Защита от ввода

  • html.escape на значениях жителя в HTML-разметке прогресс-бара
  • hmac.compare_digest на WEBHOOK_SECRET (защита от timing-oracle)
  • Rate-limit обращений: 3 в час на жителя
  • asyncio.Semaphore(32) на webhook-обработчиках
  • Idempotency через events.idempotency_key UNIQUE
  • Reply-dedup 10 секунд на (operator_id, appeal_id, text)

Защита состояния

  • pg_advisory_xact_lock(max_user_id) в update_dialog_data
  • statement_timeout = 30s на роль
  • idle_in_transaction_session_timeout = 60s
  • misfire_grace_time = 120 на все cron
  • Persistent wizard state (таблица wizard_state)

CI как security gate

Hard fail на любом из:

  • ruff check — стиль и линт
  • mypy — типы
  • bandit -ll — статический security (medium+)
  • pip-audit --strict — CVE-сканер на frozen deps
  • shellcheck — все *.sh
  • pytest --cov-fail-under=65 — coverage gate
  • alembic upgrade head
  • alembic check — drift модель ↔ миграции
  • alembic round-trip — upgrade → downgrade base → upgrade head

Известные компромиссы (честно)

HASingle-replica

Одна VPS = single point of failure. Падение хоста означает простой до восстановления. Решение — second VPS + Patroni для Postgres + DNS-failover. Архитектурный вопрос, не код.

TLSBot↔db без TLS внутри Docker network

Уровень доверия — «доверенная сеть Docker». Перехват возможен только при root на VPS. Для повышения — Postgres SSL + asyncpg ssl=require.

BackupЮридический разрыв backup ↔ erase

Backup от даты до /erase всё ещё содержит данные жителя. После restore из старой копии необходимо повторно применить erase. Формально это нарушение 152-ФЗ, если не сделать.

MAXСообщения остаются у мессенджера

Бот не может удалить уже отправленные сообщения из MAX-чата жителя. Они на серверах мессенджера, подчиняются политике MAX. В политике приватности этот факт нужно явно указать.

опц.DAST/penetration test не проведён

Static-analysis (bandit, pip-audit) есть. Динамического тестирования (фаззинг webhook, brute callback-payload'ов) нет. Рекомендуется при бюджете.

Чего хочется от ИБ-команды

  • Iptables outbound whitelist: разрешить только три исходящих (botapi.max.ru, S3, healthcheck), всё остальное — drop
  • Аудит .env на ротацию: BOT_TOKEN, POSTGRES_PASSWORD, GPG passphrase менялись когда-нибудь? Календарь ротации раз в полгода
  • Pen-test webhook (если включён): фаззинг, попытки обхода hmac.compare_digest
  • Compliance аудит: формальное соответствие 152-ФЗ и 59-ФЗ. Юрист, не разработчик
  • DR plan: что делает админ, если VPS уничтожена полностью
