## 0. БОТ ЛЕЖИТ — реанимация по шагам (инцидент 2026-06: docker выключен, cron'ы сняты)

Зайти на сервер с Windows (PowerShell или cmd):

```
ssh aemr@193.233.244.217
```

### Шаг 1. Понять, что вообще происходит (4 команды, ничего не меняют)

```bash
systemctl is-active docker          # ожидаем: active. "inactive" = причина №1: docker выключен
sudo docker ps -a                   # ожидаем 2 контейнера: aemr-bot-bot-1 и aemr-bot-db-1
sudo crontab -l                     # ожидаем 2 строки: */10 aemr-bot-autodeploy и */5 aemr-bot-healthwatch
ss -ltnp | grep 8080                # кто держит порт 8080 (должен docker-proxy, и только один)
```

### Шаг 2. Включить docker обратно

```bash
sudo systemctl enable --now docker
```

**Почему это обязательно:** в docker живёт не только бот, но и **база данных Postgres**
(`infra/docker-compose.yml:8-57`). Выключенный docker = выключенная база = бот не может
работать ни в каком виде. «Починить бота, выключив docker» невозможно по конструкции.

### Шаг 3. Убедиться, что нет «второго бота» вне docker

```bash
ps aux | grep -E 'aemr_bot|python.*main' | grep -v grep
systemctl list-units --all | grep -iE 'aemr|bot'
```

Любой python-процесс `aemr_bot` ВНЕ контейнера — чужак: `sudo kill <PID>`.
Любой systemd-юнит про бота, кроме `aemr-bot.service`, — чужак:
`sudo systemctl disable --now <имя>`. (Штатный `aemr-bot.service` — безопасная обёртка,
см. §1; он второй экземпляр создать не может.)

### Шаг 4. Поднять стек

```bash
cd ~/aemr-bot/infra && docker compose up -d
```

### Шаг 5. Проверить (через 30–60 секунд)

```bash
docker compose ps                                  # оба сервиса Up (healthy)
curl -s http://127.0.0.1:8080/livez               # {"ok": true}
curl -s http://127.0.0.1:8080/readyz              # {"ok": true} — включая пинг базы
docker compose logs --tail 50 bot
```

В логе ищем две строки успеха: `preflight: токен валидный — бот ...` (main.py:584)
и `Starting in long polling mode` (main.py:749).

**Если контейнер bot в состоянии Restarting** — иди в таблицу форс-мажоров (§3),
первым делом: `docker compose logs bot | grep 'BOT_TOKEN неверный'`.

### Шаг 6. Вернуть страховки (когда бот ожил)

```bash
sudo systemctl enable --now aemr-bot.service              # автозапуск после перезагрузки VPS
sudo bash ~/aemr-bot/scripts/install-healthwatch.sh       # вотчдог: cron */5, лечит зависания
sudo bash ~/aemr-bot/scripts/install-auto-deploy.sh       # автодеплой: cron */10 — ТОЛЬКО если
                                                          # хочешь, чтобы push в main сам доезжал до сервера
```

### Шаг 7. Контроль

В админ-группе MAX в течение часа должно прийти `🟢 Пульс: бот работает`
(каждый час в :05, `services/cron.py:540`). Пришло — инцидент закрыт.

## 1. Как устроена система (2 минуты, обязательно к прочтению)

Четыре слоя, и у каждого своя роль:

1. **Docker Compose** — единственный способ запуска. Два контейнера: `bot` (приложение)
   и `db` (Postgres 16, данные в volume `db_data`). Оба с политикой
   `restart: unless-stopped` — docker сам поднимает их после падения процесса
   (docker-compose.yml:57,112). Это ЕДИНСТВЕННЫЙ супервизор процесса.
2. **systemd-юнит `aemr-bot.service`** — НЕ второй способ запуска, а страховка
   автозапуска: `Type=oneshot`, `ExecStart=docker compose up -d` — срабатывает один раз
   при загрузке сервера и идемпотентен (уже подняты — ничего не делает). Конфликт
   с docker невозможен by design. **Ловушка:** `systemctl stop aemr-bot` выполняет
   `docker compose down` — гасит И БОТА И БАЗУ (aemr-bot.service:51-64). Рестарт бота —
   это `docker compose restart bot`, не systemctl.
3. **Два root-cron'а** (то, что было отключено в инциденте):
   - `*/5 aemr-bot-healthwatch` — вотчдог: опрашивает `/livez`; на 3-м подряд провале
     (15 минут) делает `docker compose restart bot`, с 8-го (40 минут) шлёт `⛑️`-алёрт
     в админ-группу напрямую через MAX API, мимо бота (healthwatch.sh:58-92).
     Это единственный механизм, который лечит «живой труп» — процесс жив, но завис
     (docker healthcheck лишь помечает unhealthy, НЕ рестартует).
   - `*/10 aemr-bot-autodeploy` — автодеплой из origin/main с health-gate 60с по `/livez`
     и автооткатом (auto-deploy.sh:102-147). Отключён = деплой только руками.
4. **17 внутренних задач APScheduler** (пульс, selfcheck, бэкап вс 03:00, ретенции
   152-ФЗ — `services/cron_registry.py`) живут ВНУТРИ процесса бота и от серверного
   cron не зависят.

**Кто кого перезапускает:**

| Что случилось | Кто поднимает |
|---|---|
| Процесс бота упал (любой exit) | docker `restart: unless-stopped` |
| Процесс жив, но завис (unhealthy) | ТОЛЬКО healthwatch-cron (если установлен) |
| Перезагрузка VPS | systemd-юнит → `docker compose up -d` (если docker.service enabled) |
| docker выключен + cron'ы сняты | **НИКТО. Ровно состояние инцидента 2026-06.** |

## 2. Запуск с нуля на новом сервере

Канон — вика `docs/site/index.html` → «Администратору» → установка в 9 шагов
(требования: 1–2 vCPU, 2–4 ГБ RAM, 20 ГБ SSD, Ubuntu 22.04/24.04, наружу только
исходящий 443; inbound-портов бот не требует — long polling). Выжимка:

```bash
# 1. Токен: ТОЛЬКО через max.ru/business (НЕ @MasterBot)
# 2. Docker + ufw по вике
git clone https://github.com/gaben1488/aemr-bot.git ~/aemr-bot
cd ~/aemr-bot/infra
cp .env.example .env && chmod 600 .env
# 3. В .env заполнить МИНИМУМ: BOT_TOKEN и POSTGRES_PASSWORD
#    (DATABASE_URL compose соберёт сам из POSTGRES_*; config.py:8-23)
docker compose up -d --build
curl -s http://127.0.0.1:8080/readyz        # {"ok": true}
# 4. Страховки: systemd-юнит + оба cron-установщика (см. §0 шаг 6)
```

## 3. Форс-мажоры: симптом → диагноз → лечение

### а) Бот «работает, но через раз»: теряются нажатия, отвечает не всем
Почти наверняка **два экземпляра на один BOT_TOKEN**. Ошибки в логах НЕ будет:
обработки конфликта polling'а нет нигде (maxapi знает только 401; всё прочее —
вечный retry с INFO-логом, dispatcher.py:1321-1327), а маркер апдейтов у каждого
экземпляра свой in-memory — они молча воруют события друг у друга.
Диагноз: §0 шаг 3 + вспомнить, не запущен ли бот с прод-токеном где-то ещё
(другой сервер, чей-то ноутбук, «локальный тест»). Лечение: оставить ровно один.
Нюанс: второй процесс НА ТОМ ЖЕ хосте умрёт сам при старте — порт 8080 занят
(EADDRINUSE, main.py:717 без try/except) — поэтому тихое двоение возможно только
с РАЗНЫХ машин/копий.

### б) Контейнер bot в Restarting, бот молчит
```bash
docker compose logs bot | grep -E 'BOT_TOKEN неверный|Неверный токен'
```
Нашлось — токен бит/отозван: взять действующий в max.ru/business → вписать в
`infra/.env` → `docker compose up -d`. Учти: фраза в логе «Контейнер выйдет, чтобы
избежать restart-loop» (main.py:569) фактически неверна — unless-stopped крутит
рестарты бесконечно, по одной ❌-строке на цикл.
Не нашлось — смотри полный лог: `docker compose logs --tail 200 bot` (падение
alembic = проблема базы, см. г/д).

### в) Нет сети/DNS до platform-api.max.ru
Лог: `Ошибка подключения при получении обновлений: ..., жду 30 секунд` — бот ЖИВ
и сам переподключится (вечный retry, dispatcher.py:1308-1316). Ничего не делать.
На старте при мёртвой сети процесс может упасть (окно в `__ready`) — docker
поднимет, цикл продолжится до возврата сети.

### г) `/readyz` красный при зелёном `/livez` — упала база
```bash
docker compose ps db && docker compose logs --tail 50 db && df -h
```
Бот при этом НЕ рестартует и НЕ алёртит — осознанное решение (краткий сбой БД не
должен убивать живой polling; healthwatch.sh:46-51) и известная дыра: затяжное
падение БД заметит только человек. Движок сам переподключится после возврата БД
(pool_pre_ping, session.py:14-38).

### д) База не стартует вовсе / alembic падает на старте
Логи db, пароль POSTGRES_PASSWORD в .env, место на диске. Контейнер bot будет
рестартиться, пока db не станет healthy (depends_on, docker-compose.yml:66-68).

### е) Кончился диск
Вика #admin-disk. **ЗАПРЕЩЕНО**: `docker system prune -a --volumes` — снесёт
volume с базой. Postgres на полном диске уходит в PANIC → сценарий (г).

### ж) Подозрение на OOM (бот перезапускается без ошибок в логах)
```bash
docker inspect aemr-bot-bot-1 --format '{{.State.OOMKilled}} {{.RestartCount}}'
dmesg | grep -i oom | tail
```
SIGKILL в логах бота следов не оставляет; признак в MAX — внеплановое
`🔄 Рестарт: процесс бота запущен заново` в админ-группе.

### з) «Жив, healthy, но в чате не отвечает»
```bash
docker compose logs bot | grep 'БОТ ИГНОРИРУЕТ POLLING'
```
Если есть — когда-то была включена webhook-подписка, и она молча ворует события
у polling (dispatcher.py:495-499). Снять подписку (см. вику), рестартовать.

### и) Автодеплой привёз плохой коммит
Сам откатится за ≤60с по `/livez`-гейту (auto-deploy.sh:124-147). Смотреть:
`journalctl -t aemr-bot-deploy -n 50` (строки `DEPLOY FAILED`, `ROLLBACK ok`).
Ручной откат ≤10 минут — вика #admin-deploy.

### к) После перезагрузки VPS ничего не поднялось
```bash
systemctl status aemr-bot; systemctl is-enabled docker aemr-bot
```
Оба должны быть enabled. Дальше §0.

### л) Перенос на новый сервер / восстановление из бэкапа
Унести три вещи: `infra/.env`, свежий бэкап из volume `backups`, GPG-фразу.
Вика #admin-migrate и #admin-restore-prod. Главное правило переноса: **сначала
остановить старого бота, потом запускать нового** — двух живых на один токен
быть не должно (см. а).

## 4. Чего НЕ делать (анти-чек-лист по мотивам инцидента)

- **НЕ выключать docker**, чтобы «починить бота»: там база; станет только хуже.
- **НЕ снимать оба cron'а**: вотчдог — единственный, кто лечит зависший процесс.
- **НЕ путать** `systemctl stop aemr-bot` (= `compose down`, гасит ВСЁ) с рестартом
  бота (`docker compose restart bot`).
- **НЕ запускать** второй экземпляр с тем же токеном — нигде, включая свой компьютер.
- **НЕ коммитить** `infra/.env.example` с заполненными секретами (см. §6-предупреждение).
- **НЕ выполнять** `docker system prune -a --volumes`.

## 5. Рутина и логи

- **Пульс**: `🟢 Пульс: бот работает` в админ-группе каждый час в :05. Тишина
  дольше 2 часов — зайти и проверить §0 шаг 5.
- **Логи бота**: `docker compose logs --tail 200 bot` (ротация 10MB×3).
- **Логи вотчдога**: `journalctl -t aemr-bot-watchdog -n 50`.
- **Логи деплоя**: `journalctl -t aemr-bot-deploy -n 50`.
- **Логи автозапуска**: `journalctl -u aemr-bot.service -n 50`.
- **Греп-лист дословных строк** (по коду): токен — `BOT_TOKEN неверный`,
  `Неверный токен! Останавливаю polling`; сеть — `Ошибка подключения при получении
  обновлений`, `Ошибка соединения (`; polling — `Starting in long polling mode`,
  `БОТ ИГНОРИРУЕТ POLLING!`; БД — `readyz: DB ping failed`; handlers — `Ошибка при
  обработке события`; вотчдог — `/livez unreachable (consecutive fails=`,
  `auto-restart of bot container triggered`; деплой — `new commits:`, `deploy ok:`,
  `DEPLOY FAILED`, `ROLLBACK ok`.
- **Раз в неделю**: `df -h`; `docker compose ps`; свежий бэкап в volume `backups`
  (создаётся вс 03:00, ротация 8 копий); полный срез одной командой:
  `sudo bash ~/aemr-bot/scripts/audit_vps.sh` (всё с маскированием секретов).

## 6. Открытые риски (честно, на 2026-06-13)

1. **Затяжное падение Postgres не алёртится** — `/readyz` красный, но ни рестарта,
   ни сообщения оператору (дыра by design, см. §3-г).
2. **Offsite-бэкапа нет**: потеря VPS = потеря базы и всех копий (RPO до недели).
3. **Внешнего мониторинга нет** (HEALTHCHECK_URL пуст) — здоровье сторожат только
   cron-вотчдог и человек.
4. **Bus-factor = 1**; DR-учение end-to-end не проводилось.
5. **Битые ссылки в вике** на упразднённые .md (SETUP/RUNBOOK/SYSADMIN/...) — содержание
   переехало в вику, ссылки остались.
6. **⚠️ СЕКРЕТЫ В РАБОЧЕЙ КОПИИ (ноутбук)**: в `infra/.env.example` локально вписаны
   непустые BOT_TOKEN и BACKUP_GPG_PASSPHRASE (не совпадают с боевым `.env`; в
   git-истории их НЕТ — файл изменён, но не закоммичен). Решить: если это
   действующие значения — перенести в `.env` и/или менеджер паролей; затем
   `git restore infra/.env.example`. Ни в коем случае не `git add`.
7. **consent.md / {policy_url}** — юридический P0 до боевого сбора ПДн
   (settings_store.py:463,623): владельческое действие, не код.
8. **REQUIRE_SIGNED_COMMITS выключен** — автодеплой не проверяет подпись коммитов
   (включение — owner-действие, auto-deploy.sh:27-30).

## 7. Разбор инцидента 2026-06: что объясняется кодом, а что нет

**«Бот сломался (docker был включён)»** — два кандидата по коду:
(1) второй экземпляр с тем же токеном где-то ещё → молчаливое воровство апдейтов,
без единой ошибки в логах (§3-а); (2) битый/отозванный токен → restart-loop (§3-б).
Отличить постфактум: `docker compose logs bot | grep 'BOT_TOKEN неверный'` и
`docker inspect aemr-bot-bot-1 --format '{{.RestartCount}}'`.

**«Отключил docker — поработал 5 минут и умер»** — кодом НЕ объясняется: механизма
«умереть через 5 минут» в боте нет. При выключенном docker не работает ни бот, ни
база — «поработал» мог только осиротевший процесс вне docker (без БД он падает на
первом обращении) либо это был хвост вотчдог-тика. Точный ответ — в журналах сервера:
`journalctl --since '2026-06-10' -t aemr-bot-watchdog`, `dmesg | grep -i oom`,
`last reboot`.

**«Лежит 2 дня, cron'ы отключены»** — объясняется полностью: docker выключен
(умерли restart-policy, контейнеры, база), cron'ы сняты (умерли вотчдог и
автодеплой), systemd-юнит oneshot ждёт перезагрузки и требует включённого
docker.service. Поднимать некому. Лечение — §0.

**Гипотеза по токену**: в локальной копии репо в `.env.example` лежит непустой
токен, отличающийся от того, что в `.env`. Если ты получал НОВЫЙ токен в
max.ru/business и вписал его «не туда» — бот на сервере мог остаться со старым
(отозванным) и уйти в restart-loop. Проверка: §3-б; действующее значение должно
быть в `infra/.env` НА СЕРВЕРЕ.
