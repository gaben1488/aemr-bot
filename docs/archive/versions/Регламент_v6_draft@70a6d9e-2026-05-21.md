# Регламент v6 — DRAFT для подписания распоряжением

**Статус:** черновик, не утверждено распоряжением. Подготовлено разработчиком для согласования с автором Регламента (отдел по работе с обращениями граждан + отдел цифрового развития).

**База:** утверждённый текст Регламента v5 (`файлы/Регламент_AEMR-bot_MAX_v5.docx`).

**Что меняется в v6:** только дельта относительно v5. Все пункты, не упомянутые ниже, остаются без изменений. Полный документ v6 готовится в `.docx` на основе v5 + правок из этого черновика.

---

## Источник правок

После согласования v5 в кодовую базу добавлены следующие фичи (см. PR #26, #27, #31 в `github.com/gaben1488/aemr-bot`):

- **Управление операторами через кнопочный мастер** (UI «👥 Операторы» в `/op_help` для роли `it`) — заменяет старый путь «`/whoami` в личке → psql».
- **Иерархическое меню «⚙️ Настройки бота»** (UI для роли `it`) — заменяет команду `/setting` как основной способ менять настройки.
- **Синхронизация настроек в репозиторий** через автоматический Pull Request (`services/repo_sync.py`).
- **Картинки в рассылках** — оператор может приложить до N картинок к одному сообщению рассылки.
- **Картинки в ответах оператора** — оператор может приложить одну картинку к ответу жителю.
- **Категоризированные алёрты бэкапа** — отдельные сообщения для pg_dump-fail, gpg-fail, config-fail.
- **Funnel-watchdog admin-summary** при массовых зашпиле (≥5/час).

Все фичи **расширяют** функциональность Регламента v5, не противоречат ему. Делta-правки ниже фиксируют их в нормативном тексте.

## Правки

### Глава 4 «Роли и полномочия операторов» — добавить пункт 17.1

Между текущими пунктами 17 и 18 добавить:

**17.1.** Назначение и прекращение полномочий операторов реализуется специалистом отдела цифрового развития одним из двух способов:

1) через кнопочный мастер «👥 Операторы» в панели быстрых действий `/op_help`, обеспечивающий добавление будущего оператора из участников служебной группы, изменение роли, мягкую деактивацию (с сохранением записи и журнала действий) и реактивацию;
2) через команду `/add_operators` в служебной группе — массовая регистрация по многострочному списку.

Оба способа эквивалентны по правовым последствиям и фиксируются в журнале действий (audit_log).

Деактивация единственного активного оператора с ролью `it` блокируется программно. Замена единственного `it` производится последовательностью: добавление нового `it` → деактивация прежнего.

### Глава 6 «Порядок работы оператора» — пункт 34 расширить

Текущий п. 34 (1)-(2) описывает два способа ответа жителю: свайпом и командой `/reply`. Добавить:

**34.1.** Оператор может приложить к ответу одну фотографию (визуальное пояснение, схема). Фотография направляется жителю в личном чате вместе с текстом ответа в соответствии с шаблоном Приложения 2 настоящего Регламента. Поддерживается приложение одной фотографии — Чат-бот информирует оператора, если приложено больше одной, и направляет жителю только первую.

### Глава 7 «Информационные рассылки» — пункт 47 расширить

Текущий п. 47 описывает мастер `/broadcast`: ввод текста ≤ 1000 символов, предпросмотр, подтверждение, отправка 1 сообщение в секунду. Добавить:

**47.1.** Оператор может приложить к рассылке от 1 до 20 фотографий в одном сообщении с текстом. Все приложенные фотографии направляются каждому подписчику в составе рассылки. Конкретное значение максимального количества фотографий устанавливается специалистом отдела цифрового развития через меню «⚙️ Настройки бота» → ключ `broadcast_max_images` (по умолчанию — 5).

**47.2.** Карточка предпросмотра в шаге подтверждения мастера `/broadcast` отображает счётчик приложенных фотографий и сами фотографии. При превышении установленного лимита `broadcast_max_images` Чат-бот выводит явное предупреждение «приложили N, разойдутся первые M» — лишние фотографии не направляются.

### Глава 10 «Журнал действий операторов» — пункт 68 расширить перечень действий

Дополнить перечень типов действий, фиксируемых в журнале (после пункта 68 (14) `auto_erase_pdn_retention`):

15) `operator_deactivate` — деактивация оператора через мастер «👥 Операторы»;
16) `operator_reactivate` — реактивация деактивированного оператора через мастер «👥 Операторы»;
17) `settings_repo_pr_created` — создание Pull Request на синхронизацию настроек в репозитории.

### Глава 11 «Резервное копирование и фоновые задачи» — пункт 72 уточнить

Дополнить п. 72:

При возникновении ошибки еженедельного резервного копирования Чат-бот направляет в служебную группу категоризированное уведомление с указанием стадии сбоя (создание дампа pg_dump / шифрование gpg / конфигурация). Незашифрованный дамп, оставшийся после сбоя на стадии шифрования, удаляется автоматически — Чат-бот не оставляет незашифрованных копий персональных данных на диске.

### Глава 12 «Изменение настроек и регистрация операторов» — пункт 73 уточнить

Текущая формулировка: «Изменение настраиваемых параметров Чат-бота осуществляется специалистом по цифровому развитию командой `/setting` в служебной группе либо через раздел «Настройки бота» панели быстрых действий `/op_help`».

Уточнить:

**73.** Изменение настраиваемых параметров Чат-бота осуществляется специалистом отдела цифрового развития одним из двух способов:

1) **рекомендуемый** — через иерархическое меню «⚙️ Настройки бота» панели быстрых действий `/op_help` с типизированной валидацией и предпросмотром изменений;
2) команда `/setting list` / `/setting <ключ> <значение>` в служебной группе — экспертный shortcut для тех же изменений.

После изменения настроек, влияющих на seed-файлы репозитория (URL'ы, расписания, тематики, населённые пункты, текст приёма граждан), специалист отдела цифрового развития может направить эти изменения в репозиторий через действие «🔄 Синхронизировать с репо» в том же меню. Чат-бот создаёт автоматический Pull Request в репозитории на основании текущего состояния базы данных. Прямые изменения в основную ветку (`main`) репозитория Чат-бот не осуществляет.

### Глава 12 — пункт 78 расширить (whoami)

Текущий п. 78 фиксирует: «В личной переписке с жителями команда `/whoami` не выполняется». Уточнить:

**78.** Команда `/whoami` выполняется исключительно в служебной группе. В личной переписке с жителем команда не отвечает — Чат-бот молча игнорирует её. Указанное ограничение реализовано в программном коде в целях исключения отображения внутреннего идентификатора жителю.

## Действия по согласованию

1. Автор Регламента (отдел по работе с обращениями граждан) ознакамливается с дельтой, согласовывает терминологию.
2. Юрист подтверждает, что новые формулировки не вступают в противоречие с действующим законодательством.
3. На основе v5 + дельты готовится финальный `.docx` v6.
4. Согласовывается распоряжением Администрации.
5. Запись в `docs/COMPLIANCE_WITH_REGLAMENT_v5.md` обновляется до `docs/COMPLIANCE_WITH_REGLAMENT_v6.md` с пересмотром построчной матрицы под новые требования.
