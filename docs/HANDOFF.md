## 2. Что обязательно сделать перед запуском

Это обязательные пункты до запуска. Без них запуск считаем рискованным.

1. **Заполнить `BACKUP_GPG_PASSPHRASE`** в `.env`. Это пароль для шифрования резервных копий по алгоритму GPG AES-256 (стандартный симметричный шифр). Без этого пароля локальные копии базы данных лежат как обычный текст SQL. Они содержат имена жителей, телефоны и тексты обращений. Пароль хранить в двух независимых надёжных местах. Например, в менеджере паролей администратора (1Password или KeePass) и в офлайн-копии у руководителя.
2. **Прогнать учебное восстановление из резервной копии** на тестовом стенде. Порядок действий — в [RUNBOOK.md §7](RUNBOOK.md). Резервная копия, которую ни разу не пробовали восстанавливать, — не резервная копия.
3. **Убедиться, что внутренняя самопроверка работает.** Самопроверка — это задача планировщика APScheduler (планировщик фоновых задач). Она шлёт сигнал тревоги в группу администраторов, если бот перестал отвечать. Как проверить. Выполнить `docker compose restart bot`. Дождаться истечения интервала `HEALTHCHECK_STALE_SECONDS`. Убедиться, что в группе администраторов пришло сообщение «⚠️ Бот не отвечает…». После восстановления должно прийти «✅ Бот восстановил отзывчивость». Это и есть основной мониторинг при размещении бота на собственном сервере. Внешние сервисы пингования не нужны. Адрес `/healthz` наружу не публикуется.
4. **Юристу АЕМР проверить статус оператора персональных данных** в реестре Роскомнадзора (Федеральная служба по надзору в сфере связи) по ИНН. Адрес — pd.rkn.gov.ru. Если такого статуса нет, подать уведомление с указанием новой информационной системы персональных данных (ИСПДн) «Чат-бот обратной связи в MAX». См. §3.
5. **Сменить `BOT_TOKEN`** на боевой. Боевой токен создаётся через `max.ru/business`. Через тестового бота он не выдаётся.
6. **Проверить, что сервер физически находится в РФ.** Это требование статьи 18.5 закона 152-ФЗ. При размещении на собственной площадке это означает, что серверная стойка АЕМР стоит в РФ.

## 3. Соответствие 152-ФЗ и УЗ-4: что закрыто, что нет

УЗ-4 — четвёртый уровень защищённости персональных данных по Постановлению Правительства РФ № 1119. Полный отчёт — в файле `compliance-audit-report.md`. Файл готовится агентом и хранится в текущей рабочей сессии.

**Закрыто на уровне кода:**
- ✅ Согласие на обработку (статья 9): поле `users.consent_pdn_at`. Отзыв согласия — командой `/forget`.
- ✅ Право на удаление (статья 21): команда `/forget` для жителя. Команды `/erase max_user_id=` и `/erase phone=` — для ИТ-оператора. Анонимизация затрагивает поля `first_name`, `phone`, `phone_normalized`, `consent_pdn_at`, `dialog_state`, `dialog_data`, `subscribed_broadcast`, `is_blocked`. Все изменения происходят в одной транзакции вместе с записью в `audit_log` (журнал действий).
- ✅ Меры защиты (статья 19, ПП № 1119): роли в таблице `operators`. Журнал действий `audit_log`. Шифрование резервных копий через GPG AES-256. Гарантии целостности данных PostgreSQL (свойства ACID). Идентификация по `BOT_TOKEN` и MAX `user_id`.
- ✅ Хранение в РФ (статья 18.5): задокументировано в ADR-001 §1.
- ✅ Минимизация данных (статья 5): собираем только необходимое. Имя без фамилии, телефон, адрес, тематику, текст обращения.
- ✅ Таблица `events`: автоматический срок хранения 30 дней.

**Не закрыто, ответственный — юрист АЕМР совместно с разработчиком:**

- ❌ Право на доступ (статья 14): `/whoami` показывает только `max_user_id`, `first_name` и `chat_id`. Полной выгрузки «всё, что бот хранит про меня» пока нет. Нужна команда `/mydata`. В реестре технического долга это **TD-08, P0**.
- ❌ Право на исправление (статьи 14, 21): нет команд `/setname` и `/setphone` для жителя. Реестр технического долга — **TD-09, P0**.
- ⚠️ Уведомление о цели обработки (статья 18): файл `PRIVACY.md` упоминает реагирование на сообщения. Он не покрывает рассылки, статистику и передачу третьим лицам (компании MAX как оператору связи). Требуется редакция юристом до запуска.
- ⚠️ Срок хранения данных (статья 5): таблицы `appeals`, `messages` и `audit_log` хранятся бессрочно. После закрытия обращения цель обработки исчерпана. Нужна политика анонимизации через N лет. Предлагается 5 лет. Реестр технического долга — **TD-10, P1**.
- ⚠️ Уведомление Роскомнадзора (статья 22): процедура внекодовая. Юрист АЕМР проверяет статус и подаёт дополнение.

## 4. Известные ограничения (формальный технический долг)

Полный реестр — в [DEVELOPER.md «Известные ограничения»](DEVELOPER.md). Сводка по приоритетам.

**P0 — первый месяц после запуска:**
- TD-08: команда `/mydata` для жителя (152-ФЗ статья 14).
- TD-09: команда `/setname` (152-ФЗ статьи 14, 21).
- TD-11: проверки `pip-audit` и `safety` в CI. Это поиск известных уязвимостей CVE в зависимостях.

**P1 — первый квартал:**
- TD-10: задача планировщика для анонимизации записей в `appeals` старше 5 лет.
- TD-12: регрессионные тесты на закрытые замечания. Инварианты `/erase`, проверка БД в `/healthz`, сигнал тревоги при сбое резервного копирования.
- Расширение `_diag`: один запрос с `GROUP BY` вместо восьми отдельных скаляров.

**P2 — первый год:**
- TD-01: пакетная (батчинговая) запись в `record_delivery` при рассылке. Триггер — более 1000 подписчиков.
- TD-02: словарный диспетчер для `on_callback`. Триггер — добавление пятого нового префикса коллбэка.
- TD-03: декомпозиция функции `_run_broadcast_impl`.
- `recover_stuck_funnels`: заменить `asyncio.gather` на `Semaphore(2)` или последовательный цикл с ограничителем темпа. Цель — избежать всплеска по лимитам API при крупном восстановлении.
- Корректное завершение работы: по сигналу SIGTERM делать `engine.dispose()` и отменять `_collect_timers`. Сейчас при `kill -9` соединения с БД теряются.
- Адрес Prometheus для метрик: количество обращений в минуту, p95 времени отправки сообщения, длина очереди рассылки.

**P3 — по триггеру:**
- TD-04: анонимные обработчики `_(event)` заменить на именованные функции.
- TD-05: переход с `_user_locks` на блокировки уровня сессии PostgreSQL (advisory locks). Триггер — переход на несколько реплик бота.
- TD-06: повторная загрузка PRIVACY.pdf при холодном старте.
- TD-07: вынести встроенные русские строки из `admin_commands.py` в отдельный модуль `texts.py`.

## 5. Минимум для эксплуатации

- **Логи.** Команда `docker compose logs --tail=200 -f bot`. Структурированное логирование и сквозной идентификатор запроса (trace-id) пока не настроены. Это приоритет P2. Для одного экземпляра MVP `grep` по логам справляется.
- **Проверка здоровья.** Параметр `healthcheck` в Compose каждые 30 секунд дёргает `/healthz`. Сам адрес `/healthz` теперь делает `SELECT 1` к базе. Результат кэшируется на 10 секунд, чтобы не перегружать пул соединений. Если БД зависает, ответ — HTTP 503.
- **Задачи планировщика при старте:** `db-backup` (воскресенье 03:00). `events-retention` (ежедневно 04:00). `health-selfcheck` (каждые `HEALTHCHECK_INTERVAL_MIN` минут). `monthly-stats` (1-го числа в 09:00). Проверка: `docker compose logs bot | grep "added job"`.
- **Сигналы тревоги в группе администраторов:**
  - сигнал жизни (heartbeat) задержался → «⚠️ Бот не отвечает на проверку здоровья…»
  - резервная копия не выполнилась → «⚠️ Еженедельный бэкап БД не выполнен…»

## 6. Мониторинг PostgreSQL — команды для администратора

Десять готовых команд с копированием лежат в [DEVELOPER.md] (раздел добавляется отдельно). Минимально необходимый набор:

```bash
# Размер таблиц + индексов
docker compose exec db psql -U aemr -c "SELECT schemaname||'.'||relname AS table, pg_size_pretty(pg_total_relation_size(relid)) AS total FROM pg_catalog.pg_statio_user_tables ORDER BY pg_total_relation_size(relid) DESC LIMIT 10;"

# Bloat / dead tuples
docker compose exec db psql -U aemr -c "SELECT relname, n_live_tup, n_dead_tup, last_autovacuum FROM pg_stat_user_tables ORDER BY n_dead_tup DESC LIMIT 10;"

# Активные запросы
docker compose exec db psql -U aemr -c "SELECT pid, state, wait_event, query_start, left(query,80) FROM pg_stat_activity WHERE state != 'idle';"

# Версия миграций (должно быть 0004 после деплоя)
docker compose exec bot alembic current
```

| ID | Риск | Что снижает риск | Что делать администратору |
|---|---|---|---|
| R-01 | Утечка персональных данных через незашифрованный файл резервной копии | Шифрование GPG AES-256, включается опционально | Заполнить `BACKUP_GPG_PASSPHRASE` до запуска |

## 8. Проверочный прогон (золотой путь)

Из чек-листа развёртывания:

- [ ] Житель шлёт `/start` → видит меню (5 кнопок).
- [ ] Полная воронка: имя → телефон → адрес → тема → текст → вложение → карточка падает в группу администраторов.
- [ ] Ответ свайпом оператора (жест проведения вправо по сообщению) → жителю уходит формально оформленный ответ.
- [ ] `/reply N <текст>` — запасной путь работает.
- [ ] `/broadcast` от и до. Плюс отмена в середине процесса.
- [ ] `/stats today | week | month` отдаёт XLSX-файл.
- [ ] `/erase phone=+7XXX` выполняет анонимизацию и пишет запись в `audit_log`.
- [ ] `/diag` отдаёт ожидаемые счётчики.
- [ ] `/backup` создаёт копию в `/backups/` с правами `chmod 0600`. Шифрует её, если задан пароль.

Критерий приёмки: все пункты ✓ плюс 24 часа без падений на тестовом стенде.
