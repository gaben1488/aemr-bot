Раздел 01

Резюме проекта в одном экране

То, что нужно понять за две минуты, прежде чем начать читать дальше.

Что это. Чат-бот в мессенджере MAX (отечественная платформа VK), принимающий обращения граждан Елизовского муниципального округа. Жители пишут боту в личке, операторы Администрации отвечают из служебной группы MAX через свайп-реплай или команду /reply. Бот работает по образцу бота «Солодов. Обратная связь» Правительства Камчатского края.

Как развёрнут. Два Docker-контейнера на собственном VPS заказчика (Ubuntu 20.04/22.04 LTS): bot (Python 3.12 + maxapi) и db (PostgreSQL 16-alpine). Docker Compose v2, named-volumes для данных БД и для еженедельных pg_dump. Облачные сервисы не требуются.

Режим связи. По умолчанию long polling — бот сам опрашивает MAX. Не требует домена, открытых входящих портов, TLS-сертификата. Альтернативный режим webhook (профиль Compose webhook) включается отдельно — поднимает nginx + certbot для HTTPS-приёма событий.

Состояние. Production-проект с CI на GitHub Actions, 11 миграциями Alembic, 49 тестами pytest (покрытие ≥ 65%), автоматическим деплоем через cron + deploy-key, внешним watchdog'ом и health-gate с автооткатом.

Главный принцип эксплуатации. Бот рассчитан на размещение на инфраструктуре заказчика. Облачные провайдеры намеренно не используются. Все ПДн жителей остаются на собственном сервере АЕМО.

Контейнер базы данных

Запускается с явным shared_preload_libraries=pg_stat_statements и pg_stat_statements.track=all. Это обязательно, иначе миграция 0010_pg_ops_hardening применяется, но статистика не собирается. Capabilities контейнера сжаты до минимально необходимых: CHOWN, DAC_READ_SEARCH, FOWNER, SETGID, SETUID; всё остальное сброшено через cap_drop: ALL.

Различение жителя и оператора

Бот определяет роль чата по идентификатору chat_id, который приходит в каждом событии MAX. Сообщения из чата, у которого chat_id == ADMIN_GROUP_ID, попадают в operator-handlers. Любой другой — это житель в личке. Поэтому корректное значение ADMIN_GROUP_ID в .env — обязательное условие работы. До его задания бот работает только в личке.

Что внутри контейнера bot

  • main.py — точка входа. Preflight-проверка токена, очистка /-меню (MAX не умеет per-scope команды), bootstrap первого ИТ-оператора из .env, hydrate wizard-state из БД, запуск scheduler'а, выбор polling/webhook.
  • handlers/ — 15 модулей: start, menu, appeal_funnel, appeal_geo, operator_reply, broadcast, admin_commands, admin_operators, admin_settings, admin_stats, admin_audience, admin_panel, admin_appeal_ops, callback_router, _auth.
  • services/ — 19 модулей бизнес-логики: appeals, users, operators, broadcasts, uploads, geo (point-in-polygon по 10 поселениям), db_backup, policy, cron, idempotency, progress, stats, calendar_ru, wizard_registry, wizard_persist, admin_relay, admin_events, card_format, settings_store.
  • db/ — модели SQLAlchemy, async-session, Alembic-миграции 0001 → 0011.
  • health.py — мини-aiohttp на :8080 с эндпоинтами /livez, /readyz, /healthz.

Идемпотентность

Каждое входящее событие MAX получает idempotency_key и записывается в таблицу events. Повторная доставка того же события (бывает при сетевых сбоях MAX) распознаётся по уникальному индексу и пропускается. Таблица events чистится cron-задачей раз в сутки (хранение 30 дней).

Раздел 05

Системные требования к серверу

Минимальная конфигурация, проверенная в production.

Минимум

ПараметрЗначениеКомментарий

ДоступSSH + sudoКлючевая аутентификация. Парольная — отключить
Жёсткий лимит контейнера
В docker-compose.yml для контейнера бота установлен mem_limit: 512m. При стабильном превышении контейнер OOM-перезапускается без падения хоста. Это намеренно — runaway-рассылка или патологический FSM не должны уронить машину.

Рекомендации

  • Размещение — собственный сервер заказчика. Облачные провайдеры не используются и не рекомендуются. Это требование 152-ФЗ к локализации ПДн.
  • Российский ЦОД с прозрачной собственностью. Площадка с лицензией ФСТЭК на ТЗКИ — желательно, но не обязательно (УЗ-4 не требует аттестации провайдера).
  • Раздельные пользователи: aemr — владелец репозитория и docker-group, root — только для cron-задач deploy/healthwatch.
  • Сторонний бэкап (например, rsync на отдельный хост) еженедельно. Локальный pg_dump защищает от логической ошибки в БД, но не от потери VPS целиком.

Брандмауэр (UFW)

Минимальный набор правил для режима polling:

sudo ufw default deny incoming
sudo ufw default allow outgoing
sudo ufw allow 22/tcp
sudo ufw enable

В режиме webhook добавить:

sudo ufw allow 80/tcp
sudo ufw allow 443/tcp
Важно
Порт 8080 бота биндится через docker compose как 127.0.0.1:8080:8080 — снаружи сервера он недоступен даже без UFW. Это намеренно: операционная диагностика (полные ответы /livez с heartbeat и DB-status) отдаётся только локальным запросам.

Раздел 11

Безопасность и соответствие 152-ФЗ

Уровень защищённости УЗ-4 по ПП РФ № 1119. Иные ПДн, третий тип угроз, менее 100 000 субъектов, не сотрудники.

Категория ИСПДн

УЗ-4 — четвёртый, самый низкий уровень защищённости. Установлен по совокупности признаков:

  • Иные категории ПДн (не специальные, не биометрические)
  • Третий тип актуальных угроз (отсутствуют недокументированные возможности в системном и прикладном ПО)
  • Менее 100 000 субъектов
  • Субъекты — не сотрудники оператора (граждане-заявители)

Аттестация провайдера по ФСТЭК не требуется. Достаточно соответствия типовому набору организационных и технических мер.

Какие ПДн собираются

Только то, что необходимо для отправки обратной связи:

  • Имя — вводит житель в воронке
  • Телефон — передаётся через системный диалог MAX «Поделиться контактом»
  • max_user_id — идентификатор в мессенджере
  • Адрес и населённый пункт — вводится в обращении
  • Текст обращения и вложения — фото, видео, геолокация

Согласия — три отдельных правовых сценария

Журналирование

Таблица audit_log хранит каждое действие оператора: ответ жителю, переоткрытие/закрытие обращения, удаление ПДн, изменение настройки, регистрация другого оператора, инициирование рассылки. Поля: кто (operator_max_user_id), что (action), к чему (target), детали в JSONB.

Защита токенов и секретов

  • .env с правами chmod 600, владелец — пользователь, запускающий compose
  • В git исключён (правило в .gitignore: .env, .env.*, !.env.example)
  • Скрипт scripts/audit_vps.sh при сборе диагностики маскирует BOT_TOKEN, POSTGRES_PASSWORD, BACKUP_GPG_PASSPHRASE, BACKUP_S3_*, WEBHOOK_SECRET, заголовки Authorization
  • Бэкапы в production обязательно шифровать через BACKUP_GPG_PASSPHRASE (32+ символа). Дамп лежит на диске в зашифрованном виде
  • В webhook-режиме приём событий защищён HMAC-сравнением заголовка X-Max-Secret (защита от timing-oracle через hmac.compare_digest)

Что важно знать ИБ-специалисту

  • Сервер размещается на инфраструктуре заказчика. ПДн физически находятся на серверах АЕМО.
  • Внешние соединения только к Bot API MAX (botapi.max.ru, files.max.ru) и опционально к S3-эндпоинту для бэкапа.
  • Сертификат КЭП/МЧД для платформы не требуется — Bot API работает по обычному токену.
  • Регистрация в реестре операторов ПДн (Роскомнадзор) — отдельная процедура, выполняется юристом АЕМО.
  • СКЗИ для УЗ-4 не обязательно. При повышении уровня (если число субъектов превысит 100 000) — потребуется пересмотр.
Раздел 12

Защита контейнеров и хостовый hardening

Меры применены в коде. Системному администратору — знать и не отменять.

Контейнер бота

  • read_only: true — корневая файловая система контейнера неизменяема
  • tmpfs: /tmp:size=128m — единственное место для записи временных файлов
  • mem_limit: 512m, memswap_limit: 512m — runaway-рассылка не уронит хост
  • pids_limit: 200 — защита от fork-бомб со стороны pg_dump, gpg, rclone
  • cap_drop: ALL — сброшены все Linux capabilities
  • security_opt: no-new-privileges:true — никаких setuid-эскалаций
  • Запуск под non-root пользователем botuser (UID/GID 1000)
  • Логи docker ограничены: max-size: 10m, max-file: 3 — суммарно до 30 МБ; ротация автоматическая

Контейнер БД

  • cap_drop: ALL + минимальный набор: CHOWN, DAC_READ_SEARCH, FOWNER, SETGID, SETUID — только для chown'а data-каталога при первом старте
  • security_opt: no-new-privileges:true
  • Логи docker: те же лимиты, что у бота
  • Не публикуется наружу — только во внутренней сети Compose

Образы закреплены по digest

В docker-compose.yml и Dockerfile базовые образы указаны не тегом, а полным tag@sha256:...:

  • postgres:16-alpine@sha256:4e6e670bb069...
  • python:3.12-slim@sha256:4386a385d81d...

Это значит, что docker compose pull не утянет в обход контроля версии с поломанными init-скриптами или CVE. Обновление digest — ручная процедура с прогоном smoke-теста.

Хост

  • UFW в режиме default deny incoming
  • SSH — ключевая аутентификация, парольная отключена
  • Рекомендуется fail2ban для SSH
  • Регулярный apt upgrade -y для безопасных пакетов (auto-upgrades или ручная процедура)
  • Отдельный сервисный пользователь aemr в docker-group; root — только для cron-задач

CI-проверки безопасности

На каждый PR прогоняются:

  • ruff — линтер
  • mypy — статическая типизация
  • bandit с уровнем medium и выше — поиск известных небезопасных паттернов Python
  • pip-audit в strict-режиме — поиск CVE в зависимостях. Hard fail — PR не сольётся при наличии уязвимости
  • shellcheck — проверка bash-скриптов
Раздел 13

Health-check, мониторинг, логи

Три эндпоинта с разной семантикой и внешний watchdog с двухуровневым алёртом.

Endpoint'ы (порт 8080, биндинг 127.0.0.1)

EndpointСемантикаКто использует
/livezLiveness: жив asyncio-loop и heartbeat свежий. БД не проверяетсяDocker healthcheck, healthwatch.sh, auto-deploy.sh health-gate
/readyzReadiness: /livez + SELECT 1 к Postgres. Кэш 10 секРучная диагностика, будущий балансировщик
/healthzBackward-compatible alias для /readyzСтарые runbook-команды
Почему рестарт только по /livez
Краткая проблема Postgres или Docker DNS не должна перезапускать живой polling-процесс. /readyz только диагностирует, не управляет рестартом. Авто-rollback в auto-deploy.sh тоже смотрит только на /livez.

Heartbeat

Внутри бота фоновая задача обновляет last_beat каждые HEALTHCHECK_PULSE_SECONDS (по умолч. 30 сек). Heartbeat считается свежим, если возраст ≤ HEALTHCHECK_STALE_SECONDS (по умолч. 120 сек). При зависании asyncio-loop heartbeat не обновится — /livez вернёт 503.

Внешний watchdog (scripts/healthwatch.sh)

Запускается из cron каждые 5 минут. Алгоритм:

  1. Проверяет /livez. Если ок — счётчик провалов сбрасывается.
  2. После 3 провалов подряд (≈ 15 минут недоступности) делает docker compose restart bot.
  3. После 8 провалов (≈ 40 минут) — пишет в служебную группу MAX через прямой curl к Bot API: «бот не отвечает, авторестарт не помог».

Установка однократная:

sudo bash /home/aemr/aemr-bot/scripts/install-healthwatch.sh

Логи watchdog'а:

journalctl -t aemr-bot-watchdog -n 50

Внутренние алёрты в служебную группу

  • health-selfcheck — каждые 5 минут проверяет heartbeat. При переходе fresh→stale пишет «⚠️ бот молчит». При обратном переходе — «✅ восстановлено».
  • pulse — каждые 30 минут в рабочее время и раз в час по воскресеньям пишет «бот жив».
  • backup — упавший pg_dump шлёт громкий алёрт в служебную группу.

Логи

Docker json-driver с ротацией:

docker compose logs --tail 200 -f bot          # последние строки, follow
docker compose logs --tail 100 bot | grep -i error    # фильтр ошибок
docker compose logs --since 1h bot             # за последний час

Физическое расположение: /var/lib/docker/containers/<id>/*.log. Совокупный лимит — 30 МБ на контейнер (10 МБ × 3 файла).

Раздел 14

Резервное копирование и восстановление

Еженедельный pg_dump, опциональное GPG-шифрование, опциональная копия в S3.

Что происходит автоматически

Каждое воскресенье в 03:00 по Камчатке APScheduler запускает pg_dump:

  1. Дамп пишется в файл вида aemr-YYYYMMDD-HHMMSS.sql в /backups внутри контейнера
  2. Если задан BACKUP_GPG_PASSPHRASE — дамп шифруется (gpg --symmetric) до записи на диск. Расширение .sql.gpg
  3. Если заполнены BACKUP_S3_* — параллельно копируется в S3-бакет через rclone
  4. Старые файлы сверх BACKUP_KEEP_COUNT удаляются (по mtime, сортировка убыванием)
  5. Результат логируется. При падении — алёрт в служебную группу

Где лежат файлы

На хосте — в named-volume Docker: /var/lib/docker/volumes/infra_backups/_data/. Том переживает пересборку контейнера и обновления образа.

Просмотр

docker compose exec bot ls -lh /backups
docker compose exec bot cat /backups/last-backup.log

Ручной запуск бэкапа

В служебной группе ИТ-оператор шлёт команду /backup. Бот запускает pg_dump на лету.

Восстановление

Полная процедура — в docs/BACKUP_RESTORE_TEST.md. Краткий путь:

# 1. Скопировать файл бэкапа в работу
docker cp /var/lib/docker/volumes/infra_backups/_data/aemr-YYYYMMDD.sql.gpg \
    $(docker compose ps -q db):/tmp/restore.sql.gpg

# 2. (Если зашифровано) расшифровать
docker compose exec db sh -c \
    "gpg --batch --passphrase '$BACKUP_GPG_PASSPHRASE' \
         --decrypt /tmp/restore.sql.gpg > /tmp/restore.sql"

# 3. Восстановить
docker compose stop bot
docker compose exec db psql -U aemr -d postgres -c "DROP DATABASE aemr;"
docker compose exec db psql -U aemr -d postgres -c "CREATE DATABASE aemr;"
docker compose exec db psql -U aemr -d aemr -f /tmp/restore.sql
docker compose start bot
Защищает только от логической ошибки в БД
Локальный бэкап в named-volume лежит на той же VPS. Полная потеря сервера → потеря бэкапов. Внешний бэкап — отдельный rsync или scp по cron на изолированный хост — обязателен для production. Проектируется и поддерживается ИТ-отделом отдельно.
Раздел 15

Обновление, авто-deploy и откат

Три варианта: ручное обновление, автоматический deploy по cron, откат при сбое.

Ручное обновление

cd /opt/aemr-bot   # или /home/aemr/aemr-bot
git fetch origin main
git pull --ff-only origin main
cd infra
docker compose up -d --build
docker compose logs --tail 100 -f bot

Alembic мигрирует БД сам при старте контейнера. Если миграция падает — контейнер уходит в restart-loop; смотрите логи с фильтром по alembic.

Автоматический deploy (scripts/auto-deploy.sh)

Cron-задача от root каждые 10 минут:

  1. git fetch origin main с deploy-ключом
  2. Если есть новые коммиты — git reset --hard origin/main
  3. cp .env в безопасное место перед reset (хотя git его не трогает)
  4. docker compose up -d --build от пользователя aemr
  5. Health-gate: ждёт до 60 секунд ответа от /livez
  6. Если /livez не отвечает за это время — автоматический rollback на предыдущий коммит и пересборка

Установка однократная (требует deploy-key в GitHub):

sudo bash /home/aemr/aemr-bot/scripts/install-auto-deploy.sh

Откат вручную

Полная процедура — в docs/ROLLBACK.md. Короткий путь:

cd /home/aemr/aemr-bot
git log --oneline -10
git reset --hard <prev_commit_sha>
cd infra
docker compose up -d --build

Откат на «много миграций назад» делать через alembic downgrade <target_revision>, потом git-checkout и rebuild. Round-trip downgrade base ↔ upgrade head проверяется в CI.

Раздел 16

Диагностика типовых сбоев

Что смотреть в первую очередь, когда что-то не так.

Контейнер в restart-loop

  1. docker compose ps — посмотреть статус
  2. docker compose logs --tail 200 bot — последние 200 строк
  3. Частые причины: битый BOT_TOKEN (preflight завершает процесс с явным сообщением), сломанная миграция (фильтр | grep -A20 alembic), OOM (превышен mem_limit: 512m)

Бот не отвечает в личке

  • curl http://127.0.0.1:8080/livez — если 503, бот завис
  • curl http://127.0.0.1:8080/readyz — если 503 при ok-livez, проблема с БД
  • docker compose ps — обе службы healthy?
  • Логи: docker compose logs --tail 100 bot | grep -i error

Бот в группе игнорирует сообщения

  • ADMIN_GROUP_ID в .env совпадает с тем, что бот вернул на /whoami в группе?
  • Знак минуса в начале на месте?
  • Бот перезапущен после правки .env? docker compose up -d --force-recreate bot
  • Бот не покинул группу, не был исключён?

Оператор пишет, бот не доставляет

  • Ответ оператора — это свайп-реплай на карточку обращения? Без свайпа бот не понимает, к какому обращению ответ
  • Длина ответа ≤ ANSWER_MAX_CHARS (300 по умолч.)?
  • Оператор зарегистрирован в operators с is_active=true?
  • docker compose logs bot --tail 100 | grep -i error

Безопасный сбор диагностики

bash /home/aemr/aemr-bot/scripts/audit_vps.sh
# Создаёт /tmp/aemr_audit_TIMESTAMP.tar.gz
# Все секреты маскируются ***HIDDEN*** перед записью

Файл можно безопасно передать разработчику без риска утечки токенов.

Раздел 19

Ограничения и известные риски

То, что важно знать до того, как что-то сломается.

Платформенные ограничения MAX

  • 2 RPS на исходящие сообщения от бота. BROADCAST_RATE_LIMIT_PER_SEC=1.0 намеренно ниже — оставляет запас под обычную активность (ответы, карточки)
  • Боты публикуются только верифицированными юрлицами РФ. ИП, самозанятые, физлица — нет
  • На одну организацию — максимум 5 ботов
  • API нет аналога BotCommandScopeChat Telegram — нельзя показывать разные команды жителям и операторам. Поэтому /-меню очищено для всех
  • Документация MAX расходится с реальным поведением API в нюансах. maxapi-библиотека — сообщественная, не официальная. Закладывать время на отладку при обновлении

Хранилище и масштаб

  • При числе субъектов ПДн > 100 000 уровень защищённости поднимется на ступень — потребуется пересмотр организационных и технических мер
  • БД на одной VPS без репликации. Высокая доступность (HA) — не реализована и не планировалась для МVP
  • Локальные бэкапы на той же машине — не защита от потери VPS. Внешний бэкап на изолированный хост обязателен (отдельная процедура ИТ-отдела)

Архитектурные напоминания

  • APScheduler работает внутри процесса бота. При рестарте задача, которая шла в момент рестарта, не возобновится автоматически. Для рассылок есть reap_orphaned_sending(), который помечает SENDING-рассылки как FAILED после рестарта
  • В режиме polling: входящие порты вообще не нужны — бот сам опрашивает MAX. Это значительно упрощает безопасность периметра
  • В режиме webhook: появляются 80/443, nginx, certbot, домен. Безопасность периметра становится сложнее. Используется только если требуется по причинам производительности или политики

Терминологические несостыковки

  • Пакет Python называется aemr_bot, репозиторий — aemr-bot. Сокращение АЕМР осталось от старого имени муниципалитета («Елизовский муниципальный район»). Сейчас правильное название — АЕМО (Елизовский муниципальный округ). Технические идентификаторы менять нельзя — это сломает миграции и историю

Зависимости от внешних сервисов

  • Bot API MAX — единственная критическая внешняя зависимость. Падение MAX → бот не работает
  • Docker Hub — для pull базовых образов. Кеш сборки на сервере смягчает
  • GitHub — для авто-deploy и pull исходников. Ручной deploy возможен с локального клона
  • PyPI — только при пересборке образа
