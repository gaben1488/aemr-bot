# Регламент работы координатора и ИТ-специалиста

Документ — пошаговая инструкция для людей в служебной группе MAX «Администрация ЕМР. Обратная связь». Открывается через `cat docs/RUNBOOK.md` или прямо с GitHub.

В тексте служебная группа АЕМР называется именно так: **служебная группа**. В переменных окружения и коде ей соответствует `ADMIN_GROUP_ID`. Это одно и то же.

> **Полный справочник по командам бота** — отдельный файл [BOT_COMMANDS.md](BOT_COMMANDS.md). Там собраны все команды и кнопки в MAX, кто может ими пользоваться, лимиты, запись в журнал действий, граничные случаи. RUNBOOK описывает рабочие сценарии. BOT_COMMANDS описывает синтаксис и точную семантику.

## Кто чем занимается

**Координатор АЕМР (специалист по ответам).** Отвечает жителям. Передаёт обращения смежным специалистам. Следит за сроком ответа в 4 рабочих часа. Срок ответа — это SLA (Service Level Agreement, целевой срок ответа), зафиксированный в регламенте. Команды координатора: `/reply`, `/stats`, `/reopen`, `/close`, `/diag`, `/op_help`, `/broadcast`, `/whoami`, `/help`.

**ИТ-специалист.** Настраивает бот. Добавляет операторов. Удаляет персональные данные (далее — ПДн) по запросу жителя. Снимает резервные копии базы. Разбирает сбои. У ИТ-специалиста есть всё, что у координатора, плюс «опасные» команды: `/erase`, `/setting`, `/add_operators`, `/backup`.

## Ежедневная работа координатора

### 1. Обработать новое обращение

Когда житель пишет в бота, в служебную группу приходит карточка вида:

```
📨 Новое обращение #107
👤 Имя: Алексей
📞 Телефон: 89964240723
──────────
🏠 Адрес: г. Елизово, ул. Ленина, д. 13
🏷️ Тематика: Дороги
📝 Суть:
<текст обращения>
```

**Способ 1. Ответ свайпом.** Свайп — это жест проведения вправо по сообщению. На карточке нажмите долгим тапом и выберите «Ответить». Либо проведите по карточке вправо, если ваш клиент MAX поддерживает такой жест. Напишите ответ. Лимит — 300 символов. После отправки бот доставит ответ жителю в личку. В группу придёт подтверждение «✉️ Ответ ушёл жителю. Обращение #107 закрыто».

В тексте ниже эта связь сообщения и его ответа называется **reply-связь**. В служебных полях MAX она хранится как `link.message.mid` — идентификатор сообщения, на которое отвечают.

**Способ 2. Команда `/reply`.** Если клиент MAX не передаёт reply-связь (зависит от версии и платформы), используйте команду:

```
/reply 107 Здравствуйте! Ваш вопрос рассмотрен, проблема устранена.
```

Доставка такая же. Запись в `messages` создаётся. В журнал действий (таблица `audit_log`) пишется отметка `reply_via_command`. Команда удобна тем, что номер обращения виден в строке. Реже путаются с номерами.

**Важно.** Ответ свайпом или `/reply` должен попасть в существующее обращение. Если просто написать в группу без свайпа — бот промолчит. Житель ничего не получит. Если указать `/reply` с несуществующим номером — бот ответит «Обращение #N не найдено».

**Если ответ длиннее 300 символов** — бот напишет в группу «Ответ длиннее 300 символов (фактически: …)». Жителю ничего не уходит. Сократите и повторите.

**Если житель заблокировал бота** — придёт сообщение «Не удалось доставить ответ жителю по обращению #N: …». Обращение остаётся в работе. Свяжитесь с жителем по телефону из карточки.

### 1а. Быстрые действия и панель `/op_help`

В служебной группе доступна команда `/op_help`. Бот присылает закрепляемое сообщение с тремя кнопками:

- **📊 Статистика за сегодня** — то же, что `/stats today`. Команду набирать не нужно.
- **📢 Сделать рассылку** — запускает мастер `/broadcast`.
- **📋 Все команды** — расширенная подсказка с напоминанием формата `/reply`, `/stats`, `/reopen`, `/close`.

Панель удобна тем, кто редко работает с ботом. Команды запоминать не надо: нажал кнопку — получил действие. Закрепите ответ бота вверху группы, чтобы он был под рукой.

### 2. Передать обращение специалисту ЕГП

Сейчас архитектура простая: одна общая группа для координатора и ЕГП (Елизовское городское поселение). Передача идёт так:

1. В группе скажите голосом или текстом «Передаю на ЕГП», упомяните `@username` ЕГП-специалиста.
2. ЕГП-специалист отвечает свайпом на ту же исходную карточку.
3. Бот доставит ответ жителю. В журнале действий появится запись с пометкой ЕГП.

Разделение на отдельные служебные группы по подразделениям не реализовано. Всё идёт в одну общую группу. Это одно из направлений развития ([ADR-001 §11](ADR-001-architecture.md)).

### 3. Реоткрыть закрытое обращение

Если житель написал ещё раз после ответа, бот сам реоткрывает обращение. В группу приходит сообщение-дополнение со ссылкой на исходную карточку. **Никаких ручных действий не нужно.**

Если хотите принудительно вернуть закрытое обращение в работу:

```
/reopen 107
```

### 4. Закрыть обращение без ответа

Спам, дубль, нерелевантное:

```
/close 107
```

Жителю ничего не уходит. В журнале действий запишется кто и когда закрыл.

### 5. Выгрузить статистику

```
/stats today        — за сегодня
/stats week         — за 7 дней
/stats month        — за 30 дней
```

Бот пришлёт XLSX-файл с одиннадцатью колонками: №, дата, имя, телефон, адрес, тематика, суть, статус, ответ оператора, время ответа в часах, в срок (да/нет).

Раз в месяц 1-го числа в 9:00 по Камчатке отчёт за прошлый месяц приходит автоматически. Запрашивать отдельно не нужно.

## Действия ИТ-специалиста

### 1. Удалить ПДн жителя по запросу (ст. 21 закона 152-ФЗ «О персональных данных»)

Команда принимает один из двух идентификаторов жителя. `max_user_id` — идентификатор пользователя в MAX:

```
/erase max_user_id=165729385
/erase phone=+79964240723
```

`max_user_id` обычно есть в карточке обращения. Если житель сам обращается с просьбой «удалите мои данные» — попросите его написать `/whoami` боту в личке. Если ID назвать не получается, ищите по телефону. Бот поднимет жителя по `users.phone`. Нормализация цифр идёт в `users.find_by_phone`.

После команды:

- `users.first_name` → «Удалено»
- `users.phone` → NULL
- `users.consent_pdn_at` → NULL (отзыв согласия)
- `users.dialog_state` → `idle`, `dialog_data` → `{}` (стирание состояния воронки)
- `users.subscribed_broadcast` → false, `users.is_blocked` → true (исключает из рассылок: после отзыва согласия мы не имеем права отправлять жителю что-либо)
- В журнал действий пишется запись: кто инициировал, когда, по какому идентификатору

**Что не удаляется и почему.** Запись в `users` остаётся. Она нужна, чтобы при возвращении жителя в бот не плодить дубль и чтобы старые `appeals` не висели с битым FK. Текст обращений и ответов в `messages` тоже остаётся. Если житель уже подал претензию в Роскомнадзор и аудит спрашивает «что вы ему ответили» — без истории нечего показать. Это согласовано с юристом АЕМР и описано в политике обработки ПДн.

### 1а. Сделать рассылку подписавшимся жителям

Доступно ролям `coordinator` и `it`.

В служебной группе пишете:

```
/broadcast
```

Бот спрашивает текст одним сообщением (лимит 1000 символов). Пишете текст. Бот показывает предпросмотр «как увидит житель» и спрашивает «Разослать N подписчикам?» с кнопками `✅ Разослать` и `❌ Отмена`.

После нажатия `✅` бот в служебной группе постит сообщение с прогресс-баром «⏳ X/N доставлено» и кнопкой `⛔ Экстренно остановить`. Прогресс обновляется каждые пять секунд. По завершении бот пришлёт итог «Доставлено: X из N. Не доставлено: Y.»

Кнопка `⛔ Экстренно остановить` доступна **любому оператору в группе** на случай, если рассылку запустили по ошибке. По нажатию рассылка прерывается на следующем шаге. Итог фиксирует, сколько успело отправиться.

`/broadcast list` — последние десять рассылок со статусами и счётчиками.

Подписчиков ведёт сам житель. Командой `/subscribe` или кнопкой «🔔 Подписаться на новости» в подменю «Полезная информация». Заблокированные ботом и анонимизированные через `/erase` жители из получателей исключаются автоматически.

### 2. Зарегистрировать нового оператора

Сначала пусть человек напишет боту в личке `/whoami`. Бот пришлёт его `max_user_id`.

Далее в служебной группе вызывает ИТ-специалист. Только роль `it` имеет доступ к этой команде. Координатор делегирует:

```
/add_operators
123456 coordinator Иванова Анна Петровна
789012 aemr Петров Пётр Иванович
345678 it Сидоров Сергей Сергеевич
```

Команда принимает несколько строк за один вызов. Формат строки: `<max_user_id> <role> <ФИО>`. Бот ответит сводкой «добавлено / обновлено / ошибки». Каждое действие запишется в журнал.

Доступные роли:

- `coordinator` — координатор АЕМР. Может всё кроме `/erase`, `/setting`, `/add_operators`, `/backup`.
- `aemr` — рядовой специалист АЕМР по ответам.
- `egp` — специалист ЕГП по ответам.
- `it` — ИТ. Может всё, включая удаление ПДн и настройки.

После регистрации добавьте человека в служебную группу через клиент MAX.

Прямой `psql ... INSERT` остаётся как запасной путь на случай, если бот недоступен:

```bash
ssh user@feedback.elizovomr.ru
docker compose exec db psql -U aemr -d aemr -c \
  "INSERT INTO operators (max_user_id, full_name, role, is_active) VALUES (<id>, '<ФИО>', '<роль>', true);"
```

### 3. Изменить настройки без выкладки новой версии

```
/setting list                                 — список доступных ключей
/setting policy_url https://elizovomr.ru/...  — ссылка на политику ПДн
/setting electronic_reception_url https://... — электронная приёмная
/setting udth_schedule_url https://...        — расписание УДТХ
/setting appointment_text "Приём по записи..." — текст приёма граждан
/setting emergency_contacts [{"name":"112","phone":"112"}, ...]  — экстренные службы
/setting topics ["Дороги", "Мусор", ...]      — тематики обращений
/setting welcome_text "Здравствуйте! ..."     — приветствие
/setting consent_text "..."                   — текст согласия
```

JSON разбирается. Валидатор отбрасывает мусор: пустые ссылки, не-список для тематик и тому подобное. Если бот ответил «⚠️ Настройка не обновлена: …» — посмотрите причину и поправьте формат.

### 4. Диагностика

```
/diag
```

Возвращает:

- количество пользователей и обращений в БД
- метку времени последнего полученного события от MAX
- режим работы (длинный опрос или webhook)
- лимит длины ответа
- целевой срок ответа

### 5. Что делать, если бот молчит

**Алерт уже был** в группе («⚠️ Бот не отвечает на проверку здоровья…»):

1. По SSH на сервер:
   ```bash
   ssh user@feedback.elizovomr.ru
   cd ~/aemr-bot/infra
   docker compose ps
   docker compose logs --tail 200 bot
   ```
2. Контейнер упал — `docker compose up -d bot`.
3. Контейнер живой, бот молчит — `docker compose restart bot`.
4. БД недоступна — `docker compose logs --tail 100 db`, затем `docker compose restart db`, затем `docker compose restart bot`.
5. После перезапуска проверьте `curl http://localhost:8080/healthz`. Должно вернуться `{"ok": true}`.
6. Когда бот восстановится, в группу автоматически придёт «✅ Бот восстановил отзывчивость».

**Алерта не было, но обращения не приходят.** Возможен сбой со стороны MAX. Сначала проверьте, можете ли вы сами написать боту в личке. Если нет — либо токен отозван, либо MAX недоступен.

### 6. Выкладка новой версии бота

```bash
ssh user@feedback.elizovomr.ru
cd ~/aemr-bot
git pull
cd infra
docker compose up -d --build bot
docker compose exec bot alembic upgrade head    # если были миграции (изменения схемы БД с версионностью)
docker compose logs -f bot                       # убедиться что стартовал
```

Откат, если что-то сломалось:

```bash
git log --oneline -5
git checkout <предыдущий-коммит>
cd infra
docker compose up -d --build bot
```

Перед каждой выкладкой ставьте тег с предыдущей рабочей версией: `git tag -a release-2026-04-29 -m "..."`.

### 7. Работа с резервными копиями

> Все CLI-команды этого раздела с поясняющими комментариями и расширенный набор сценариев восстановления — в едином файле [COMMANDS.md](COMMANDS.md), разделы §5 «Резервное копирование» и §6 «Восстановление из бэкапа». Здесь — рабочий минимум.

**Расписание.** Каждое воскресенье в 03:00 по Камчатке APScheduler (планировщик фоновых задач) делает `pg_dump` и сохраняет файл в именованный том `backups`. Внутри контейнера путь — `/backups/aemr-YYYYMMDD_HHMMSS.sql`. Хранятся последние 8 файлов, примерно 2 месяца истории. Ротация автоматическая. Если задан `BACKUP_GPG_PASSPHRASE`, файлы шифруются GPG AES256 (`*.sql.gpg`). Без него — обычный SQL. Если дополнительно заданы `BACKUP_S3_*`, копия льётся в хранилище, совместимое с S3. Для self-hosted без облака переменные оставляйте пустыми.

**Снять резервную копию вручную.** В служебной группе ИТ-специалист пишет:

```
/backup
```

Бот вызывает `pg_dump` и отвечает «✅ Бэкап готов: aemr-20260504_113000.sql (12 КБ).» Полезно перед миграциями БД и перед удалением тестовых данных.

**Где лежат файлы.**

```bash
# Список бэкапов в именованном томе:
docker compose exec bot ls -lh /backups/

# Размер тома:
docker volume inspect infra_backups
```

**Скопировать бэкап на хост.**

```powershell
# В корне проекта. Создаёт папку backups\ на хосте.
docker compose -f infra\docker-compose.yml cp bot:/backups/aemr-20260504_113000.sql.gpg .\backups\
```

**Восстановление из резервной копии.**

```powershell
# Если обычный SQL:
docker compose -f infra\docker-compose.yml down
docker volume rm infra_db_data
docker compose -f infra\docker-compose.yml up -d db
Start-Sleep -Seconds 10
Get-Content backups\aemr-20260504_113000.sql | docker compose -f infra\docker-compose.yml exec -T db psql -U aemr -d aemr
docker compose -f infra\docker-compose.yml up -d bot
```

```bash
# Если зашифрован GPG:
gpg --batch --passphrase "$BACKUP_GPG_PASSPHRASE" \
    --decrypt backups/aemr-20260504_113000.sql.gpg > backups/aemr.sql
# Дальше как с обычным SQL выше.
```

Перед боевым восстановлением **обязательно тренируйте процедуру** на тестовом стенде. Потеря данных при ошибке восстановления страшнее любого инцидента.

**Если сервер был выключен в момент еженедельного запуска** — APScheduler с in-memory jobstore не наверстывает пропущенные задачи. На следующее воскресенье цикл продолжится. На критических периодах (после выкладки, перед миграцией) делайте `/backup` руками.

### 8. Сбросить тестовые данные перед запуском в продакшен

Между проверочными прогонами (или после длительной отладки на стенде) полезно очистить БД от мусора. Операторы и редактируемые настройки остаются. Готовый скрипт — `scripts/reset_test_data.sql`. Он стирает все обращения, сообщения, рассылки, idempotency-логи, журнал действий и тестовых пользователей. Не трогает таблицы `operators` и `settings`.

```powershell
# Из корня проекта:
Get-Content scripts\reset_test_data.sql | docker compose -f infra\docker-compose.yml exec -T db psql -U aemr -d aemr
```

После завершения скрипт сам выводит итоговое количество строк по каждой таблице. Операторы и настройки на месте, остальное в нулях.

**Когда НЕ запускать.** В продакшене. После публикации жителям эта команда удалит реальные обращения и историю переписки. Только тестовая среда.
