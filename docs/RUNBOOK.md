# Регламент работы координатора и ИТ-специалиста

Этот документ — пошаговая инструкция для людей, сидящих в служебной группе MAX «Администрация ЕМР. Обратная связь». Открывается через `cat docs/RUNBOOK.md` или прямо с GitHub.

## Кто чем занимается

**Координатор АЕМР (специалист по ответам)** — отвечает жителям, передаёт обращения смежным специалистам, следит за SLA в 4 рабочих часа. Команды: `/reply`, `/stats`, `/reopen`, `/close`, `/diag`, `/op_help`, `/broadcast`, `/whoami`, `/help`.

**ИТ-специалист** — настраивает бот, добавляет операторов, удаляет ПДн по запросу жителя, снимает бэкапы, диагностирует сбои. Всё, что у координатора, плюс «опасные»: `/erase`, `/setting`, `/add_operators`, `/backup`.

## Ежедневная работа координатора

### 1. Обработать новое обращение

Когда житель пишет в бота, в админ-группу прилетает карточка вида:

```
📨 Новое обращение #107
👤 Имя: Алексей
📞 Телефон: 89964240723
──────────
🏠 Адрес: г. Елизово, ул. Ленина, д. 13
🏷️ Тематика: Дороги
📝 Суть:
<текст обращения>
```

**Способ 1 — свайп-реплай.** Нажмите долгим тапом на карточку → «Ответить» (или свайпом-вправо в зависимости от клиента MAX) и напишите ответ. Лимит — 300 символов. После отправки бот доставит ответ жителю в личку и пришлёт в группу подтверждение «✉️ Ответ ушёл жителю. Обращение #107 закрыто».

**Способ 2 — команда `/reply`.** Если клиент MAX не передаёт reply-связь (зависит от версии и платформы), используйте альтернативу:

```
/reply 107 Здравствуйте! Ваш вопрос рассмотрен, проблема устранена.
```

Тот же путь доставки и аудит — формальная обёртка письма от АЕМР, запись в `messages`, лог в `audit_log` с пометкой `reply_via_command`. Удобно ещё и тем, что номер обращения виден явно: меньше шансов перепутать.

**Важно:** реплай (или `/reply`) должен попасть в существующее обращение. Если просто написать в группу без reply — бот промолчит, житель ничего не получит. Если указать `/reply` несуществующий номер — бот ответит «Обращение #N не найдено».

**Если ответ длиннее 300 символов** — бот напишет в группу «Ответ длиннее 300 символов (фактически: …)». Жителю ничего не уходит. Сократите и повторите.

**Если житель заблокировал бота** — придёт сообщение «Не удалось доставить ответ жителю по обращению #N: …». Обращение остаётся в работе. Свяжитесь с жителем по телефону из карточки.

### 1а. Быстрые действия и панель `/op_help`

В админ-группе доступна команда `/op_help` — она присылает закрепляемое сообщение с тремя кнопками:

- **📊 Статистика за сегодня** — эквивалент `/stats today`, без необходимости печатать команду.
- **📢 Сделать рассылку** — стартует мастер `/broadcast`.
- **📋 Все команды** — расширенная подсказка по операторским командам с напоминанием формата `/reply`, `/stats`, `/reopen`, `/close`.

Эта панель — ближайший аналог Telegram-style menu button в MAX: операторы, которые редко работают с ботом, не запоминают команд — нажали кнопку и получили нужное действие. Закрепите ответ бота вверху группы, чтобы он был под рукой.

### 2. Передать обращение специалисту ЕГП

Сейчас архитектура — одна общая группа для координатора и ЕГП. Для передачи:

1. В группе скажите голосом или текстом «Передаю на ЕГП», упомяните `@username` ЕГП-специалиста.
2. ЕГП-специалист отвечает реплаем на ту же исходную карточку.
3. Бот доставит ответ жителю с пометкой ЕГП в `audit_log`.

Разделение на отдельные admin-группы по подразделениям не реализовано: всё идёт в одну общую группу. Это одно из возможных направлений развития ([ADR-001 §11](ADR-001-architecture.md)).

### 3. Реоткрыть закрытое обращение

Если житель написал ещё раз после ответа, бот сам реоткрывает обращение и присылает в группу followup-сообщение со ссылкой на исходную карточку. **Никаких ручных действий не нужно.**

Если хотите принудительно вернуть закрытое обращение в работу:

```
/reopen 107
```

### 4. Закрыть обращение без ответа

Спам, дубль, нерелевантное:

```
/close 107
```

Жителю ничего не уходит. В `audit_log` запишется кто и когда закрыл.

### 5. Выгрузить статистику

```
/stats today        — за сегодня
/stats week         — за 7 дней
/stats month        — за 30 дней
```

Бот пришлёт XLSX-файл с одиннадцатью колонками: №, дата, имя, телефон, адрес, тематика, суть, статус, ответ оператора, время ответа в часах, в SLA (да/нет).

Раз в месяц 1-го числа в 9:00 по Камчатке отчёт за прошлый месяц приходит автоматически — отдельно запрашивать не нужно.

## Действия ИТ-специалиста

### 1. Удалить ПДн жителя по запросу (ст. 21 152-ФЗ)

Команда принимает один из двух идентификаторов жителя:

```
/erase max_user_id=165729385
/erase phone=+79964240723
```

`max_user_id` обычно есть в карточке обращения; если житель сам обращается с просьбой «удалите мои данные» — попросите его написать `/whoami` боту в личке. Если ID назвать не получается, ищите по телефону — бот поднимет жителя по `users.phone` (нормализация цифр идёт в `users.find_by_phone`).

После команды:
- `users.first_name` → «Удалено»
- `users.phone` → NULL
- `users.consent_pdn_at` → NULL (отзыв согласия)
- `users.dialog_state` → `idle`, `dialog_data` → `{}` (стирание состояния воронки)
- `users.subscribed_broadcast` → false, `users.is_blocked` → true (исключает из рассылок: после отзыва согласия мы не имеем права отправлять жителю что-либо)
- В `audit_log` запись: кто инициировал, когда, по какому идентификатору

**Что не удаляется и почему.** Запись в `users` остаётся — она нужна, чтобы при возвращении жителя в бот не плодить дубль и чтобы старые `appeals` не висели с битым FK. Текст обращений и ответов в `messages` тоже остаётся: если житель уже подал претензию в Роскомнадзор и аудит спрашивает «что вы ему ответили» — без истории нечего показать. Это согласовано с юристом АЕМР и описано в политике обработки ПДн.

### 1а. Сделать рассылку подписавшимся жителям

Доступно ролям `coordinator` и `it`.

В админ-группе пишете:

```
/broadcast
```

Бот спрашивает текст одним сообщением (лимит 1000 символов). Пишете текст. Бот показывает предпросмотр «как увидит житель» и спрашивает «Разослать N подписчикам?» с кнопками `✅ Разослать` и `❌ Отмена`.

После нажатия `✅` бот в админ-группу постит сообщение с прогресс-баром «⏳ X/N доставлено» и кнопкой `⛔ Экстренно остановить`. Прогресс обновляется каждые пять секунд. По завершении — финальный итог «Доставлено: X из N. Не доставлено: Y.»

Кнопка `⛔ Экстренно остановить` доступна **любому оператору в группе** на случай, если рассылку запустили по ошибке. По нажатию — рассылка прерывается на следующем шаге, итог фиксирует сколько успело отправиться.

`/broadcast list` — последние десять рассылок со статусами и счётчиками.

Подписчиков на рассылку ведёт сам житель — командой `/subscribe` или кнопкой «🔔 Подписаться на новости» в подменю «Полезная информация». Заблокированные ботом и анонимизированные через `/erase` жители автоматически исключаются из получателей.

### 2. Зарегистрировать нового оператора

Сначала пусть человек напишет боту в личке `/whoami` — бот пришлёт его `max_user_id`.

Далее в админ-группе вызывает ИТ-специалист (только роль `it` имеет доступ к этой команде — координатор делегирует):

```
/add_operators
123456 coordinator Иванова Анна Петровна
789012 aemr Петров Пётр Иванович
345678 it Сидоров Сергей Сергеевич
```

Команда принимает несколько строк за один вызов. Формат строки: `<max_user_id> <role> <ФИО>`. Бот ответит сводкой «добавлено / обновлено / ошибки» и запишет каждое действие в `audit_log`.

Доступные роли:
- `coordinator` — координатор АЕМР, может всё кроме `/erase`, `/setting`, `/add_operators`, `/backup`
- `aemr` — рядовой специалист АЕМР по ответам
- `egp` — специалист ЕГП по ответам
- `it` — ИТ, может всё включая удаление ПДн и настройки

После регистрации добавьте человека в админ-группу через MAX-клиент.

Прямой `psql ... INSERT` остаётся как fallback на случай, если бот недоступен:

```bash
ssh user@feedback.elizovomr.ru
docker compose exec db psql -U aemr -d aemr -c \
  "INSERT INTO operators (max_user_id, full_name, role, is_active) VALUES (<id>, '<ФИО>', '<роль>', true);"
```

### 3. Изменить настройки без релиза

```
/setting list                                 — список доступных ключей
/setting policy_url https://elizovomr.ru/...  — URL политики ПДн
/setting electronic_reception_url https://... — э-приёмная
/setting udth_schedule_url https://...        — расписание УДТХ
/setting appointment_text "Приём по записи..." — текст приёма граждан
/setting emergency_contacts [{"name":"112","phone":"112"}, ...]  — экстренные службы
/setting topics ["Дороги", "Мусор", ...]      — тематики обращений
/setting welcome_text "Здравствуйте! ..."     — приветствие
/setting consent_text "..."                   — текст согласия
```

JSON парсится, валидатор отбрасывает мусор (пустые URL, не-list для тематик и т.п.). Если бот ответил «⚠️ Настройка не обновлена: …» — посмотрите причину и поправьте формат.

### 4. Диагностика

```
/diag
```

Возвращает:
- количество пользователей и обращений в БД
- timestamp последнего полученного события от MAX
- режим работы (polling/webhook)
- лимит длины ответа
- SLA-таргет

### 5. Что делать если бот молчит

**Алерт уже был** в группе («⚠️ Бот не отвечает на проверку здоровья…»):

1. По SSH на сервер:
   ```bash
   ssh user@feedback.elizovomr.ru
   cd ~/aemr-bot/infra
   docker compose ps
   docker compose logs --tail 200 bot
   ```
2. Если контейнер упал — `docker compose up -d bot`.
3. Если контейнер живой, но бот молчит — `docker compose restart bot`.
4. Если БД недоступна — `docker compose logs --tail 100 db` и `docker compose restart db`, потом `docker compose restart bot`.
5. После рестарта проверьте `curl http://localhost:8080/healthz` — должно вернуться `{"ok": true}`.
6. Когда бот восстановится, в группу автоматически придёт «✅ Бот восстановил отзывчивость».

**Алерта не было, но обращения не приходят** — может быть упал MAX-API. Сначала проверьте, можете ли вы сами написать боту в личке. Если нет — это либо токен отозван, либо MAX недоступен.

### 6. Развёртывание новой версии бота

```bash
ssh user@feedback.elizovomr.ru
cd ~/aemr-bot
git pull
cd infra
docker compose up -d --build bot
docker compose exec bot alembic upgrade head    # если были миграции
docker compose logs -f bot                       # убедиться что стартовал
```

Откат, если что-то поломалось:

```bash
git log --oneline -5
git checkout <предыдущий-коммит>
cd infra
docker compose up -d --build bot
```

Тег с предыдущей рабочей версией советую проставлять перед каждым деплоем: `git tag -a release-2026-04-29 -m "..."`.

### 7. Работа с бэкапами

**Расписание.** Каждое воскресенье в 03:00 по Камчатке APScheduler делает `pg_dump` и сохраняет файл в named-volume `backups` (внутри контейнера — `/backups/aemr-YYYYMMDD_HHMMSS.sql`). Хранятся последние 8 файлов (≈2 месяца истории), ротация автоматическая. Если задан `BACKUP_GPG_PASSPHRASE` — файлы шифруются gpg AES256 (`*.sql.gpg`), без этого — plain SQL. Если дополнительно заданы `BACKUP_S3_*` — копия льётся в S3-совместимое хранилище (для self-hosted без облака — оставить пустыми).

**Снять бэкап вручную.** В админ-группе ИТ:

```
/backup
```

Бот выдаёт `pg_dump` и отвечает «✅ Бэкап готов: aemr-20260504_113000.sql (12 КБ).» Полезно перед миграциями БД и перед удалением тестовых данных.

**Где лежат файлы.**

```bash
# Список бэкапов в named-volume:
docker compose exec bot ls -lh /backups/

# Размер volume (общий):
docker volume inspect infra_backups
```

**Скопировать бэкап на хост.**

```powershell
# В корне проекта. Создаёт папку backups\ на хосте.
docker compose -f infra\docker-compose.yml cp bot:/backups/aemr-20260504_113000.sql.gpg .\backups\
```

**Восстановление из бэкапа.**

```powershell
# Если plain SQL:
docker compose -f infra\docker-compose.yml down
docker volume rm infra_db_data
docker compose -f infra\docker-compose.yml up -d db
Start-Sleep -Seconds 10
Get-Content backups\aemr-20260504_113000.sql | docker compose -f infra\docker-compose.yml exec -T db psql -U aemr -d aemr
docker compose -f infra\docker-compose.yml up -d bot
```

```bash
# Если зашифрован gpg:
gpg --batch --passphrase "$BACKUP_GPG_PASSPHRASE" \
    --decrypt backups/aemr-20260504_113000.sql.gpg > backups/aemr.sql
# Дальше как с plain SQL выше.
```

Перед боевым восстановлением **обязательно тренируйте процедуру** на тестовом стенде — потеря данных при ошибке в восстановлении страшнее любого инцидента.

**Если сервер был выключен в момент еженедельного триггера** — APScheduler с in-memory jobstore не наверстывает пропущенные запуски. На следующее воскресенье цикл продолжится. На критических периодах (после деплоя, перед миграцией) делайте `/backup` руками.

### 8. Сбросить тестовые данные перед запуском в продакшен

Между прогонами smoke-тестов (или после длительной отладки на стенде) полезно очистить БД от мусора, оставив только операторов и редактируемые настройки. Готовый скрипт — `scripts/reset_test_data.sql`. Он стирает все обращения, сообщения, рассылки, idempotency-логи, audit_log и тестовых пользователей. Не трогает таблицы `operators` и `settings`.

```powershell
# Из корня проекта:
Get-Content scripts\reset_test_data.sql | docker compose -f infra\docker-compose.yml exec -T db psql -U aemr -d aemr
```

После завершения скрипт сам выводит итоговый count по каждой таблице — операторы и настройки на месте, остальное в нулях.

**Когда НЕ запускать:** в production. После публикации жителям эта команда удалит реальные обращения и историю переписки. Только тестовая среда.
