# Команды бота — полный справочник

Все команды и кнопки, которые понимает aemr-bot в мессенджере MAX. Каждая запись говорит: кто может вызвать, в каком чате, какой формат ввода, что бот делает, какие лимиты, что попадает в `audit_log`, и что бывает при ошибках.

> **Соглашения.**
> - **«Личка»** — личный диалог жителя с ботом.
> - **«Админ-группа»** — служебная группа MAX, чей `chat_id` равен `ADMIN_GROUP_ID` в `.env`.
> - В админ-группе citizen-команды (`/start`, главное меню, воронка) автоматически отбрасываются guard'ом `is_admin_chat`. Бот не путает оператора с жителем.
> - `<...>` в синтаксисе — обязательный плейсхолдер.
> - Все CLI-аналоги (через psql / docker compose) — в [COMMANDS.md](COMMANDS.md).

## 1. Команды жителя в личке

Все команды доступны без авторизации. Если житель пишет команду в админ-группе — бот игнорирует (citizen-flow guard). Команда `/whoami` — единственное исключение, работает в обеих группах.

### `/start`

| | |
|---|---|
| **Кто может** | любой житель |
| **Где** | в личке боту |
| **Формат** | `/start` без аргументов |
| **Что делает** | приветствие + главное меню (5 кнопок). Создаёт запись в `users` (если нет) с `dialog_state=idle`. Если житель раньше уже общался — состояние не сбрасывается, история сохраняется. |
| **Audit** | нет (это служебное действие) |
| **Текст** | из `settings.welcome_text` либо `seed/welcome.md` |

`@dp.bot_started()` срабатывает с тем же эффектом, когда житель нажимает «Старт» в карточке бота.

### `/menu`

| | |
|---|---|
| **Что делает** | то же, что `/start` — заново показать главное меню. Полезно, если житель удалил inline-кнопки и хочет вернуть их. |

### `/help`

| | |
|---|---|
| **Что делает** | список доступных команд жителя + их назначение |
| **Текст** | `texts.HELP_USER` |

### `/policy`

| | |
|---|---|
| **Что делает** | присылает PDF политики обработки персональных данных (`docs/PRIVACY.pdf`, токен закеширован в `settings.policy_pdf_token` после первого старта). Если кеш пуст — пытается загрузить заново; если и это не сработало — присылает URL из `settings.policy_url`. |

### `/subscribe`

| | |
|---|---|
| **Что делает** | подписывает на broadcast-рассылки. Ставит `users.subscribed_broadcast=true`. Доставка возможна только при `is_blocked=false`. |
| **Альтернатива** | кнопка «🔔 Подписаться на новости» в подменю «Полезная информация». |
| **Audit** | нет (действие самого жителя) |

### `/unsubscribe`

| | |
|---|---|
| **Что делает** | `users.subscribed_broadcast=false`. После следующей рассылки житель ничего не получит. |
| **Альтернатива** | кнопка «🔕 Отписаться от рассылки» под каждым broadcast-сообщением (one-tap unsubscribe). |

### `/forget`

| | |
|---|---|
| **Что делает** | анонимизация (152-FZ ст. 21, право на удаление). Полностью эквивалентна `/erase max_user_id=<свой>` от ИТ-оператора. |
| **Эффект в БД** | `first_name='Удалено'`, `phone=NULL`, `phone_normalized=NULL`, `consent_pdn_at=NULL`, `dialog_state='idle'`, `dialog_data={}`, `subscribed_broadcast=false`, `is_blocked=true`. |
| **Что не удаляется** | `appeals.summary`, `messages.text` — остаются как муниципальные документы (см. PRIVACY.md). |
| **Audit** | `action='self_erase'`, `target='user max_id=<id>'` |

### `/whoami`

| | |
|---|---|
| **Где** | работает и в личке, и в админ-группе (исключение из citizen-flow guard) |
| **Что делает** | возвращает `max_user_id` пользователя, его `first_name` (как видит MAX) и `chat_id` текущего чата. |
| **Использование** | для оператора — узнать `chat_id` админ-группы при cold start; для жителя — узнать свой `max_user_id`, чтобы продиктовать ИТ-специалисту при просьбе об удалении ПДн. |

## 2. Главное меню жителя

Inline-кнопки под приветственным сообщением. Реализованы как callback'и (`on_callback` в `appeal.py`).

| Кнопка | Payload | Что делает |
|---|---|---|
| 📝 Написать обращение | `menu:new_appeal` | Запускает воронку (см. §3). Сначала проверяется `consent_pdn_at`; если согласия нет — шаг с PDF-политикой. |
| 📂 Мои обращения | `menu:my_appeals` | Список своих обращений (по 5 на страницу) с эмодзи статуса и датой. По клику — карточка обращения. |
| 🌐 Электронная приёмная | LinkButton | Прямая ссылка на `settings.electronic_reception_url`. Не callback — открывается во встроенном браузере. |
| 📋 Приём граждан | `menu:appointment` | Показывает текст `settings.appointment_text` (расписание + телефон записи). |
| 📚 Полезная информация | `menu:useful_info` | Подменю (см. §4). |

В подменю и в карточке обращения везде есть кнопка «↩️ В меню» (`menu:main`), которая возвращает на главное.

## 3. Воронка приёма обращения

FSM (состояния жителя в `users.dialog_state`). Переход — на каждое нажатие или сообщение.

```
idle
  → awaiting_consent      (если consent_pdn_at пуст)
  → awaiting_contact      (если телефона нет)
  → awaiting_name
  → awaiting_address
  → awaiting_topic
  → awaiting_summary
  → idle                  (финализация → создание appeal)
```

Шаги:

1. **`awaiting_consent`** — текст согласия + PDF + кнопки:
   - `consent:yes` — записывает `consent_pdn_at`, переходит к `awaiting_contact` (или дальше, если телефон уже есть).
   - `consent:no` — сбрасывает в `idle`, возвращает в меню.

2. **`awaiting_contact`** — нативная кнопка `RequestContactButton` («📲 Поделиться контактом»). MAX отдаёт VCF; бот парсит телефон и имя через `utils/attachments.extract_phone`. Кнопка «Отмена» (`cancel`) сбрасывает в `idle`.

3. **`awaiting_name`** — свободный ввод имени. Лимит `NAME_MAX_CHARS=120`.

4. **`awaiting_address`** — свободный ввод адреса. Лимит `ADDRESS_MAX_CHARS=500`.

5. **`awaiting_topic`** — inline-клавиатура из `settings.topics` (по 2 кнопки в ряд). Payload `topic:<idx>`. Выход из диапазона — silent ignore.

6. **`awaiting_summary`** — свободный ввод текста и/или фото / геолокации. Можно посылать несколько сообщений подряд — все накапливаются в `dialog_data.summary_chunks` и `dialog_data.attachments`. Финализация:
   - кнопка «✅ Отправить» (`appeal:submit`),
   - либо автоматически после `APPEAL_TIMEOUT=60` секунд тишины.
   - Лимиты: `SUMMARY_MAX_CHARS=2000`, `ATTACHMENTS_MAX_PER_APPEAL=20`.

После финализации — создание `appeals.NNN` со статусом `new`, отправка карточки в админ-группу, ack жителю «Обращение #N принято». FSM возвращается в `idle`.

**Кнопка «Отмена»** на любом шаге возвращает в `idle` и в главное меню. Per-user `asyncio.Lock` снимается; накопленные `dialog_data` стираются.

## 5. Команды оператора в админ-группе

Доступны всем 4 ролям (`coordinator`, `aemr`, `egp`, `it`), если оператор вызывает команду из `ADMIN_GROUP_ID`.

### `/reply`

| | |
|---|---|
| **Формат** | `/reply <номер_обращения> <текст ответа>` |
| **Пример** | `/reply 42 Здравствуйте! Ваш вопрос рассмотрен, проблема устранена.` |
| **Что делает** | находит обращение по `id=N`, проверяет лимит ответа `ANSWER_MAX_CHARS=300`, проверяет `users.is_blocked`, оборачивает текст в формальное письмо («📬 Ответ Администрации Елизовского муниципального района…»), доставляет жителю. Меняет `appeals.status='answered'`, ставит `answered_at=now()`. |
| **Альтернатива** | свайп-reply на карточку (см. §9). |
| **Audit** | `action='reply_via_command'`, `target='appeal #N'`, `details={"chars": <длина>}` |
| **Failure** | обращение не найдено → «Обращение #N не найдено». Превышение 300 символов → «Ответ длиннее 300 символов (фактически: …)». Житель `is_blocked=true` → «⚠️ Не могу доставить ответ … житель отозвал согласие на ПДн». MAX недоступен → «⚠️ Не удалось доставить ответ жителю по обращению #N (<тип_исключения>).» |

### `/stats`

| | |
|---|---|
| **Формат** | `/stats today` / `/stats week` / `/stats month` (без аргумента — `today`) |
| **Что делает** | формирует XLSX-выгрузку обращений за период, отправляет файлом в админ-группу. 11 колонок: номер, дата, имя, телефон, адрес, тематика, суть, статус, ответ оператора, время ответа в часах, в SLA (да/нет). |
| **Альтернатива** | кнопка «📊 Статистика за сегодня» в `/op_help` (только today). |
| **Лимит** | период жёсткий: today / week / month. Произвольные даты не поддерживаются. |
| **Audit** | нет (read-only операция) |

Раз в месяц 1-го числа в 09:00 (по `TZ`) бот сам шлёт в админ-группу отчёт `month` — настраивать не надо.

### `/reopen`

| | |
|---|---|
| **Формат** | `/reopen <N>` |
| **Что делает** | возвращает обращение в `status='in_progress'`, обнуляет `answered_at`, `closed_at`. Ничего не отправляет жителю. |
| **Audit** | `action='reopen'`, `target='appeal #N'` |
| **Когда нужно** | если оператор хочет вручную вернуть закрытое обращение. Авто-реоткрытие при повторном сообщении жителя — встроено, ручное `reopen` не требуется. |

### `/close`

| | |
|---|---|
| **Формат** | `/close <N>` |
| **Что делает** | закрывает обращение без ответа: `status='closed'`, `closed_at=now()`. Жителю ничего не уходит. |
| **Audit** | `action='close'`, `target='appeal #N'` |
| **Использование** | спам, дубль, нерелевантное обращение. |

### `/diag`

| | |
|---|---|
| **Что делает** | счётчики и конфиг-снимок: жителей (всего/подписаны/заблокированы), обращений (всего/в работе), рассылок (✅ done / ⚠️ failed), события (всего/последнее), режим бота, лимит ответа, SLA. |
| **Использование** | первая проверка при инцидентах. Если цифры выглядят разумно — БД и event-loop живы. |

### `/op_help`

См. §7. Команда показывает закрепляемое сообщение с тремя кнопками-быстрыми-действиями.

### `/whoami`

| | |
|---|---|
| **Что делает** | возвращает `max_user_id` оператора, `first_name`, `chat_id` админ-группы. |
| **Использование** | при cold start — взять `chat_id` для `ADMIN_GROUP_ID` в `.env`; узнать `max_user_id` нового оператора перед `/add_operators`. |

### `/help`

| | |
|---|---|
| **Что делает** | краткая справка по командам оператора. Расширенная версия — кнопка «📋 Все команды» в `/op_help`. |

### `/broadcast`

См. §8. Двухшаговый wizard, доступен ролям `coordinator` и `it`.

## 6. Команды только для роли `it`

`_ensure_role(event, OperatorRole.IT)` отбивает остальных. Аудит каждой команды записывается в `audit_log`.

### `/erase`

| | |
|---|---|
| **Формат** | `/erase max_user_id=<id>` или `/erase phone=+7XXXXXXXXXX` |
| **Что делает** | анонимизирует жителя в одной транзакции с записью в `audit_log` (152-FZ ст. 21). Эффект тот же, что у `/forget` от самого жителя. Поиск по телефону идёт по индексированной колонке `phone_normalized` (нормализация — только цифры, с обрезанием 7/8 в начале для 11-значных номеров). |
| **Audit** | `action='erase'`, `target='user max_id=<id>'` |
| **Failure** | пользователь не найден → «Пользователь не найден.» Битый формат → пример из usage-сообщения. |
| **Безопасность** | self-erase IT-оператора не блокируется специально, но `users` обычно не содержит запись для оператора (он же не житель), поэтому случайное самоудаление практически невозможно. |

### `/setting`

| | |
|---|---|
| **Формат** | `/setting list` или `/setting <key> <value>` |
| **Что делает** | редактирует ключ из таблицы `settings` (см. §10). `value` парсится как JSON если возможно, иначе как сырая строка. Валидируется по schema (`settings_store.SCHEMA`); при ошибке — «⚠️ Настройка не обновлена: <причина>». |
| **Audit** | `action='setting_update'`, `target=<key>`, `details={"kind": "<type>", "chars": <len>}` (без сохранения значения, чтобы не дублировать ПДн в audit_log). |
| **`/setting list`** | возвращает список доступных ключей. |

### `/add_operators`

| | |
|---|---|
| **Формат** | многострочный — первая строка `/add_operators`, дальше по одной записи на строку: `<max_user_id> <role> <ФИО полностью>` |
| **Пример** | <pre>/add_operators<br>234567 aemr Иванова Анна Петровна<br>345678 egp Петров Пётр Иванович</pre> |
| **Что делает** | upsert-ит каждого оператора в таблицу `operators` (создаёт или обновляет). Роли: `coordinator`, `aemr`, `egp`, `it`. Возвращает сводку «Добавлено / Обновлено / Ошибок». |
| **Audit** | `action='operator_upsert'`, `target='max_id=<id>'`, `details={"role": "<role>", "full_name": "<ФИО>"}` |
| **Защита от self-promote** | оператор не может изменить через эту команду свою собственную роль (`actor_id == target_id`). Изменение роли себе — только через psql. |

### `/backup`

| | |
|---|---|
| **Что делает** | вне расписания запускает `pg_dump` (через тот же `_backup_db`, что и cron-job). Возвращает имя файла и размер. |
| **Когда нужно** | перед миграцией БД, перед сбросом тестовых данных, перед опасной операцией. |
| **Failure** | при провале — admin-alert «⚠️ Бэкап упал…», подробности в логах бота. |

## 7. Панель `/op_help` и быстрые действия

`/op_help` присылает в админ-группу сообщение с тремя callback-кнопками. Закрепи это сообщение — будет всегда под рукой.

| Кнопка | Payload | Что делает |
|---|---|---|
| 📊 Статистика за сегодня | `op:stats_today` | Эквивалент `/stats today`, без необходимости печатать команду. |
| 📢 Сделать рассылку | `op:broadcast` | Запускает wizard `/broadcast`. |
| 📋 Все команды | `op:help_full` | Развёрнутая справка по всем операторским командам с примерами синтаксиса. |

«Все команды» полезно скинуть новому оператору в первый раз. Закреплённое сообщение он сможет открыть и без помощи.

## 8. Wizard `/broadcast` — пошагово

Доступен ролям `coordinator` и `it`. Двухшаговый, защищён от случайной отправки и от двойного запуска.

```
1. /broadcast (или кнопка «📢 Сделать рассылку» из /op_help)
   → Бот: «Введите текст рассылки (≤1000 символов)»

2. <текст рассылки>
   → Бот: предпросмотр + «Разослать N подписчикам?»
   → две кнопки: ✅ Разослать  ❌ Отмена

3a. ✅ Разослать → старт цикла отправки:
    - в админ-группу постится сообщение с прогресс-баром
      «⏳ X/N доставлено» и кнопкой «⛔ Экстренно остановить»
    - адаптивный шаг обновления (короткая рассылка обновляет
      бар чаще, длинная — реже, чтобы не упереться в edit-rate-limit MAX)
    - rate-limit = BROADCAST_RATE_LIMIT_PER_SEC=1.0 (1 msg/sec)
    - под каждым полученным жителем сообщением — кнопка
      «🔕 Отписаться от рассылки»

3b. ❌ Отмена → wizard сброшен, рассылка не запущена.
```

`/broadcast list` — последние 10 рассылок со статусами и счётчиками.

**Экстренный стоп.** Кнопка `⛔` доступна **любому** оператору в группе. Прерывает цикл на следующей итерации; счётчики `delivered_count` / `failed_count` фиксируются. Статус → `cancelled`.

**Кому доставится.** Только живым подписчикам: `subscribed_broadcast=true AND is_blocked=false AND first_name <> 'Удалено'`. Отозвавшие согласие через `/forget` или `/erase` — автоматически исключены.

**Recovery после рестарта.** Если процесс упал в середине — на старте `reap_orphaned_sending` находит висящие в `SENDING` строки и помечает их `FAILED` (со статусом, без выдуманного `finished_at`). Оператор должен вручную перезапустить рассылку, если нужно.

**Audit.** `action='broadcast_send'`, `target='broadcast #<id>'`, `details={"chars": <len>, "subscriber_count": <N>}`. Текст сообщения хранится в `broadcasts.text`, не дублируется в audit_log.

## 9. Свайп-reply и `/reply` — два пути ответа

Обе ведут к одному коду доставки (`_deliver_operator_reply`), отличается только способ привязки к обращению.

### Способ 1 — свайп-reply

Долгий тап на карточку обращения в админ-группе → «Ответить» (или свайпом-вправо в зависимости от клиента MAX) → ввести текст ответа.

Бот ловит `message_created` с `link.type=REPLY`, читает `link.message.mid`, по нему через `appeals.get_by_admin_message_id` находит обращение.

Если клиент MAX **не передаёт reply-link** (зависит от версии и платформы) — бот логирует «no reply-link in event.message — message ignored». В этом случае использовать §2.

### Способ 2 — `/reply N <текст>`

Прямая команда. Работает на всех клиентах, не зависит от swipe-link. Номер виден явно в команде — меньше шансов перепутать.

### Общая часть

После любого пути:

1. `len(text) > 300` → отказ в админ-группу, ничего жителю.
2. `users.is_blocked=true` (после `/erase` / `/forget`) → отказ в админ-группу, ничего жителю.
3. Вызов `bot.send_message(user_id, formatted_text)` с формальной обёрткой.
4. Запись `messages` (`direction='from_operator'`).
5. `appeals.status='answered'`, `appeals.answered_at=now()`.
6. Запись в `audit_log` (`reply` для swipe, `reply_via_command` для команды).
7. Подтверждение в админ-группу: «✉️ Ответ ушёл жителю. Обращение #N закрыто».

## 10. Ключи `/setting`

Список редактируемых из админки настроек. Все изменения через `/setting list` и `/setting <key> <json_or_string>`. Валидатор — в `services/settings_store.py::SCHEMA`.

| Ключ | Тип | Лимиты | Что это |
|---|---|---|---|
| `welcome_text` | string | 1–4000 символов | Приветственное сообщение в `/start`. |
| `consent_text` | string | 1–4000 | Текст согласия на обработку ПДн (показывается перед PDF). |
| `policy_url` | string | URL (http/https) | Ссылка на PDF политики на сайте АЕМР. |
| `electronic_reception_url` | string | URL | Ссылка на электронную приёмную (LinkButton). |
| `udth_schedule_url` | string | URL | PDF расписания муниципальных автобусов УДТХ. |
| `udth_schedule_intermunicipal_url` | string | URL | Страница межмуниципальных маршрутов на kamgov.ru. |
| `appointment_text` | string | 1–2000 | Расписание приёма граждан главой района + телефон записи. |
| `emergency_contacts` | list[object] | min 1 элемент, ключи `name` + `phone` | Экстренные службы. Пример: `[{"name":"112","phone":"112"}, ...]` |
| `transport_dispatcher_contacts` | list[object] | min 1, ключи `routes` + `phone` | Диспетчерские автотранспорта. Пример: `[{"routes":"101,102","phone":"+7..."}, ...]` |
| `topics` | list[string] | 1–30 элементов | Тематики обращения для шага FSM. Пример: `["Дороги","Мусор","ЖКХ", ...]` |

Примеры:

```
/setting policy_url https://elizovomr.ru/storage/.../privacy.pdf
/setting topics ["Дороги","Мусор","ЖКХ","Освещение","Транспорт"]
/setting emergency_contacts [{"name":"Полиция","phone":"02"},{"name":"МЧС","phone":"01"}]
```

Битый JSON / неверный тип / нарушение лимитов → «⚠️ Настройка не обновлена: <причина>». Пользовательский опыт жителя не ломается — старое значение остаётся.

## 11. Что попадает в audit_log

`audit_log` — append-only-таблица для МСУ-аудита. Каждая запись: `operator_max_user_id` (кто), `action` (что), `target` (над кем/чем), `details` (метаданные), `created_at`.

| Действие | `action` | `target` | Что в `details` |
|---|---|---|---|
| Свайп-reply оператора | `reply` | `appeal #N` | `{"chars": <len>}` |
| Команда `/reply N` | `reply_via_command` | `appeal #N` | `{"chars": <len>}` |
| `/reopen N` | `reopen` | `appeal #N` | `{}` |
| `/close N` | `close` | `appeal #N` | `{}` |
| `/erase` (любая форма) | `erase` | `user max_id=<id>` | `{}` |
| `/forget` от жителя | `self_erase` | `user max_id=<id>` | `{}` |
| `/setting <key> <value>` | `setting_update` | `<key>` | `{"kind": "<тип>", "chars": <len> или "items": <count>}` |
| `/add_operators` (на каждого) | `operator_upsert` | `max_id=<id>` | `{"role": "<role>", "full_name": "<ФИО>"}` |
| `/broadcast` confirm | `broadcast_send` | `broadcast #<id>` | `{"chars": <len>, "subscriber_count": <N>}` |

В `details` **не пишутся** тексты обращений, ответов, рассылок и значений `/setting` — только метаданные (длина, тип, количество). PII полная — только в основных таблицах (`messages.text`, `broadcasts.text`, `settings.value`).

## 12. Лимиты и валидация ввода

Все лимиты — env-tunable через `infra/.env.example`.

| Лимит | Значение по умолчанию | Что ограничивает |
|---|---|---|
| `ANSWER_MAX_CHARS` | 300 | Длина ответа оператора жителю (свайп / `/reply`). |
| `SUMMARY_MAX_CHARS` | 2000 | Накопленная длина суммы в `awaiting_summary`. |
| `NAME_MAX_CHARS` | 120 | Имя жителя на шаге `awaiting_name`. |
| `ADDRESS_MAX_CHARS` | 500 | Адрес на шаге `awaiting_address`. |
| `ATTACHMENTS_MAX_PER_APPEAL` | 20 | Количество вложений в обращении. |
| `ATTACHMENTS_PER_RELAY_MESSAGE` | 10 | Сколько вложений уходит в админ-группу одним сообщением (chunks `(1/3)`, `(2/3)`…). |
| `BROADCAST_MAX_CHARS` | 1000 | Длина текста рассылки. |
| `BROADCAST_RATE_LIMIT_PER_SEC` | 1.0 | Скорость отправки сообщений рассылки (msg/sec). |
| `APPEAL_TIMEOUT` | 60 | Секунды тишины в `awaiting_summary` до автофинализации. |
| `RECOVER_BATCH_SIZE` | 1000 | Сколько застрявших FSM-сессий обрабатывает recovery при старте. |

Превышение лимита на ответ оператора → отказ в админ-группу. Превышение на стороне жителя — silent drop с записью в лог.

## 13. Edge cases и failure modes

**Житель блокирует бота в MAX.** Свайп-reply / `/reply` падают с `Exception` от MAX, оператор видит «⚠️ Не удалось доставить ответ жителю по обращению #N (<имя_исключения>). Возможно, житель удалил диалог или заблокировал бота.» Обращение остаётся в работе — координатор связывается по телефону из карточки.

**Житель отозвал согласие через `/forget` или `/erase`, но обращение ещё открыто.** Любой оператор-reply увидит «⚠️ Не могу доставить ответ … житель отозвал согласие на ПДн». Обращение можно закрыть через `/close N`.

**Двойной клик «✅ Отправить» на финализации воронки.** Per-user `asyncio.Lock` + проверка `dialog_state == idle` гарантируют ровно одну запись в `appeals`.

**Бот рестартанул, пока житель в `awaiting_summary` дольше 60 сек.** На старте `recover_stuck_funnels` пакетами по `RECOVER_BATCH_SIZE` финализирует таких жителей. Пустые сессии (нет ни текста, ни вложений) сбрасываются в `idle`.

**Бот рестартанул в середине `/broadcast`.** На старте `reap_orphaned_sending` помечает зависшие в `SENDING` рассылки как `FAILED`. Оператор должен запустить новую при необходимости (текст сохранён в `broadcasts.text`).

**Оператор пишет в админ-группу без reply-link.** Лог: «operator_reply: no reply-link in event.message — message ignored». Бот ничего не делает (сообщение не считается ответом). Если оператор хотел ответить — использовать `/reply N <текст>`.

**Дубль Update от MAX.** `IdempotencyMiddleware` отбивает по unique-индексу `events.idempotency_key`. Дубликат тихо отбрасывается до любого хендлера.

**Оператор в админ-группе нажал старую inline-кнопку обращения.** Citizen-flow callback'ы (`menu:*`, `consent:*`, `topic:*`, `appeal:*`, `info:*`, `cancel`) в админ-чате игнорируются на уровне `on_callback` — оператор не попадает в воронку как «житель».

**Изменился набор тематик в `settings.topics`, у жителя в `dialog_data` старый индекс.** Range-check `0 <= idx < len(topics)` отрезает невалидный выбор. Житель просто переоткрывает меню.
