# Команды бота — полный справочник

Все команды и кнопки, которые бот aemr-bot понимает в мессенджере MAX. Каждая запись отвечает на одни и те же вопросы. Кто может вызвать команду. В каком чате. Как правильно её ввести. Что бот делает в ответ. Какие установлены лимиты. Что бот записывает в журнал действий (таблица `audit_log`). Что увидит оператор, если что-то пошло не так.

> **Соглашения по терминам.**
> - **«Личка»** — личный диалог жителя с ботом.
> - **«Админ-группа»** — служебная группа MAX. Её идентификатор задан в файле `.env` как `ADMIN_GROUP_ID`.
> - В админ-группе команды жителя (`/start`, главное меню, шаги анкеты на обращение) отбрасываются автоматически. За это отвечает функция-проверка `is_admin_chat`. Бот не путает оператора с жителем.
> - Запись `<...>` в синтаксисе команды означает обязательный параметр.
> - Аналоги через командную строку (через `psql` и `docker compose`) собраны в [COMMANDS.md](COMMANDS.md).

## 1. Команды жителя в личке

Все команды доступны без входа в систему. Если житель отправит команду в админ-группу, бот её проигнорирует. Единственное исключение — команда `/whoami`. Она работает в обоих типах чатов.

### `/start`

| | |
|---|---|
| **Кто может** | любой житель |
| **Где** | в личном диалоге с ботом |
| **Формат** | `/start` без параметров |
| **Что делает** | присылает приветствие и главное меню из пяти кнопок. Если жителя ещё нет в таблице `users`, создаёт запись с состоянием `dialog_state=idle`. Если житель уже общался с ботом раньше, его состояние и история не сбрасываются. |
| **Журнал действий** | не пишет (это служебное действие) |
| **Текст приветствия** | берётся из настройки `settings.welcome_text`. Если её нет, из файла `seed/welcome.md`. |

Когда житель нажимает «Старт» в карточке бота, срабатывает событие `@dp.bot_started()`. Эффект тот же, что у команды `/start`.

### `/menu`

| | |
|---|---|
| **Что делает** | то же, что `/start`. Заново показывает главное меню. Полезно, если житель удалил кнопки и хочет вернуть их. |

### `/help`

| | |
|---|---|
| **Что делает** | присылает список доступных команд жителя и краткое описание каждой. |
| **Текст** | берётся из константы `texts.HELP_USER`. |

### `/policy`

| | |
|---|---|
| **Что делает** | присылает PDF политики обработки персональных данных (файл `docs/PRIVACY.pdf`). После первой загрузки токен файла кешируется в `settings.policy_pdf_token`. Если кеш пустой, бот пробует загрузить PDF заново. Если и это не получилось, присылает прямую ссылку из `settings.policy_url`. |

### `/subscribe`

| | |
|---|---|
| **Что делает** | подписывает жителя на массовые рассылки. Ставит в таблице `users` поле `subscribed_broadcast=true`. Доставка работает, только если у жителя `is_blocked=false`. |
| **Альтернатива** | кнопка «🔔 Подписаться на новости» в подменю «Полезная информация». |
| **Журнал действий** | не пишет (это действие самого жителя). |

### `/unsubscribe`

| | |
|---|---|
| **Что делает** | ставит `users.subscribed_broadcast=false`. После следующей рассылки житель ничего не получит. |
| **Альтернатива** | кнопка «🔕 Отписаться от рассылки» под каждым сообщением рассылки. Отписаться можно одним нажатием. |

### `/forget`

| | |
|---|---|
| **Что делает** | обезличивает данные жителя в базе. Это право, гарантированное статьёй 21 Федерального закона 152-ФЗ. Команда полностью равнозначна `/erase max_user_id=<свой>`, которую вызывает ИТ-оператор. |
| **Что меняется в базе** | `first_name='Удалено'`, `phone=NULL`, `phone_normalized=NULL`, `consent_pdn_at=NULL` (отметка согласия на обработку ПДн), `dialog_state='idle'`, `dialog_data={}`, `subscribed_broadcast=false`, `is_blocked=true`. |
| **Что не удаляется** | поле `appeals.summary` (суть обращения) и поле `messages.text` (тексты сообщений). Они хранятся как муниципальные документы. Подробности в `PRIVACY.md`. |
| **Журнал действий** | `action='self_erase'`, `target='user max_id=<id>'`. |

### `/whoami`

| | |
|---|---|
| **Где** | работает и в личке, и в админ-группе. Это исключение из правила про игнор команд жителя в админ-чате. |
| **Что делает** | возвращает `max_user_id` пользователя, его имя в MAX (`first_name`) и идентификатор текущего чата `chat_id`. |
| **Зачем нужно** | оператору — узнать `chat_id` админ-группы при первой настройке. Жителю — узнать свой `max_user_id`, чтобы продиктовать его ИТ-специалисту, если житель просит удалить персональные данные. |

## 2. Главное меню жителя

Под приветственным сообщением житель видит inline-кнопки (кнопки, прикреплённые к сообщению). Каждое нажатие на такую кнопку обрабатывает функция `on_callback` в файле `appeal.py`. Каждой кнопке соответствует своё значение, которое бот получает после нажатия. Это значение ниже названо «значением кнопки».

| Кнопка | Значение кнопки | Что делает |
|---|---|---|
| 📝 Написать обращение | `menu:new_appeal` | Запускает пошаговую анкету (см. §3). Сначала бот проверяет поле `consent_pdn_at`. Если согласие на обработку ПДн ещё не получено, добавляется шаг с PDF политики. |
| 📂 Мои обращения | `menu:my_appeals` | Показывает обращения жителя, по 5 на страницу. Рядом с каждым — эмодзи статуса и дата. По клику открывается карточка обращения. |
| 🌐 Электронная приёмная | LinkButton (кнопка-ссылка) | Прямая ссылка на адрес из `settings.electronic_reception_url`. Это не нажатие, обрабатываемое ботом, а открытие ссылки во встроенном браузере MAX. |
| 📋 Приём граждан | `menu:appointment` | Показывает текст из `settings.appointment_text`. Внутри — расписание приёма и телефон записи. |
| 📚 Полезная информация | `menu:useful_info` | Открывает подменю (см. §4). |

В подменю и в карточке обращения всегда есть кнопка «↩️ В меню» со значением `menu:main`. Она возвращает жителя на главный экран.

## 3. Пошаговая анкета приёма обращения

Анкета — это последовательность состояний. Текущее состояние жителя хранится в поле `users.dialog_state`. На каждое нажатие кнопки или входящее сообщение бот переводит жителя в следующее состояние.

```
idle
  → awaiting_consent      (если consent_pdn_at пустой)
  → awaiting_contact      (если телефон ещё не записан)
  → awaiting_name
  → awaiting_address
  → awaiting_topic
  → awaiting_summary
  → idle                  (анкета завершена, обращение создано)
```

Шаги по порядку.

1. **`awaiting_consent`** — текст согласия, PDF политики и две кнопки.
   - `consent:yes` записывает текущее время в `consent_pdn_at` и переводит на шаг `awaiting_contact`. Если телефон уже сохранён, шаг с контактом пропускается.
   - `consent:no` сбрасывает состояние в `idle` и возвращает в главное меню.

2. **`awaiting_contact`** — нативная кнопка «📲 Поделиться контактом» (`RequestContactButton`). MAX отдаёт боту визитку формата VCF. Бот разбирает её и достаёт телефон и имя через функцию `utils/attachments.extract_phone`. Кнопка «Отмена» (значение `cancel`) сбрасывает состояние в `idle`.

3. **`awaiting_name`** — свободный ввод имени. Лимит длины — 120 символов (константа `NAME_MAX_CHARS`).

4. **`awaiting_address`** — свободный ввод адреса. Лимит — 500 символов (`ADDRESS_MAX_CHARS`).

5. **`awaiting_topic`** — клавиатура с тематиками из настройки `settings.topics`. Кнопки идут по две в ряд. Значение каждой имеет вид `topic:<номер>`. Если индекс вышел за допустимый диапазон, бот молча игнорирует нажатие.

6. **`awaiting_summary`** — свободный ввод текста. Можно слать фотографии и геолокацию. Можно слать несколько сообщений подряд. Бот накапливает текст в `dialog_data.summary_chunks`, а вложения — в `dialog_data.attachments`. Анкета завершается двумя способами.
   - Житель нажимает кнопку «✅ Отправить» (значение `appeal:submit`).
   - Либо бот сам завершает анкету, если житель молчит дольше времени, заданного в `APPEAL_TIMEOUT` (по умолчанию 60 секунд).
   - Лимиты на этом шаге. Текст — до 2000 символов (`SUMMARY_MAX_CHARS`). Вложений — до 20 на одно обращение (`ATTACHMENTS_MAX_PER_APPEAL`).

После завершения анкеты бот создаёт запись в таблице `appeals` со статусом `new`. Карточка обращения уходит в админ-группу. Жителю приходит подтверждение «Обращение #N принято». Состояние возвращается в `idle`.

**Кнопка «Отмена»** работает на любом шаге. Она возвращает жителя в `idle` и в главное меню. На время анкеты бот держит для жителя блокировку `asyncio.Lock`. Эта блокировка снимается, накопленные данные в `dialog_data` стираются.

## 5. Команды оператора в админ-группе

Эти команды доступны всем четырём ролям: `coordinator`, `aemr`, `egp`, `it`. Условие — оператор вызывает команду из чата с идентификатором `ADMIN_GROUP_ID`.

### `/reply`

| | |
|---|---|
| **Формат** | `/reply <номер_обращения> <текст ответа>` |
| **Пример** | `/reply 42 Здравствуйте! Ваш вопрос рассмотрен, проблема устранена.` |
| **Что делает** | находит обращение по полю `id=N`. Проверяет, что текст ответа не длиннее лимита (`ANSWER_MAX_CHARS=300`). Проверяет поле `users.is_blocked`. Оборачивает текст в формальное письмо с шапкой «📬 Ответ Администрации Елизовского муниципального района…». Доставляет письмо жителю. Меняет `appeals.status='answered'` и `appeals.answered_at=now()`. |
| **Альтернатива** | ответ свайпом по карточке обращения (см. §9). |
| **Журнал действий** | `action='reply_via_command'`, `target='appeal #N'`, `details={"chars": <длина>}`. |
| **Что может пойти не так** | обращение не найдено — оператор увидит «Обращение #N не найдено». Текст длиннее 300 символов — «Ответ длиннее 300 символов (фактически: …)». У жителя `is_blocked=true` — «⚠️ Не могу доставить ответ … житель отозвал согласие на ПДн». MAX недоступен — «⚠️ Не удалось доставить ответ жителю по обращению #N (<тип_исключения>)». |

### `/stats`

| | |
|---|---|
| **Формат** | `/stats today`, `/stats week` или `/stats month`. Без параметра считается `today`. |
| **Что делает** | формирует выгрузку обращений за выбранный период в формате XLSX и отправляет файл в админ-группу. В файле 11 колонок: номер, дата, имя, телефон, адрес, тематика, суть, статус, ответ оператора, время ответа в часах, попадание в нормативный срок (да или нет). |
| **Альтернатива** | кнопка «📊 Статистика за сегодня» в панели `/op_help`. Эта кнопка делает только `today`. |
| **Лимит** | период строго фиксирован: `today`, `week` или `month`. Произвольные даты команда не принимает. |
| **Журнал действий** | не пишет (это операция только на чтение). |

Раз в месяц первого числа в 09:00 по часовому поясу из настройки `TZ` бот сам отправляет в админ-группу отчёт за месяц. Отдельно настраивать ничего не нужно.

### `/reopen`

| | |
|---|---|
| **Формат** | `/reopen <N>` |
| **Что делает** | возвращает обращение в статус `in_progress`. Обнуляет поля `answered_at` и `closed_at`. Жителю ничего не отправляется. |
| **Журнал действий** | `action='reopen'`, `target='appeal #N'`. |
| **Когда нужно** | если оператор хочет вернуть закрытое обращение в работу вручную. Когда житель сам присылает новое сообщение по старому обращению, бот переоткрывает обращение автоматически. Ручная команда тогда не нужна. |

### `/close`

| | |
|---|---|
| **Формат** | `/close <N>` |
| **Что делает** | закрывает обращение без ответа жителю. Ставит `status='closed'` и `closed_at=now()`. Жителю ничего не уходит. |
| **Журнал действий** | `action='close'`, `target='appeal #N'`. |
| **Когда нужно** | спам, дубликат уже принятого обращения, обращение не по адресу. |

### `/diag`

| | |
|---|---|
| **Что делает** | присылает счётчики и текущие настройки. В выводе: количество жителей (всего, подписанных, заблокированных), обращений (всего, в работе), рассылок (✅ выполнено и ⚠️ упало с ошибкой), события (всего и время последнего), режим работы бота, лимит длины ответа, нормативный срок. |
| **Когда нужно** | первая проверка при инциденте. Если цифры выглядят разумно, значит, база и обработчик событий живы. |

### `/op_help`

См. §7. Команда присылает в админ-группу сообщение с тремя кнопками быстрых действий и закрепляет его.

### `/whoami`

| | |
|---|---|
| **Что делает** | возвращает `max_user_id` оператора, его `first_name` и `chat_id` админ-группы. |
| **Когда нужно** | при первой настройке бота — взять `chat_id` для строки `ADMIN_GROUP_ID` в `.env`. Когда добавляется новый оператор — узнать его `max_user_id`, чтобы передать в `/add_operators`. |

### `/help`

| | |
|---|---|
| **Что делает** | присылает короткую справку по командам оператора. Расширенная версия открывается через кнопку «📋 Все команды» в панели `/op_help`. |

### `/broadcast`

См. §8. Это мастер из двух шагов. Доступен только ролям `coordinator` и `it`.

## 6. Команды только для роли `it`

Эти команды защищены проверкой `_ensure_role(event, OperatorRole.IT)`. Остальные роли получают отказ. Каждый успешный вызов попадает в журнал действий `audit_log`.

### `/erase`

| | |
|---|---|
| **Формат** | `/erase max_user_id=<id>` или `/erase phone=+7XXXXXXXXXX`. |
| **Что делает** | обезличивает данные жителя в одной транзакции с записью в `audit_log`. Это требование статьи 21 Федерального закона 152-ФЗ. Эффект совпадает с тем, что делает команда `/forget` от самого жителя. Поиск по телефону идёт по индексированной колонке `phone_normalized`. Нормализация — только цифры, у одиннадцатизначных номеров обрезается ведущая 7 или 8. |
| **Журнал действий** | `action='erase'`, `target='user max_id=<id>'`. |
| **Что может пойти не так** | пользователь не найден — «Пользователь не найден.». Битый формат — пример из подсказки. |
| **Защита от ошибок** | если ИТ-оператор случайно вызовет `/erase` на самого себя, специальной блокировки нет. На практике в таблице `users` записи об операторе обычно нет (он не житель), поэтому случайно удалить себя не получится. |

### `/setting`

| | |
|---|---|
| **Формат** | `/setting list` или `/setting <ключ> <значение>`. |
| **Что делает** | редактирует ключ в таблице `settings` (полный список — в §10). Значение бот пробует разобрать как JSON. Если не получилось, сохраняет как обычную строку. Каждое значение проверяется по правилам из `settings_store.SCHEMA`. Если проверка не прошла, оператор увидит «⚠️ Настройка не обновлена: <причина>». |
| **Журнал действий** | `action='setting_update'`, `target=<ключ>`, `details={"kind": "<тип>", "chars": <длина>}`. Само значение в журнал не пишется, чтобы там не дублировались персональные данные. |
| **`/setting list`** | возвращает список доступных ключей. |

### `/add_operators`

| | |
|---|---|
| **Формат** | многострочный. Первая строка — `/add_operators`. Дальше по одной записи на строку: `<max_user_id> <роль> <ФИО полностью>`. |
| **Пример** | <pre>/add_operators<br>234567 aemr Иванова Анна Петровна<br>345678 egp Петров Пётр Иванович</pre> |
| **Что делает** | добавляет или обновляет каждого оператора в таблице `operators`. Доступные роли: `coordinator`, `aemr`, `egp`, `it`. По итогу команда возвращает сводку: «Добавлено / Обновлено / Ошибок». |
| **Журнал действий** | `action='operator_upsert'`, `target='max_id=<id>'`, `details={"role": "<роль>", "full_name": "<ФИО>"}`. |
| **Защита от смены своей роли** | оператор не может через эту команду поменять собственную роль. Проверка делается по равенству `actor_id == target_id`. Изменить свою роль можно только напрямую через `psql`. |

### `/backup`

| | |
|---|---|
| **Что делает** | запускает `pg_dump` вне расписания. Внутри вызывается та же функция `_backup_db`, что работает по расписанию. Команда возвращает имя файла дампа и его размер. |
| **Когда нужно** | перед миграцией базы. Перед сбросом тестовых данных. Перед любой опасной операцией. |
| **Что может пойти не так** | если дамп упал, бот пришлёт «⚠️ Бэкап упал…». Подробности в логах контейнера. |

## 7. Панель `/op_help` и быстрые действия

Команда `/op_help` присылает в админ-группу сообщение с тремя кнопками. Бот автоматически закрепляет это сообщение, чтобы оно всегда было под рукой.

| Кнопка | Значение кнопки | Что делает |
|---|---|---|
| 📊 Статистика за сегодня | `op:stats_today` | То же, что `/stats today`. Не нужно набирать команду. |
| 📢 Сделать рассылку | `op:broadcast` | Запускает мастер `/broadcast`. |
| 📋 Все команды | `op:help_full` | Полная справка по командам оператора с примерами. |

«Все команды» удобно скинуть новому оператору в первый день. В дальнейшем он откроет закреплённое сообщение сам.

## 8. Мастер `/broadcast` пошагово

Команда доступна ролям `coordinator` и `it`. Мастер состоит из двух шагов. Он защищён от случайной отправки и от двойного запуска.

```
1. /broadcast (или кнопка «📢 Сделать рассылку» в /op_help)
   → Бот: «Введите текст рассылки (≤1000 символов)»

2. <текст рассылки>
   → Бот: предпросмотр и вопрос «Разослать N подписчикам?»
   → две кнопки: ✅ Разослать  ❌ Отмена

3а. ✅ Разослать → бот запускает цикл отправки.
    - В админ-группе появляется сообщение с прогрессом:
      «⏳ X/N доставлено» и кнопкой «⛔ Экстренно остановить».
    - Шаг обновления прогресса подстраивается под объём
      рассылки. Короткая рассылка обновляется чаще, длинная
      реже. Это нужно, чтобы не упереться в ограничение MAX
      на частоту правок сообщения.
    - Скорость отправки задана через `BROADCAST_RATE_LIMIT_PER_SEC`
      (по умолчанию 1.0, то есть одно сообщение в секунду).
    - Под каждым сообщением, которое получает житель, есть
      кнопка «🔕 Отписаться от рассылки».

3б. ❌ Отмена → мастер сброшен, рассылка не запущена.
```

Команда `/broadcast list` показывает последние 10 рассылок с их статусами и счётчиками.

**Экстренный стоп.** Кнопка «⛔» доступна **любому** оператору в админ-группе. Цикл прерывается на следующей итерации. Счётчики `delivered_count` и `failed_count` фиксируются в базе. Статус рассылки меняется на `cancelled`.

**Кому уходит рассылка.** Только живым подписчикам. Условие в SQL: `subscribed_broadcast=true AND is_blocked=false AND first_name <> 'Удалено'`. Жители, отозвавшие согласие через `/forget` или `/erase`, исключаются автоматически.

**Восстановление после сбоя.** Если процесс бота упал в середине рассылки, на старте функция `reap_orphaned_sending` находит висящие в статусе `SENDING` записи и помечает их как `FAILED`. Время `finished_at` при этом не подделывается. Если рассылку нужно завершить, оператор запускает её заново вручную.

**Журнал действий.** `action='broadcast_send'`, `target='broadcast #<id>'`, `details={"chars": <длина>, "subscriber_count": <N>}`. Сам текст рассылки лежит в `broadcasts.text`. В журнал он не дублируется.

## 9. Ответ свайпом и команда `/reply`

Оба способа в итоге попадают в один и тот же код доставки — функцию `_deliver_operator_reply`. Различается только то, как оператор указывает, на какое обращение он отвечает.

### Способ 1 — ответ свайпом

«Ответ свайпом» — это жест проведения вправо по карточке обращения в админ-группе. На некоторых клиентах MAX вместо свайпа работает долгое нажатие и пункт «Ответить» в меню. Дальше оператор пишет текст ответа.

Бот ловит событие `message_created` с полем `link.type=REPLY`. Из него он читает `link.message.mid` — идентификатор исходного сообщения. По этому идентификатору функция `appeals.get_by_admin_message_id` находит обращение.

Если клиент MAX **не передаёт ссылку на исходное сообщение** (это зависит от версии и платформы), бот пишет в лог «no reply-link in event.message — message ignored». В таком случае нужно использовать второй способ.

### Способ 2 — команда `/reply N <текст>`

Прямая команда. Работает на всех клиентах. Не зависит от свайпа. Номер обращения виден явно. Меньше шансов ответить не на то обращение.

### Общий путь обоих способов

Дальше шаги одинаковые.

1. Если длина текста больше 300 символов — отказ в админ-группу. Жителю ничего не уходит.
2. Если у жителя `is_blocked=true` (после `/erase` или `/forget`) — отказ в админ-группу. Жителю ничего не уходит.
3. Бот вызывает `bot.send_message(user_id, formatted_text)`. Текст обёрнут в формальную шапку.
4. В таблицу `messages` пишется запись с `direction='from_operator'`.
5. У обращения проставляется `status='answered'` и `answered_at=now()`.
6. В журнал действий пишется запись. Для свайпа — `action='reply'`. Для команды — `action='reply_via_command'`.
7. В админ-группу уходит подтверждение: «✉️ Ответ ушёл жителю. Обращение #N закрыто».

## 10. Ключи команды `/setting`

Это полный список настроек, которые редактируются из админ-группы. Все изменения — через `/setting list` и `/setting <ключ> <значение>`. Правила проверки описаны в `services/settings_store.py::SCHEMA`.

| Ключ | Тип | Лимиты | Что это |
|---|---|---|---|
| `welcome_text` | string | 1–4000 символов | приветственное сообщение для команды `/start`. |
| `consent_text` | string | 1–4000 символов | текст согласия на обработку ПДн. Показывается перед PDF политики. |
| `policy_url` | string | URL (http или https) | ссылка на PDF политики на сайте АЕМР. |
| `electronic_reception_url` | string | URL | ссылка на электронную приёмную (используется в кнопке-ссылке). |
| `udth_schedule_url` | string | URL | PDF с расписанием муниципальных автобусов УДТХ. |
| `udth_schedule_intermunicipal_url` | string | URL | страница с межмуниципальными маршрутами на kamgov.ru. |
| `appointment_text` | string | 1–2000 символов | расписание приёма граждан главой района и телефон записи. |
| `emergency_contacts` | list[object] | минимум 1 элемент, ключи `name` и `phone` | экстренные службы. Пример: `[{"name":"112","phone":"112"}, ...]`. |
| `transport_dispatcher_contacts` | list[object] | минимум 1, ключи `routes` и `phone` | диспетчерские автотранспорта. Пример: `[{"routes":"101,102","phone":"+7..."}, ...]`. |

Примеры использования.


Битый JSON, неверный тип, нарушение лимитов — оператор увидит «⚠️ Настройка не обновлена: <причина>». Старое значение остаётся прежним. Житель не заметит сбоя.

## 11. Что записывается в журнал действий

Таблица `audit_log` работает только на дозапись. Это нужно для аудита со стороны муниципалитета. Каждая запись содержит пять полей: `operator_max_user_id` (кто), `action` (что), `target` (над чем или над кем), `details` (метаданные действия), `created_at` (время).

| Действие | `action` | `target` | Что в `details` |
|---|---|---|---|
| Ответ свайпом | `reply` | `appeal #N` | `{"chars": <длина>}` |
| Команда `/reply N` | `reply_via_command` | `appeal #N` | `{"chars": <длина>}` |
| `/reopen N` | `reopen` | `appeal #N` | `{}` |
| `/close N` | `close` | `appeal #N` | `{}` |
| `/erase` (любая форма) | `erase` | `user max_id=<id>` | `{}` |
| `/forget` от жителя | `self_erase` | `user max_id=<id>` | `{}` |
| `/setting <ключ> <значение>` | `setting_update` | `<ключ>` | `{"kind": "<тип>", "chars": <длина>}` или `{"kind": "<тип>", "items": <количество>}` |
| `/add_operators` (на каждого) | `operator_upsert` | `max_id=<id>` | `{"role": "<роль>", "full_name": "<ФИО>"}` |
| Подтверждение `/broadcast` | `broadcast_send` | `broadcast #<id>` | `{"chars": <длина>, "subscriber_count": <N>}` |

В поле `details` **не пишутся** тексты обращений, ответов, рассылок и значений `/setting`. Только метаданные: длина, тип, количество элементов. Сами тексты лежат в основных таблицах: `messages.text`, `broadcasts.text`, `settings.value`.

## 12. Лимиты и проверка ввода

Все лимиты можно изменить через `.env`. Образец — в файле `infra/.env.example`.

| Лимит | По умолчанию | Что ограничивает |
|---|---|---|
| `ANSWER_MAX_CHARS` | 300 | длина ответа оператора жителю (ответ свайпом или `/reply`). |
| `SUMMARY_MAX_CHARS` | 2000 | накопленная длина текста на шаге `awaiting_summary`. |
| `NAME_MAX_CHARS` | 120 | длина имени жителя на шаге `awaiting_name`. |
| `ADDRESS_MAX_CHARS` | 500 | длина адреса на шаге `awaiting_address`. |
| `ATTACHMENTS_MAX_PER_APPEAL` | 20 | количество вложений в одном обращении. |
| `ATTACHMENTS_PER_RELAY_MESSAGE` | 10 | сколько вложений уходит в админ-группу одним сообщением. Длинные обращения режутся на части `(1/3)`, `(2/3)` и так далее. |
| `BROADCAST_MAX_CHARS` | 1000 | длина текста рассылки. |
| `BROADCAST_RATE_LIMIT_PER_SEC` | 1.0 | скорость отправки сообщений рассылки (сообщений в секунду). |
| `APPEAL_TIMEOUT` | 60 | сколько секунд тишины на шаге `awaiting_summary` приводит к автоматическому завершению анкеты. |
| `RECOVER_BATCH_SIZE` | 1000 | размер пакета застрявших анкет, который восстанавливается за один проход при старте. |

Превышение лимита на ответ оператора — отказ в админ-группу. Превышение со стороны жителя — бот молча отбрасывает ввод и пишет об этом в лог.

## 13. Нетипичные ситуации и возможные ошибки

**Житель заблокировал бота в MAX.** Ответ свайпом или `/reply` упадут с исключением от MAX. Оператор увидит сообщение «⚠️ Не удалось доставить ответ жителю по обращению #N (<имя_исключения>). Возможно, житель удалил диалог или заблокировал бота.». Обращение остаётся в работе. Координатор связывается с жителем по телефону из карточки.

**Житель отозвал согласие через `/forget` или `/erase`, но обращение всё ещё открыто.** Любая попытка ответа от оператора покажет «⚠️ Не могу доставить ответ … житель отозвал согласие на ПДн». Обращение закрывается командой `/close N`.

**Двойной клик по «✅ Отправить» при завершении анкеты.** Защищает блокировка `asyncio.Lock` на пользователя плюс проверка `dialog_state == idle`. В таблице `appeals` создаётся ровно одна запись.

**Бот перезапустился, пока житель был на шаге `awaiting_summary` дольше 60 секунд.** На старте функция `recover_stuck_funnels` достаёт таких жителей пакетами по `RECOVER_BATCH_SIZE` и завершает их анкеты. Пустые сессии (нет ни текста, ни вложений) сбрасываются в `idle`.

**Бот перезапустился в середине рассылки.** На старте функция `reap_orphaned_sending` помечает зависшие в статусе `SENDING` рассылки как `FAILED`. Если рассылку нужно завершить, оператор запускает новую. Текст исходной рассылки сохранён в `broadcasts.text`.

**Оператор пишет в админ-группу без ссылки на исходное сообщение.** В лог пишется «operator_reply: no reply-link in event.message — message ignored». Бот ничего не делает. Если оператор хотел ответить, нужно использовать команду `/reply N <текст>`.

**Дубль входящего события от MAX.** Промежуточный слой обработки `IdempotencyMiddleware` (защита от повторов — это значит, что одно и то же событие, обработанное дважды, не приведёт к двойной записи) отбивает дубликаты по уникальному индексу `events.idempotency_key`. Дубль молча отбрасывается ещё до обработчика.

**Оператор в админ-группе нажал старую кнопку из карточки обращения жителя.** Нажатия с префиксами `menu:*`, `consent:*`, `topic:*`, `appeal:*`, `info:*` и `cancel` в админ-чате игнорируются на уровне `on_callback`. Оператор не попадает в анкету как «житель».

**В настройке `settings.topics` поменялся набор тематик, а у жителя в `dialog_data` сохранён старый индекс.** Проверка диапазона `0 <= idx < len(topics)` отрезает невалидный выбор. Жителю достаточно переоткрыть меню.
