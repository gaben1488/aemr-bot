# ADR-001: Архитектура MVP «Администрация ЕМР. Обратная связь»

**Статус:** v6 — готовая к эксплуатации сборка MVP
**Дата:** 2026-05-04
**Автор:** filat (с участием Claude Code)

## 1. Контекст

Администрация Елизовского муниципального района (АЕМР, Камчатский край) хочет канал обратной связи от жителей в мессенджере MAX. Канал дополняет существующий Telegram-чат. В качестве образца взят бот «Солодов. Обратная связь» Камчатского правительства. Там административная панель устроена как обычный групповой чат: туда падают обращения, оператор отвечает обычным ответом на сообщение, бот пересылает ответ в личку жителю и публикует в группе подтверждение.

Управляющие ограничения проекта:

- **Платформа.** Только MAX (программный интерфейс ботов на `platform-api.max.ru`).
- **Юрлицо АЕМР верифицировано** на dev.max.ru. Токен бота можно получить.
- **Скорость важнее перфекционизма.** Старт MVP за два-три рабочих дня вместо двух недель.
- **Инфраструктура.** Российский виртуальный сервер, центр обработки данных в РФ (статья 18.5 закона 152-ФЗ «О персональных данных»).
- **Уровень защищённости персональных данных.** УЗ-4 (четвёртый уровень защищённости персональных данных по Постановлению Правительства РФ № 1119, далее ПП №1119). Аттестация провайдера не требуется.
- **Бюджет.** Хостинг около 1 700 ₽ в месяц. Свой код, не подписочный сервис.

Что переиспользуется из готовых наработок. Из каталога `telegram_analytics/agent_export/` берём `operator_templates.json` (33 шаблона ответов и ранжированный каталог). Файл кладём в `seed/templates.json`. В MVP оператору эти подсказки не показываются (см. §11 «Возможные направления развития»), но материал лежит под рукой. Системный текст-инструкция и чек-лист качества в MVP не используются. На обращения отвечает живой оператор без помощника на основе языковой модели.

## 2. Решение в одной фразе

Один сервис на Python поверх библиотеки `maxapi`. Внутри одного процесса — PostgreSQL и планировщик фоновых задач APScheduler. Связь с MAX — опросный режим (бот сам периодически спрашивает MAX о новых сообщениях). Никаких отдельных серверной части, клиентской части, мини-приложения. Административная панель — это групповой чат в MAX.

## 3. Архитектурные блоки

### 3.1 Бот

Единственный сервис. Делает всё сам. Разговаривает с жителями в личных диалогах. Разговаривает с операторами в служебных группах. Запускает APScheduler (планировщик фоновых задач) для задач по расписанию. Ходит в базу через библиотеку SQLAlchemy (Python-библиотека для работы с БД).

Режим связи на старте — опросный (long polling: бот сам периодически спрашивает MAX о новых сообщениях). Преимущества для MVP:

- Не нужен публичный HTTPS-адрес, домен и сертификат. Это экономит первую неделю.
- Бот разворачивается одной командой `docker compose up`.
- Локальная разработка идёт с любой машины с интернетом.

У опросного режима есть один недостаток. При простое процесса MAX держит очередь до 8 часов. Если перерыв дольше — события теряются. Для MVP с одним работающим экземпляром это допустимо. Когда сборка стабилизируется (через две-четыре недели после публикации), переключаемся на серверный режим связи (webhook: MAX шлёт события на наш сервер). Код переписывать не понадобится: добавляются Nginx, обычный обработчик HTTP-запросов и подписка через `POST /subscriptions`.

Защита от повторов реализована через ограничение уникальности на колонку `events.idempotency_key`. В качестве ключа берём `update_id` или комбинацию `update_type + message_id + timestamp`.

Команды бота для жителя: `/start`, `/menu`, `/help`, `/policy`, `/subscribe`, `/unsubscribe`, `/forget`, `/whoami`. Главное меню — пять кнопок прямо под сообщением (inline).

Команды бота в административной группе:

- `/stats today`, `/stats week`, `/stats month` — выгрузка обращений за период в формате XLSX.
- `/reopen NNN` — вернуть обращение в статус «в работе».
- `/close NNN` — закрыть обращение без ответа.
- `/op_help` — закрепляемая памятка с кнопками для операторов.

Индексы: `users.max_user_id` (уникальный), `events.idempotency_key` (уникальный), `appeals.status`, `appeals.created_at`, `messages.appeal_id`, `messages.created_at`.

### 3.3 Административная панель = групповой чат в MAX

**Это не отдельное приложение.** Административная панель — это служебная группа в MAX. В неё добавлены бот и операторы.

Поток обработки обращения:

1. Житель в личном диалоге с ботом отправляет обращение (текст с фото или геолокацией по желанию).
2. После 60 секунд тишины (или нажатия кнопки «Отправить» в боте) обращение получает статус `new`. Бот публикует в административную группу карточку:

   ```
   📨 Новое обращение #107
   👤 Имя: Алексей
   📞 Телефон: 89964240723
   ──────────
   <текст обращения>
   <фото вложением>
   ```

3. Координатор открывает карточку и **отвечает на сообщение бота через цитирование** (в MAX это обычный жест ответа на сообщение).
4. Бот ловит событие `message_created` в административной группе с полем `link.type=reply`. По `link.mid` находит обращение, проверяет длину ответа (не больше 300 символов) и передаёт ответ жителю через `POST /messages?user_id=…`.
5. После успешной доставки бот публикует в административной группе подтверждение:

   ```
   ✉️ Ответ успешно отправлен пользователю
   Обращение #107 помечено как завершённое
   ```

6. В БД ставится `appeals.status=answered`, `appeals.answered_at=now()`. В таблицу `messages` добавляется запись с `direction=from_operator`.

Если оператор пишет ответ длиннее 300 символов, бот возвращает в группу: «Ответ длиннее 300 символов, отредактируйте и пришлите снова». Жителю при этом ничего не уходит.

Если житель присылает следующее сообщение в обращение со статусом `answered`, бот заново открывает обращение (`status=in_progress`) и публикует новое сообщение в группе как продолжение цепочки (с цитатой исходной карточки).

**Роли** в первой итерации проще, чем в первоначальной редакции ADR:

- **Координатор АЕМР** (вы). Добавлен в административную группу. Отвечает на обращения. Имеет доступ к командам `/stats`, `/reopen`, `/close`.
- **ИТ-специалист.** Добавлен в группу. Видит всё. Использует те же команды плюс `/diag` для диагностики бота, `/setting`, `/erase`, `/add_operators`, `/backup`.
- **Специалист ЕГП по ответам.** Возможны два варианта: отдельная служебная группа (`admin_group_id_egp`) либо общая группа с указанием на конкретные обращения через упоминание `@username`. На MVP делаем простейший вариант: одна общая административная группа, специалист ЕГП добавлен как обычный оператор. Если потребуется разделение — это направление развития (см. §11).

### 3.4 Инфраструктура

```
                Интернет ◄──── опросный режим (исходящий) ──── bot ───► PostgreSQL
                                                              │
                                                              └─► APScheduler
                                                                  (еженедельный
                                                                   pg_dump в named-volume,
                                                                   опционально gpg+S3)
```

Один виртуальный сервер. Два контейнера в `docker-compose`:

- `bot` — Python 3.12, библиотеки `maxapi`, SQLAlchemy, APScheduler, `openpyxl`.
- `db` — PostgreSQL 16, том на хосте.

Никакого Nginx. Никакого обратного прокси. Никаких внешних портов наружу. Бот ходит только исходящими запросами к `platform-api.max.ru`. На сервере открыт только SSH (порт 22) для администратора.

Резервное копирование PostgreSQL устроено так. Задача APScheduler раз в сутки запускает `pg_dump`, шифрует результат через `gpg` (пароль хранится в `.env`), отправляет в Yandex Object Storage через `rclone` (около 50 ₽ в месяц за 50 ГБ). Срок хранения — 30 дней с ротацией.

Готовая инфраструктура серверного режима связи (Nginx, утилита Let's Encrypt для сертификата, обработчик `/max/webhook` с проверкой `WEBHOOK_SECRET`) лежит в опциональном профиле `webhook` в `docker-compose`. По умолчанию профиль не активен. Переключение — `BOT_MODE=webhook` в `.env` плюс `docker compose --profile webhook up -d`. См. §11 «Возможные направления развития».

### 3.5 Поток данных от жителя до ответа (пошаговая анкета)

Главное меню жителя — пять кнопок: «Написать обращение», «Мои обращения», «Электронная приёмная», «Приём граждан», «Полезная информация». Согласие на обработку персональных данных запрашивается **внутри** воронки «Написать обращение», а не сразу после `/start`. Так житель может посмотреть контакты и свои прошлые обращения без согласия.

Воронка приёма обращения работает как пошаговая анкета (конечный автомат, состояние хранится в БД в колонке `users.dialog_state`, накопленные данные — в `users.dialog_data` JSONB). Шаги:

1. **Согласие (`AWAITING_CONSENT`).** Текст политики со ссылкой и две кнопки: «Согласен» и «Отказаться». Шаг пропускается, если `users.consent_pdn_at IS NOT NULL`. Отказ возвращает в главное меню.
2. **Контакт (`AWAITING_CONTACT`).** Кнопка `request_contact`, телефон извлекается из вложения формата vCF. Шаг пропускается, если телефон уже есть.
3. **Имя (`AWAITING_NAME`).** Свободный ввод. Только имя, без фамилии (как в боте-образце).
4. **Адрес (`AWAITING_ADDRESS`).** Свободный ввод улицы и дома. Отдельного шага «район» нет — Елизовский МО единственный.
5. **Тематика (`AWAITING_TOPIC`).** Кнопки прямо под сообщением, по две в ряд, из списка `settings.topics` (11 пунктов из стартового набора). Выбор пишется в `dialog_data.topic`.
6. **Суть (`AWAITING_SUMMARY`).** Свободный ввод текста плюс необязательные фото или геолокация. Каждое сообщение в этом состоянии добавляется в `dialog_data.summary_chunks` и `dialog_data.attachments`. Завершение — кнопкой «Отправить» либо автоматически по тишине в 60 секунд (`APPEAL_TIMEOUT`).
7. **Финализация.** Бот создаёт `appeals.NNN` со структурированными полями `address/topic/summary/attachments`, публикует карточку в административной группе (формат идентичен карточке в «Мои обращения»), сохраняет `appeals.admin_message_id` для последующих обновлений, обнуляет состояние анкеты (`dialog_state = idle`), отвечает жителю «Обращение #N принято, ответим в течение 4 рабочих часов».

После ответа оператора (через цитирование в группе) и доставки жителю обращение получает статус `answered`. Если житель пишет в состоянии `idle` новое сообщение, и у него есть активное обращение со статусом `answered`, бот открывает его заново (`status = in_progress`) и публикует в группе сообщение-продолжение с цитатой исходной карточки.

На любом шаге воронки доступна кнопка «Отмена». Она обнуляет состояние анкеты и возвращает житель в главное меню без сохранения.

## 4. Лимит ответа 200–300 символов

Жёсткий лимит **300 символов** в программном интерфейсе бота. При попытке отправить более длинный ответ — отказ с сообщением в административной группе (см. 3.3, пункт 4). Без модального окна и без автоматической разбивки. Так упрощается интерфейс, оператор сам решает, что делать.

Обоснование цифры — в требованиях куратора (200–300). Берём верхнюю границу, чтобы не ломать стандартные шаблоны на 250–280 символов из `agent_export/operator_templates.json`.

## 5. Безопасность и соответствие закону 152-ФЗ

**Класс информационной системы персональных данных: УЗ-4** по ПП №1119:

- Категория персональных данных — иные (имя, фамилия, телефон, текст обращений, фото и геолокация в качестве вложений).
- Тип угроз — третий.
- Количество субъектов — менее 100 000.
- Категория субъектов — не сотрудники оператора.

Обязательные меры:

- Хранение данных на территории РФ (статья 18.5 закона 152-ФЗ). Сервер — в Москве или Санкт-Петербурге.
- Согласие пользователя на обработку персональных данных при первом взаимодействии (фиксация в `users.consent_pdn_at`).
- Публикация политики на сайте АЕМР со ссылкой из бота. Текст политики лежит в `docs/Политика.md`, согласован с куратором.
- АЕМР зарегистрирована в реестре операторов персональных данных Роскомнадзора (проверить по ИНН в открытом реестре).
- Базовая гигиена информационной безопасности на сервере: межсетевой экран `ufw`, доступ по SSH-ключам, отключение входа под `root`, `fail2ban`, `unattended-upgrades`, шифрование резервных копий через `gpg`.

**Не требуется** для УЗ-4: аттестация провайдера, средства криптографической защиты, аттестация информационной системы персональных данных, расширенная модель угроз.

**Удаление по запросу.** Команда `/erase max_user_id=NNN` или `/erase phone=+7...` от ИТ-специалиста в административной группе. Либо текстовая команда жителя в боте `/forget`. Алгоритм: `users.first_name='Удалено'`, `users.phone=NULL`, `users.consent_pdn_at=NULL`, сброс состояния анкеты, `is_blocked=true`, `subscribed_broadcast=false`, запись в журнал действий. Удаление завершается за минуту, в пределах 24 часов, как требует закон 152-ФЗ.

## 7a. Эксплуатационные компоненты

**Проверка живости.** Файл `bot/aemr_bot/health.py` поднимает HTTP-сервер на основе `aiohttp` на порту `WEBHOOK_PORT` (по умолчанию 8080) с обработчиком `/healthz`. В `docker-compose` порт публикуется на `127.0.0.1:8080`. Это нужно только для локальной проверки живости контейнера. Наружу `/healthz` не выставляется (модель самостоятельного хостинга, никаких входящих портов). Внутри — единый объект `Heartbeat` обновляется фоновой задачей `heartbeat_pulse` каждые `HEALTHCHECK_PULSE_SECONDS`. Если основной цикл завис, сигнал жизни перестаёт обновляться, и `/healthz` отдаёт 503 после `HEALTHCHECK_STALE_SECONDS`. Основной потребитель — внутренняя задача `selfcheck` по расписанию. Она публикует в административную группу сообщения-уведомления «упало» и «восстановилось». Если у администратора есть собственный сборщик состояния внутри корпоративной сети, можно настроить исходящий пинг на `HEALTHCHECK_URL`.

**Защита от повторов.** Функция `services/idempotency.py::claim` строит ключ из `update_type`, `callback_id`, `mid`, `seq`, `timestamp`, `chat`, `user`. Делает `INSERT ON CONFLICT DO NOTHING` в таблицу `events` и возвращает True или False. Подключается через внешний промежуточный слой обработки в `handlers/__init__.py::IdempotencyMiddleware`. Дубликаты от MAX (особенно при повторных доставках в серверном режиме) отбрасываются молча, ещё до того как попадут в любой обработчик.

**Восстановление зависших анкет.** Функция `handlers/appeal.py::recover_stuck_funnels` запускается через `asyncio.create_task` при старте. Через `users_service.find_stuck_in_summary(idle_seconds)` находит пользователей в состоянии `AWAITING_SUMMARY`, у которых `updated_at` старше таймаута. Параллельно через `asyncio.gather` финализирует их обращения. Пустые отправки (нет ни текста, ни вложений) сбрасываются в `IDLE`, чтобы не зацикливаться.

**Асинхронное резервное копирование.** Функция `services/cron.py::_backup_db` теперь полностью асинхронная. Она использует `asyncio.create_subprocess_exec` для конвейера `pg_dump | gpg | rclone`. Цикл событий `asyncio` не блокируется на время дампа.

**Загрузка вложений.** Файл `services/uploads.py` — единый адаптер над `bot.upload_media`. Используется и в `services/policy.py` (PDF на старте), и в `_build_admin_senders` в `main.py` (XLSX по `/stats` и в ежемесячном отчёте). Клиент `services/cron.py::send_admin_document` теперь действительно доставляет файл, а не текстовую заглушку.

**Обработчики анкеты.** Функция `handlers/appeal.py::on_message` — это диспетчер по справочнику `_STATE_HANDLERS`. Каждое состояние — отдельная функция (`_on_awaiting_contact`, `_on_awaiting_name`, и так далее) с одной и той же сигнатурой `(event, body, text_body, max_user_id, op_reply)`. Добавление нового состояния — это новая функция и одна строка в справочнике.

## 8. Что меняется при изменении вводных

- Безопасник АЕМР требует аттестации хостинга. Тогда пересматриваем серверное окружение в рамках самостоятельного хостинга (другая стойка, другая конфигурация ОС). Облачные провайдеры в этом проекте не используются.
- Куратор хочет жёсткое разделение АЕМР и ЕГП. Тогда создаём вторую административную группу. В таблице `settings` указываем `admin_group_id_egp`. Добавляем правило маршрутизации в `settings.egp_keywords` (ключевые слова темы ЕГП).
- Появятся подписки и рассылки. Тогда добавляем поля `consent_chs`, `consent_events` в `users` и таблицы `broadcasts`, `broadcast_deliveries`. Это чистая надстройка, ломать MVP не понадобится.
- Бот вырастает до федеральных функций (приём обращений по 59-ФЗ). Тогда пересматриваем уровень защищённости и провайдера.

## 9. Дополнения и уточнения после первичной реализации

Эти решения внесены поверх архитектуры разделов 1–8 без её переписывания. Каждый пункт — фиксация фактического состояния кода и обоснование выбора. Содержание раздела отражает то, что действительно работает в производственной сборке MVP.

### 9.1 Хранилище состояний анкеты: Postgres JSONB, не Redis

`users.dialog_state` (`String(32)`) и `users.dialog_data` (JSONB) остаются основным хранилищем состояния пошаговой анкеты. Перенос в Redis рассматривался как умозрительная оптимизация при росте числа одновременно активных воронок выше тысячи. Для АЕМР это сценарий «никогда»: население района около 70 тысяч человек, проникновение государственных цифровых сервисов 3–5%. Активных воронок одновременно — десятки в часы пик, единицы ночью. Пиковая нагрузка 1–2 операции UPDATE в секунду. Postgres проглатывает не замечая. Триггер пересмотра решения — устойчивые показатели свыше 500 одновременных воронок по метрикам диагностики (`/diag`).

### 9.2 Промежуточный слой защиты от повторов — обязательный, не «страховка»

В разделе 7a защита от повторов описывалась как опциональный компонент. По факту `IdempotencyMiddleware` подключён как внешний промежуточный слой (`outer_middleware`) в `handlers/__init__.py::register_handlers` и срабатывает на каждое обновление до любого обработчика. Дубликаты обновлений от MAX (редкие в опросном режиме, частые при повторных доставках в серверном режиме) молча отбрасываются на уровне `INSERT ON CONFLICT` в `events`. Без этого слоя переход на серверный режим связи дал бы дубли карточек в административной группе.

### 9.3 Восстановление зависших анкет — фоновая задача при старте

Функция `handlers/appeal.py::recover_stuck_funnels` запускается через `asyncio.create_task` в `main.py::main()` и не блокирует старт диспетчера (объект из `maxapi`). Пустые отправки (нет ни текста, ни вложений) сбрасываются в `IDLE`, чтобы не зацикливаться при каждом перезапуске. Пакетное восстановление ограничено: `RECOVER_BATCH_SIZE=1000`. Это защита от патологии: 10 тысяч застрявших воронок после многочасового простоя дали бы 10 тысяч вызовов программного интерфейса при старте.

### 9.4 Время ожидания опроса — настраиваемый параметр, не интервал

Параметр `POLLING_TIMEOUT_SECONDS` (по умолчанию `30`, диапазон `0..90`) задаёт **серверное** время ожидания вызова `getUpdates` в MAX. Это сколько MAX держит соединение, ожидая событий. Это не интервал между запросами клиента. Опросный режим возвращает ответ либо при появлении события, либо по истечении этого времени. Чем выше значение — тем меньше пустых обращений в простое (это важно при ограничении частоты в 2 запроса в секунду на бота, действующем с 11 мая 2026 года), и тем медленнее реакция на остановку. Значение 30 секунд по умолчанию — баланс. Реализация — подмена связанного метода `bot.get_updates` в `main.py::_install_polling_timeout`. Это сделано потому что `maxapi.Dispatcher.start_polling` не пробрасывает параметр времени ожидания внутрь. Если когда-то понадобится свой форк `maxapi`, это первый кандидат на запрос изменений в исходный репозиторий.

### 9.5 Зависимость maxapi — выбор между сообществом и официальным форком

В экосистеме MAX для Python есть три варианта:

| Репозиторий | Тип | Активность | Звёзды |
|---|---|---|---|
| `love-apples/maxapi` | сообщество | активно обновляется (последний коммит — апрель 2026) | 166+ |
| `max-messenger/max-botapi-python` | официальный форк предыдущего, проверенный командой MAX | заморожен на ревизии `2025-07-30`, без обновлений с тех пор | 54 |
| `green-api/max-api-client-python` | клиент через шлюз green-api.com | требует подписку на green-api | — |

Остаёмся на **`love-apples/maxapi`** через ограничение версий `~=0.6` (совместимый выпуск). Аргументация:

- Официальный форк `max-messenger/max-botapi-python` — это **снимок исходного репозитория девятимесячной давности**, а не самостоятельная активная ветка. Переход на него означал бы откат назад с потерей исправлений, накопленных в исходном репозитории после июля 2025 года (включая то, что мы используем для `process_update_webhook`, `TypeAdapter[Attachments]`, `MessageLinkType`).
- `green-api/max-api-client-python` создаёт зависимость от внешнего платного сервиса. Это нарушает пункт «свой код, не подписочный» из контекста ADR §1.

Решение о **создании своего форка `love-apples/maxapi`** принимается по одному из двух условий: (а) три и более месяцев без коммитов в исходный репозиторий и при этом открыты критические задачи; либо (б) обнаружена уязвимость безопасности без реакции мейнтейнера в течение 14 дней. Ежеквартальная проверка активности репозитория — задача координатора. При срабатывании условия форк делается в организацию `gaben1488/maxapi` под контролем муниципалитета.

Дополнительно для предотвращения галлюцинаций ИИ-помощников при работе с программным интерфейсом MAX через Claude, Copilot, Cursor мы взяли на учёт `PronikFire/Max-API-Guide` (реверс-инженерный гайд по WSS и TLS) и подход с файлами `cursorrules` из `max-bot-aio-template`. На MVP их не подключаем. Но при росте кодовой базы — это кандидат на использование как «шпаргалок» для ИИ-помощников.

### 9.6 Пересылка вложений в административную группу

Фото, видео, геолокация и файлы из обращения теперь пересылаются в административную группу вторым сообщением (после текстовой карточки). Сообщение привязывается к карточке через `NewMessageLink(type=REPLY, mid=admin_card_mid)`. Сохранённые в `appeals.attachments` (JSONB) словари десериализуются обратно в pydantic-модели через `TypeAdapter(Attachments)` из библиотеки `maxapi` (Pydantic — Python-библиотека для валидации данных) и передаются в `bot.send_message(attachments=[...])`. Контактные вложения исключены: телефон уже в карточке, дублировать персональные данные не нужно. Каждое сообщение с вложениями ограничено `ATTACHMENTS_PER_RELAY_MESSAGE=10`. Партии пронумерованы «(2/3)» и так далее для удобства чтения.

### 9.7 Массовая регистрация операторов: команда `/add_operators` доступна только ИТ-специалисту

Команда принимает многострочный ввод формата `<max_user_id> <role> <ФИО>`. Для каждой строки — создание или обновление записи оператора с записью в журнал действий. После второго прохода проверки безопасности команда переведена с прав `IT|COORDINATOR` на **только `IT`**. Координатор через эту команду мог бы выдать себе роль ИТ-специалиста и затем дёрнуть `/erase` или `/setting` (повышение привилегий). Дополнительно запрещено самоизменение (`actor_id == target_id`) даже для ИТ-специалиста. Изменения собственной роли идут только через эскалацию прямым SQL-запросом по инструкции из RUNBOOK.

### 9.8 Команда `/policy` для жителя

Житель в любой момент пишет `/policy` и получает PRIVACY.pdf файлом. Цепочка запасных вариантов: (1) кешированный токен из `settings.policy_pdf_token` (заполняется один раз при старте через `policy_service.ensure_uploaded`); (2) загрузка PDF по запросу и кеширование токена; (3) ссылка на адрес из `settings.policy_url`; (4) сообщение об ошибке с предложением написать координатору. Команда добавлена в `/help`.

### 9.9 Стандарт качества ответов АЕМР

Тексты в `texts.py` приведены в соответствие с Приложением 2 к Постановлению Администрации Елизовского муниципального района № 2129 от 19.12.2025 (Стандарт качества ответов на сообщения из открытых источников). Тексты прошли через инструменты humanizer и clarify: убраны склейки через длинное тире, обороты-смягчители, канцеляризмы, аббревиатуры (АЕМР раскрыто в строках, обращённых к жителю), страдательный залог заменён на действительный. Автоматический проверщик ответов оператора против Стандарта — возможное направление развития (см. §11).

### 9.10 Защита от гонок при финализации обращения

Функция `_persist_and_dispatch_appeal` обёрнута в блокировку на пользователя (`asyncio.Lock`, словарь `_user_locks: dict[int, asyncio.Lock]`). Это защищает от двойного клика по «Отправить», от гонки между срабатыванием таймера и нажатием «Отправить», от гонки между отменой и срабатыванием таймера. Внутри блокировки делается проверка на повтор по полю `user.dialog_state == IDLE`. Второй параллельный вызов после успешной финализации первого видит состояние `IDLE` и выходит без создания дубля обращения и карточки. Реализация рассчитана только на один экземпляр сервиса. Горизонтальное масштабирование (более одного контейнера бота) сломает эту схему. Для развёртывания с несколькими экземплярами понадобится `pg_advisory_xact_lock` или блокировка на основе Redis.

### 9.14 Команды `/reply N <текст>` и `/op_help` с кнопками

Команда `/reply <appeal_id> <текст>` доставляет ответ оператора жителю, не полагаясь на ответ свайпом (свайп — жест проведения вправо по сообщению). Команда полезна, когда клиент MAX по каким-то причинам не помечает сообщение как ответ (поле `Message.link` приходит пустым). Также её удобно использовать тем операторам, кто предпочитает явные команды. Логика доставки и журналирования та же, что и у ответа свайпом. Общая функция — `_deliver_operator_reply` в `handlers/operator_reply.py`.

Команда `/op_help` присылает в административную группу памятку с тремя кнопками («📊 Статистика за сегодня», «📢 Сделать рассылку», «📋 Все команды») и автоматически закрепляет это сообщение через `bot.pin_message`. Это ближайший аналог кнопки меню из Telegram. Нативный `set_my_commands` в программном интерфейсе MAX признан устаревшим, поэтому вместо него используем закреплённое сообщение в роли меню.

### 9.15 Защита потока жителя в административной группе

Команды `/start`, `/menu`, `/help`, `/policy`, `/forget`, `/subscribe`, `/unsubscribe` и нажатия кнопок из меню жителя (согласие, тема, информация, отмена) молча игнорируются, если событие пришло из `ADMIN_GROUP_ID`. Реализовано через функцию `is_admin_chat(event)` в `utils/event.py` и проверки в `handlers/start.py` плюс ранний контроль в `handlers/appeal.on_callback`. Без этого оператор, по привычке набравший `/start` в административной группе, получал бы приветственное меню. Его `max_user_id` создавал бы запись в `users`, и оператор мог бы застрять в состоянии `AWAITING_CONSENT`.

### 9.16 Локальное еженедельное резервное копирование

`pg_dump` запускается каждое воскресенье в 03:00 по камчатскому времени и сохраняется в именованный том `backups` (внутри контейнера — `/backups/aemr-YYYYMMDD_HHMMSS.sql`). Ротация — последние `BACKUP_KEEP_COUNT=8` файлов (около двух месяцев истории). Шифрование `gpg` (`*.sql.gpg`) подключается по желанию через `BACKUP_GPG_PASSPHRASE`. Загрузка в S3 подключается по желанию через переменные `BACKUP_S3_*`. Команда `/backup` (доступна только роли `it`) — снятие снимка по запросу, например, перед миграцией БД.

Значение по умолчанию рассчитано на самостоятельный хостинг: для одного сервера без облачного хранилища достаточно именованного тома и ротации. Том переживает пересборки контейнера: команда `docker compose up --build` не пересоздаёт именованные тома.

## 10. Рассылка по подписавшимся жителям

Реализованная возможность. Координатор АЕМР объявляет о чрезвычайных ситуациях, публичных слушаниях, плановых работах коммунальных служб через одну команду в административной группе. Решение появилось из запроса «нужна возможность разослать важное объявление всем подписавшимся».

### 10.1 Согласие

Подписка явная (житель сам нажимает «🔔 Подписаться на новости» в подменю «Полезная информация» либо пишет `/subscribe`). Это отдельное согласие на обработку данных в целях информирования, рядом с согласием на обработку персональных данных в воронке обращения. Без подписки житель не получит ни одной рассылки.

По закону 38-ФЗ «О рекламе» муниципальное информирование рекламой не является (это не коммерческое продвижение). Поэтому отдельной формальной галочки «согласен на рекламу» не требуется. Подписка через кнопку — это согласие на конкретный канал коммуникации, не на персональные данные.

В каждом сообщении рассылки есть кнопка «🔕 Отписаться от рассылки». Однократное нажатие переключает флаг `users.subscribed_broadcast = false`. Команда `/unsubscribe` делает то же самое.

### 10.2 Технический контур

- Таблица `broadcasts`: `id`, `created_by_operator_id`, `text`, `subscriber_count_at_start`, `started_at`, `finished_at`, `status` (`draft`, `sending`, `done`, `cancelled`, `failed`), `delivered_count`, `failed_count`, `admin_message_id`.
- Таблица `broadcast_deliveries`: `id`, `broadcast_id`, `user_id`, `delivered_at`, `error`.
- Поле `users.subscribed_broadcast` (булево, по умолчанию `false`).
- Команда `/broadcast` в административной группе — двухшаговая анкета. Бот запрашивает текст (предел 1000 символов), показывает предпросмотр с числом подписчиков, ждёт подтверждения. На «✅ Разослать» создаётся запись в `broadcasts`, запускается фоновая задача через `asyncio.create_task`.
- Рассылка асинхронная. Ограничение скорости — 1 сообщение в секунду (`BROADCAST_RATE_LIMIT_PER_SEC`). Это вписывается в лимит MAX в 2 запроса в секунду. На 1000 подписчиков рассылка идёт около 17 минут.
- Состояние анкеты-рассылки хранится в памяти процесса (`_wizards: dict[int, _WizardState]`) с временем жизни 300 секунд. Реализация рассчитана только на один экземпляр сервиса. Горизонтальное масштабирование сломает её.

### 10.3 Интерфейс жителя

```
📢 Объявление от Администрации Елизовского муниципального района

<текст рассылки>

[ 🔕 Отписаться от рассылки ]
```

Кнопка отписки — `payload="broadcast:unsubscribe"`. Бот ставит флаг и шлёт «Подписка отключена. Вернуть — командой /subscribe.»

### 10.4 Интерфейс оператора

В административной группе после команды `/broadcast` и подтверждения появляется сообщение прогресса:

```
Рассылка #5 запущена.
⏳ 0/47

[ ⛔ Экстренно остановить ]
```

Каждые 5 секунд бот обновляет это же сообщение через `bot.edit_message`: счётчик доставленных, число упавших. По завершении — итоговое «✅ Рассылка #5 завершена. Доставлено: 46. Не доставлено: 1.» Кнопка «⛔ Экстренно остановить» доступна **любому оператору в группе**. Это страховка на случай скомпрометированной учётной записи координатора.

### 10.5 Команды

- `/broadcast` — создать рассылку (анкета).
- `/broadcast list` — последние 10 рассылок со статусами и счётчиками.

Доступ — `coordinator` и `it`. Рядовым `aemr` и `egp` команда недоступна.

### 10.6 Защита и фильтры

- Каждая рассылка пишется в журнал действий с полным текстом и инициатором (`action=broadcast_send`).
- Заблокированные ботом жители (`is_blocked=true`) и анонимизированные через `/erase` или `/forget` (`first_name='Удалено'`) исключаются из списка получателей автоматически.
- Предел длины текста рассылки — 1000 символов (`BROADCAST_MAX_CHARS`). Длинные объявления имеет смысл публиковать на сайте АЕМР, а в рассылку давать краткую выжимку со ссылкой.
- Рассылку можно остановить кнопкой «⛔». Фоновая задача проверяет флаг между каждой отправкой и прерывается.

## 11. Возможные направления развития

Идеи, которые рассматривались, но не вошли в производственный периметр MVP. Без сроков и плановых обозначений. Это перспективы, а не отложенные задачи. Если в будущем появится конкретный сценарий или потребность, каждая из них обсуждается отдельно и реализуется как новый блок поверх текущей архитектуры.

### Расширения механизма рассылок

- **Шаблоны рассылок.** Заготовленные тексты «ЧС», «праздник», «работы», чтобы координатор не писал каждое объявление с нуля.
- **Планирование на будущее.** Команда `/broadcast schedule «завтра в 10:00»` для постановки в очередь.
- **Сегментация подписчиков.** Рассылка только в определённое поселение, по тематикам или по адресу.
- **Загрузка резервной копии в S3.** Облачное резервное копирование как дополнение к локальному. Инфраструктура в коде уже есть (поля `BACKUP_S3_*`). Если в будущем потребуется, нужна настройка под конкретное хранилище и ключи.

### Маршрутизация и роли операторов

- **Маршрутизация обращений по адресу.** Жители, попадающие в Елизовское ГП, направляются операторам с ролью `egp`, остальные — в АЕМР. Шаблон таблицы поселений с улицами и индексами уже есть в `RUNBOOK.md`. Реализация имеет смысл, если сотрудник ЕГП устанет видеть в общей ленте обращения не из своего поселения.
- **Разделение ролей внутри одной группы.** Сейчас все авторизованные считаются операторами с одинаковым набором команд (кроме `/erase`, `/setting`, `/add_operators`, `/broadcast`). Гранулярные роли могли бы ограничить, например, кто закрывает обращения, а кто только отвечает.
- **Адресные сигналы для ИТ в личку.** Если бот падает или резервная копия не сделана, отправлять сигнал не только в административную группу, но и в личку конкретному ИТ-специалисту.

### Качество ответов и помощь оператору

- **Автоматический проверщик против Стандарта качества АЕМР.** Подсветка нарушений пунктов 5, 6, 9, 11 (страдательный залог, канцелярит, лишние восклицательные знаки, многоточия) при нажатии «Отправить ответ». Не блокирующая, рекомендательная.
- **Помощник оператору на основе языковой модели** («перепиши короче, мягче, по чек-листу»). На MVP отказались. Оператор отвечает живым языком, без посредника на основе ИИ.
- **Подсказки шаблонов ответа** оператору. Файл JSON примерно с тридцатью шаблонами от прошлого анализа Telegram-чата лежит в `telegram_analytics/agent_export/` как образец.
- **Автоматическая классификация тематик** обращения. Жители выбирают тематику из списка вручную. Машинное обучение для классификации сэкономило бы шаг, но не критично.

### Инфраструктура

- **Серверный режим связи** вместо опросного. Готовая инфраструктура (Nginx, утилита Let's Encrypt, обработчик `/max/webhook` с проверкой секрета) лежит в опциональном профиле `webhook` в `docker-compose`. Переключение — `BOT_MODE=webhook` и подъём профиля. Имеет смысл при росте нагрузки или при введении в MAX более жёстких лимитов на опросный режим.
- **Свой форк `love-apples/maxapi`.** Сейчас исходный репозиторий активный (последний коммит — апрель 2026). Условия для форка: три и более месяцев без активности исходного репозитория и при этом открыты критические задачи; либо уязвимость безопасности без реакции мейнтейнера в течение 14 дней. При срабатывании — форк в `gaben1488/maxapi` под контролем муниципалитета.
- **Точечное восстановление БД (PITR, point-in-time recovery).** Сейчас `pg_dump` работает раз в неделю. PITR через архивирование журнала упреждающей записи (WAL — Write-Ahead Log, журнал, в который Postgres пишет изменения до их применения к таблицам) в S3 даёт восстановление на любую секунду между резервными копиями. Цена — постоянная отправка WAL-файлов в облако. Имеет смысл после значимого роста объёма данных.
- **Управляемый Postgres** вместо самостоятельного хостинга. Гипотетически снимает с нас резервные копии, мониторинг и обновления версий. Но входит в противоречие с принятой моделью самостоятельного хостинга у заказчика и не рассматривается без отдельного решения АЕМР.

### Аналитика

- **Дашборды и графики.** Когда наберётся примерно 200–300 обращений, имеет смысл собрать одноразовый HTML-дашборд с разбивкой по тематикам, операторам и времени отклика. Шаблон есть в `telegram_analytics/`.
- **Многоязычие.** Идея интересная, но для муниципального сервиса потребность не очевидна — все жители Елизовского района говорят по-русски.
