# ADR-001: Архитектура MVP «Администрация ЕМР. Обратная связь»

**Статус:** v4 (после фаз A и B пред-релизной шлифовки: idempotency middleware, async-backup, healthcheck-сторож, recover_stuck_funnels, единый `_persist_and_dispatch_appeal`, FSM в state-handler-функциях, реальный upload XLSX/PDF)
**Дата:** 2026-04-29
**Автор:** filat (с участием Claude Code)

## 1. Контекст

Администрация Елизовского муниципального района (АЕМР, Камчатский край) хочет канал обратной связи от жителей в мессенджере MAX, дополняющий существующий Telegram-чат. Эталонный референс — бот «Солодов. Обратная связь» Камчатского правительства: админка живёт в групповом чате, куда падают обращения, оператор отвечает реплаем, бот пересылает ответ в личку жителю и присылает в группу подтверждение.

Управляющие ограничения:

- **Платформа.** Только MAX (Bot API на `platform-api.max.ru`).
- **Юрлицо АЕМР верифицировано** на dev.max.ru — токен бота можно получить.
- **Скорость важнее перфекционизма.** Старт MVP — 2–3 рабочих дня вместо двух недель.
- **Инфраструктура.** Российский VPS, ЦОД в РФ (статья 18.5 закона 152-ФЗ).
- **Уровень защищённости ПДн.** УЗ-4 по ПП №1119. Аттестация провайдера не требуется.
- **Бюджет.** Хостинг ~1 700 ₽/мес. Свой код, не подписочный сервис.

Что переиспользуется из готового. Из `telegram_analytics/agent_export/` берём `operator_templates.json` (33 шаблона + ранжированный каталог) — складываем в `seed/templates.json`, на MVP не показываем оператору как подсказки (это вторая фаза), но имеем под рукой. Системный промпт и чек-лист качества в MVP не используются — отвечает живой оператор без LLM-помощника.

## 2. Решение в одной фразе

Один Python-сервис на `maxapi` с PostgreSQL и APScheduler внутри одного процесса. Long polling. Никаких отдельных бэкенда, фронтенда, мини-приложения. Админка — групповой чат в MAX.

## 3. Архитектурные блоки

### 3.1 Bot

Единственный сервис. Отвечает за всё: общается с жителями в личных диалогах, общается с операторами в служебных группах, держит APScheduler для крона, ходит в БД через SQLAlchemy.

Режим доставки событий — **long polling** на старте. Преимущества для MVP:

- Не требуется публичный HTTPS-эндпойнт, домен и сертификат — экономия первой недели.
- Бот разворачивается одной командой `docker compose up`.
- Локальная разработка — с любой машины с интернетом.

Минус long polling: при простое процесса MAX держит очередь до 8 часов; если перерыв дольше — события теряются. Для MVP с одним инстансом приемлемо. Когда стабилизируется (~2–4 недели после публикации) — переключаем на webhook без переписывания кода: добавляется Nginx, regular endpoint и `POST /subscriptions`.

Идемпотентность — через unique constraint на `events.idempotency_key` (берём `update_id` или комбинацию `update_type + message_id + timestamp`).

Команды бота для жителя: `/start`, `/menu`, `/help`. Главное меню — три inline-кнопки.

Команды бота в админ-группе:

- `/stats today` / `/stats week` / `/stats month` — XLSX-выгрузка обращений за период.
- `/reopen NNN` — вернуть обращение в статус «в работе».
- `/close NNN` — закрыть обращение без ответа.
- `/help` — список команд для операторов.

### 3.3 Админка = групповой чат в MAX

**Не отдельное приложение.** Админка — это служебная группа в MAX, куда добавлены бот и операторы.

Поток:

1. Житель в личном диалоге с ботом отправляет обращение (текст ± фото ± геолокация).
2. После 60 секунд тишины (или нажатия «Завершить» в боте, если такая кнопка появится в фазе 2) обращение помечается `status=new` и бот шлёт в админ-группу карточку:

   ```
   📨 Новое обращение #107
   👤 Имя: Алексей
   📞 Телефон: 89964240723
   ──────────
   <текст обращения>
   <фото вложением>
   ```

3. Координатор открывает карточку, **отвечает реплаем** на сообщение бота (просто пишет ответ как обычное сообщение с реплаем).
4. Бот ловит событие `message_created` в админ-группе с полем `link.type=reply`, по `link.mid` находит обращение, валидирует длину ответа (≤300 символов), передаёт ответ жителю через `POST /messages?user_id=…`.
5. После успешной доставки бот шлёт в админ-группу подтверждение:

   ```
   ✉️ Ответ успешно отправлен пользователю
   Обращение #107 помечено как завершённое
   ```

6. В БД: `appeals.status=answered`, `appeals.answered_at=now()`, в `messages` добавляется запись `direction=from_operator`.

Если оператор пишет реплай длиннее 300 символов — бот отвечает в группу: «Ответ длиннее 300 символов, отредактируйте и пришлите снова». Без отправки жителю.

Если житель присылает следующее сообщение в обращение со статусом `answered` — бот реоткрывает обращение (`status=in_progress`) и шлёт новое сообщение в группу как продолжение цепочки (с цитатой исходной карточки).

**Роли** в первой итерации проще, чем в v1 ADR:

- **Координатор АЕМР** (вы) — добавлен в админ-группу, отвечает на обращения, имеет доступ к `/stats`, `/reopen`, `/close`.
- **ИТ-специалист** — добавлен в группу, видит всё, использует те же команды плюс /diag для диагностики бота (фаза 2).
- **Специалист ЕГП по ответам** — отдельная служебная группа (`admin_group_id_egp`), либо в основной группе с указанием на конкретные обращения через упоминание `@username`. На MVP делаем простейший вариант: одна общая админ-группа, ЕГП-специалист добавлен как обычный оператор. Если потребуется разделение — отдельную группу включим во вторую неделю.

### 3.4 Инфраструктура

```
                Internet ◄──── long polling (исходящий) ──── bot ───► PostgreSQL
                                                              │
                                                              └─► APScheduler
                                                                  (ежесуточный
                                                                   бэкап в S3)
```

Один VPS. Два контейнера в `docker-compose`:

- `bot` — Python 3.12, `maxapi`, SQLAlchemy, APScheduler, `openpyxl`.
- `db` — PostgreSQL 16, том на хосте.

Никакого Nginx, никакого reverse-proxy, никаких внешних портов наружу — бот ходит только исходящими запросами к `platform-api.max.ru`. На VPS открыт только SSH (порт 22) для админа.

Бэкап PostgreSQL: APScheduler-задача раз в сутки запускает `pg_dump`, шифрует через `gpg` (passphrase в `.env`), заливает в Yandex Object Storage через rclone (~50 ₽/мес за 50 ГБ). Хранение 30 дней с ротацией.

Когда переключим на webhook (фаза 2) — добавится Nginx-контейнер и сертификат через certbot.

### 3.5 Поток данных от жителя до ответа (FSM-воронка)

Главное меню жителя — три кнопки: «Написать обращение», «Мои обращения», «Контакты». Согласие на ПДн запрашивается **внутри** воронки «Написать обращение», а не сразу после `/start` — пользователь может посмотреть контакты и свои прошлые обращения без согласия.

Воронка приёма обращения — пошаговый wizard, состояние хранится в `users.dialog_state`, накопленные данные — в `users.dialog_data` JSONB. Шаги:

1. **Согласие (`AWAITING_CONSENT`).** Текст политики со ссылкой и две кнопки: «Согласен» / «Отказаться». Пропускается, если `users.consent_pdn_at IS NOT NULL`. Отказ возвращает в главное меню.
2. **Контакт (`AWAITING_CONTACT`).** Кнопка `request_contact`, телефон извлекается из vCF-вложения. Пропускается, если телефон уже есть.
3. **Имя (`AWAITING_NAME`).** Свободный ввод. Только имя, без фамилии (как в боте-референсе).
4. **Адрес (`AWAITING_ADDRESS`).** Свободный ввод улицы и дома. Район как отдельный шаг убран — Елизовский МО единственный.
5. **Тематика (`AWAITING_TOPIC`).** Inline-клавиатура по 2 кнопки в ряд из списка `settings.topics` (11 пунктов из seed). Выбор пишется в `dialog_data.topic`.
6. **Суть (`AWAITING_SUMMARY`).** Свободный ввод текста + опциональные фото/геолокация. Каждое сообщение в этом состоянии добавляется в `dialog_data.summary_chunks` и `dialog_data.attachments`. Завершение — кнопкой «Отправить» либо автоматически по таймауту тишины 60 секунд (`APPEAL_TIMEOUT`).
7. **Финализация.** Бот создаёт `appeals.NNN` со структурированными полями `address/topic/summary/attachments`, постит карточку в админ-группу (формат идентичен карточке в «Мои обращения»), сохраняет `appeals.admin_message_id` для последующих апдейтов, обнуляет FSM (`dialog_state = idle`), отвечает жителю «Обращение #N принято, ответим в течение 4 рабочих часов».

После ответа оператора (через реплай в группе) и доставки жителю — статус `answered`. Если житель пишет в idle-состоянии новое сообщение, и у него есть активное `answered`-обращение, бот реоткрывает его (`status = in_progress`) и шлёт в группу followup-сообщение с цитатой исходной карточки.

На любом шаге воронки доступна кнопка «Отмена» — обнуляет FSM и возвращает в главное меню без сохранения.

## 4. Лимит ответа 200–300 символов

Жёсткий лимит **300 символов** в API бота. При попытке отправить длиннее — отбой с сообщением в админ-группу (см. 3.3 пункт 4). Без модалки, без авторазбиения — упрощаем UX, оператор сам решает.

Обоснование цифры — в требованиях куратора (200–300). Берём верхнюю границу, чтобы не ломать стандартные шаблоны на 250–280 символов из `agent_export/operator_templates.json`.

## 5. Безопасность и соответствие 152-ФЗ

**Класс ИСПДн: УЗ-4** по ПП №1119:

- Категория ПДн — иные (имя, фамилия, телефон, текст обращений, фото и геолокация — приложения).
- Тип угроз — третий.
- Количество субъектов — менее 100 000.
- Категория субъектов — не сотрудники оператора.

Обязательные меры:

- Локализация хранения в РФ (статья 18.5) — VPS в Москве/Питере.
- Согласие пользователя на обработку ПДн при первом взаимодействии (фиксация в `users.consent_pdn_at`).
- Публикация политики на сайте АЕМР — ссылка из бота. Текст политики — `docs/PRIVACY.md`, согласован с куратором.
- АЕМР зарегистрирована в реестре операторов ПДн Роскомнадзора (проверить по ИНН в открытом реестре).
- Базовая ИБ-гигиена сервера: ufw, SSH по ключам, отключение root-логина, fail2ban, `unattended-upgrades`, шифрование бэкапов через gpg.

**Не требуется** для УЗ-4: аттестация провайдера, СКЗИ, аттестация ИСПДн, расширенная модель угроз.

**Удаление по запросу.** Команда `/erase max_user_id=NNN` от ИТ-специалиста в админ-группе или текстовая команда жителя в боте `/forget`. Алгоритм: `users.first_name='Удалено'`, `users.last_name=''`, `users.phone=NULL`, удаление `messages.attachments`, запись в `audit_log`. Завершается за минуту, в течение 24 часов как требует 152-ФЗ.

## 6. Развёртывание и CI/CD

На MVP — без CI/CD. Деплой ручной:

```bash
ssh feedback@<vps>
cd ~/aemr-bot
git pull
docker compose up -d --build
docker compose exec bot alembic upgrade head
```

Когда стабилизируется — добавим минимальный workflow на GitHub Actions с прогоном тестов и `docker compose pull && up -d` через SSH.

Тесты на старте — pytest для сервисного слоя бота с моком `maxapi.Bot`. Проверяем сценарии: согласие → запрос контакта → меню; приём обращения с серией сообщений; ответ оператора реплаем; отказ при ответе длиннее 300; статистика за период; удаление ПДн.

## 7a. Эксплуатационные компоненты (после фазы A/B)

**Healthcheck.** `bot/aemr_bot/health.py` поднимает aiohttp-сервер на `WEBHOOK_PORT` с эндпойнтом `/healthz`. Внутри — singleton `Heartbeat` обновляется фоновой задачей `heartbeat_pulse` каждые `HEALTHCHECK_PULSE_SECONDS`. Если main-loop завис, heartbeat перестаёт обновляться, и `/healthz` отдаёт 503 после `HEALTHCHECK_STALE_SECONDS`. На это завязаны два потребителя: внешний UptimeRobot (направляется на публичный URL) и внутренний `selfcheck` cron-job, постящий в админ-группу транзишн-алерты «упало/восстановилось».

**Idempotency.** `services/idempotency.py::claim` строит ключ из update_type + `callback_id`/`mid`/`seq` + timestamp + chat/user, делает `INSERT ON CONFLICT DO NOTHING` в `events` и возвращает True/False. Подключается через outer-middleware в `handlers/__init__.py::IdempotencyMiddleware`. Дубликаты от MAX (особенно при webhook-retries) молча отбрасываются перед тем, как попадут в любой обработчик.

**Recovery.** `handlers/appeal.py::recover_stuck_funnels` запускается через `asyncio.create_task` при старте. Через `users_service.find_stuck_in_summary(idle_seconds)` находит пользователей в `AWAITING_SUMMARY`, чей `updated_at` старше таймаута, и параллельно через `asyncio.gather` финализирует их обращения. Empty-submissions (нет ни текста, ни вложений) сбрасываются в `IDLE`, чтобы не зацикливаться.

**Async backup.** `services/cron.py::_backup_db` теперь полностью async через `asyncio.create_subprocess_exec` для `pg_dump | gpg | rclone`. Event-loop не блокируется на время дампа.

**Upload.** `services/uploads.py` — единый адаптер над `bot.upload_media`. Используется и `services/policy.py` (PDF на старте) и `_build_admin_senders` в `main.py` (XLSX по `/stats` и в ежемесячном отчёте). Клейн `services/cron.py::send_admin_document` теперь реально доставляет файл, а не текстовый fallback.

**FSM-handlers.** `handlers/appeal.py::on_message` — чистый диспатчер по `_STATE_HANDLERS` map. Каждое состояние — отдельная функция (`_on_awaiting_contact`, `_on_awaiting_name`, ...) с одинаковой сигнатурой `(event, body, text_body, max_user_id, op_reply)`. Добавление нового состояния — это новая функция и одна строка в map.

## 8. Что меняется при изменении вводных

- Безопасник АЕМР требует аттестации провайдера → меняем VPS на Selectel/VK Cloud, остальное не трогаем.
- Куратор хочет жёсткое разделение АЕМР↔ЕГП → создаём вторую админ-группу, в `settings` указываем `admin_group_id_egp`, добавляем правило маршрутизации в `settings.egp_keywords` (ключевые слова темы ЕГП).
- Появятся подписки и broadcast → добавляем поля `consent_chs`, `consent_events` в `users` и таблицы `broadcasts`, `broadcast_deliveries`. Это чистая надстройка, ломать MVP не придётся.
- Бот вырастает до федеральных функций (приём обращений по 59-ФЗ) → пересматриваем УЗ и провайдера.
