# ADR-001: Архитектура MVP «Администрация ЕМР. Обратная связь»

**Статус:** v6 — production-ready MVP
**Дата:** 2026-05-04
**Автор:** filat (с участием Claude Code)

## 1. Контекст

Администрация Елизовского муниципального района (АЕМР, Камчатский край) хочет канал обратной связи от жителей в мессенджере MAX, дополняющий существующий Telegram-чат. Эталонный референс — бот «Солодов. Обратная связь» Камчатского правительства: админка живёт в групповом чате, куда падают обращения, оператор отвечает реплаем, бот пересылает ответ в личку жителю и присылает в группу подтверждение.

Управляющие ограничения:

- **Платформа.** Только MAX (Bot API на `platform-api.max.ru`).
- **Юрлицо АЕМР верифицировано** на dev.max.ru — токен бота можно получить.
- **Скорость важнее перфекционизма.** Старт MVP — 2–3 рабочих дня вместо двух недель.
- **Инфраструктура.** Российский VPS, ЦОД в РФ (статья 18.5 закона 152-ФЗ).
- **Уровень защищённости ПДн.** УЗ-4 по ПП №1119. Аттестация провайдера не требуется.
- **Бюджет.** Хостинг ~1 700 ₽/мес. Свой код, не подписочный сервис.

Что переиспользуется из готового. Из `telegram_analytics/agent_export/` берём `operator_templates.json` (33 шаблона + ранжированный каталог) — складываем в `seed/templates.json`, на MVP не показываем оператору как подсказки (см. §11 «Возможные направления развития»), но имеем под рукой. Системный промпт и чек-лист качества в MVP не используются — отвечает живой оператор без LLM-помощника.

## 2. Решение в одной фразе

Один Python-сервис на `maxapi` с PostgreSQL и APScheduler внутри одного процесса. Long polling. Никаких отдельных бэкенда, фронтенда, мини-приложения. Админка — групповой чат в MAX.

## 3. Архитектурные блоки

### 3.1 Bot

Единственный сервис. Отвечает за всё: общается с жителями в личных диалогах, общается с операторами в служебных группах, держит APScheduler для крона, ходит в БД через SQLAlchemy.

Режим доставки событий — **long polling** на старте. Преимущества для MVP:

- Не требуется публичный HTTPS-эндпойнт, домен и сертификат — экономия первой недели.
- Бот разворачивается одной командой `docker compose up`.
- Локальная разработка — с любой машины с интернетом.

Минус long polling: при простое процесса MAX держит очередь до 8 часов; если перерыв дольше — события теряются. Для MVP с одним инстансом приемлемо. Когда стабилизируется (~2–4 недели после публикации) — переключаем на webhook без переписывания кода: добавляется Nginx, regular endpoint и `POST /subscriptions`.

Идемпотентность — через unique constraint на `events.idempotency_key` (берём `update_id` или комбинацию `update_type + message_id + timestamp`).

Команды бота для жителя: `/start`, `/menu`, `/help`. Главное меню — три inline-кнопки.

Команды бота в админ-группе:

- `/stats today` / `/stats week` / `/stats month` — XLSX-выгрузка обращений за период.
- `/reopen NNN` — вернуть обращение в статус «в работе».
- `/close NNN` — закрыть обращение без ответа.
- `/help` — список команд для операторов.

### 3.3 Админка = групповой чат в MAX

**Не отдельное приложение.** Админка — это служебная группа в MAX, куда добавлены бот и операторы.

Поток:

1. Житель в личном диалоге с ботом отправляет обращение (текст ± фото ± геолокация).
2. После 60 секунд тишины (или нажатия «Отправить» в боте) обращение помечается `status=new` и бот шлёт в админ-группу карточку:

   ```
   📨 Новое обращение #107
   👤 Имя: Алексей
   📞 Телефон: 89964240723
   ──────────
   <текст обращения>
   <фото вложением>
   ```

3. Координатор открывает карточку, **отвечает реплаем** на сообщение бота (просто пишет ответ как обычное сообщение с реплаем).
4. Бот ловит событие `message_created` в админ-группе с полем `link.type=reply`, по `link.mid` находит обращение, валидирует длину ответа (≤300 символов), передаёт ответ жителю через `POST /messages?user_id=…`.
5. После успешной доставки бот шлёт в админ-группу подтверждение:

   ```
   ✉️ Ответ успешно отправлен пользователю
   Обращение #107 помечено как завершённое
   ```

6. В БД: `appeals.status=answered`, `appeals.answered_at=now()`, в `messages` добавляется запись `direction=from_operator`.

Если оператор пишет реплай длиннее 300 символов — бот отвечает в группу: «Ответ длиннее 300 символов, отредактируйте и пришлите снова». Без отправки жителю.

Если житель присылает следующее сообщение в обращение со статусом `answered` — бот реоткрывает обращение (`status=in_progress`) и шлёт новое сообщение в группу как продолжение цепочки (с цитатой исходной карточки).

**Роли** в первой итерации проще, чем в v1 ADR:

- **Координатор АЕМР** (вы) — добавлен в админ-группу, отвечает на обращения, имеет доступ к `/stats`, `/reopen`, `/close`.
- **ИТ-специалист** — добавлен в группу, видит всё, использует те же команды плюс `/diag` для диагностики бота, `/setting`, `/erase`, `/add_operators`, `/backup`.
- **Специалист ЕГП по ответам** — отдельная служебная группа (`admin_group_id_egp`), либо в основной группе с указанием на конкретные обращения через упоминание `@username`. На MVP делаем простейший вариант: одна общая админ-группа, ЕГП-специалист добавлен как обычный оператор. Если потребуется разделение — отдельную группу включим во вторую неделю.

### 3.4 Инфраструктура

```
                Internet ◄──── long polling (исходящий) ──── bot ───► PostgreSQL
                                                              │
                                                              └─► APScheduler
                                                                  (ежесуточный
                                                                   бэкап в S3)
```

Один VPS. Два контейнера в `docker-compose`:

- `bot` — Python 3.12, `maxapi`, SQLAlchemy, APScheduler, `openpyxl`.
- `db` — PostgreSQL 16, том на хосте.

Никакого Nginx, никакого reverse-proxy, никаких внешних портов наружу — бот ходит только исходящими запросами к `platform-api.max.ru`. На VPS открыт только SSH (порт 22) для админа.

Бэкап PostgreSQL: APScheduler-задача раз в сутки запускает `pg_dump`, шифрует через `gpg` (passphrase в `.env`), заливает в Yandex Object Storage через rclone (~50 ₽/мес за 50 ГБ). Хранение 30 дней с ротацией.

Готовая инфраструктура webhook-режима (Nginx + certbot + endpoint `/max/webhook` с проверкой `WEBHOOK_SECRET`) лежит в опциональном compose-профиле `webhook`, не активна по умолчанию. Переключение — `BOT_MODE=webhook` в `.env` плюс `docker compose --profile webhook up -d`. См. §11 «Возможные направления развития».

### 3.5 Поток данных от жителя до ответа (FSM-воронка)

Главное меню жителя — три кнопки: «Написать обращение», «Мои обращения», «Контакты». Согласие на ПДн запрашивается **внутри** воронки «Написать обращение», а не сразу после `/start` — пользователь может посмотреть контакты и свои прошлые обращения без согласия.

Воронка приёма обращения — пошаговый wizard, состояние хранится в `users.dialog_state`, накопленные данные — в `users.dialog_data` JSONB. Шаги:

1. **Согласие (`AWAITING_CONSENT`).** Текст политики со ссылкой и две кнопки: «Согласен» / «Отказаться». Пропускается, если `users.consent_pdn_at IS NOT NULL`. Отказ возвращает в главное меню.
2. **Контакт (`AWAITING_CONTACT`).** Кнопка `request_contact`, телефон извлекается из vCF-вложения. Пропускается, если телефон уже есть.
3. **Имя (`AWAITING_NAME`).** Свободный ввод. Только имя, без фамилии (как в боте-референсе).
4. **Адрес (`AWAITING_ADDRESS`).** Свободный ввод улицы и дома. Район как отдельный шаг убран — Елизовский МО единственный.
5. **Тематика (`AWAITING_TOPIC`).** Inline-клавиатура по 2 кнопки в ряд из списка `settings.topics` (11 пунктов из seed). Выбор пишется в `dialog_data.topic`.
6. **Суть (`AWAITING_SUMMARY`).** Свободный ввод текста + опциональные фото/геолокация. Каждое сообщение в этом состоянии добавляется в `dialog_data.summary_chunks` и `dialog_data.attachments`. Завершение — кнопкой «Отправить» либо автоматически по таймауту тишины 60 секунд (`APPEAL_TIMEOUT`).
7. **Финализация.** Бот создаёт `appeals.NNN` со структурированными полями `address/topic/summary/attachments`, постит карточку в админ-группу (формат идентичен карточке в «Мои обращения»), сохраняет `appeals.admin_message_id` для последующих апдейтов, обнуляет FSM (`dialog_state = idle`), отвечает жителю «Обращение #N принято, ответим в течение 4 рабочих часов».

После ответа оператора (через реплай в группе) и доставки жителю — статус `answered`. Если житель пишет в idle-состоянии новое сообщение, и у него есть активное `answered`-обращение, бот реоткрывает его (`status = in_progress`) и шлёт в группу followup-сообщение с цитатой исходной карточки.

На любом шаге воронки доступна кнопка «Отмена» — обнуляет FSM и возвращает в главное меню без сохранения.

## 4. Лимит ответа 200–300 символов

Жёсткий лимит **300 символов** в API бота. При попытке отправить длиннее — отбой с сообщением в админ-группу (см. 3.3 пункт 4). Без модалки, без авторазбиения — упрощаем UX, оператор сам решает.

Обоснование цифры — в требованиях куратора (200–300). Берём верхнюю границу, чтобы не ломать стандартные шаблоны на 250–280 символов из `agent_export/operator_templates.json`.

## 5. Безопасность и соответствие 152-ФЗ

**Класс ИСПДн: УЗ-4** по ПП №1119:

- Категория ПДн — иные (имя, фамилия, телефон, текст обращений, фото и геолокация — приложения).
- Тип угроз — третий.
- Количество субъектов — менее 100 000.
- Категория субъектов — не сотрудники оператора.

Обязательные меры:

- Локализация хранения в РФ (статья 18.5) — VPS в Москве/Питере.
- Согласие пользователя на обработку ПДн при первом взаимодействии (фиксация в `users.consent_pdn_at`).
- Публикация политики на сайте АЕМР — ссылка из бота. Текст политики — `docs/PRIVACY.md`, согласован с куратором.
- АЕМР зарегистрирована в реестре операторов ПДн Роскомнадзора (проверить по ИНН в открытом реестре).
- Базовая ИБ-гигиена сервера: ufw, SSH по ключам, отключение root-логина, fail2ban, `unattended-upgrades`, шифрование бэкапов через gpg.

**Не требуется** для УЗ-4: аттестация провайдера, СКЗИ, аттестация ИСПДн, расширенная модель угроз.

**Удаление по запросу.** Команда `/erase max_user_id=NNN` от ИТ-специалиста в админ-группе или текстовая команда жителя в боте `/forget`. Алгоритм: `users.first_name='Удалено'`, `users.last_name=''`, `users.phone=NULL`, удаление `messages.attachments`, запись в `audit_log`. Завершается за минуту, в течение 24 часов как требует 152-ФЗ.

## 6. Развёртывание и CI/CD

На MVP — без CI/CD. Деплой ручной:

```bash
ssh feedback@<vps>
cd ~/aemr-bot
git pull
docker compose up -d --build
docker compose exec bot alembic upgrade head
```

Когда стабилизируется — добавим минимальный workflow на GitHub Actions с прогоном тестов и `docker compose pull && up -d` через SSH.

Тесты на старте — pytest для сервисного слоя бота с моком `maxapi.Bot`. Проверяем сценарии: согласие → запрос контакта → меню; приём обращения с серией сообщений; ответ оператора реплаем; отказ при ответе длиннее 300; статистика за период; удаление ПДн.

## 7. Что не вошло в первичный MVP

Этот раздел сохранён исторически — он описывает первоначальный scope «что отрезали ради скорости». По факту часть этих идей уже реализована в более поздних итерациях (broadcast-рассылки, /op_help, /reply, локальный бэкап с ротацией), часть переехала в раздел «Возможные направления развития» (см. §11). Полный актуальный список — там.

## 7a. Эксплуатационные компоненты

**Healthcheck.** `bot/aemr_bot/health.py` поднимает aiohttp-сервер на `WEBHOOK_PORT` с эндпойнтом `/healthz`. Внутри — singleton `Heartbeat` обновляется фоновой задачей `heartbeat_pulse` каждые `HEALTHCHECK_PULSE_SECONDS`. Если main-loop завис, heartbeat перестаёт обновляться, и `/healthz` отдаёт 503 после `HEALTHCHECK_STALE_SECONDS`. На это завязаны два потребителя: внешний UptimeRobot (направляется на публичный URL) и внутренний `selfcheck` cron-job, постящий в админ-группу транзишн-алерты «упало/восстановилось».

**Idempotency.** `services/idempotency.py::claim` строит ключ из update_type + `callback_id`/`mid`/`seq` + timestamp + chat/user, делает `INSERT ON CONFLICT DO NOTHING` в `events` и возвращает True/False. Подключается через outer-middleware в `handlers/__init__.py::IdempotencyMiddleware`. Дубликаты от MAX (особенно при webhook-retries) молча отбрасываются перед тем, как попадут в любой обработчик.

**Recovery.** `handlers/appeal.py::recover_stuck_funnels` запускается через `asyncio.create_task` при старте. Через `users_service.find_stuck_in_summary(idle_seconds)` находит пользователей в `AWAITING_SUMMARY`, чей `updated_at` старше таймаута, и параллельно через `asyncio.gather` финализирует их обращения. Empty-submissions (нет ни текста, ни вложений) сбрасываются в `IDLE`, чтобы не зацикливаться.

**Async backup.** `services/cron.py::_backup_db` теперь полностью async через `asyncio.create_subprocess_exec` для `pg_dump | gpg | rclone`. Event-loop не блокируется на время дампа.

**Upload.** `services/uploads.py` — единый адаптер над `bot.upload_media`. Используется и `services/policy.py` (PDF на старте) и `_build_admin_senders` в `main.py` (XLSX по `/stats` и в ежемесячном отчёте). Клейн `services/cron.py::send_admin_document` теперь реально доставляет файл, а не текстовый fallback.

**FSM-handlers.** `handlers/appeal.py::on_message` — чистый диспатчер по `_STATE_HANDLERS` map. Каждое состояние — отдельная функция (`_on_awaiting_contact`, `_on_awaiting_name`, ...) с одинаковой сигнатурой `(event, body, text_body, max_user_id, op_reply)`. Добавление нового состояния — это новая функция и одна строка в map.

## 8. Что меняется при изменении вводных

- Безопасник АЕМР требует аттестации провайдера → меняем VPS на Selectel/VK Cloud, остальное не трогаем.
- Куратор хочет жёсткое разделение АЕМР↔ЕГП → создаём вторую админ-группу, в `settings` указываем `admin_group_id_egp`, добавляем правило маршрутизации в `settings.egp_keywords` (ключевые слова темы ЕГП).
- Появятся подписки и broadcast → добавляем поля `consent_chs`, `consent_events` в `users` и таблицы `broadcasts`, `broadcast_deliveries`. Это чистая надстройка, ломать MVP не придётся.
- Бот вырастает до федеральных функций (приём обращений по 59-ФЗ) → пересматриваем УЗ и провайдера.

## 9. Дополнения и уточнения после первичной реализации

Эти решения внесены поверх архитектуры разделов 1–8 без её переписывания. Каждый пункт — фиксация фактического состояния кода и обоснование выбора. Содержание раздела отражает то, что реально работает в production-сборке MVP.

### 9.1 FSM-хранилище: Postgres JSONB, не Redis

`users.dialog_state` (`String(32)`) + `users.dialog_data` (JSONB) остаются основным хранилищем FSM. Перенос в Redis рассматривался как теоретическая оптимизация при росте >1k одновременных активных воронок. Для АЕМР это сценарий «никогда»: население района ~70 тысяч человек, проникновение гос-цифровых сервисов 3–5%, активных воронок одновременно — десятки в часы пик, единицы ночью. Пиковая нагрузка 1–2 UPDATE/секунду, Postgres проглатывает не замечая. Триггер пересмотра — устойчивые >500 одновременных воронок по метрикам диагностики (`/diag`).

### 9.2 Idempotency-middleware — обязательный, не «страховка»

Раздел 7a описывал idempotency как опциональный компонент. По факту `IdempotencyMiddleware` подключён как `outer_middleware` в `handlers/__init__.py::register_handlers` и срабатывает на каждый update до любого handler'а. Дубликаты Update от MAX (редкие в polling, частые при webhook-retries) молча отбрасываются на уровне INSERT-ON-CONFLICT в `events`. Без middleware переход на webhook давал бы дубли карточек в админ-группе.

### 9.3 Recovery stuck funnels — fire-and-forget при старте

`handlers/appeal.py::recover_stuck_funnels` запускается через `asyncio.create_task` в `main.py::main()` и не блокирует старт диспетчера. Empty-submissions (нет ни текста, ни вложений) сбрасываются в IDLE, чтобы не зацикливаться при каждом перезапуске. Ограничение batch-recovery: `RECOVER_BATCH_SIZE=1000` — защита от патологии «10k застрявших воронок после многочасового downtime → 10k API-вызовов при старте».

### 9.4 Long-polling timeout — настраиваемый параметр, не интервал

`POLLING_TIMEOUT_SECONDS` (default `30`, диапазон `0..90`) задаёт **серверный** timeout `getUpdates` MAX — сколько MAX держит соединение ожидая событий. Это не интервал между запросами клиента; long-polling возвращает либо при появлении события, либо по timeout. Чем выше timeout — тем меньше пустых round-trip'ов в idle (важно при rate-limit 2 RPS на бота с 11 мая 2026), тем медленнее реакция на shutdown. Default 30s — баланс. Реализация — monkey-patch на bound-метод `bot.get_updates` в `main.py::_install_polling_timeout`, потому что `maxapi.Dispatcher.start_polling` не пробрасывает timeout-параметр дальше. Если когда-то будем форкать maxapi — это первый кандидат на upstream-PR.

### 9.5 Зависимость maxapi — выбор между community и официальным форком

В экосистеме MAX для Python три варианта:

| Репозиторий | Тип | Активность | Стрелки звёзд |
|---|---|---|---|
| `love-apples/maxapi` | community-фреймворк | активно обновляется (последний коммит — апрель 2026) | 166+ |
| `max-messenger/max-botapi-python` | официальный форк предыдущего, проверенный командой MAX | заморожен на ревизии `2025-07-30`, с тех пор без обновлений | 54 |
| `green-api/max-api-client-python` | клиент через шлюз green-api.com | требует подписку на green-api | — |

Остаёмся на **`love-apples/maxapi`** через `~=0.6` (compatible release). Аргументация:

- Официальный форк `max-messenger/max-botapi-python` — это **снапшот upstream девятимесячной давности**, не самостоятельная активная ветка. Переход на него означал бы откат назад с потерей фиксов, накопленных в upstream после июля 2025 (включая то, что мы используем для `process_update_webhook`, `TypeAdapter[Attachments]`, `MessageLinkType`).
- `green-api/max-api-client-python` создаёт зависимость от внешнего платного сервиса — нарушает пункт «свой код, не подписочный» из контекста ADR §1.

Решение о **форке `love-apples/maxapi`** принимается по одному из триггеров: `(a)` 3+ месяца без коммитов в upstream И открытые critical issues, либо `(b)` обнаруженная security-уязвимость без реакции мейнтейнера в течение 14 дней. Ежеквартальная проверка активности репозитория — задача на координатора. При срабатывании триггера форк делается в организацию `gaben1488/maxapi` под МСУ-контролем.

Дополнительно для предотвращения AI-галлюцинаций при работе с MAX API через Claude / Copilot / Cursor — взяли на учёт `PronikFire/Max-API-Guide` (реверс-инженерный гайд по WSS/TLS) и подход с cursorrules-файлами из `max-bot-aio-template`. На MVP не подключаем, но при росте кодовой базы — кандидат на использование как «шпаргалки» для AI-помощников.

### 9.6 Attachments relay в админ-группу

Фото / видео / геолокация / файлы из обращения теперь пересылаются в админ-группу вторым сообщением (после текстовой карточки), привязанным `NewMessageLink(type=REPLY, mid=admin_card_mid)`. Сохранённые в `appeals.attachments` (JSONB) словари десериализуются обратно в pydantic-модели через `TypeAdapter(Attachments)` из maxapi и передаются в `bot.send_message(attachments=[...])`. Контакт-вложения исключены — телефон уже в карточке, дублирование PII не нужно. Каждое сообщение с вложениями ограничено `ATTACHMENTS_PER_RELAY_MESSAGE=10`, batch'и пронумерованы «(2/3)» и т. д. для читаемости.

### 9.7 Bulk-регистрация операторов: `/add_operators` IT-only

Команда принимает многострочный input `<max_user_id> <role> <ФИО>`, upsert каждого оператора с записью в `audit_log`. После второго прохода security-review команда переведена с `IT|COORDINATOR` на **только `IT`** — coordinator-tier мог через эту команду выдать себе IT-роль и потом дёрнуть `/erase`/`/setting` (privilege escalation). Дополнительно запрещена самомодификация (`actor_id == target_id`) даже для IT — изменения собственной роли идут только через psql-эскалацию из RUNBOOK.

### 9.8 Команда `/policy` жителю

Житель в любой момент пишет `/policy` и получает PRIVACY.pdf файлом. Fallback-цепочка: `(1)` cached token из `settings.policy_pdf_token` (заполняется один раз при старте через `policy_service.ensure_uploaded`) → `(2)` on-demand upload PDF и кеширование токена → `(3)` ссылка на URL из `settings.policy_url` → `(4)` сообщение об ошибке с предложением написать координатору. Команда добавлена в `/help`.

### 9.9 Стандарт качества ответов АЕМР

Тексты в `texts.py` приведены в соответствие с Приложением 2 к Постановлению Администрации Елизовского муниципального района № 2129 от 19.12.2025 (Стандарт качества ответов на сообщения из открытых источников). Прошёл humanizer + clarify: убраны em-dash склейки, hedging-обороты, канцеляризмы, аббревиатуры (АЕМР раскрыто в citizen-facing строках), passive voice заменён на active. Auto-validator на ответы оператора против Стандарта — возможное направление развития, см. §11.

### 9.10 Race protection в финализации обращения

`_persist_and_dispatch_appeal` обёрнут в per-user `asyncio.Lock` (`_user_locks: dict[int, asyncio.Lock]`). Защищает от двойного клика «Отправить», race между timer-firing и user-clicking submit, race между cancel и timer. Внутри lock — idempotency-проверка по `user.dialog_state == IDLE`: второй concurrent-вызов после успешной финализации первого видит IDLE и bail без создания дубля Appeal/admin-карточки. Single-instance only — горизонтальное масштабирование (>1 контейнер бота) сломает эту схему; для много-инстансной деплойки понадобится `pg_advisory_xact_lock` или Redis-lock.

### 9.11 Hard caps на пользовательский ввод

| Параметр | Default | Назначение |
|---|---|---|
| `SUMMARY_MAX_CHARS` | 2000 | Cap на cumulative chunks `summary_chunks` — оставляет headroom внутри 4000-char admin-карточки |
| `ATTACHMENTS_MAX_PER_APPEAL` | 20 | Cap на attachments per appeal — выше уровня — silent drop с `log.info` |
| `ATTACHMENTS_PER_RELAY_MESSAGE` | 10 | Batch size для relay — обходит недокументированный server-cap MAX |
| `VCF_INFO_MAX_CHARS` | 10000 | Cap на vcf_info parsing в `extract_phone` — против hostile multi-MB contact |

Все env-tunable через `Field(alias=...)` в `config.py`.

### 9.12 Pagination в «Мои обращения»

Список обращений жителя выводится по 5 на страницу с inline-кнопками навигации `⬅️ N/M ➡️`. В каждом ряду — emoji статуса + текстовый статус («Новое», «В работе», «Завершено», «Закрыто без ответа») + дата + первые 32 символа summary. Callback `appeals:page:N` обрабатывается в `handlers/menu.py::handle_callback`. Размер страницы — константа `MY_APPEALS_PAGE_SIZE = 5`.

### 9.13 Реальные данные АЕМР

`seed/welcome.md`, `seed/contacts.json`, `seed/transport_dispatchers.json` и `settings_store.DEFAULTS` содержат данные, присланные администрацией: kamgov.ru/questions для электронной приёмной, PDF муниципальных маршрутов УДТХ, kamgov.ru/mintrans для межмуниципальных, текст приёма граждан А.С. Гончарова (1 и 3 среда месяца, запись по 8 (415-31) 7-25-29), номера экстренных служб (112, ЕДДС, Камчатскэнерго ЦЭС и горячая линия, Теплоэнерго, Водоканал), 5 номеров диспетчерских автотранспорта по группам маршрутов. В SCHEMA `settings_store` добавлены ключи `udth_schedule_intermunicipal_url` и `transport_dispatcher_contacts`. Подменю «Полезная информация» содержит ссылки на оба расписания, callback-список диспетчерских и кнопку подписки на рассылку.

### 9.14 Команды `/reply N <текст>` и `/op_help` с inline-клавиатурой

Команда `/reply <appeal_id> <текст>` доставляет ответ оператора жителю без зависимости от swipe-reply механизма. Полезна когда клиент MAX по каким-то причинам не помечает сообщение как reply (`Message.link` приходит пустым) либо оператор предпочитает явные команды. Та же логика доставки и audit log, что у swipe-reply, общая функция `_deliver_operator_reply` в `handlers/operator_reply.py`.

`/op_help` присылает в админ-группу памятку с тремя inline-кнопками («📊 Статистика за сегодня», «📢 Сделать рассылку», «📋 Все команды») и автоматически закрепляет это сообщение через `bot.pin_message`. Это ближайший аналог Telegram menu-button'а — нативный `set_my_commands` в API MAX deprecated, поэтому используем pinned-сообщение как меню.

### 9.15 Citizen-flow guard в админ-группе

Команды `/start`, `/menu`, `/help`, `/policy`, `/forget`, `/subscribe`, `/unsubscribe` и inline-callback'и из citizen-меню (consent, topic, info, cancel) молча игнорируются если событие пришло из `ADMIN_GROUP_ID`. Реализовано через `is_admin_chat(event)` в `utils/event.py` и проверки в `handlers/start.py` + ранний guard в `handlers/appeal.on_callback`. Без этого оператор по привычке набравший `/start` в админ-группе получал welcome-меню, его `max_user_id` создавал запись в `users`, и он мог застрять в состоянии `AWAITING_CONSENT`.

### 9.16 Локальный еженедельный бэкап

`pg_dump` каждое воскресенье в 03:00 (camchatka), сохраняется в named-volume `backups` (внутри контейнера — `/backups/aemr-YYYYMMDD_HHMMSS.sql`). Ротация — последние `BACKUP_KEEP_COUNT=8` файлов (≈2 месяца истории). gpg-шифрование (`*.sql.gpg`) опционально через `BACKUP_GPG_PASSPHRASE`. S3-загрузка опционально через `BACKUP_S3_*`. Команда `/backup` (только для роли `it`) — снятие снимка по запросу, например, перед миграцией БД.

Self-hosted-friendly default: для одного сервера без облачного хранилища достаточно named-volume + ротации. Volume переживает пересборки контейнера (`docker compose up --build` не пересоздаёт named volumes).

## 10. Broadcast — массовая рассылка подписавшимся жителям

Реализованная фича. Координатор АЕМР объявляет о ЧС, публичных слушаниях, плановых работах коммунальных служб через одну команду в админ-группе. Дизайн возник из запроса «нужна возможность разослать важное объявление всем подписавшимся».

### 10.1 Согласие

Подписка явная (opt-in) — житель сам нажимает «🔔 Подписаться на новости» в подменю «Полезная информация» либо пишет `/subscribe`. Это отдельное согласие на обработку данных для целей информирования, рядом с согласием на обработку ПДн в воронке обращения. Без подписки житель не получит ни одной рассылки.

По 38-ФЗ «О рекламе» муниципальное информирование рекламой не является (это не коммерческое продвижение), поэтому отдельной формальной галочки «согласен на рекламу» не требуется. Подписка через кнопку — это согласие на конкретный канал коммуникации, не на ПДн.

В каждом broadcast-сообщении inline-кнопка «🔕 Отписаться от рассылки» — однократное нажатие переключает флаг `users.subscribed_broadcast = false`. Команда `/unsubscribe` делает то же самое.

### 10.2 Технический контур

- Таблица `broadcasts`: `id`, `created_by_operator_id`, `text`, `subscriber_count_at_start`, `started_at`, `finished_at`, `status` (`draft` / `sending` / `done` / `cancelled` / `failed`), `delivered_count`, `failed_count`, `admin_message_id`.
- Таблица `broadcast_deliveries`: `id`, `broadcast_id`, `user_id`, `delivered_at`, `error`.
- Поле `users.subscribed_broadcast` (boolean, default false).
- Команда `/broadcast` в админ-группе — двухшаговый wizard: бот запрашивает текст (лимит 1000 символов), показывает предпросмотр с числом подписчиков, ждёт подтверждения. На «✅ Разослать» — создаётся запись `broadcasts`, запускается фоновая задача через `asyncio.create_task`.
- Рассылка асинхронная, rate-limit 1 сообщение в секунду (`BROADCAST_RATE_LIMIT_PER_SEC`), вписывается в MAX-лимит 2 RPS. На 1000 подписчиков — ~17 минут.
- Wizard-state хранится в памяти процесса (`_wizards: dict[int, _WizardState]`) с TTL 300 секунд. Single-instance only — горизонтальное масштабирование сломает.

### 10.3 UX жителя

```
📢 Объявление от Администрации Елизовского муниципального района

<текст рассылки>

[ 🔕 Отписаться от рассылки ]
```

Кнопка отписки — `payload="broadcast:unsubscribe"`. Бот ставит флаг и шлёт «Подписка отключена. Вернуть — командой /subscribe.»

### 10.4 UX оператора

В админ-группе после `/broadcast` и подтверждения появляется прогресс-сообщение:

```
Рассылка #5 запущена.
⏳ 0/47

[ ⛔ Экстренно остановить ]
```

Каждые 5 секунд бот обновляет это же сообщение через `bot.edit_message`: счётчик доставленных, число упавших. По завершении — финальный итог «✅ Рассылка #5 завершена. Доставлено: 46. Не доставлено: 1.» Кнопка «⛔ Экстренно остановить» доступна **любому оператору в группе** — страховка на случай скомпрометированной учётки координатора.

### 10.5 Команды

- `/broadcast` — создать рассылку (wizard).
- `/broadcast list` — последние 10 рассылок со статусами и счётчиками.

Доступ — `coordinator` и `it`. Рядовые `aemr`/`egp` — нет.

### 10.6 Защита и фильтры

- Каждая рассылка пишется в `audit_log` с полным текстом и инициатором (`action=broadcast_send`).
- Заблокированные ботом жители (`is_blocked=true`) и анонимизированные через `/erase` или `/forget` (`first_name='Удалено'`) исключаются из списка получателей автоматически.
- Лимит длины текста рассылки — 1000 символов (`BROADCAST_MAX_CHARS`). Длинные объявления стоит публиковать на сайте АЕМР, а в broadcast давать краткую выжимку со ссылкой.
- Рассылку можно остановить кнопкой «⛔» — фоновая задача проверяет флаг между каждой отправкой и прерывается.

## 11. Возможные направления развития

Идеи, которые рассматривались, но не вошли в production-периметр MVP. Без сроков и плановых обозначений — это перспективы, а не отложенные задачи. Если со временем появится конкретный сценарий или потребность — каждая из них обсуждается отдельно и реализуется как новый блок поверх текущей архитектуры.

### Расширения broadcast-механизма

- **Шаблоны рассылок** — заготовленные тексты «ЧС», «праздник», «работы», чтобы координатор не писал каждое объявление с нуля.
- **Планирование на будущее** — `/broadcast schedule «завтра в 10:00»` для постановки в очередь.
- **Сегментация подписчиков** — рассылка только в определённое поселение, по тематикам или по адресу.
- **Загрузка резервной копии в S3** — облачный backup как дополнение к локальному. Инфраструктура в коде уже есть (поля `BACKUP_S3_*`), нужна настройка под конкретное хранилище и ключи.

### Маршрутизация и роли операторов

- **Маршрутизация обращений по адресу.** Жители, попадающие в Елизовское ГП → операторам с ролью `egp`, остальные → АЕМР. Шаблон таблицы поселений с улицами и индексами уже есть в `RUNBOOK.md`. Реализация имеет смысл когда сотрудник ЕГП устанет видеть в общей ленте обращения не из своего поселения.
- **Разделение ролей внутри одной группы.** Сейчас все авторизованные считаются операторами с одинаковым набором команд (кроме `/erase`, `/setting`, `/add_operators`, `/broadcast`). Гранулярные роли могли бы ограничить, например, кто может закрывать обращения, кто только отвечает.
- **Адресные алерты IT в личку.** Если бот падает или бэкап не сделан, отправлять не только в админ-группу, но и в личку конкретному ИТ-специалисту.

### Качество ответов и помощь оператору

- **Auto-validator против Стандарта качества АЕМР.** Подсветка нарушений п. 5/6/9/11 (страдательный залог, канцелярит, лишние восклицательные знаки, многоточия) при нажатии «Отправить ответ». Не блокирующая, рекомендательная.
- **LLM-помощник оператору** («перепиши короче / мягче / по чек-листу»). На MVP отказались — оператор отвечает живым языком, без AI-посредника.
- **Подсказки шаблонов ответа** оператору. JSON с ~30 шаблонами от прошлого анализа Telegram-чата лежит в `telegram_analytics/agent_export/` как референс.
- **Автоматическая классификация тематик** обращения. Жители выбирают тематику из списка вручную; ML-классификация сэкономила бы шаг, но не критична.

### Инфраструктура

- **Webhook-режим** вместо long-polling. Готовая инфраструктура (nginx + certbot + endpoint `/max/webhook` с валидацией секрета) лежит в опциональном compose-профиле `webhook`. Переключение — изменение `BOT_MODE=webhook` и подъём профиля. Имеет смысл при росте нагрузки или при введении MAX'ом более жёстких лимитов на polling.
- **Форк `love-apples/maxapi`.** Сейчас upstream активный (последний коммит — апрель 2026). Триггер форка: 3+ месяца без активности upstream И открытые critical issues, либо security-уязвимость без реакции мейнтейнера в течение 14 дней. При срабатывании — форк в `gaben1488/maxapi` под МСУ-контролем.
- **Точечное восстановление БД (PITR).** Сейчас pg_dump раз в неделю. PITR через WAL archiving в S3 даёт восстановление на любую секунду между бэкапами, ценой постоянной отправки WAL-файлов в облако. Имеет смысл после значимого роста объёма данных.
- **Managed Postgres** вместо self-hosted. Yandex Cloud Managed PostgreSQL или подобное — снимает с нас бэкапы, мониторинг, обновления версий. Цена — порядка 1500-3000 ₽/мес.

### Аналитика

- **Дашборды и графики.** Когда наберётся ~200-300 обращений — собрать одноразовый HTML-дашборд с разбивкой по тематикам, операторам, времени отклика. Шаблон есть в `telegram_analytics/`.
- **Многоязычие.** Теоретически интересно, но для МСУ-сервиса не ясна потребность — все жители Елизовского района говорят по-русски.
