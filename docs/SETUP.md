# Настройка бота и админ-группы — от А до Я

Этот документ — пошаговый гайд для ИТ-специалиста АЕМР: что сделать, чтобы бот заработал в производственной служебной группе. Ориентирован на сценарий «бот развёрнут, операторы добавлены, нужно, чтобы он отличал жителя от оператора».

## Картинка целиком

```
житель в личке → бот ловит /start, ведёт по воронке → создаёт #N
                                   │
                                   ▼
              текстовая карточка #N в служебную группу
                  + reply'ом фото/гео если есть
                                   │
                                   ▼
            оператор делает reply на карточку → бот валидирует
                  → доставляет жителю в личку
                                   │
                                   ▼
           бот шлёт в группу подтверждение «✉️ Ответ ушёл жителю»
```

Бот различает жителя и оператора по `chat_id`. Сообщение, пришедшее из чата, чей `chat_id == ADMIN_GROUP_ID`, попадает в `handle_operator_reply`. Любой другой `chat_id` — это житель в личке. Поэтому правильный `ADMIN_GROUP_ID` в `.env` — обязательное условие.

## Шаг 1 — получить токен бота

ИТ открывает в MAX чат с `@MasterBot`, набирает `/newbot`, проходит мастер. На выходе бот выдаёт токен — Base64-подобную строку. Сохрани в безопасное место (1Password, Vaultwarden, либо просто `.env` в защищённой директории сервера).

Если бот уже создан и токен утерян — можно сбросить его через `@MasterBot` командой `/revoke`. Прежний токен после revoke перестаёт работать.

## Шаг 2 — поднять бот с пустым ADMIN_GROUP_ID

Оставь `ADMIN_GROUP_ID` пустым в `.env`, остальное заполни. Запусти контейнер:

```bash
cd ~/aemr-bot/infra
docker compose up -d --build bot
docker compose logs -f bot
```

Должно появиться `Starting in long polling mode`. Бот уже отвечает в личке — но реакции на сообщения в группе пока нет: `cfg.admin_group_id is None`, и весь блок группы пропускается.

## Шаг 3 — создать служебную группу и добавить бота

В MAX создай группу — название, например, «АЕМР · Обратная связь · служебка». Добавь туда бота и операторов (координатора + ИТ + всех, кто будет отвечать). Только верифицированные боты добавляются в группы — если получаешь ошибку на этом шаге, проверь, что юрлицо АЕМР подтверждено на dev.max.ru.

После добавления бот ничего не делает в группе — он не настроен на этот `chat_id`.

## Шаг 4 — узнать `chat_id` группы

В группе напиши `/whoami`. Бот ответит:

```
🛠 whoami
max_user_id: <твой id>
first_name: <твоё имя>
chat_id: <chat_id группы>
```

`chat_id` группы — отрицательное число вида `-1001234567890`. Сохрани его. Заодно зафиксируй свой `max_user_id` — пригодится в шаге 6.

Скажи каждому будущему оператору написать боту в личку `/whoami` — соберёшь их `max_user_id` в табличке для bulk-регистрации.

## Шаг 5 — прописать `ADMIN_GROUP_ID` и перезапустить

Открой `.env` в `infra/`, впиши:

```
ADMIN_GROUP_ID=-1001234567890
COORDINATOR_MAX_USER_ID=<max_user_id координатора>
```

`COORDINATOR_MAX_USER_ID` опционален — нужен только для адресной отправки алертов. Перезапусти бота:

```bash
docker compose up -d --force-recreate bot
docker compose logs -f bot
```

С этого момента сообщения в группе с `chat_id == -1001234567890` обрабатываются как операторские: бот ждёт reply на карточки и команды.

## Шаг 6 — зарегистрировать операторов

В админ-группе ИТ-специалист (тот, чей `max_user_id` уже в БД с `role=it` — об этом ниже) пишет `/add_operators`:

```
/add_operators
123456 it Иванов Иван Иванович
234567 coordinator Петров Пётр Петрович
345678 aemr Сидорова Анна Сергеевна
```

Команда доступна только роли `it`. Если в БД ещё нет ни одного оператора с ролью `it` — раскрути cold start через psql:

```bash
docker compose exec db psql -U aemr -d aemr -c \
  "INSERT INTO operators (max_user_id, full_name, role, is_active) \
   VALUES (123456, 'Иванов И.И.', 'it', true);"
```

После этого тот же IT может добавлять остальных через `/add_operators`. Самомодификация заблокирована и для IT — изменения собственной роли только через psql, чтобы избежать privilege-escalation сценариев (см. ADR-001 §9.7).

## Шаг 7 — публикация политики обработки ПДн

Бот шлёт жителю PDF файлом по команде `/policy`, плюс PDF прикладывается на шаге согласия в воронке. Файл загружается один раз при старте бота из `seed/PRIVACY.pdf` через `policy_service.ensure_uploaded`, токен кешируется в `settings.policy_pdf_token`.

Дополнительно нужна публичная HTML-ссылка на политику — на сайте АЕМР. Когда страница опубликована, IT в админ-группе обновляет URL:

```
/setting policy_url https://elizovomr.ru/privacy-policy
```

Это используется как fallback, если по какой-то причине отправка PDF файлом не сработала.

## Шаг 8 — проверка

Финальный smoke-тест из конца в конец:

1. **Житель.** В личке с ботом нажми «Старт» → должно прийти приветствие про администрацию + три кнопки.
2. **Воронка.** Нажми «Написать обращение» → согласие (PDF приложен) → «Согласен» → запрос контакта → имя → адрес → тематика → суть текстом + одно фото → «Отправить». Бот отвечает «Обращение #N принято».
3. **Админ-группа.** Через секунду в группе появится текстовая карточка с #N, ниже reply'ом — фото жителя. Если фото не пришло — проверь логи бота на `failed to relay attachments` (см. RUNBOOK).
4. **Reply оператора.** Координатор делает reply на карточку, пишет ответ ≤300 символов, отправляет. В группе появляется «✉️ Ответ ушёл жителю. Обращение #N закрыто.»
5. **Доставка.** В личке у жителя должно быть сообщение от бота с текстом ответа.
6. **Реоткрытие.** Житель пишет в личке ещё одно сообщение → в группе появляется «💬 Дополнение к обращению #N» с цитатой исходной карточки.

Если все 6 шагов прошли — бот готов к публикации.

## Что делать, если бот в группе игнорирует сообщения

Чек-лист в порядке вероятности:

- `ADMIN_GROUP_ID` в `.env` совпадает с `chat_id`, который бот вернул на `/whoami` в группе? Знаки (минус вначале) на месте?
- Бот после правки `.env` перезапущен (`docker compose up -d --force-recreate bot`)?
- Бот не покидал группу, не был исключён? Проверь список участников.
- Бот видит себя как участника? `docker compose logs bot | grep -i "first_name"` должен показывать имя бота.
- Команда `/whoami` в группе всё ещё отвечает? Если да — бот видит группу, но `chat_id` не матчится с `ADMIN_GROUP_ID`. Если нет — проблема глубже, бот не получает события из группы.

## Что делать, если оператор пишет, а бот не доставляет

- Reply действительно стоит на карточке бота? Если оператор просто пишет в группу без reply — это не считается ответом на обращение. Бот выведет `⚠️ Не понял, к какому обращению ответ. Сделайте реплай на карточку.`
- Длина ответа ≤300 символов? Превышение — бот просит сократить, жителю ничего не уходит.
- Оператор зарегистрирован в `operators` с `is_active=true`? Если нет — выводится `Эта команда — только для операторов в служебной группе.` (для команд) либо игнор для reply.
- Логи бота — `docker compose logs bot --tail 100 | grep -i error`. Самые частые: пользователь заблокировал бота (нечего доставлять), либо MAX-сторона рвёт TLS.

## Что лежит в БД, что в .env, что в seed

Чтобы не путаться, что где меняется:

| Параметр | Где лежит | Чем меняется |
|---|---|---|
| `BOT_TOKEN`, `DATABASE_URL`, `ADMIN_GROUP_ID`, `WEBHOOK_*` | `.env` | вручную, требует перезапуска контейнера |
| Тематики обращений, тексты приветствия, контакты экстренных служб, URL электронной приёмной, расписание УДТХ, диспетчерские автотранспорта | таблица `settings` | команда `/setting <key> <value>` от IT в группе |
| Список операторов | таблица `operators` | команда `/add_operators` от IT в группе |
| PRIVACY.pdf, тексты и контакты при первом запуске | `seed/*.json`, `seed/welcome.md`, `seed/PRIVACY.pdf` | редактируешь файлы в репо, перезапускаешь контейнер; `seed_if_empty` обновит БД только при отсутствующих ключах |

Полный список редактируемых ключей — `/setting list` в админ-группе.
