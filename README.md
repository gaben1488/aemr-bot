# Администрация ЕМР. Обратная связь

[![CI](https://github.com/gaben1488/aemr-bot/actions/workflows/ci.yml/badge.svg?branch=main)](https://github.com/gaben1488/aemr-bot/actions/workflows/ci.yml)

Чат-бот в мессенджере MAX для приёма обратной связи от жителей Елизовского муниципального района. Админ-панель — служебный групповой чат в MAX, куда падают карточки обращений и где операторы отвечают свайп-реплаем или командой `/reply`.

Образец UX — бот «Солодов. Обратная связь» (правительство Камчатского края).

## Quick start за пять минут

Полный гайд для разработчика — в [DEVELOPER.md](docs/DEVELOPER.md). Для производственного развёртывания и регистрации операторов — [SETUP.md](docs/SETUP.md). Здесь только кратчайший путь до «работает локально».

**Prerequisites:** Docker Desktop с включённым WSL2 (для Windows), Git с доступом к этому репозиторию, аккаунт в MAX. Python 3.12 на хосте опционален — нужен только для `pytest` и помощников из `scripts/`.

```bash
git clone https://github.com/gaben1488/aemr-bot.git
cd aemr-bot/infra
cp .env.example .env
```

Открой `.env` и впиши минимум три значения:

```ini
BOT_TOKEN=<токен с max.ru/business → раздел «Боты»>
POSTGRES_PASSWORD=local-test-pass
DATABASE_URL=postgresql+asyncpg://aemr:local-test-pass@db:5432/aemr
```

`BOT_TOKEN` берётся на <https://max.ru/business> → войти как админ организации → раздел «Боты» → создать бота или открыть существующего → скопировать Bot API token. Документация: <https://dev.max.ru/docs-api>. `ADMIN_GROUP_ID` оставь пустым — заполнишь после первого старта (см. [SETUP.md](docs/SETUP.md)).

Запустить:

```bash
docker compose up --build bot
```

В логах должно появиться `Starting in long polling mode` и `dispatcher Бот: @<твой_бот>`. После этого открой бот в MAX, нажми «Старт» — должно прилететь приветствие и пять кнопок главного меню.

**Прогнать тесты** (требуется живой Postgres):

```bash
docker compose up -d db
cd ../bot
DATABASE_URL=postgresql+asyncpg://aemr:local-test-pass@localhost:5432/aemr pytest -q
```

**Что делать дальше:**

- [DEVELOPER.md](docs/DEVELOPER.md) — детальный гайд: вход в воронку, отладка, миграции, тесты с моком Bot, известные особенности `maxapi`.
- [SETUP.md](docs/SETUP.md) — как добавить бота в админ-группу, получить `ADMIN_GROUP_ID`, выполнить cold-start первого ИТ-оператора через `BOOTSTRAP_IT_*`, зарегистрировать остальных через `/add_operators`.
- [RUNBOOK.md](docs/RUNBOOK.md) — операционные процедуры: ответить на обращение, запустить рассылку, выгрузить статистику, удалить ПДн, снять и восстановить бэкап.
- [ADR-001](docs/ADR-001-architecture.md) и [PRD-mvp](docs/PRD-mvp.md) — архитектура и спецификация продукта.
- [docs/architecture-diagrams.md](docs/architecture-diagrams.md) — BPMN жизненного цикла обращения, flowchart обработки событий, sequence-диаграммы доставки ответа и рассылки, схема развёртывания.

## Стек

- **Python 3.12** + библиотека `maxapi` (community SDK от love-apples).
- **PostgreSQL 16** через SQLAlchemy 2.x + Alembic.
- **APScheduler** в том же процессе: ежемесячная статистика, healthcheck-pulse, еженедельный `pg_dump`.
- **openpyxl** для XLSX-выгрузки статистики по команде из админ-группы.
- **Docker Compose** — по умолчанию два контейнера: `bot` и `db`. Профили `webhook` (поднимают `nginx` + `certbot`) включаются по требованию для production-развёртывания на собственном HTTPS-эндпоинте.
- **Long polling** — режим доставки событий по умолчанию. Webhook поддерживается опционально.

Никакого FastAPI, Redis, React, мини-приложения — на MVP не нужны.

## Главное меню жителя — пять кнопок

1. **Написать обращение** — после согласия на обработку ПДн (один раз) и нативного запроса контакта (имя + телефон) житель пишет свободным текстом + опционально фото и геолокация.
2. **Мои обращения** — список обращений жителя со статусами и последними ответами оператора.
3. **Электронная приёмная** — прямая ссылка на сайт АЕМР (видна, если URL задан в настройках).
4. **Приём граждан** — расписание и контакты.
5. **Полезная информация** — подменю: телефоны экстренных и аварийных служб, муниципальные и межмуниципальные маршруты УДТХ, диспетчерские автотранспорта, кнопка подписки на рассылку.

Дополнительные команды для жителя: `/help`, `/menu`, `/policy` (PDF политики ПДн), `/subscribe`, `/unsubscribe`, `/forget` (удалить мои данные), `/whoami`.

## Админ-группа в MAX

Создаётся служебная группа, куда добавлены бот и операторы (координатор АЕМР, ИТ, при необходимости — специалисты ЕГП). При новом обращении бот шлёт в группу карточку: номер `#NNN`, имя, телефон, текст обращения, фото и геолокация (если есть). Оператор отвечает на эту карточку — двумя путями:

- **Свайп-реплаем** на карточку: текст ответа отправляется жителю, в группу падает подтверждение «✉️ Ответ ушёл жителю. Обращение #NNN закрыто.»
- **Командой `/reply NNN <текст>`** — альтернатива, работает на всех клиентах MAX даже там, где свайп не передаёт reply-link.

Команды для оператора: `/stats today|week|month` (XLSX-выгрузка), `/reopen NNN`, `/close NNN`, `/op_help` (закреплённое сообщение с кнопками-быстрыми-действиями), `/diag`, `/broadcast` (мастер рассылки). Для роли `it` дополнительно: `/erase max_user_id=… | phone=…` (удаление ПДн), `/setting list|<key> <value>` (редактируемые настройки), `/add_operators` (регистрация операторов), `/backup` (внеплановый pg_dump).

## Бэкап

Каждое воскресенье в 03:00 по Камчатке APScheduler делает `pg_dump` и сохраняет файл в named-volume `backups`. Хранятся последние восемь файлов, ротация автоматическая. Опционально — gpg-шифрование (`BACKUP_GPG_PASSPHRASE`) и копия в S3-совместимое хранилище (`BACKUP_S3_*`). По умолчанию для self-hosted без облака достаточно локального бэкапа. Подробнее — [RUNBOOK.md §7](docs/RUNBOOK.md#7-работа-с-бэкапами).

## Уровень защищённости персональных данных

УЗ-4 по ПП №1119: иные ПДн, третий тип угроз, менее 100 000 субъектов, не сотрудники. Аттестация провайдера по ФСТЭК не требуется. Подробнее — [ADR-001 §6 «Безопасность»](docs/ADR-001-architecture.md).

## С чего начать в production

1. Зарегистрировать тестового бота на <https://max.ru/business> в разделе «Боты», получить токен.
2. Создать служебную группу в MAX, добавить туда бота и будущих операторов.
3. Поднять собственный сервер для self-host (минимум 2 vCPU / 4 ГБ RAM / 60 ГБ диска). Облачные провайдеры не требуются и не рекомендуются для этого проекта — заказчик self-host'ит.
4. Развернуть бот по [SETUP.md](docs/SETUP.md): cold-start первого ИТ через `BOOTSTRAP_IT_MAX_USER_ID`, остальных операторов — командой `/add_operators` в группе.
5. Прогнать smoke-тест из конца в конец (раздел «Шаг 8 — проверка» в SETUP.md).
